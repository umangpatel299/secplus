01Core Security Principles for InfrastructureFoundation · Design · Defense

Applying security principles to enterprise infrastructure means building security into every layer of the architecture — not bolting it on afterward. These principles guide every infrastructure decision from network design to device configuration.

Fundamental Infrastructure Security Principles

🏰
Defense in Depth
Layer multiple security controls so that if one fails, others remain. No single control is relied upon. Perimeter firewall fails → internal firewall still blocks. Credential stolen → MFA still required. Endpoint compromised → network segmentation limits lateral movement. The more layers, the higher the attacker's cost and time. Apply at: perimeter, network, host, application, data, and identity layers simultaneously.
🔒
Least Privilege
Every user, process, and system gets only the minimum access required to perform its function — nothing more. A web server should not have database admin rights. A developer should not have production admin access. A service account should only access the specific resources it needs. Reduces blast radius when anything is compromised. Apply to: user accounts, service accounts, API keys, firewall rules, network access, IAM policies.
🎯
Attack Surface Reduction
Minimize the number of entry points an attacker can use. Disable unused services, close unused ports, remove unnecessary software, decommission unused systems. Every open port is an opportunity. Every installed service that isn't needed is an unnecessary risk. Hardening guides (CIS Benchmarks) define which services to disable. "If it's not needed — remove it."
🔐
Zero Trust
Never trust, always verify. No implicit trust based on network location — being "inside the network" doesn't mean trusted. Every access request verified: identity + device health + context + behavior. Micro-segmentation isolates resources. ZTNA replaces VPN. Continuous verification — not just at login. Assumes breach: "the attacker is already inside — what does that mean for our architecture?"
🧩
Separation of Duties
No single person or system should control an entire critical process end-to-end. Developer cannot deploy to production without a second approval. Firewall rule changes require security review. Admin who creates accounts cannot also delete audit logs. Prevents both insider fraud and reduces blast radius of compromised accounts. Technical enforcement: RBAC, workflow approvals, two-person integrity rules.
⚙️
Secure by Default
Systems should ship in the most secure state possible — security enabled, unnecessary services disabled, default passwords changed, permissions restrictive. The default configuration should be safe — users must explicitly relax security, not explicitly enable it. Contrast with "secure by design": default = what ships out of the box. Design = the architecture choices made during development. Both are required for truly secure infrastructure.
🔄
Fail Secure (Fail Closed)
When a security control fails, it should fail in the most secure state — denying access rather than allowing it. Firewall goes down: all traffic should be blocked (fail closed), not allowed through (fail open). Authentication server unreachable: deny access, don't bypass authentication. The alternative (fail open) means failures become security vulnerabilities. Design all controls with a defined failure mode that defaults to security.

Applying Principles to Architecture Decisions

# Principles guide EVERY infrastructure decision: Decision: "Should our web server have direct database access?" Least privilege: web server should only access the DB it needs Attack surface: if web server is compromised, DB access is gained Defense in depth: add: app layer firewall, WAF, DB firewall Answer: no direct access — API gateway + DB firewall rules Decision: "Should we use a single admin account for all infrastructure?" Least privilege: separate accounts by function Separation of duties: no one admin controls everything Zero trust: shared accounts = no accountability Answer: individual named accounts, MFA, PAM vault Decision: "Should the IoT thermostat be on our corporate network?" Attack surface: IoT = weak security = attack vector Defense in depth: isolate on separate VLAN Zero trust: thermostat doesn't need corporate AD access Answer: IoT VLAN → only allowed to reach HVAC controller

02Network Segmentation & ZonesVLAN · DMZ · Micro-seg

Network segmentation divides the network into isolated zones with controlled communication between them. It limits lateral movement — an attacker who compromises one segment cannot freely reach all others.

Why Segmentation Matters

# Flat network vs segmented network — the difference: Flat network (no segmentation): ALL devices on same broadcast domain IoT thermostat → can reach domain controller Marketing laptop → can ping finance database Attacker on ANY device: reach everything Lateral movement: trivial — no barriers Segmented network: IoT VLAN: only reaches HVAC controller Corporate VLAN: users reach apps via firewall rules Finance VLAN: only finance apps + audit logging Server VLAN: servers only reachable from specific sources Attacker on IoT: stuck in IoT VLAN ✓ Attacker on marketing laptop: cannot reach finance DB ✓ Lateral movement: blocked at every segment boundary

Network Zones Architecture

UNTRUSTED
Internet / Untrusted Zone
The public internet and any untrusted external networks. No trust — all traffic from here is potentially hostile. Filtered at perimeter firewall. Only specific inbound services allowed (HTTP/S to web server). All other inbound: DENY by default.
DMZ
Demilitarized Zone (DMZ)
Buffer zone between internet and internal network. Houses publicly accessible servers: web servers, email relays, DNS servers, reverse proxies. Semi-trusted — internet can reach DMZ servers, but DMZ cannot freely reach internal network. Compromised DMZ server should NOT be able to access internal Active Directory. Two-firewall DMZ model: one firewall between internet and DMZ, another between DMZ and internal.
INTERNAL
Internal / Corporate Zone
Employee workstations, internal applications, printers, corporate services. Trusted — but not implicitly. Access controlled by internal firewalls and ACLs. Sub-segmented by function: Finance VLAN, HR VLAN, Engineering VLAN, IT Admin VLAN. Zero Trust: internal trust is earned, not assumed based on being inside the network.
RESTRICTED
Restricted / Secure Zone
Most sensitive resources: financial systems, HR data, PCI-scoped systems, security management infrastructure (SIEM, PAM). Highest access controls — MFA + approval + session recording. Minimal connectivity — only specific authorized systems can communicate. Audit logging on all access. Often air-gapped from general corporate network.
GUEST/IOT
Guest / IoT Zone
Isolated zone for untrusted devices. Guest WiFi: internet access only — cannot reach any corporate resources. IoT VLAN: devices only reach their management systems — not the corporate network. Prevents IoT or guest devices from being used as pivot points. Separate SSID, separate VLAN, firewall blocks access to internal zones.

VLAN — Virtual Local Area Network

# VLANs create logical network segments on same physical infrastructure: Switch VLAN configuration: Port 1-20: VLAN 10 (Corporate users) Port 21-30: VLAN 20 (Finance systems) Port 31-40: VLAN 30 (IoT devices) Port 41: VLAN 99 (IT Management - trunk) Traffic rules: VLAN 10 → VLAN 20: DENY (users can't reach finance directly) VLAN 10 → Internet: ALLOW via proxy VLAN 30 → Any: DENY except HVAC controller IP VLAN 20 → Internet: DENY (finance never touches internet) # Inter-VLAN routing through firewall only: Traffic between VLANs must pass through L3 device (router/firewall) Firewall applies rules to VLAN-to-VLAN traffic No VLAN-to-VLAN traffic bypasses security inspection

Micro-segmentation

🔬
Micro-segmentation — Zero Trust Network
Takes segmentation to the individual workload level. Traditional: perimeter + VLANs. Micro-seg: every server, VM, container has its own security policy. East-west traffic between workloads controlled at workload level, not just at VLAN boundary. Software-defined: policy applied in software — scales to thousands of workloads. Example: VMware NSX, Illumio. Even if attacker reaches a server, they cannot reach adjacent servers without explicit policy.

// Exam Keywords

VLAN →Logical network segment on same physical infra — inter-VLAN through firewall only
Micro-segmentation →Workload-level isolation — east-west control — Zero Trust network — VMware NSX
Flat network risk →No segmentation — lateral movement trivial — compromise one = compromise all
Guest VLAN →Internet only — no corporate access — prevents IoT/guest pivot
03Security Control SelectionPreventive · Detective · Corrective

Control selection means choosing the right security controls for each risk — matching the control type, category, and method to the threat. No control is perfect alone; selecting complementary controls creates effective defense in depth.

Control Classification — Two Dimensions

By Function (What It Does)

  • Preventive: stops attack from occurring — firewall, MFA, encryption, access controls
  • Detective: identifies attack in progress or after — IDS, SIEM, audit logs, security cameras
  • Corrective: fixes damage after attack — backups/restore, patch, IR plan
  • Deterrent: discourages attack — warning banners, visible cameras, security policies
  • Compensating: alternative when primary control isn't feasible — used when can't patch, add network isolation instead
  • Directive: tells people what to do — policies, procedures, training, signs

By Category (What It Is)

  • Technical (Logical): technology enforces — firewalls, EDR, encryption, MFA systems, access control lists
  • Administrative (Managerial): policies, procedures, training, risk assessments, vendor management
  • Physical: hardware and environment — locks, cameras, badge readers, guards, mantraps
  • Operational: day-to-day practices — patch management, backups, monitoring, incident response

Control Matrix — Matching Controls to Threats

Firewall Preventive Blocks unauthorized traffic — technical control — first line of network defense — stateful packet inspection
IDS Detective Detects intrusion attempts — alerts only — out-of-band — signature and anomaly detection
IPS Preventive Detects AND blocks — inline — can create false positive outages — more aggressive than IDS
Encryption Preventive Protects data confidentiality — in transit (TLS) and at rest (AES-256) — also provides integrity
Audit Logs Detective Record all actions for investigation — also deterrent — must be protected from tampering — retention policy
SIEM Detective Aggregate and correlate logs — real-time alerting — central investigation platform
Backups Corrective Restore data after incident — 3-2-1 rule — immutable backups protect against ransomware
Security Camera Detective Physical monitoring — also deterrent (visible cameras) — CCTV — access recording
Badge Reader Preventive Physical access control — also detective (logs who accessed where) — combined function
AUP Policy Directive Tells users what is and isn't acceptable — administrative control — sets legal foundation for discipline
Warning Banner Deterrent "Unauthorized access is prohibited..." — discourages casual attackers — also sets legal consent for monitoring
Network Isolation Compensating Used when can't patch — isolate vulnerable system — reduce attack surface — temporary until real fix

Control Selection Framework

# How to select the right controls for a risk: Risk: Ransomware encrypting file servers Step 1 — Preventive controls (stop it happening): EDR on all endpoints (behavioral detection) Email sandbox (stop delivery of ransomware) User training + phishing simulation Least privilege: users can't write to share they don't need Step 2 — Detective controls (know when it's happening): SIEM rule: mass file modification alert EDR behavioral: shadow copy deletion = ransomware File integrity monitoring on critical shares Step 3 — Corrective controls (recover when it happens): Immutable backups (3-2-1: 3 copies, 2 media, 1 offsite) Tested DR procedure: restore from backup Incident response plan with ransomware playbook Step 4 — Compensating controls (if primary fails): Network segmentation: isolate file servers from workstations Air-gapped backup: no connection to network = can't be encrypted

// Exam Keywords

Preventive →Stops attack — firewall, MFA, encryption, access control, patch management
Detective →Identifies attacks — IDS, SIEM, audit logs, cameras — alerts not blocks
Corrective →Fixes after attack — backups, IR plan, patching after breach
Deterrent →Discourages attacks — warning banners, visible cameras, security policies
Compensating →Alternative when primary can't be implemented — network isolation for unpatched system
Technical vs administrative →Technical = technology enforces / Administrative = policies and procedures
04Secure Communication ProtocolsTLS · SSH · IPsec · HTTPS

Secure protocols protect data in transit across networks. Using the wrong protocol — or the right protocol with weak settings — exposes communications to interception, tampering, and impersonation.

Protocol Quick Reference — Ports and Security

HTTP TCP 80 — INSECURE Plaintext — no encryption — no authentication of server. Everything visible to MITM. Never use for anything with credentials or sensitive data. Replace with HTTPS. Redirect HTTP → HTTPS.
HTTPS TCP 443 — TLS encrypted HTTP over TLS. Encrypts in transit + authenticates server via certificate. Minimum: TLS 1.2. Recommended: TLS 1.3. Verify: cipher suite strength, certificate validity, HSTS enabled. HSTS: forces HTTPS even if user types HTTP.
Telnet TCP 23 — INSECURE Plaintext remote access. Username, password, and all commands visible in plaintext. NEVER use on any production system. Replaced by SSH. If you see Telnet in production: immediate remediation required.
SSH TCP 22 — Encrypted Secure Shell — encrypted remote access, file transfer (SFTP), port forwarding. Key-based auth stronger than password. Disable root login. Restrict to management VLAN. Use version 2 (SSHv1 is broken). Harden: AllowUsers, MaxAuthTries, disable PasswordAuthentication.
FTP TCP 20/21 — INSECURE File Transfer Protocol — plaintext credentials and data. Replaced by SFTP (SSH-based) or FTPS (TLS-based). SFTP uses TCP 22. FTPS uses TCP 21 with explicit TLS or TCP 990 implicit TLS. Never run plain FTP in enterprise.
SFTP/FTPS TCP 22 / TCP 990 SFTP: SSH-based file transfer — uses port 22 — encrypted. FTPS: FTP over TLS — port 21 (explicit) or 990 (implicit). Both replace plain FTP. SFTP preferred — simpler firewall rules (single port).
SNMP v1/v2 UDP 161 — Weak auth Simple Network Management Protocol — monitors network devices. v1/v2: community string = plaintext "password". Anyone sniffing = read/write access to all managed devices. Use only SNMPv3 which adds authentication and encryption.
SNMPv3 UDP 161 — Auth + Encrypt Adds: authentication (SHA/MD5) and privacy (AES/DES encryption). Only version acceptable in modern environments. Supports: authNoPriv (auth only) and authPriv (auth + encryption — use this). Replace all SNMPv1/v2 configurations.
IPsec IP protocol 50/51 + UDP 500 IP Security — encrypts and authenticates IP packets. Used for: VPN tunnels (site-to-site), network-layer encryption. Two modes: Transport (encrypts payload only) and Tunnel (encrypts entire packet, new IP header). AH (Authentication Header): integrity + auth, no encryption. ESP (Encapsulating Security Payload): encryption + integrity + auth. ESP preferred.
TLS 1.3 Various — depends on app Transport Layer Security — current standard. Removed: weak cipher suites, CBC mode, compression, renegotiation. Forward Secrecy: even if key is compromised later, past sessions can't be decrypted. 1-RTT and 0-RTT handshakes — faster than TLS 1.2. Disable TLS 1.0/1.1 (deprecated). Minimum: TLS 1.2 with strong ciphers.
LDAP TCP 389 — Cleartext Directory query protocol — AD authentication. Plaintext by default. Credentials visible on network. Replace with LDAPS (TCP 636) or LDAP with StartTLS. Attackers on network can harvest credentials from LDAP. All AD-joining systems should use LDAPS.
LDAPS TCP 636 — TLS encrypted LDAP over SSL/TLS. Encrypts all directory communications. Required for: any environment where AD credentials must be protected. Enable on domain controllers. Configure all clients to use LDAPS. Also: LDAP with StartTLS (upgrades TCP 389 to encrypted).
DNSSEC UDP/TCP 53 (DNS) DNS Security Extensions — cryptographically signs DNS records. Prevents DNS poisoning/cache poisoning. Doesn't encrypt DNS (use DoH/DoT for that). Verifies: response came from authoritative server and wasn't modified. Response validated via chain of trust from root zone.
DoH / DoT TCP 443 / TCP 853 DNS over HTTPS / DNS over TLS — encrypts DNS queries. ISP/network can't see what domains you're resolving. DoH (port 443): hard to block, used by browsers. DoT (port 853): easier to filter at firewall. Enterprise concern: DoH bypasses DNS filtering — disable in browsers via GPO.

05Secure Remote AccessVPN · ZTNA · Jump Server

Remote access allows employees to work from outside the corporate network. Every remote access method must balance usability with security — a remote access solution that's too complex gets bypassed.

VPN — Virtual Private Network

# VPN types and security implications: Site-to-site VPN: Connects two office networks permanently HQ ←→ Branch office encrypted tunnel Protocol: IPsec (most common), OpenVPN Gateway devices: VPN concentrators / firewalls Users: transparent — no client needed Remote access VPN (client VPN): Individual users connect from remote locations Client software: Cisco AnyConnect, GlobalProtect Protocol: SSL/TLS VPN (TCP 443) or IPsec MFA: required — credential theft bypasses VPN otherwise Split tunnel: some traffic through VPN, rest directly Full tunnel: ALL traffic through VPN # Split tunneling security concern: Split tunnel: employee connected to work VPN Work traffic → VPN tunnel (encrypted, monitored) Personal browsing → direct internet (unmonitored) Risk: malware on home computer has path to VPN-connected network Risk: no corporate proxy/DNS filtering on non-VPN traffic Defense: full tunnel OR strict split tunnel with DLP # VPN limitations: Traditional VPN: "trust the tunnel" — connected = trusted If attacker gets VPN credentials → access to ENTIRE internal network Modern answer: ZTNA replaces VPN for most use cases

ZTNA — Zero Trust Network Access

🛡️
ZTNA vs Traditional VPN
Traditional VPN: authenticate once → access everything on the network. ZTNA: authenticate continuously → access only specific authorized applications. User never gets network access — only application access. Device posture checked at every request. Example: Cloudflare Access, Zscaler ZPA, Prisma Access. Replaces VPN for most remote access scenarios. Even compromised credentials give access to ONE app, not the whole network.

Jump Server / Bastion Host

# Jump server: hardened intermediate access point for admin tasks Without jump server: Admin laptop (internet/home) → direct SSH to prod server Attack path: compromise admin laptop → direct access to prod Risk: admin's home network is insecure With jump server: Admin → Authenticates to jump server (MFA required) Jump server → SSH to production servers Production: only accepts connections FROM jump server IP Admin's machine: can't directly reach production servers Jump server security: Hardened OS: CIS benchmark, minimal software MFA: required for all jump server access Session recording: everything done via jump server is logged No internet access from jump server: only to managed systems Separate jump servers: for different security zones

RDP — Remote Desktop Protocol

# RDP security — TCP 3389 — commonly targeted: RDP risks: Default port 3389 — heavily scanned by attackers BlueKeep (CVE-2019-0708): wormable RDP exploit DejaBlue: similar critical RDP vulnerabilities Brute force: RDP login = unlimited password attempts Credential stuffing: leaked credentials + RDP = access RDP hijacking: attacker takes over disconnected session RDP hardening: Never expose RDP directly to internet VPN or ZTNA required before RDP is accessible MFA: Network Level Authentication (NLA) minimum Change default port: obscures from automated scanners Account lockout: lock after 3-5 failed attempts Jump server: RDP only accessible through jump server Firewall: restrict RDP to management IPs only Logging: all RDP sessions logged and monitored # Alternative: RDP Gateway (built into Windows Server) # Proxies RDP over HTTPS (443) — no direct 3389 exposure

Out-of-Band Management

📡
Out-of-Band (OOB) Management
Separate management network physically isolated from production traffic. Even if production network is down or compromised, admin can still access and fix systems. Uses: IPMI, iDRAC (Dell), iLO (HP), console servers, cellular OOB modems. When ransomware encrypts production network: OOB allows recovery actions. OOB network should have: separate credentials, MFA, strict access control, no internet access.

06Network Infrastructure DevicesSwitches · Routers · Firewalls · Proxies

Network devices are the hardware and software components that move and filter traffic. Securing these devices is as important as securing servers — a compromised switch or router gives an attacker ability to intercept, redirect, or block all network traffic.

Device Security Hardening

# Universal device hardening — applies to ALL network devices: Access security: Change default credentials immediately Use strong unique passwords or certificate auth MFA: for management access where supported Management access: only from dedicated mgmt VLAN SSH (not Telnet) for remote management HTTPS (not HTTP) for web management Service reduction (attack surface): Disable unused services: Telnet, HTTP, finger, CDP* *CDP (Cisco Discovery Protocol): reveals network topology to attacker Disable unused ports: shutdown all unused switch ports Assign unused ports to "dead" VLAN Monitoring and logging: Syslog: forward all logs to SIEM NTP: synchronize time (critical for log correlation) SNMP v3: monitoring with encryption and auth NetFlow: traffic data for anomaly detection Configuration management: Backup running config: before any change Version control: track all config changes Change management: formal process for network changes

Key Network Device Types

Managed Switch

Layer 2 device — forwards frames based on MAC addresses. Security features: port security (limit MAC addresses per port), 802.1X (authenticate before network access), VLAN assignment, BPDU Guard (prevent rogue switches), Storm Control (mitigate broadcast storms), Private VLAN (isolate ports within same VLAN).

Router / Layer 3 Switch

Routes packets between networks. Access Control Lists (ACLs): filter traffic between subnets — stateless (check each packet independently). Anti-spoofing: uRPF (Unicast Reverse Path Forwarding) — drops packets with spoofed source IPs. Routing protocol security: MD5 auth on OSPF/EIGRP/BGP to prevent route injection.

Next-Generation Firewall (NGFW)

Stateful packet inspection + deep packet inspection. App identification: block Dropbox, allow Office 365. User identity: rules by AD username not just IP. SSL inspection: decrypt HTTPS to inspect content. IPS integration: block exploits inline. URL filtering. Most important perimeter security device.

Web Application Firewall (WAF)

Specifically protects web applications. Understands HTTP — inspects URLs, headers, request body. Blocks: SQL injection, XSS, CSRF, path traversal, API abuse. Positive/negative security model. Deployed: in front of web servers (reverse proxy). Required for: PCI-DSS scope web apps. Examples: AWS WAF, Imperva, ModSecurity.

Proxy Server

Forward proxy: sits between users and internet. Inspects outbound web requests. Enforces URL category filtering. SSL inspection: decrypt outbound HTTPS. Logs all web access. Blocks malware downloads. Reverse proxy: sits in front of servers — hides server IPs, load balancing, WAF. Protects internal servers from direct internet exposure.

Load Balancer

Distributes traffic across multiple servers. Security benefits: DDoS mitigation, connection rate limiting, SSL offloading (centralize certificate management), session persistence, health monitoring. Active/Active: both handle traffic. Active/Passive: standby takes over on failure. Inspect incoming connections before forwarding to backend servers.

Wireless Security

# Enterprise wireless security standards: WPA3 (current standard): SAE (Simultaneous Authentication of Equals) replaces PSK Forward secrecy: new encryption key per session Protection against offline dictionary attacks WPA3-Enterprise: 802.1X + EAP — per-user authentication WPA2-Enterprise (still widely used): 802.1X authentication: RADIUS server validates users Per-user credentials: not shared PSK EAP methods: PEAP, EAP-TLS (certificate-based = strongest) VLAN assignment per user/group based on auth WPA2-Personal / WPA3-Personal: Shared Pre-Shared Key (PSK) All users share same password One compromised device → sniff all traffic (WPA2) Rogue AP easy to set up with same SSID + PSK Only acceptable for: home networks, guest WiFi WEP — NEVER USE: Broken encryption (RC4 flaws) Crackable in minutes with basic tools Should not exist in any modern environment

07Secure Infrastructure DesignArchitecture · Hardening · Baselines

Secure design means building security into infrastructure from the ground up — choosing the right architecture patterns, applying security baselines, and ensuring the design supports ongoing security operations.

Infrastructure Hardening

# System hardening: remove what isn't needed, secure what remains OS hardening (CIS Benchmark Level 1/2): Remove: unnecessary packages and services Disable: unneeded network services (NFS, FTP, rsh) Configure: audit logging (specific event categories) Enable: host-based firewall (ufw/firewalld/Windows Firewall) Restrict: filesystem permissions (least privilege) Secure: bootloader (GRUB password) Enable: SELinux/AppArmor (mandatory access control) Set: strong password policy via PAM Application hardening: Remove: default accounts, sample applications, help files Change: default ports where possible Disable: verbose error messages (leak internal info) Enable: security headers (HSTS, CSP, X-Frame-Options) Configure: input validation, parameterized queries Network hardening: Disable: CDP/LLDP on external-facing ports Enable: BPDU Guard on access ports Configure: DHCP snooping (prevent rogue DHCP) Enable: Dynamic ARP Inspection (prevent ARP poisoning) Implement: IP Source Guard (prevent IP spoofing)

Security Baselines and CIS Benchmarks

CIS Benchmarks

Consensus-based configuration standards for hundreds of technologies. Level 1: basic security — minimal performance impact. Level 2: more restrictive — may impact usability. Scored: every control has pass/fail check. Free download. Covers: Windows, Linux, AWS, Azure, Docker, Kubernetes, network devices. Gold standard for system hardening.

STIG — Security Technical Implementation Guide

US DoD hardening guidance. More restrictive than CIS. Required for: DoD systems, many federal contractors. Automated: STIG Viewer, Evaluate-STIG tool. SCAP (Security Content Automation Protocol): machine-readable compliance checking. Used alongside CIS in high-security environments.

Network Architecture Patterns

1
Perimeter Architecture (Traditional)
Internet → Perimeter Firewall → DMZ → Internal Firewall → Internal Network. Strong perimeter — limited internal controls. Modern weakness: assumes internal = trusted. Works for: organizations with clear internal/external boundary. Insufficient alone: once inside, lateral movement is easy.
2
Defense in Depth Architecture
Perimeter firewall + internal segmentation + host controls + identity + data controls. Every layer adds cost to attacker. Compromising perimeter doesn't mean owning the network. Most enterprise networks should use this model. Complexity: more to manage, but significantly more resilient.
3
Zero Trust Architecture
No implicit trust anywhere. Identity is the control plane. Every access request: verify identity + device + context. Micro-segmentation at workload level. ZTNA replaces VPN. Comprehensive logging: all access events. Modern cloud-native organizations are building this from scratch. Legacy organizations migrating toward it gradually.
4
SASE Architecture
Secure Access Service Edge — network and security converged in the cloud. Users connect to cloud-based security stack (not corporate datacenter). SD-WAN + ZTNA + CASB + FWaaS + SWG — all delivered as a service. Ideal for: remote-first organizations, distributed offices, cloud-heavy workloads. Gartner term — vendors: Zscaler, Cloudflare, Palo Alto Prisma.

Infrastructure as Code Security (Revisited)

# Secure infrastructure design implemented via IaC: Security group (Terraform) — least privilege: Inbound: only HTTPS from load balancer SG Inbound: SSH only from jump server SG Outbound: only to required services No 0.0.0.0/0 rules → automatic DENY all else Network ACL (subnet-level defense): Subnet A (web tier): allow 443 inbound, 443/3306 to app tier Subnet B (app tier): allow from web tier only, to DB tier Subnet C (DB tier): allow from app tier only, deny all else Security baked into code = consistent + auditable + reviewable Every change reviewed in pull request before deployment

08Infrastructure Resilience & AvailabilityHA · Redundancy · Recovery

Resilient infrastructure ensures services remain available despite hardware failures, attacks, or disasters. Security can't just be about keeping attackers out — it also means ensuring availability for legitimate users.

High Availability Concepts

# Availability = MTBF / (MTBF + MTTR) MTBF: Mean Time Between Failures (how reliable is hardware) MTTR: Mean Time To Repair (how fast can we restore) Availability targets: 99%: 3.65 days downtime/year (unacceptable for most) 99.9%: 8.76 hours/year (three nines) 99.99%: 52.6 minutes/year (four nines) 99.999%:5.26 minutes/year (five nines — carrier grade) Achieving high availability: Eliminate single points of failure (SPOF) Redundancy: N+1 (at least one spare) Clustering: multiple nodes, any can take over Geographic distribution: survive site-level failure

Redundancy Strategies

Network Redundancy

Dual ISP connections — if one fails, traffic routes over other. Redundant switches — spanning tree / LACP. Multiple paths with OSPF/BGP failover. No single network device is a SPOF. BGP: internet routing protocol for multihoming to multiple ISPs.

Server Redundancy

Clustering: Active/Active (both serve traffic), Active/Passive (standby). Load balancing: distribute across multiple servers. RAID: disk redundancy within a server. Failover clustering: automatic takeover on failure. Heartbeat: nodes monitor each other — detect failure in seconds.

Power Redundancy

UPS: uninterruptible power supply — bridges power to generator startup. Generator: sustains power during extended outage. Dual power feeds: from separate utility circuits. PDU redundancy: A+B power feeds to servers. N+1 UPS: one UPS beyond minimum required.

Backup and Recovery

# 3-2-1 Backup Rule: 3 copies of data 2 different media types (disk + tape, disk + cloud) 1 copy offsite (geographically separate) # 3-2-1-1-0 Extended Rule (modern): 3 copies | 2 media types | 1 offsite | 1 immutable | 0 errors Backup types: Full: all data — slowest — most space — fastest restore Incremental: changes since LAST backup — fastest — complex restore Differential: changes since LAST FULL — moderate — faster restore Recovery testing (CRITICAL): Backup without test = false confidence Regular restore tests: verify data integrity Tabletop exercises: test recovery procedures Documented RTO/RPO: clear recovery objectives # Immutable backups — ransomware defense: Write Once Read Many (WORM) storage AWS S3 Object Lock: data cannot be modified or deleted Once written: cannot be encrypted by ransomware Air-gapped backup: physical separation from network Tape backup: ransomware cannot encrypt offline tapes
