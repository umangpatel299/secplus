01Enterprise Security Controls — OverviewDefense in Depth

Enterprise security controls are the tools and technologies that enforce security policy across an organization. No single control stops every attack — the goal is layered defense where each control catches what the others miss.

Why Multiple Controls?

# Defense in depth — every layer catches something different: Attacker sends phishing email with malicious link: Layer 1 — Email Gateway: scans attachment → blocked (known malware) Attacker tries different payload — no signature match Layer 1 — Email Gateway: passes through (unknown payload) Layer 2 — DNS Filtering: user clicks link DNS query to malicious domain → blocked (known bad domain) Attacker uses new domain — not yet blacklisted Layer 2 — DNS Filtering: passes through (new domain) Layer 3 — Firewall: connection to attacker C2 Rule blocks port 4444 → blocked Attacker switches to HTTPS/443 Layer 3 — Firewall: passes (looks like normal HTTPS) Layer 4 — IPS: HTTPS traffic inspection Signature matches exploit pattern → blocked Attacker uses custom obfuscation Layer 4 — IPS: passes (novel technique) Layer 5 — EDR: payload executes on endpoint Behavior: Word → PowerShell → network → C2 EDR: suspicious behavior chain → KILL + ALERT ✓ Five layers — attacker stopped at layer 5 Each missed by earlier layers — caught by later ones # This is why you need ALL these controls

The Six Enterprise Controls — At a Glance

🔥 Firewall

Controls network traffic by IP, port, protocol, application, and user identity. First and most fundamental control. Perimeter and internal enforcement.

🚨 IDS / IPS

Detects (IDS) or blocks (IPS) malicious traffic patterns in real time. Signature-based and behavioral. Inline or out-of-band.

🌐 DNS Filtering

Blocks access to malicious, inappropriate, or unauthorized domains at the DNS resolution layer. First layer of web access control.

🛑 DLP

Prevents sensitive data from leaving the organization. Monitors email, web uploads, USB transfers, cloud storage, and printing.

🔐 NAC

Controls who and what can access the network. Verifies device health before granting access. Quarantines non-compliant devices.

🖥️ EDR / XDR

Behavior-based endpoint monitoring and response. Detects threats by what they do. Automated response capabilities.

Where Each Control Lives in the Network

# Network topology — where controls sit: INTERNET ↓ [Perimeter Firewall] ← filters by IP/port/protocol ↓ [IPS — inline] ← blocks malicious traffic patterns ↓ [DNS Sinkhole/Filter] ← blocks bad domains before connection ↓ [DMZ] └─ Web/Email servers ↓ [Internal Firewall] ← controls segment-to-segment traffic ↓ INTERNAL NETWORK ↓ [NAC] ← checks device before network access ↓ [End User Devices] └─ EDR agent ← behavior monitoring on every device ↓ [DLP — endpoint] ← controls what data leaves the device # Also: [DLP — network] ← monitors email and web uploads [DLP — cloud] ← monitors SaaS data movement [SIEM] ← correlates logs from ALL the above

Defense in depth →Layered controls — each catches what others miss
No single control →Every enterprise control has gaps — layers compensate
Security architecture →Right control at the right layer — perimeter, network, endpoint, data
02Modifying Firewall Rules & PoliciesRules · Zones · ACLs

Firewalls are only as good as their rules. Modifying firewalls means adding, changing, or removing rules to respond to new threats, new services, business changes, or discovered misconfigurations — all through a formal change management process.

Implicit Deny — Most Important Firewall Concept

# Default firewall behavior: Rules evaluated TOP to BOTTOM First matching rule wins → action applied No rule matches → IMPLICIT DENY — traffic blocked # This means: Everything NOT explicitly allowed is DENIED Firewall starts in "deny all" state You add ALLOW rules for legitimate traffic Anything you forget to allow → blocked Secure by default ✓ # Common mistake: Admin creates: ALLOW any → any (for testing) Forgets to remove it All traffic allowed — firewall is useless ✗

Types of Firewall Modifications

Adding New Rules

New service deployed, new business requirement, incident response requires blocking threat actor IPs. Must be specific — avoid overly broad rules. Proper change request required. Test in staging first.

# New web app deployed — add rule: # Before: DENY any → WebServer:8080 # Change request approved: INSERT rule 150: ALLOW 10.1.0.0/24 → WebServer:8080 TCP # Only allow internal subnet, not internet

Blocking Threats

Incident response: block attacker IP immediately. Block countries with no business relationship. Block known-malicious IP ranges (threat intelligence feeds). Block specific attack patterns via application firewall rules.

# Active attack — emergency rule: INSERT rule 10 (highest priority): DENY 185.220.101.0/24 → any any # Block known Tor exit node range # Log every attempt

Rule Cleanup / Optimization

Unused rules accumulate over time. Shadowed rules (higher rule already covers it — lower rule never fires). Overly broad rules that should be tightened. Firewall rule review should be quarterly.

# Rule audit findings: Rule 250: ALLOW any→any TCP 80 (2018) Last match: never → DELETE Rule 300: ALLOW 10.0.0.0/8→DB TCP 3306 Should be: 10.1.5.20→DB TCP 3306 → TIGHTEN — least privilege

Zone-Based Changes

Moving a server from DMZ to internal. Creating new network segment. Changing trust relationships between zones. Updating inter-VLAN routing policies. Adjusting what DMZ can reach internally.

Firewall Zones and Traffic Flow

# Standard enterprise zone model: UNTRUST (Internet): No trust — treat all as hostile Deny by default Only allow inbound to DMZ specific services DMZ (Demilitarized Zone): Semi-trusted — publicly accessible servers Web servers, email servers, DNS servers Allow: Internet → DMZ (specific ports only) Allow: DMZ → Internal (limited, specific) DENY: DMZ → Internal (all others) DENY: Internet → Internal (all) TRUST (Internal): Trusted — employees and corporate systems Allow: Internal → Internet (via proxy) Allow: Internal → DMZ (management) Restrict: Internal segment → segment (microseg) OT/IoT Zone (if applicable): Isolated — factory floor, building systems Allow: Monitoring only → internal DENY: OT → Internet (strict) Air gap or data diode preferred # Key rule: DMZ → Internal should be minimal # Compromised DMZ server should NOT reach AD

NGFW-Specific Modifications

Application Control

Block specific apps regardless of port. Block Dropbox personal (allow Dropbox Business). Block TikTok, gaming, streaming. Block Tor browser. Identify by deep packet inspection not just port number.

User-Based Rules

Rules tied to AD identity not IP address. Finance group → allow finance servers. Marketing → block code repositories. Contractors → block sensitive file shares. IP changes — policy follows user.

SSL Inspection Policy

Decrypt and inspect HTTPS traffic. Bypass: banking sites, health sites (privacy). Inspect: social media, file sharing, general browsing. Certificate installed on endpoints to prevent browser warnings.

Firewall Rule Change Management Process

1
Request Submitted
Business owner or IT submits change request — what rule, why needed, impact assessment, risk level, rollback plan.
2
Security Review
Security team reviews: does this open unnecessary risk? Does it follow least privilege? Is there a safer way to achieve the goal?
3
CAB Approval
Change Advisory Board approves. Emergency changes use emergency CAB process (faster). Standard changes go through weekly CAB.
4
Test in Staging
Apply rule to test firewall first. Verify legitimate traffic still works. Verify blocked traffic is blocked. No surprises in production.
5
Production Implementation
Apply during maintenance window (or immediately for emergencies). Document exact rule applied. Screenshot before/after configuration.
6
Validate & Document
Test that new rule works as expected. Close change ticket. Update firewall rule documentation. Set review date if temporary rule.

03Modifying IDS / IPSSignatures · Rules · Tuning

IDS/IPS systems require continuous tuning to remain effective. Out-of-the-box, they generate massive false positive noise. Over time, rules become stale. New threats need new signatures. Tuning is the difference between a useful tool and a noisy distraction.

IDS vs IPS — Key Difference Recap

IDS — Detection Only
Out of band — copy of traffic
Alerts only — cannot block
No impact on performance
False positives = wasted alerts
Placed on SPAN port or TAP
IPS — Detection + Prevention
Inline — all traffic through it
Alerts AND blocks traffic
Performance impact possible
False positives = blocked legit traffic!
Placed between network zones

Detection Methods

Signature-Based Detection

Matches traffic against known attack patterns. Fast and accurate for known threats. Zero detection for new/unknown attacks. Signatures must be continuously updated.

Signature example (Snort rule): alert tcp any any → $HOME_NET 445 ( msg:"MS17-010 EternalBlue exploit"; content:"|FF|SMB|73|"; sid:2024373; rev:1; ) # Matches WannaCry exploit pattern → alert/block

Anomaly/Behavioral Detection

Establishes baseline of normal traffic. Alerts when traffic deviates significantly. Catches unknown/zero-day attacks. Higher false positive rate than signature-based. Requires good baseline period.

Baseline: Normal DNS queries: 500/hour from 192.168.1.0/24 Normal SMB traffic: workstations to file server Anomaly alert: DNS queries: 50,000/hour from one host = DNS tunneling or malware DGA ← ALERT

IDS/IPS Modifications — What Gets Changed

Adding Signatures

New CVE disclosed → add signature to detect exploit attempts. Threat intelligence reveals new attack technique → custom signature. Pen test revealed blind spot → add detection for that technique.

Workflow: 1. New CVE announced (Log4Shell) 2. Vendor releases signature update 3. Test signature against traffic samples 4. Deploy in DETECT mode first 5. Monitor: FP rate acceptable? 6. Switch to BLOCK mode after validation

Updating Signatures

Vendor releases updated signature set regularly. Keep up to date — stale signatures miss new variants. Auto-update where possible for signature sets. Manual approval for significant changes that might impact production.

Tuning — Reducing False Positives

Vulnerability scanner triggers IPS signatures → whitelist scanner IP. Legitimate pentester → temporary bypass. Application generates unusual-looking legitimate traffic → add exception. Business-critical app blocked → tune rule to allow.

# Nessus scanner triggering IPS SQL injection rule: IPS alert: SQL injection from 10.0.0.25 Reality: 10.0.0.25 is vulnerability scanner Solution: add exception suppress rule 2024001 for src 10.0.0.25 # Scanner whitelisted — no more false alerts ✓

Mode Changes — Detect vs Block

New rule always starts in detect (IDS) mode first. Validate it's not causing false positives. After validation period — promote to block (IPS) mode. Rollback plan ready if blocking breaks legitimate traffic.

IPS Placement Options

# Where to place IPS in the network: Perimeter IPS (most common): Internet → [Perimeter IPS] → [Firewall] → Internal Catches: internet-sourced attacks Misses: internal lateral movement Volume: high — all internet traffic Internal IPS (east-west traffic): [Segment A] → [Internal IPS] → [Segment B] Catches: lateral movement between segments Detects: attacker who already got past perimeter More targeted — specific traffic flows Host-based IPS (HIPS): Software on each endpoint Catches: local exploitation attempts Works even offline/on travel Integral part of modern EDR solutions # Best practice: deploy all three layers

Snort / Suricata — Open Source IDS/IPS

# Snort is the most tested IDS/IPS for Security+ exam Snort rule format: action proto src_ip src_port → dst_ip dst_port (options) Examples: alert tcp any any → 192.168.1.0/24 22 ( msg:"SSH brute force attempt"; threshold:type threshold, track by_src, count 5, seconds 60; sid:1000001; ) # Alert if 5+ SSH connections from same source in 60 seconds drop tcp $EXTERNAL_NET any → $HOME_NET 3389 ( msg:"RDP access from internet — block"; sid:1000002; ) # Drop all inbound RDP from internet Actions: alert | drop | reject | pass | log
// Scenario — IPS Tuning After False Positive Wave
# Monday morning — 847 IPS alerts overnight Analyst reviews alerts: Alert: "SQL Injection Attempt" ×847 Source: 10.0.0.50 Destination: webapp.internal:443 Investigation: 10.0.0.50 = Nessus vulnerability scanner Running scheduled scan overnight Scanner's SQL injection tests → triggering IPS Problem: 847 false positives buried real alerts Risk: real SQL injection missed in the noise Fix 1 — Add scanner exception: IPS policy → exceptions: suppress rule 2024SQL for src 10.0.0.50 Result: scanner traffic no longer triggers rule ✓ Fix 2 — Schedule scans during business hours: Alerts during day → analyst sees them immediately Overnight → only critical real threats should alert Fix 3 — Lower threshold for after-hours: Daytime: alert on 10+ SQL patterns After-hours: alert on 1 SQL pattern Result: real attacks at night get immediate attention ✓

IDS →Out of band — alerts only — no performance impact — false positive = wasted time
IPS →Inline — alerts AND blocks — false positive = blocks legitimate traffic
Signature-based →Known attacks only — fast — must update continuously
Anomaly-based →Detects unknown attacks — higher false positives — needs good baseline

04DNS FilteringBlock · Sinkhole · Categories

DNS filtering intercepts DNS queries and blocks resolution of malicious or unauthorized domains before the connection is ever made. It's the fastest and most efficient way to block access to bad destinations — stop it at the DNS layer.

How DNS Filtering Works

# Normal DNS resolution: User types: www.malware-c2.com Browser: DNS query → DNS resolver → answer: 185.220.x.x Browser: connects to 185.220.x.x → malware downloads # With DNS filtering: User types: www.malware-c2.com Browser: DNS query → [DNS FILTER] → checks category/reputation DNS Filter: domain is in MALWARE category DNS Filter: returns SINKHOLE IP (e.g. 0.0.0.0 or block page IP) Browser: connects to block page → user sees "BLOCKED" ✓ Malware: never downloads ✓ # DNS filter sits between clients and resolvers: Client → DNS Filter → (if allowed) → External DNS → (if blocked) → Sinkhole/Block page

What DNS Filtering Blocks

Security Categories

  • Known malware distribution sites
  • Phishing and credential harvesting
  • Command and control (C2) domains
  • Ransomware payment portals
  • Botnet infrastructure
  • Newly registered domains (high risk)
  • Domain generation algorithm (DGA) domains

Policy Categories

  • Adult content
  • Gambling sites
  • Social media (work hours)
  • Streaming video (bandwidth)
  • Peer-to-peer / torrenting
  • Personal cloud storage
  • Gaming sites

Compliance Categories

  • Proxy/anonymizer bypass tools
  • Cryptocurrency sites
  • Dark web access tools
  • Hacking/security tools download
  • Competitor sites (custom policy)
  • Unauthorized SaaS applications

DNS Filtering Modifications

Blocking a Domain

Threat intel identifies new C2 domain. Add to custom blocklist immediately. No waiting for vendor to categorize it. Proactive blocking based on your own threat intel.

# Add to custom blocklist: evil-c2-domain.ru → BLOCK Category: Custom/Threat Intelligence Action: SINKHOLE → block page Reason: Active C2 for observed malware Added by: SecOps team 2024-01-15

Whitelisting (Allow) a Domain

Business partner domain incorrectly categorized. New SaaS tool deployment. Security researcher needs access to security tool sites. Override vendor categorization for your specific needs.

# Allow a domain blocked by vendor categorization: security-research-tools.com → ALLOW Overrides: "Hacking Tools" category Scope: Security team VLAN only (10.0.5.0/24) Reason: Legitimate security team use Approved by: CISO

Category Adjustments

Enable stricter categories for certain departments. Relax for others. Time-based policies — block social media 9am-5pm, allow after hours. Role-based — IT team gets broader access than general staff.

Sinkholing

Redirect queries for known-bad domains to internal server you control. Capture: which devices are trying to reach C2 domains. Identifies already-infected machines. Logging provides forensic evidence of infections.

# DNS sinkhole configuration: *.evil-malware-c2.* → 10.0.0.1 (sinkhole server) Sinkhole server logs all connections Device 10.1.4.22 → sinkhole at 02:14 = 10.1.4.22 is infected with malware ← investigate

DNS Security Protocols

# DNS has security problems — these protocols address them: DNSSEC — DNS Security Extensions: Problem: DNS responses can be spoofed (DNS poisoning) Solution: cryptographically signs DNS records Verifies: response came from authoritative server Verifies: response not modified in transit Limitation: doesn't encrypt queries — names visible DoH — DNS over HTTPS: DNS queries sent over HTTPS (port 443) Encrypted: ISP can't see what you're resolving Problem: bypasses corporate DNS filters! Browsers (Firefox/Chrome) do this by default Enterprise: disable DoH in browsers via GPO/MDM Or: force all DNS through corporate resolver DoT — DNS over TLS: DNS queries encrypted with TLS (port 853) Can be filtered at firewall (block port 853) More enterprise-friendly than DoH Used by: Android, some clients # Enterprise concern with DoH: Employee browser uses DoH → bypasses DNS filter Malware uses DoH → C2 traffic bypasses DNS filter Both cases: your DNS filter is blind Solution: block DoH at firewall (block external 443 DNS)

DNS Filtering Products

Cisco Umbrella (OpenDNS)

Most widely deployed. Cloud-based DNS security. Blocks threats before connection. Roaming client for laptops off-network. Predictive intelligence — blocks domains before they become malicious.

Infoblox

Enterprise DNS management + security. On-premises or cloud. Strong integration with SIEM. Threat intelligence feeds built in. Popular in large enterprises and government.

Zscaler / Cloudflare Gateway

DNS filtering as part of SASE (Secure Access Service Edge) platform. DNS filtering + firewall + CASB in one. Cloud-delivered — works for remote users natively.

05Data Loss Prevention (DLP)Prevent · Detect · Respond

DLP systems identify, monitor, and control sensitive data movement to prevent unauthorized exfiltration. DLP knows what data is sensitive, watches where it goes, and takes action when policy is violated.

DLP Deployment Types

Network DLP

Sits inline on network — monitors email, web uploads, FTP transfers. Scans all outbound traffic for sensitive content. Can block before data leaves org. Best for: email exfiltration, web upload to personal cloud.

Placement: between internal and internet Inspects: SMTP, HTTP/S, FTP, DNS Action: block | quarantine | encrypt | alert

Endpoint DLP

Agent on every device. Controls: USB drives, printing, clipboard, screenshots, application usage. Works offline — enforces policy even without network. Best for: USB exfiltration, printing sensitive docs.

Agent monitors: USB writes → check content Print jobs → check content Clipboard → prevent paste to unapproved apps Screen capture → alert if sensitive content visible

Cloud DLP / CASB

Monitors SaaS applications — Microsoft 365, Google Workspace, Salesforce, Box. CASB (Cloud Access Security Broker) extends DLP to cloud. Detects: sharing sensitive files externally, uploading PII to unapproved services.

DLP Content Detection Methods

# How DLP identifies sensitive data: 1. Regular Expression (Regex) patterns: Credit card: \b(?:\d{4}[-\s]?){3}\d{4}\b SSN: \b\d{3}-\d{2}-\d{4}\b Canadian SIN: \b\d{3}\s\d{3}\s\d{3}\b Email: \b[A-Za-z0-9._%+−]+@[A-Za-z0-9.−]+\.[A-Z]{2,}\b → Simple, fast, high false positive rate 2. Data fingerprinting (exact matching): Upload exact sensitive documents to DLP system DLP creates "fingerprint" (hash) of content Match any document containing that content → Very accurate, low false positives Used for: specific contracts, source code, reports 3. Keyword/dictionary matching: Alert on: "confidential", "HIPAA", "top secret" Alert on custom terms: product codenames, exec names → Simple, easy to configure, context-free 4. ML/AI classification: Trained model classifies content (medical, financial, legal) No need to define exact patterns → Most accurate, requires training data 5. File metadata: Check file classification label Documents labeled "Confidential" get extra controls Microsoft Purview sensitivity labels

DLP Monitored Channels

📧 Email Outbound email scanned for PII, credit cards, classified keywords. Attachments analyzed. Recipient domain checked (external = higher risk). Block · Quarantine · Encrypt
🌐 Web Upload HTTP/S POST requests inspected. File uploads to personal cloud (Dropbox, Google Drive) blocked. SSL inspection required to see HTTPS uploads. Block · Alert
💾 USB / Removable USB write operations scanned. Can allow read-only USB. Block unapproved USB devices. Allow only encrypted/managed USB drives. Block · Encrypt · Allow
🖨️ Printing Print jobs scanned before printing. Watermarking applied automatically. Block printing of classified documents on non-secure printers. Block · Watermark
☁️ Cloud / SaaS SharePoint external sharing. OneDrive personal sync. Salesforce data export. Box external collaboration. All monitored and controlled. Block · Alert · Log
📋 Clipboard Prevent copy-paste of sensitive data from corporate apps to personal apps. Block pasting financial data into personal note-taking apps. Block · Alert

Finance team legitimately emails financial data to auditors. Add trusted recipient domain exception. Set threshold — single credit card number vs 100+. Context-aware — HR can email SSNs to payroll processor.

# Too many FPs — finance email to auditor: Policy: block SSN in outbound email Exception: If recipient: @deloitte.com AND sender: [email protected] AND approved: true → ALLOW with encryption + logging

New Data Categories

M&A activity — add rules for merger codename detection. New regulation — GDPR requires protecting EU PII. New product launch — protect IP with document fingerprinting. Healthcare acquisition — add HIPAA rules.

06NAC — Network Access ControlPosture · Quarantine · Identity

NAC ensures that only authorized, healthy, compliant devices can access the network. It checks device health (posture) at connection time and enforces access policies based on what it finds — not just who is connecting, but in what state.

NAC Components

# NAC three-party model: SUPPLICANT (device trying to connect) → Runs NAC agent (or uses 802.1X) → Reports: OS version, patches, AV status, config POLICY ENFORCEMENT POINT → Switch, wireless AP, VPN gateway → Blocks traffic until NAC decision made → Enforces: allow, quarantine, or deny POLICY DECISION POINT (NAC server) → Evaluates posture against policy → Consults: AD for identity, MDM for device status → Returns: allow full access | quarantine | deny # Flow: Device connects → 802.1X auth → identity verified → Posture check → device health verified → Policy evaluated → access decision → Correct VLAN assigned

Posture Assessment — What NAC Checks

✅
COMPLIANT — Full Network Access
OS up to date + patches current + AV running + AV definitions current + MDM enrolled + disk encrypted + no prohibited software + firewall enabled → All checks passed → Full access to production VLAN
⚠️
NON-COMPLIANT — Quarantine VLAN
Missing patches OR AV definitions outdated OR disk not encrypted → Quarantine VLAN only → Access to: patch server, AV update server, help desk portal → Cannot reach: file servers, email, internet until remediated
🚫
UNKNOWN / GUEST — Guest VLAN
No MDM enrollment, no corporate certificate, no 802.1X credentials → Guest VLAN → Internet access only → Cannot reach any corporate resources → Must enroll to get corporate access

NAC Modifications — Policy Changes

Adding New Posture Checks

New security requirement — all devices must have EDR agent. New regulation — all devices accessing cardholder data must be encrypted. Zero trust initiative — verify device trust on every connection not just first.

# New policy: require CrowdStrike agent Old check: AV running (any AV) New check: CrowdStrike Falcon agent version 6.50+ running last sync < 24 hours If fails → quarantine VLAN "Please contact IT to install Falcon"

VLAN Assignment Rules

Role-based VLAN assignment — AD group membership determines VLAN. Finance users → Finance VLAN. Contractors → Restricted VLAN. IoT devices → IoT isolation VLAN. Security posture → quarantine VLAN if needed.

Identity: jsmith — member of Finance-AD-Group Device: compliant Result: Finance VLAN (VLAN 20) assigned Identity: [email protected] Device: unmanaged Result: Contractor VLAN (VLAN 50) assigned # Can reach: vendor-specific servers only

Agentless NAC (for IoT/OT)

IoT devices can't run NAC agent. NAC uses: MAC address OUI lookup (device type), network behavior profiling, traffic pattern analysis to categorize device. Auto-assign IoT VLAN based on device fingerprint.

Guest Network Configuration

Visitors get captive portal authentication. Time-limited access. Internet only — no corporate network. Bandwidth throttled. Logged for compliance. Isolated from corporate VLAN completely.

NAC Enforcement Methods

# 802.1X — primary NAC enforcement mechanism: Wired: Device → switch port → 802.1X challenge Device: presents certificate OR username/password RADIUS: validates against AD RADIUS → switch: "Assign VLAN 20, full access" Switch: opens port, assigns VLAN Wireless: Device → AP → 802.1X (WPA2/WPA3 Enterprise) Same process — RADIUS validates RADIUS → AP: "Assign SSID Corp, full access" # What happens when non-802.1X device connects: Unknown MAC connects to switch port Switch: MAB (MAC Authentication Bypass) RADIUS: MAC not in approved list Result: Guest VLAN assigned automatically IoT device fingerprinted → IoT VLAN # NAC products: Cisco ISE (Identity Services Engine) — most widely used Forescout eyeSight — agentless, good for IoT Aruba ClearPass — strong wireless integration Microsoft NPS (Network Policy Server) — Windows environments

07Modifying EDR / XDR PoliciesBehavioral · Response · Tuning

EDR/XDR systems require continuous policy modification to stay effective — new software causes false positives, new threats need detection rules, response actions need tuning. Managing EDR is an ongoing security operations task.

EDR vs XDR Scope

# What each covers: EDR — Endpoint Detection and Response: Coverage: individual endpoint behavior Process creation + parent/child relationships File system changes Registry modifications Network connections from endpoint Memory analysis XDR — Extended Detection and Response: Coverage: endpoint + network + cloud + email + identity Everything EDR sees PLUS: Network traffic patterns (NDR) Cloud API activity (CSPM) Email threat data Identity anomalies (UEBA) Correlated across ALL sources # XDR = unified platform replacing separate tools # Single console, correlated detections, unified response

EDR Policy Modifications

Detection Rule Creation

Create custom behavioral rules for threats specific to your environment. Pen test found technique not caught by default rules. Threat intel reveals novel attack technique. Industry-specific attack patterns.

# Custom EDR rule (CrowdStrike IOA): Trigger: Process creation Where: parent = excel.exe And: child = powershell.exe OR cmd.exe And: child has encoded command (-enc flag) Response: BLOCK + alert analyst Name: "Excel macro spawning encoded shell"

Exclusions / Whitelisting

Legitimate software triggers EDR false positives. Software deployment tools look like malware. IT scripts trigger behavioral rules. Security tools themselves trigger detections. Careful whitelisting prevents blocking business tools.

# SCCM deployment triggering EDR: CcmExec.exe running encoded PowerShell EDR: BLOCKED — looks like malware! IT: that's Microsoft SCCM! legitimate! Fix: add path exclusion C:\Windows\CCM\* → trusted process Signed by Microsoft → allow Result: SCCM works, real threats still detected ✓

Response Policy Tuning

Adjust what happens automatically when threat detected. High-confidence detections → auto-isolate. Lower confidence → alert only for analyst review. Time-based policies — less aggressive during maintenance.

# Response tiering: Severity Critical (100% confidence): Auto: isolate host Auto: kill process Auto: alert P1 Severity High (80% confidence): Auto: kill process Auto: alert analyst Manual: isolation decision Severity Medium (60% confidence): Auto: alert analyst Manual: all response actions

Threat Hunting Rules

Create scheduled queries to proactively hunt for threats that haven't triggered alerts. "Show me all processes that have made connections to newly registered domains." Run weekly, investigate results manually.

XDR Correlation Rules

# XDR correlates across sources — example rule: Rule: "Multi-stage attack — email to endpoint to cloud" Trigger when ALL of these occur within 60 minutes: [Email] Suspicious attachment opened by user X AND [EDR] User X's endpoint spawns PowerShell AND [Network] User X's endpoint connects to new external IP AND [Cloud] User X's account accesses cloud storage for first time Each alone: low confidence All four together: HIGH CONFIDENCE — active attack Response: isolate endpoint + disable cloud account + P1 alert # This catches what individual tools miss # The correlation is the detection

EDR Rollback — Unique Feature

# Some EDR platforms can UNDO ransomware damage: Ransomware starts encrypting files: file1.docx → file1.docx.locked file2.xlsx → file2.xlsx.locked file3.pptx → file3.pptx.locked EDR (SentinelOne) detects ransomware behavior EDR: kills ransomware process EDR ROLLBACK: uses VSS/shadow copies to restore file1.docx.locked → file1.docx (restored!) file2.xlsx.locked → file2.xlsx (restored!) file3.pptx.locked → file3.pptx (restored!) Result: no files permanently lost ✓ Zero data recovery cost ✓ # Available in: SentinelOne, some CrowdStrike tiers # Requires: EDR detect BEFORE VSS deletion

EDR Deployment Considerations

Agent Deployment

EDR agent required on every endpoint — Windows, macOS, Linux, servers. Deploy via SCCM/Intune/Ansible. Verify coverage — unprotected endpoints = blind spots. Mobile devices: use MDM + MAM instead.

Audit vs Protect Mode

New deployments start in Audit mode — detects but doesn't block. Build confidence — verify FP rate acceptable. After validation — switch to Protect mode. Same as detect vs block in IDS/IPS.

Performance Impact

EDR agent uses CPU/RAM. Tune scanning aggressiveness for servers vs workstations. Exclude known-good high-frequency paths (database transaction logs). Balance security vs performance.
