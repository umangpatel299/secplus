01The CIA TriadConfidentiality · Integrity · Availability

The CIA triad is the foundational model for information security. Every security control, policy, and risk decision maps back to protecting one or more of these three properties. It is the most important concept in Security+.

C
Confidentiality
Ensuring data is accessible only to those who are authorized to access it. Keeping secrets secret.
I
Integrity
Ensuring data is accurate, complete, and has not been altered by unauthorized parties.
A
Availability
Ensuring systems and data are accessible when legitimate users need them.

Confidentiality — Keep Secrets Secret

Confidentiality means only authorized parties can read or access sensitive information. The threat to confidentiality is unauthorized disclosure — someone who shouldn't see the data sees it.

# Threats to Confidentiality: Eavesdropping (sniffing network traffic) Man-in-the-middle (MITM) attack Unauthorized access (stolen credentials) Shoulder surfing (looking at screen) Dumpster diving (finding printed documents) Data exfiltration (attacker steals data) Social engineering (tricks someone into revealing) # Controls that protect Confidentiality: Encryption: renders data unreadable without key at rest: BitLocker, AES-256 in transit: TLS 1.3, HTTPS, SSH Access Control: only authorized users see the data ACLs, RBAC, need-to-know principle MFA: blocks unauthorized account access Data masking: hides real values (CC: **** **** **** 4321) Steganography: hides data inside another file (covert channel) Physical security: locks, screen shields, clear-desk policy DLP: prevents data from leaving the org

Integrity — Ensure Data Accuracy

Integrity means data has not been tampered with, corrupted, or altered without authorization. You can be confident the data you receive is exactly what was sent — and that it was sent by who you think sent it.

# Threats to Integrity: Man-in-the-middle (modifying data in transit) SQL injection (modifying database records) Insider threat (malicious data alteration) Ransomware (encrypts/corrupts files) Hardware failure (bit rot, storage corruption) Accidental modification (human error) # Controls that protect Integrity: Hashing: creates a fingerprint of data MD5 (weak/broken), SHA-256, SHA-3 if hash changes → data changed Digital signatures: proves sender identity + data not modified private key signs → public key verifies HMAC: hash + secret key → proves source + integrity Checksums: simple integrity check for transmission Code signing: software hasn't been tampered since release Version control: tracks all changes with attribution FIM: File Integrity Monitoring — alerts on file changes Input validation: reject malformed/malicious input # Integrity ≠ Confidentiality: Encrypted file: confidential (can't read it) but integrity unknown Hashed file: integrity checked but still readable (not confidential) Best: encrypt AND hash → both properties

Availability — Systems Work When Needed

Availability means authorized users can access systems and data when they need them. The threat to availability is denial — legitimate users cannot do their work.

# Threats to Availability: DDoS (Distributed Denial of Service) attack Ransomware (encrypts files — no access) Hardware failure (disk crash, power failure) Natural disaster (flood, earthquake, fire) Software bug causing system crash Accidental deletion by user Network outage (ISP failure, cable cut) # Controls that protect Availability: Redundancy: duplicate systems, no SPOF Backups: 3-2-1 rule, tested restores Failover clustering: automatic switchover on failure Load balancing: distribute across multiple servers UPS + generator: survive power failures DDoS mitigation: Cloudflare, Akamai, AWS Shield Patch management: prevents crashes from vulnerabilities DR site: Cold/Warm/Hot site for major outages # Availability vs Security tension: More security controls → sometimes less convenient (availability ↓) Less security → more available but more vulnerable Goal: balance — security should not unnecessarily impair availability

CIA Triad — Attacks vs Controls Summary

PropertyThreatened ByProtected ByPrimary Goal
Confidentiality Eavesdropping, data theft, unauthorized access Encryption, MFA, access control, DLP Data visible only to authorized parties
Integrity Tampering, MITM, SQL injection, corruption Hashing, digital signatures, FIM, input validation Data accurate and unmodified
Availability DDoS, ransomware, hardware failure, disaster Redundancy, backups, failover, DDoS mitigation Systems accessible when needed

Extra Properties Beyond CIA

Non-Repudiation

Ensuring a party cannot deny having performed an action. Digital signatures prove you sent a message. Audit logs prove you accessed a system. (Full coverage — Tab 2)

Authenticity

Ensuring data or communications genuinely originate from the claimed source. Verified by digital signatures, certificates, and hash-based MACs. Closely related to integrity.

Privacy

Individuals' rights to control their personal information. Extends confidentiality to legal/regulatory domain. GDPR, HIPAA, CCPA protect privacy. Overlaps with confidentiality but addresses rights, not just protection.

// Real-World CIA Mapping
# Each incident breaks a specific CIA property: Ransomware attack: Availability ↓ Files encrypted — users cannot access them Integrity ↓ Files have been modified (encrypted) Often NOT confidentiality (attacker may not have read the data) Data breach (attacker steals customer SSNs): Confidentiality ↓ Unauthorized party read the data Integrity: maybe unchanged Availability: maybe unaffected Man-in-the-middle attack (attacker modifies data in transit): Integrity ↓ Data was altered between sender and receiver Confidentiality ↓ Attacker also read the data Availability: likely unaffected (data still arrives) DDoS attack: Availability ↓ Service unreachable Confidentiality: unchanged Integrity: unchanged SQL injection deletes database records: Integrity ↓ Data destroyed/modified without authorization Availability ↓ Data no longer accessible

Ransomware →Primarily Availability attack (can't access files) + Integrity (files modified)
Data breach →Confidentiality attack — unauthorized party accessed data
DDoS →Availability attack — legitimate users denied service
02Non-RepudiationProof · Accountability · Cannot Deny

Non-repudiation means a person or system cannot deny having performed an action — there is irrefutable proof that the action occurred and who performed it. It's accountability enforced by cryptography and logging.

Why Non-Repudiation Matters

# Without non-repudiation: Employee: "I never sent that email authorizing the $500K transfer" Admin: "I never deleted those files" Vendor: "We never signed that contract" User: "Someone else must have accessed that account" # With non-repudiation: Email: digital signature proves it came from their private key Files: audit log + hash proves admin account deleted them at 14:32 Contract: digital signature on PDF = legally binding Access: certificate-based auth + log = irrefutable # Non-repudiation serves TWO parties: Sender: "I can prove I sent this message" (sender) Receiver: "They cannot deny having sent it" (receiver) # Legal importance: Financial transactions: cannot deny authorizing payment Contract signing: cannot deny agreeing to terms Security incidents: cannot deny accessing a system Data modifications: cannot deny changing records

How Non-Repudiation Is Achieved

✍️
Digital Signatures — The Primary Mechanism
"A cryptographic mechanism that proves a message was created by a specific private key holder and has not been modified since signing."
Process: 1. Sender hashes the message → 2. Encrypts hash with private key = signature → 3. Sends message + signature → 4. Receiver decrypts signature with sender's public key → 5. Hashes received message → 6. Compares: if hashes match → authenticated + integrity confirmed + non-repudiation achieved.

Why non-repudiation: Only the private key holder could have produced that signature. Private key is secret to one person. Therefore that person cannot deny signing.
# Digital signature process: Alice wants to sign a contract: 1. hash = SHA256(contract_text) // "fingerprint" of document 2. signature = RSA_encrypt(hash, Alice_private_key) 3. sends: contract_text + signature Bob receives and verifies: 4. hash_received = SHA256(contract_text) // hash what arrived 5. hash_from_sig = RSA_decrypt(signature, Alice_public_key) 6. if hash_received == hash_from_sig: ✓ Integrity: document not modified ✓ Authentication: Alice's key was used ✓ Non-repudiation: Alice cannot deny signing
📋
Audit Logs — Operational Non-Repudiation
"Tamper-evident records of who did what, when, on which system — providing an irrefutable trail of actions."
For non-repudiation to hold, logs must be: tamper-evident (stored where subject can't modify), authenticated (tied to a specific identity — not shared accounts!), time-stamped with synchronized NTP, and retained long enough for legal/compliance purposes. Shared accounts destroy non-repudiation — "admin" could be anyone.
📜
Certificates (PKI) — Identity Foundation
"Digital certificates bind a public key to a verified identity — enabling trust in who you are communicating with."
A certificate from a trusted CA says: "This public key belongs to this identity, and the CA has verified this." When someone signs with a private key and you verify with the certificate's public key, you're relying on the CA's verification of identity. The chain: Certificate Authority → verifies identity → issues cert → cert binds identity to public key → private key signs → non-repudiation achieved.

Non-Repudiation vs Authentication

Authentication
Proves WHO you are
Verifies identity at a point in time. "This is Alice logging in." Can use: passwords, MFA, certificates. Doesn't necessarily prove what you DID after authenticating.
Non-Repudiation
Proves what you DID
Proves a specific action was performed by a specific party — and they cannot deny it. "Alice signed this document with her private key at 14:32:07." Requires authentication + cryptographic proof of action.
// Scenario — Wire Transfer Dispute
# CFO disputes authorizing a $250,000 wire transfer: Without non-repudiation: CFO: "I never approved that transfer" Attacker used CFO credentials (stolen via phishing) System log: "CFO account authorized transfer" Password-only auth: shared creds possible → dispute valid With non-repudiation controls: Financial system requires: digital signature on all transfers Transfer signed with CFO's private key (stored in smart card) Smart card requires PIN + physical possession Log: signature verified → CFO_cert → issued by Corporate CA CFO: "I never approved that transfer" Evidence: signature only producible by CFO's private key smart card was in CFO's possession (physical log) CFO's PIN was required to access smart card Result: non-repudiation achieved → dispute resolved Lesson: password alone ≠ non-repudiation digital signature with cert = non-repudiation

// Exam Keywords

Non-repudiation →Cannot deny an action occurred — cryptographic proof — digital signatures are primary mechanism
Digital signature →Private key signs (hash) → public key verifies — proves identity + integrity + non-repudiation
Shared accounts →Destroy non-repudiation — "admin" could be anyone — always use named individual accounts
Audit logs →Operational non-repudiation — must be tamper-evident + authenticated + timestamped
Password alone →Does NOT provide non-repudiation — someone else could know your password
Private key →Non-repudiation relies on private key secrecy — only key holder could have signed

04Authentication — Deep DiveWho Are You? · Factors · Methods

Authentication answers "Who are you?" by verifying claimed identity. Strong authentication requires multiple independent factors — reducing the probability that a compromised single factor grants access.

Three Authentication Factors

Something You Know

Knowledge factor. Based on a secret only you know. Easy to use, easy to forget, can be shared or stolen.

  • Password / passphrase
  • PIN number
  • Security questions
  • Cognitive password (mother's maiden name)

Something You Have

Possession factor. Based on a physical object you possess. Harder to steal remotely, but can be lost or physically taken.

  • TOTP app (Google Authenticator)
  • Hardware token (RSA SecurID)
  • Smart card / PIV card
  • FIDO2 security key (YubiKey)
  • Mobile phone (SMS OTP)

Something You Are

Inherence factor. Based on a physical characteristic of you — biometrics. Hard to steal remotely but can have false acceptance/rejection issues.

  • Fingerprint
  • Facial recognition
  • Iris / retinal scan
  • Voice recognition
  • Keystroke dynamics

Additional Context Factors

Somewhere You Are

Location factor. Geolocation — IP-based or GPS. "Only allow login from within the US." Conditional access policies check location. Zero Trust uses location as one signal. Not a standalone factor — used to strengthen others.

Something You Do

Behavioral factor. Behavioral biometrics — typing rhythm, mouse movement, gait analysis. Continuous authentication — verifies throughout session, not just at login. Risk-adaptive auth — unusual behavior triggers re-authentication.

Multi-Factor Authentication (MFA)

# MFA = two or more DIFFERENT factor TYPES: Valid MFA: Password (know) + TOTP code (have) ← ✓ MFA PIN (know) + fingerprint (are) ← ✓ MFA Smart card (have) + PIN (know) ← ✓ MFA Password (know) + YubiKey (have) ← ✓ MFA NOT valid MFA (same factor type twice): Password + security question ← ✗ both = "know" Two passwords ← ✗ both = "know" Fingerprint + iris scan ← ✗ both = "are" (though more secure) # MFA strength comparison: Weakest: SMS OTP (SIM swapping attack possible) Better: TOTP app (authenticator) — harder to intercept Strong: FIDO2 / WebAuthn (hardware key or passkey) phishing-resistant — bound to specific site Strongest:PIV / Smart card with PIN (government standard) # Why MFA matters: Microsoft: MFA blocks 99.9% of automated account attacks Even if password stolen → attacker still needs second factor

Authentication Methods

Password-Based Auth

Most common but weakest alone. Vulnerabilities: phishing, credential stuffing, brute force, dictionary attacks, password spraying. Best practices (NIST 800-63B): long passphrases >14 chars, no forced rotation, check against breach databases, no complexity theater.

Certificate-Based Auth

PKI certificates instead of passwords. User presents certificate → server validates against trusted CA. Most secure — no secret transmitted. Used in: government (PIV/CAC cards), enterprise SSO, mutual TLS (mTLS). Provides non-repudiation — private key uniquely identifies person.

SSO — Single Sign-On

Authenticate once → access many systems without re-authenticating. Protocols: SAML 2.0 (XML-based, enterprise), OIDC (OAuth 2.0 extension, web/mobile), Kerberos (on-premises AD). SSO reduces password fatigue but creates single point of failure — SSO compromise = all apps compromised.

Biometric Considerations

FAR (False Acceptance Rate): unauthorized person accepted — security concern. FRR (False Rejection Rate): authorized person rejected — availability concern. CER (Crossover Error Rate): where FAR=FRR — overall accuracy metric. Lower CER = better biometric system. Cannot be changed if compromised (unlike password).

// Exam Keywords

Something you know →Knowledge factor — password, PIN, security question
Something you have →Possession factor — TOTP, smart card, YubiKey, phone
Something you are →Inherence/biometric factor — fingerprint, facial, iris
MFA →TWO OR MORE DIFFERENT factor types — password + security question is NOT MFA (both knowledge)
FAR →False Acceptance Rate — wrong person admitted — security risk
FRR →False Rejection Rate — right person denied — availability risk
FIDO2 →Phishing-resistant MFA — hardware security key — bound to specific website
SAML →SSO protocol — XML-based — enterprise federation — IdP to SP
05Authorization — Deep DiveWhat Can You Do? · Access Models

After authentication establishes identity, authorization determines what that identity is permitted to access and what actions they may perform. The authorization model defines the rules used to make these decisions.

Core Authorization Models

DAC — Discretionary Access Control
"Resource owners decide who can access their resources — access is at the owner's discretion."
The resource owner sets permissions — not a central authority. Example: Windows NTFS permissions — the file creator can share with others. Most flexible but least secure — resource owners may set overly permissive access. Data leakage risk: owner might grant access inappropriately. Common in: Windows file systems, most consumer operating systems.
Alice creates a document → Alice is the owner Alice gives: Bob (read), Carol (read/write), Dave (full control) Alice's discretion → Alice decides → not centrally controlled Risk: Alice could accidentally share with everyone
MAC — Mandatory Access Control
"Access is based on classification labels assigned by a central authority — neither owner nor user can override."
Government/military model. Every subject (user/process) has a clearance level. Every object (file/resource) has a classification label. Access: subject clearance ≥ object classification → access allowed. No-write-down rule (Bell-LaPadula): prevents leaking classified data to lower levels. Most secure but least flexible. No individual can override the policy.
Labels: Top Secret > Secret > Confidential > Unclassified Alice has: Secret clearance File A has: Confidential label → Alice CAN access File B has: Top Secret label → Alice CANNOT access Alice cannot: grant access to File A to anyone else Central authority: controls ALL access decisions
RBAC — Role-Based Access Control
"Access is based on a user's role within the organization — users inherit permissions from their assigned role."
Most common enterprise model. Assign users to roles → roles have permissions → users inherit permissions. Easy to manage at scale — change a role, affect all role members. Principle of least privilege: each role has only the permissions needed for that job. Examples: "Finance-User" role, "IT-Admin" role, "HR-Manager" role. Role explosion: too many granular roles → management burden.
Roles defined: Finance-User, IT-Admin, HR-Manager, Developer Alice is in: Finance-User → gets: accounting apps, read-only reports Bob is in: IT-Admin → gets: server access, AD, monitoring Carol moves: HR → Finance → remove HR-Manager, add Finance-User One change → all permissions update instantly
ABAC — Attribute-Based Access Control
"Access decisions based on multiple attributes of the user, resource, and environment — the most flexible and granular model."
Combines: who you are (identity attributes), what you're accessing (resource attributes), and context (environmental attributes). "Finance-User in the UK, during business hours, on a managed device, can access EU financial reports." Very granular — can implement complex policies. Foundation of Zero Trust — every request evaluated against multiple attributes. More complex to manage than RBAC.
Policy: "Allow access IF: user.department = Finance AND user.clearance >= Confidential AND resource.classification <= Confidential AND environment.time = business_hours AND device.compliant = true AND location.country != high_risk" This single policy replaces dozens of individual RBAC rules

06Accounting — Deep DiveWhat Did You Do? · Logging · Auditing

Accounting records what authenticated, authorized users actually do — providing an audit trail for forensics, compliance, billing, and non-repudiation. Without accounting, authentication and authorization alone are insufficient.

What Accounting Captures

# Accounting answers: "What happened? Who did it? When? Where?" Per session: Identity: who authenticated (username + auth method) Source: where from (IP address, device, location) Time: when session started and ended Duration: how long the session lasted Resources: what was accessed (files, systems, data) Actions: what was done (read, write, delete, exec) Data volume: bytes transferred (useful for exfil detection) Outcome: success or failure (auth failures are important!) # Why accounting matters: Forensics: reconstruct what happened during an incident Compliance: HIPAA/PCI/SOX require specific log retention Non-repudiation: "you WERE in that system at that time" Billing: ISPs/cloud charge per usage Anomaly det.: unusual patterns → alerts (SIEM) Insider threat: mass download before resignation

Logging Requirements

What to Log

  • All authentication events (success AND failure)
  • Privilege use (admin actions, sudo, runas)
  • Object access (file open, read, write, delete)
  • Policy changes (firewall rules, user accounts)
  • Process creation (new executable launched)
  • Network connections (source, dest, port, protocol)
  • System events (boot, shutdown, service start/stop)

Log Integrity Requirements

  • Tamper-evident: logs on separate, write-protected system
  • Time sync: NTP — all devices synchronized for correlation
  • Retention: HIPAA (6 yr), PCI (1 yr online + 1 yr archived), SOX (7 yr)
  • Centralized: SIEM collects all — single pane of glass
  • Protected from deletion: WORM storage or SIEM-controlled
  • Reviewed: logs only have value if someone reads them

Syslog — Centralized Log Collection

# Syslog: standard protocol for sending log data Syslog (legacy): UDP 514 — plaintext — no reliability guarantee Syslog-TLS: TCP 6514 — encrypted — reliable delivery Use this — never plaintext logs over network Syslog severity levels (0 = most critical): 0: Emergency — system unusable 1: Alert — immediate action needed 2: Critical — critical condition 3: Error — error condition 4: Warning — warning condition 5: Notice — normal but significant 6: Informational — informational 7: Debug — debug messages (very verbose — usually not sent to SIEM) # SIEM — Security Information and Event Management: Aggregates logs from: firewall, AD, EDR, servers, cloud Normalizes: different formats → single schema Correlates: "failed login from 5 IPs = brute force?" Alerts: rule-based + behavioral analytics Stores: long-term retention for compliance
// Scenario — Accounting Solves an Insider Theft
# Employee suspected of stealing customer data before resignation: Accounting records show: Monday 17:42: jsmith authenticated to CRM system Monday 17:43-18:15: accessed 45,000 customer records Monday 18:14: exported CSV file "customers_all.csv" (234 MB) Monday 18:17: connected to personal Dropbox (DLP alert triggered) Monday 18:18: uploaded 234 MB to dropbox.com Monday 18:21: deleted "customers_all.csv" from workstation Monday 18:22: session ended Forensics: SIEM alert: mass data export → DLP → triggered at 18:14 DLP log: Dropbox upload blocked? → policy misconfigured → allowed EDR: file deletion logged + deleted file recovered from MFT Result: jsmith cannot deny any action — accounting provides proof Each action timestamped, logged, correlated in SIEM Lesson: accounting is the record of what actually happened

// Exam Keywords

Syslog →Log forwarding protocol — UDP 514 (legacy) / TCP 6514 with TLS (secure)
SIEM →Aggregates, correlates, alerts on logs — Splunk, Sentinel, QRadar

07Zero TrustNever Trust · Always Verify · Assume Breach

Zero Trust is a security philosophy and architecture where no user, device, or network location is trusted by default — every access request must be verified, regardless of whether it originates inside or outside the corporate network.

The Problem Zero Trust Solves

# Traditional "castle and moat" security model: OUTSIDE network: untrusted → blocked by firewall INSIDE network: trusted → free to communicate Problem: this model fails when: Remote work: employees work from home, coffee shops → "inside" doesn't mean anything Cloud: apps are in AWS/Azure → no "inside" perimeter to protect Lateral movement: attacker breaches perimeter → inside = trusted → moves freely Supply chain: SolarWinds → vendor is "trusted" → compromised # Real impact of flat trusted-inside networks: Target 2013: HVAC vendor → inside network → POS systems SolarWinds: trusted vendor update → inside → all systems Colonial: VPN cred stolen → inside → operational systems # Zero Trust answer: NEVER trust based on network location — inside OR outside ALWAYS verify: identity + device + context before every request ASSUME breach: act as if attacker is already inside

Zero Trust Core Principles

🔐
1. Verify Explicitly
Always authenticate and authorize based on ALL available data points: identity, location, device health, service/workload, data classification, anomalies. Strong identity verification — MFA required. Certificate-based device identity. Never assume prior authentication is still valid indefinitely.
🎯
2. Use Least Privilege Access
Limit user access with just-in-time (JIT) and just-enough-access (JEA). Risk-based adaptive policies. Protect both data and productivity — not just block everything. ABAC enables granular access decisions. Privileged Access Management (PAM) for admin accounts. No persistent admin access — request, use, revoke.
💥
3. Assume Breach
Design as if the attacker is already inside. Minimize blast radius: micro-segmentation, network isolation. Encrypt all internal traffic — not just external. Comprehensive logging and analytics: detect lateral movement. End-to-end encryption even on internal network — east-west traffic encrypted too.

Zero Trust Architecture Components

Policy Engine + Policy Administrator

The "brain" of Zero Trust. Policy engine evaluates every access request against all signals (identity, device, threat intel, behavior). Policy administrator sends the access decision. Grant, deny, or revoke access in real time. Continuously re-evaluates — not just at login time.

Identity as the Control Plane

In Zero Trust, identity replaces the network perimeter. Who you are (verified identity + MFA) is more important than where you are (network location). Strong identity + device health = access. Weak identity + corporate network = still denied. Azure AD, Okta, Ping are identity providers (IdPs).

Device Health / Posture

Is the device allowed to access resources? Check: OS version, patch status, EDR installed and active, disk encryption enabled, compliance with security policies. Unhealthy device = reduced or denied access even if identity is valid. MDM/MAM: enforce device compliance before granting access.

Micro-Segmentation

Divide network into tiny zones — workload-level isolation. East-west traffic controlled between every workload. Even if attacker reaches one workload, cannot freely reach others. VMware NSX, Illumio, Guardicore. Replace VPN with ZTNA — app-level access not network-level. Zero lateral movement path.

Zero Trust vs Traditional Perimeter Security

Traditional Perimeter
Castle and Moat
• Trust based on network location
• Inside = trusted, outside = untrusted
• Perimeter firewall as primary control
• VPN to "join" the trusted network
• Lateral movement possible once inside
• Fails for cloud, remote work, BYOD
Zero Trust
Never Trust Always Verify
• No implicit trust anywhere
• Identity + device + context = access decision
• Micro-segmentation throughout
• ZTNA replaces VPN — app-level access
• Assume attacker is already inside
• Works for cloud, remote, BYOD, hybrid

// Exam Keywords

Zero Trust principle →Never trust, always verify — no implicit trust from network location
Assume breach →Design as if attacker is already inside — micro-segmentation — encrypt east-west
Identity as perimeter →Who you are replaces where you are as the access control boundary
ZTNA →Zero Trust Network Access — replaces VPN — app-level not network-level access
Least privilege + JIT →Just-in-time access — request, use, revoke — no persistent privilege
Micro-segmentation →Workload-level isolation — limits lateral movement — east-west control
Device posture →Device health check — patch status, EDR, encryption — required before access
08Deception & Disruption TechnologyHoneypots · Honeynets · Honeytokens

Deception technologies lure attackers into interacting with fake assets, revealing their presence, tactics, and tools while disrupting their operations and gathering threat intelligence. Any interaction with a deception resource is itself evidence of malicious intent.

Core Principle of Deception

# Why deception works: Legitimate users: NEVER interact with fake/unlisted resources "Why would an employee access a server that's not in their workflow?" Attackers: interact with everything they find during reconnaissance Attacker scans network → finds "interesting" unlisted server Attacker accesses it → IMMEDIATE alert — no false positives Zero false positives: No legitimate user should EVER touch a honeypot Any access = attacker activity = high-confidence alert Contrast with IDS: legitimate traffic triggers false alarms Honeypot: if you see traffic → it's malicious # Deception answers: "Is there an attacker in my network?" Traditional detection: "I see suspicious traffic — is it an attacker?" Deception: "Someone touched the honeypot — it IS an attacker"

Deception Technology Types

🍯
Honeypot
A single decoy system intentionally configured to appear as a valuable target — but is not a real production system. Any access to a honeypot is immediately suspicious. Low-interaction honeypot: simulates services (fake SSH, fake HTTP) — safer, easier to deploy. High-interaction honeypot: full real OS — more realistic, captures more attacker behavior — but riskier if attacker uses it as pivot. Purpose: detect intruders, study attack techniques, delay attackers wasting time on fake asset.
Single decoy
🌐
Honeynet
A network of honeypots — multiple fake systems forming a fake network segment. More realistic than a single honeypot — appears to be a full network environment. Captures: network-level attack patterns, lateral movement, attacker tool sets, C2 communications. Used by: security researchers, large enterprises, threat intelligence teams. Requires careful isolation — real risk of attacker using honeynet resources to attack others.
Network of decoys
🔑
Honeytoken
Fake data embedded in real systems — not a fake server, but a fake data item. When an attacker steals and uses a honeytoken, it triggers an alert. Examples: fake AWS API key planted in code repo — when used, immediate alert; fake database record with a unique email address — if that email receives spam, the database was stolen; fake credential in password manager — if used for login, credential theft detected. Can be planted anywhere real data exists.
Fake data item
🗄️
Honeyfile
A decoy file placed in a likely exfiltration path. Named to attract attention: "salaries_2024.xlsx", "passwords.txt", "executive_contracts.pdf". Appears valuable — actually fake. Any access or exfiltration of the honeyfile triggers an alert. Commonly deployed in: file servers with sensitive-sounding directories, SharePoint, S3 buckets. Variant of honeytoken — focused on file systems.
Decoy file
🌿
Fake Telemetry / DNS Sinkhole
DNS Sinkhole: redirect malicious domain lookups to a controlled server instead of the real C2. Malware calls home → DNS returns sinkhole IP → security team's server instead of attacker's. Effect: malware can't communicate with C2, attacker's infected hosts are identified (they're contacting sinkhole). Also used to block known malicious domains for all users. Disrupts attacker C2 infrastructure without malware removal.
Disruption
🚪
Fake Network Services / Ports
Open fake ports that appear as running services — but are actually traps. Port 22 (SSH) responds with a convincing banner but logs all attempts. Any connection attempt = reconnaissance detected. Fake services: fake web server, fake database, fake admin portal. Attacker wastes time on fake services → delayed, detected, profiled. Also known as "tarpits" — slow connections down to waste attacker's time.
Trap services

// Exam Scenarios → Answers

"Data only accessible to authorized users"→ Confidentiality — protect with encryption + access control
"Data hasn't been altered without authorization"→ Integrity — protect with hashing, digital signatures, FIM
"Legitimate users can access the system when needed"→ Availability — protect with redundancy, backups, DDoS mitigation
"Ransomware attack — which CIA property primarily affected?"→ Availability (can't access files) AND Integrity (files modified)
"Attacker steals customer data — which CIA property?"→ Confidentiality — unauthorized party read the data
"DDoS takes the website offline — which CIA property?"→ Availability — legitimate users denied access
"Attacker modifies financial records in transit — which CIA?"→ Integrity — data altered without authorization
"Party cannot deny having sent a document"→ Non-repudiation — digital signature provides proof
"What provides non-repudiation for email?"→ Digital signature — private key used to sign = only that person could have
"Shared admin account problem"→ Destroys non-repudiation — "admin" could be anyone — use named accounts
"Verify who you are"→ Authentication — first step of AAA
"Determine what you can access"→ Authorization — second step of AAA — comes AFTER authentication
"Record what actions were performed"→ Accounting — third step of AAA — audit trail
"AAA protocol for VPN and WiFi — UDP"→ RADIUS (UDP 1812/1813) — encrypts password only
"AAA protocol for network devices — TCP — encrypts all"→ TACACS+ (TCP 49) — Cisco — separates AAA functions
"Password + security question — is this MFA?"→ NO — both are "something you know" — same factor type
"Password + TOTP code — is this MFA?"→ YES — know + have — two different factor types
"Strongest phishing-resistant MFA"→ FIDO2 / WebAuthn / hardware security key — bound to specific site
"False Acceptance Rate — security concern"→ FAR — wrong person admitted — biometrics — lower is better
"False Rejection Rate — usability concern"→ FRR — right person denied — availability impact — lower is better
"Owner decides who accesses their files"→ DAC — Discretionary Access Control — Windows NTFS
"Classification labels — Top Secret, Secret, Confidential"→ MAC — Mandatory Access Control — government/military
"Job role determines permissions — most common enterprise"→ RBAC — Role-Based Access Control
"Access based on user, resource, AND environmental attributes"→ ABAC — Attribute-Based Access Control — Zero Trust foundation
"Trust based on network location is a security problem"→ Zero Trust addresses this — never trust, always verify
"Never trust, always verify — design as if breach occurred"→ Zero Trust architecture
"Replace VPN with app-level access"→ ZTNA — Zero Trust Network Access
"Workload-level network isolation"→ Micro-segmentation — Zero Trust component
"Decoy system — any access = confirmed attacker"→ Honeypot — no legitimate user should access it
"Network of decoy systems for threat intelligence"→ Honeynet
"Fake AWS API key planted in code repository"→ Honeytoken — use triggers alert and identifies attacker
"File named 'salaries.xlsx' that alerts if accessed"→ Honeyfile — variant of honeytoken for file systems
"Malware calls home but connects to security team server"→ DNS Sinkhole — redirects malicious domains — disrupts C2
"Log forwarding protocol — UDP 514 vs TCP 6514"→ Syslog — UDP 514 (plain/legacy) / TCP 6514 (with TLS, use this)
"Aggregates, correlates, and alerts on logs from all sources"→ SIEM — Security Information and Event Management
