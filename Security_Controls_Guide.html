01 What Are Security Controls? Foundation · Domain 1

Security controls are safeguards or countermeasures used to reduce risk, protect assets, and ensure CIA — Confidentiality, Integrity, and Availability. Every firewall rule, security policy, guard station, and training program is a security control.

Two Ways to Classify Every Control

# Security+ classifies controls along TWO independent axes: AXIS 1 — CATEGORY (what KIND of control is it?): Technical Technology enforces it — firewall, AV, MFA Managerial Policy and process — risk assessments, audits Operational People and day-to-day procedures Physical Tangible hardware — locks, cameras, guards AXIS 2 — FUNCTION (what does it DO?): Preventive Stops the attack from happening Detective Identifies an attack in progress or after Corrective Fixes damage after an attack Deterrent Discourages the attacker from trying Compensating Alternative when primary control isn't feasible Directive Guides behavior — tells people what to do # Every control has BOTH a category AND a function: Firewall: Technical + Preventive Security Policy: Managerial + Directive Backup + Restore: Operational + Corrective Security Camera: Physical + Detective (also Deterrent)

Why This Matters for the Exam

Scenario Questions

The exam presents a scenario and asks "Which type of control is this?" You must identify BOTH the category (Technical/Managerial/Operational/Physical) AND the function (Preventive/Detective/Corrective etc.). A single control can satisfy multiple classifications simultaneously.

Defense in Depth

Effective security uses controls from ALL categories and functions together. No single control is sufficient. The exam tests whether you can select the right combination of controls to address a given threat scenario.

The CIA Triad — What Controls Protect

Confidentiality

Ensure data is only accessible to authorized parties. Controls: encryption, access control, MFA, data classification, DLP. Threat: eavesdropping, credential theft, unauthorized access.

Integrity

Ensure data is accurate and hasn't been tampered with. Controls: hashing, digital signatures, file integrity monitoring, audit logs. Threat: man-in-the-middle, data tampering, unauthorized modification.

Availability

Ensure systems and data are accessible when needed. Controls: redundancy, backups, DDoS protection, UPS, DR sites. Threat: DDoS, ransomware, hardware failure, natural disaster.

02 Control Categories — What Kind Is It? Technical · Managerial · Operational · Physical

The category answers "what mechanism implements the control?" — is it technology, a policy, a human process, or a physical object? The same function (e.g., "Preventive") can be implemented through any of the four categories.

T
💻 Technical Controls Also: Logical Controls
"Security controls implemented through technology — hardware, software, or firmware — that automatically enforce policy without relying on human action."
Examples: Firewall rules, antivirus / EDR, encryption (TLS, AES-256), MFA system, intrusion detection system (IDS), access control lists (ACLs), SIEM, DLP software, password complexity requirements enforced by AD, full disk encryption (BitLocker), certificate-based authentication, VPN, network segmentation (VLANs).
# Why "logical" as well as "technical"? Technical: implemented through hardware/software/firmware Logical: same thing — older terminology still appears on exams Both terms refer to technology-enforced controls Key characteristic: enforces automatically The firewall doesn't ask "should I block this?" — it just does MFA doesn't optionally require a second factor — it mandates it If the rule says DENY → traffic is DENIED regardless of who asks
M
📋 Managerial Controls Also: Administrative Controls
"Security controls implemented through organizational policies, procedures, guidelines, and governance — the administrative framework that guides how security should work."
Examples: Information security policy, risk assessment process, security awareness training program, acceptable use policy (AUP), background checks, vendor risk management program, incident response policy, data classification policy, change management process, business continuity plan, penetration testing authorization, security audit schedule.
# Key characteristic: governance and oversight Managerial controls answer: "what SHOULD we do?" They create the framework that everything else operates within Without managerial controls: technical controls have no direction # Risk assessment = classic managerial control: Identify assets → identify threats → assess risk Decision: accept / mitigate / transfer / avoid This GUIDES which technical controls to deploy # Policy is the foundation: "Employees must use strong passwords" (managerial/directive) Active Directory password policy enforces it (technical/preventive) Managerial sets the requirement → Technical enforces it
O
🔧 Operational Controls People + Day-to-Day
"Security controls implemented through people, day-to-day procedures, and ongoing operational practices — the human element of security execution."
Examples: Security awareness training (delivery, not just the policy), patch management execution (IT staff applying patches), backup procedures (staff running and verifying backups), incident response activities (SOC analyst investigating an alert), two-person integrity rule (requires two people to complete sensitive tasks), employee termination procedure (revoking access on last day), physical security patrols, change management review meetings.
# Operational vs Managerial — the distinction: Managerial: "We shall maintain a patch management POLICY" Operational: IT team that actually APPLIES patches each Tuesday Managerial: "We shall train employees on security" Operational: The actual security awareness training SESSIONS # Key characteristic: execution by people Operational controls depend on people doing things correctly Weakness: human error, fatigue, shortcuts Strength: adaptable, can handle novel situations
P
🔒 Physical Controls Tangible · Environmental
"Security controls that use physical barriers, hardware, and environmental measures to protect assets from physical threats — unauthorized physical access, theft, damage, and environmental hazards."
Examples: Locks and deadbolts (door, cabinet, rack), security cameras (CCTV), badge/card readers, biometric scanners (fingerprint, retinal), mantraps / airlocks, security guards and receptionists, fencing and bollards, lighting, alarms (motion, door, glass-break), cable locks on devices, server room with keypad access, UPS and generator (power continuity), fire suppression system, shredder (destroy physical documents).
# Physical controls are the FIRST line of defense for hardware If attacker has physical access → all logical controls in question Cold boot attack: physical access → extract RAM → get encryption keys Evil maid attack: physical access → tamper with device Theft: physical access → device + data gone # Physical controls are often DUAL function: Security camera: Physical + Detective (records) + Deterrent (visible) Badge reader: Physical + Preventive (denies entry) + Detective (logs) Mantrap: Physical + Preventive (traps tailgaters) Guards: Physical + Preventive + Detective + Deterrent

Managerial/Administrative →Policies, governance, risk assessment — sets the direction for all other controls
Operational →People and procedures in practice — patching, training delivery, incident response

03 Control Functions — What Does It Do? Prevent · Detect · Correct · Deter · Compensate · Direct

The function answers "at what point in the attack lifecycle does this control act?" — before the attack, during it, or after it. Understanding timing is critical to building defense in depth.

Control Functions Mapped to the Attack Timeline

# Where each control type acts on the attack timeline: BEFORE ATTACK: Preventive — blocks the attack from succeeding Deterrent — discourages attacker from trying at all Directive — guides behavior to avoid creating vulnerabilities DURING ATTACK: Detective — identifies attack in progress → triggers response Preventive — (also acts during) continues blocking AFTER ATTACK: Corrective — fixes damage and restores normal state Compensating— alternative when other controls have failed/unavailable Timeline: Deterrent Preventive Detective Corrective ↓ ↓ ↓ ↓ [attacker [attack [attack [attack decides NOT blocked] detected] remediated] to try]

Preventive

🛡️ Preventive Controls Before Attack · Blocks
"Controls that stop a security incident from occurring in the first place — they act before the attack reaches its goal."
Exam examples: Firewall (blocks unauthorized traffic), MFA (blocks credential-only access), encryption (blocks data readable without key), access control list (blocks unauthorized access), antivirus (blocks known malware execution), cable lock (blocks laptop theft), mantrap (blocks tailgating), background check (blocks hiring of malicious insider), patch management (blocks exploitation of known CVEs), security awareness training (blocks social engineering success).

Detective

🔍 Detective Controls During/After Attack · Identifies
"Controls that identify and alert on security incidents — they do NOT prevent or stop the attack, but they reveal that something has occurred so a response can begin."
Exam examples: IDS — Intrusion Detection System (alerts on suspicious traffic — does NOT block), SIEM (correlates logs and generates alerts), security cameras/CCTV (records activity), audit logs (records what happened), file integrity monitoring (detects changes to critical files), motion sensors (detects unauthorized movement), smoke detectors (detects fire), honeypot (detects attacker activity by baiting them to attack it), antivirus in detection-only mode, security audit findings.
# Critical exam distinction: IDS vs IPS IDS (Intrusion DETECTION System): Detective → Monitors and alerts — does NOT block → Out-of-band — copies of packets analyzed → False positive: alert on legitimate traffic (only an alert) IPS (Intrusion PREVENTION System): Preventive → Monitors AND actively blocks inline → In-band — all traffic passes through → False positive: BLOCKS legitimate traffic (business impact)

Corrective

🔧 Corrective Controls After Attack · Restores
"Controls that reduce the impact of an incident and restore systems to normal operation after a security event has occurred."
Exam examples: Backup and restore (restores data after ransomware/failure), incident response plan (structured recovery process), patching a vulnerability after exploitation, quarantining an infected endpoint (EDR auto-isolation), applying a hotfix, restoring from a snapshot, reimaging a compromised workstation, activating DR site after primary site failure, revoking compromised certificates, resetting compromised passwords across the org.

Deterrent

⚠️ Deterrent Controls Before Attack · Discourages
"Controls that discourage a threat actor from attempting an attack by making the perceived risk or cost of attacking higher than the potential reward."
Exam examples: Warning login banner ("Unauthorized access is prohibited and monitored"), visible security cameras (presence discourages), security guard station at entrance, "Property protected by security system" sign, parking lot lighting (criminals prefer darkness), barbed wire fence topping, security awareness: "you will be caught" culture, published security policies with consequences, military presence, "No trespassing" signs.
# Deterrent vs Preventive — the key distinction: Deterrent: attacker DECIDES NOT TO TRY — works on psychology Preventive: attacker tries but is BLOCKED Visible camera = Deterrent (criminal sees it and walks away) CCTV recording = Detective (records if they proceed anyway) Warning banner = Deterrent (you MIGHT decide not to log in) also = Directive (tells users what's allowed) Firewall = Preventive (you TRY to connect and are BLOCKED) # Deterrents can FAIL: determined attacker ignores warning # That's why preventive + detective also needed

Compensating

🔄 Compensating Controls Alternative · Workaround
"Alternative controls used when the primary or ideal control cannot be implemented — they reduce risk to an acceptable level without using the preferred method."
Exam examples: Network isolation for a legacy system that can't be patched (compensates for missing patch management), enhanced monitoring instead of fixing a vulnerability during a freeze window, encrypted communications instead of physically securing a cable, additional logging + alerting when MFA can't be deployed on a legacy app, manual approval process compensating for a missing automated control, increased physical security when remote access controls are unavailable.
# Classic compensating control scenario: Problem: Windows XP machine running critical manufacturing app Cannot be patched (vendor won't certify updated OS) Cannot be replaced (app only runs on XP, replacement: 18 months) Ideal control: PATCH the OS ← not possible Compensating: Network isolation: XP machine on isolated VLAN Enhanced monitoring: IDS watches its traffic closely Firewall ACLs: only specific allowed communications No internet: zero internet access from that VLAN Physical access: only authorized technicians in that room # PCI-DSS explicitly allows compensating controls for requirements # that can't be met — must be documented and approved

Directive

📜 Directive Controls Guides Behavior · Instructs
"Controls that direct, guide, or instruct people on how to behave in order to maintain security — they inform and require specific actions through documented rules and communication."
Exam examples: Acceptable Use Policy (AUP), security awareness training program (content, not just delivery), password policy documentation, data classification policy, warning banners ("This system is for authorized users only"), "Employees must lock screens when leaving desks" sign, emergency procedures posted on the wall, compliance requirements documentation, security handbook, "No tailgating" sign, "Do not plug in unknown USB devices" poster.
# Directive controls by themselves don't ENFORCE anything: Policy: "Use strong passwords" ← Directive Does not BLOCK weak passwords — just says to use strong ones Active Directory password complexity ← Technical + Preventive ENFORCES the policy — blocks passwords that don't comply # Directive provides the REQUIREMENT # Technical/Physical/Operational ENFORCES the requirement # Warning banner is BOTH: Directive: tells users what behavior is acceptable/unacceptable Deterrent: might discourage unauthorized access attempts (Control can serve multiple functional roles)

Commonly Confused Pairs

PreventivevsDeterrent
Preventive = attacker TRIES → gets BLOCKED. Deterrent = attacker DOESN'T TRY because of perceived risk. A lock is Preventive. A "Protected by alarm" sign is Deterrent. Both can protect the same asset.
DetectivevsPreventive
IDS = Detective (alerts, does not block). IPS = Preventive (blocks inline). SIEM = Detective (identifies, alerts). Firewall = Preventive (blocks traffic). A camera RECORDS (Detective). A lock BLOCKS (Preventive).
ManagerialvsOperational
Managerial = the POLICY document ("We shall have a patch management program"). Operational = the IT staff who APPLY patches every Tuesday. Writing the backup policy = Managerial. Running the backups = Operational.
CorrectivevsPreventive
Backup = Corrective (used AFTER incident to restore). Patch = Preventive (stops exploit BEFORE it happens). Both involve fixing things but at different points in the timeline. Patching BEFORE an attack is Preventive. Applying a hotfix AFTER a breach is Corrective.
DirectivevsDeterrent
Warning banner: "Unauthorized access is prohibited" — Directive (instructs) AND Deterrent (discourages). "No tailgating" sign — Directive (instructs behavior) AND Deterrent (reminds of consequences). Many controls fulfill multiple functions simultaneously.

07 Control Layering Strategy Defense in Depth

Effective security uses multiple controls from multiple categories and functions working together. No single control is sufficient — the art is selecting complementary controls that cover each other's weaknesses.

Decision Framework — Selecting Controls

1️⃣
Identify the Threat
What are you protecting against? Unauthorized access? Malware? Data loss? Physical theft? Insider threat? Each threat may need different controls.
↓
2️⃣
Select Preventive Controls First
Try to stop the attack before it succeeds. What technical, physical, managerial, and operational controls can prevent this threat? Ideal: stop it before it happens.
↓
3️⃣
Add Detective Controls
Assume the preventive control might fail. How will you KNOW if it does? Every preventive control needs a corresponding detective control. If firewall rule is bypassed — the IDS should catch it.
↓
4️⃣
Add Corrective Controls
Assume the attack succeeds. How do you recover? Backup and restore, IR plan, DR site. Recovery capability prevents a bad event from becoming catastrophic.
↓
5️⃣
Add Deterrents and Directives
Layer in deterrents (visible controls, warnings) and directives (policies, training) to reduce attack attempts and guide employee behavior. The human element of security.
↓
6️⃣
Use Compensating Where Needed
When ideal controls can't be implemented (cost, legacy system, technical constraint), apply compensating controls that reduce risk to an acceptable level. Always document why the primary control isn't possible.

Complete Defense — Ransomware Example

# Threat: Ransomware via phishing email # Apply controls from ALL categories across ALL functions: PREVENTIVE controls: Email sandbox (Technical+Preventive): detonate attachments Anti-phishing filter (Technical+Preventive): block known bad MFA on all accounts (Technical+Preventive): blocks credential theft EDR behavioral block (Technical+Preventive): stops execution Security awareness training (Operational+Preventive): train to spot Patching policy (Managerial+Preventive): removes exploit path Least privilege (Technical+Preventive): limits spread if executes DETECTIVE controls: SIEM rule: mass file modification alert (Technical+Detective) EDR behavioral: shadow copy deletion flag (Technical+Detective) Email reporting button (Operational+Detective): staff report Phishing simulation metrics (Operational+Detective): spot trends CORRECTIVE controls: Immutable backups (Technical+Corrective): restore without paying IR plan with ransomware playbook (Managerial+Corrective): structured response Network segmentation (Technical+Corrective/Compensating): contains blast radius DR site (Technical+Corrective): continue operations during recovery DETERRENT controls: Published security policy with consequences (Managerial+Deterrent) Visible security team presence (Operational+Deterrent) DIRECTIVE controls: AUP: prohibits clicking unknown links (Managerial+Directive) "Don't open unexpected attachments" poster (Physical+Directive) Security awareness content: how to report (Operational+Directive) Result: attacker must defeat ALL layers — each layer raises cost and time

Control Coverage Assessment

Technical Controls

Most scalable and consistent. Don't rely on human behavior. Can fail due to: misconfiguration, zero-day bypasses, management plane compromise. Strength: enforces policy automatically at scale.

Managerial Controls

Sets the entire direction — without policy, technical controls have no purpose. Can fail due to: leadership not prioritizing security, poorly written policies, no enforcement. Strength: creates legal and organizational accountability.

Operational Controls

Human adaptability — can handle novel situations. Can fail due to: human error, fatigue, training gaps, shortcuts. Strength: human judgment in ambiguous situations. Security culture is an operational control.

// Exam Keywords

Control selection sequence →Preventive first → Detective → Corrective → Deterrent/Directive → Compensating if needed
Compensating control requirement →Document why primary control can't be used + must be formally approved
Detective requires response →Detection alone is useless without a process to act on alerts (SIEM + SOC)
