01IAM OverviewIdentity is the New Perimeter

Identity and Access Management (IAM) is the framework of policies, processes, and technologies that ensure the right people have the right access to the right resources at the right time — and that every access event is verified, logged, and auditable.

Why IAM Is the New Security Perimeter

# Traditional security model: TRUSTED: Inside the corporate network UNTRUSTED: Outside the corporate network Defense: strong perimeter firewall # Modern reality: Employees: work from home, coffee shops, airports Applications: in the cloud (SaaS, IaaS, PaaS) Data: in Microsoft 365, Salesforce, AWS, Dropbox Network perimeter: doesn't exist anymore # Result: Attacker steals ONE set of credentials → Access to email ✓ Access to SharePoint ✓ Access to Salesforce ✓ Access to cloud servers ✓ Access to financial systems ✓ ALL from ANY location — no perimeter stops them # IAM becomes the perimeter: Verify IDENTITY before access Verify DEVICE health before access Verify CONTEXT (location, time, risk) Grant MINIMUM required access Monitor ALL access continuously Identity = new security boundary ✓

Core IAM Functions

Authentication

Proving you are who you claim to be. Passwords, MFA, certificates, biometrics. Answering: "Who are you?" before any access is granted.

Authorization

Determining what you're allowed to do after authentication. RBAC, ABAC, ACLs. Answering: "What can you do?" — access models covered in Tab 7.

Accounting / Auditing

Tracking who did what, when, from where. Logs, audit trails, access reviews. Answering: "What did you do?" — non-repudiation and forensics.

Provisioning

Creating, managing, and revoking access throughout the user lifecycle. Joiner-mover-leaver process. Covered in Tab 2.

Privileged Access

Special controls for admin/privileged accounts. PAM vaults, just-in-time access. Covered in Tab 6 — highest risk accounts need strongest controls.

Federation & SSO

Extending identity across systems and organizations. SAML, OIDC, SSO. Covered in Tabs 3 and 5 — one identity across many systems.

The AAA Framework — Core Exam Concept

Authentication — Who are you? Prove identity: password, MFA, certificate, biometric Result: identity verified (or denied) Authorization — What can you do? After identity confirmed: what resources can be accessed? Access control models: RBAC, ABAC, MAC, DAC Result: permissions granted (minimum necessary) Accounting — What did you do? Log every action: login time, resource accessed, changes made Non-repudiation: user cannot deny their actions Forensics: reconstruct what happened # RADIUS protocol implements AAA for network access # TACACS+ implements AAA for device administration # Remember: Authentication THEN Authorization THEN Accounting

IAM Architecture Components

Identity Provider (IdP)

The authoritative source of identity — where user accounts live. Users authenticate TO the IdP. IdP issues tokens/assertions to other systems. Examples: Microsoft Azure AD (Entra ID), Okta, Google Workspace, Active Directory.

Service Provider (SP)

The application or service the user wants to access. Trusts the IdP to verify identity. Grants access based on IdP assertion. Examples: Salesforce, GitHub, Jira, any web app using SSO.

Directory Service

Stores identity objects — users, groups, computers, policies. LDAP protocol for querying. Active Directory = most common enterprise directory. Azure AD = cloud directory. Source of truth for identities.

Access Management Platform

Enforces access policies, manages sessions, handles SSO. Examples: Okta, Microsoft Entra ID, Ping Identity, ForgeRock. Sits between users and all their applications.

02Identity Provisioning & LifecycleJoiner · Mover · Leaver

Provisioning is the process of creating, modifying, and removing user access throughout their relationship with the organization. Poor provisioning creates orphaned accounts, excess privilege accumulation, and access that outlives employment — all major security risks.

The Identity Lifecycle — Joiner-Mover-Leaver

A
Active

Promotion, transfer, role change — access updated
L
Leaver

Joiner — New Employee Provisioning

# Automated provisioning triggered by HR system: HR creates record: Sarah Chen, Finance Analyst, Start: 2024-03-01 Automated provisioning (Day 0 — before she arrives): AD account created: [email protected] Email mailbox: provisioned Role assigned: Finance-Analyst-Group Permissions via RBAC: Finance shared drive: READ + WRITE Accounts payable app: Standard user SAP Finance module: Viewer access Corporate intranet: Employee access NOT provisioned (least privilege): HR database: ✗ (not needed) Source code repos: ✗ (not a developer) Admin consoles: ✗ (not IT admin) Executive reports: ✗ (not authorized) MFA enrollment email sent: [email protected] Welcome package: IT contacts, setup instructions Manager notified: account ready for Day 1 # Provisioning from ROLE not individual decisions # Finance-Analyst-Group = predefined access set # Less chance of error, consistent, auditable

Mover — Role Change Access Update

# Sarah promoted from Finance Analyst to Finance Manager: Trigger: HR system updates Sarah's title and department Access Management System receives change: Old role: Finance-Analyst-Group → REMOVE New role: Finance-Manager-Group → ADD Changes applied: Added: Finance team reports: ADMIN access Budget approval system: Approver role HR salary visibility: Direct reports only Team resource booking: Full access Removed: Accounts payable (data entry): ✗ (managers review, not enter) ACCESS CREEP problem — what often goes wrong: Old analyst access KEPT (should remove) New manager access ADDED Result: Sarah has both analyst AND manager access Over time: more and more accumulated permissions Solution: role-based provisioning with transition rules Remove old role COMPLETELY before adding new role ✓

Leaver — Offboarding & Deprovisioning

# Employee John Smith resigns — final day: Friday Immediate actions (on last day / termination for cause — IMMEDIATELY): 09:00 — AD account: DISABLED (not deleted — preserve audit trail) 09:00 — Email: OUT OF OFFICE set, access revoked 09:00 — VPN certificate: REVOKED 09:00 — All OAuth tokens: REVOKED (Salesforce, GitHub, etc) 09:00 — MFA tokens: DEREGISTERED 09:00 — SSH keys: REMOVED from all servers 09:01 — Slack access: DEACTIVATED 09:01 — GitHub access: REMOVED from org 09:01 — Privileged accounts: DISABLED FIRST (before regular) Preserve (do NOT delete immediately): AD account object: keep 90 days (email continuity, investigation) Email: redirect to manager 90 days Files: transfer to manager's review folder Audit logs: retain per policy (legal/compliance) For cause termination (immediate — security escort): Start with: DISABLE account BEFORE notifying employee Reason: prevents data destruction in anger # Orphaned accounts = account still active after person leaves # Attacker finds orphaned account = easy target (nobody monitors it)

Provisioning Methods

Manual Provisioning

IT admin manually creates accounts per ticket. Slow — days before access granted. Error-prone — wrong permissions set. No consistency — different admin, different setup. Only acceptable for very small organizations.

Automated Provisioning (IGA)

Identity Governance and Administration — triggered by HR system changes. Account created automatically on new hire record. Role-based — join Finance group = get Finance access. Fast, consistent, auditable. Examples: SailPoint, Saviynt, Microsoft Entra ID Governance.

SCIM — Automated SaaS Provisioning

System for Cross-domain Identity Management. Standard protocol for pushing user accounts to cloud apps. When AD user created → SCIM pushes to Salesforce, Slack, GitHub, etc. No manual SaaS account creation needed.

# SCIM provisioning flow: HR creates employee → AD → SCIM protocol → Salesforce account created automatically ✓ Slack workspace added automatically ✓ GitHub org membership added automatically ✓

Self-Service Provisioning

Users request additional access via portal. Manager approves request. Workflow-based — approvals required before access granted. Audit trail of who approved what. Reduces IT workload while maintaining control.

Access Reviews — Recertification

📋
What Is Access Recertification?
Periodic review of all user access — managers confirm each employee still needs their current access. Typically quarterly or annually. Catches: role creep, orphaned access after moves, access no longer needed. Required by: SOX, PCI-DSS, HIPAA, ISO 27001. Automated by IGA platforms — managers get email "Does Sarah still need access to X? Yes/No"

03Single Sign-On (SSO)One Login · Many Systems

SSO allows users to authenticate once and access multiple applications without re-entering credentials. One trusted identity provider authenticates the user, then vouches for them to all connected applications via tokens or assertions.

The Problem SSO Solves

# Without SSO — typical enterprise employee: Email: username + password A CRM (Salesforce): username + password B HR system: username + password C Project mgmt (Jira): username + password D Cloud storage: username + password E Security tools: username + password F Financial system: username + password G ...total: 12-15 different credentials Result: Employees reuse passwords → credential stuffing risk Employees forget passwords → IT helpdesk burden (30-40% of tickets) No central visibility → who has access to what? Deprovisioning → must remove from ALL 15 systems manually # With SSO: Employee logs in ONCE to IdP (Okta/Azure AD): Username: [email protected] Password: [strong password] + MFA token SSO session established → access ALL connected apps: Salesforce: auto-logged in ✓ Jira: auto-logged in ✓ Slack: auto-logged in ✓ GitHub: auto-logged in ✓ No additional passwords for any of these

SSO Protocols — SAML vs OIDC vs Kerberos

📜 SAML 2.0 — Security Assertion Markup Language XML-based · Enterprise

Older protocol, XML-based. Industry standard for enterprise SSO. Works well for web applications. Does NOT work well for mobile apps or REST APIs. Exchanges XML "assertions" between IdP and SP.

# SAML flow: 1. User accesses Salesforce (Service Provider) 2. Salesforce: "no session — redirect to IdP" 3. Browser redirected to Okta (IdP) login page 4. User authenticates to Okta (password + MFA) 5. Okta creates SAML Assertion (XML): "I confirm user [email protected] She is in roles: [Finance, Manager, Salesforce-User] Valid for: 8 hours Signed with Okta private key" 6. SAML Assertion sent to Salesforce 7. Salesforce: verifies Okta's signature → trusts assertion 8. Salesforce: grants access based on roles in assertion 9. User: in Salesforce — no second login needed
🔑 OIDC — OpenID Connect JSON/OAuth2 · Modern · Mobile

Modern protocol built on top of OAuth 2.0. JSON-based (not XML) — works great for mobile apps and APIs. Uses JWT (JSON Web Tokens). Most new applications use OIDC. "Sign in with Google/Microsoft/Apple" = OIDC.

# OIDC flow (simplified): 1. App redirects to Google (IdP) with client_id 2. User logs in to Google (password + MFA) 3. Google returns: Authorization Code 4. App exchanges code for tokens: access_token: short-lived, grants API access id_token: JWT with user identity claims refresh_token: get new access_token when expired 5. App reads id_token: {"sub":"12345","email":"[email protected]", "name":"Sarah Chen","groups":["finance"]} 6. User: logged in — JWT verified ✓ # OIDC vs OAuth 2.0: OAuth 2.0: Authorization only ("can this app access your data?") OIDC: Authentication + Authorization ("who are you AND what can you do?")
🎟️ Kerberos Ticket-based · Windows/AD

Used by Windows Active Directory for internal SSO. Ticket-based system — KDC (Key Distribution Center) issues tickets. Users log in to Windows domain once → access all domain resources without re-entering credentials. The backbone of Windows enterprise authentication.

# Kerberos components: KDC (Key Distribution Center): Authentication Server (AS): verifies password → issues TGT Ticket Granting Server (TGS): exchanges TGT for service tickets # Flow: 1. User logs into Windows → AS verifies password 2. AS issues: TGT (Ticket Granting Ticket) — encrypted 3. User wants file server access → sends TGT to TGS 4. TGS issues: Service Ticket for file server 5. User presents Service Ticket to file server → access granted 6. File server: never sees password! Only encrypted ticket Key security note: TGT valid ~8-10 hours Pass-the-ticket attack: steal TGT → impersonate user Golden ticket attack: forge TGT → admin access to everything

SSO — Security Considerations

SSO Benefits

  • One strong password + MFA everywhere
  • Fewer passwords = less reuse
  • Central visibility and control
  • Instant deprovisioning — disable IdP = lose all access
  • Reduced helpdesk password reset calls
  • Better user experience — less friction
  • Consistent policy enforcement

SSO Risks

SSO introduces a single point of failure for identity. If IdP goes down — nobody can log in anywhere. If SSO account compromised — attacker gets everything.

  • IdP availability = critical infrastructure
  • Credential theft = blast radius is EVERYTHING
  • Makes MFA on the IdP absolutely critical
  • Session token theft = bypass password+MFA
  • Must monitor SSO login events closely

04Multi-Factor Authentication (MFA)Something You Know · Have · Are

MFA requires users to prove identity using two or more factors from different categories. Even if a password is stolen, MFA prevents access without the second factor. Microsoft says MFA blocks 99.9% of automated credential attacks.

The Three Factor Categories

🧠
Something You KNOW
Knowledge factors — memorized information
Password PIN Security questions Passphrase

Weakness: can be stolen (phishing), guessed (brute force), shared, or forgotten. Alone = insufficient for sensitive access.

📱
Something You HAVE
Possession factors — physical object in your control
Authenticator app (TOTP) SMS OTP Hardware token (RSA SecurID) FIDO2 key (YubiKey) Smart card / PIV Push notification

Weakness: can be lost or stolen. SIM swapping compromises SMS. Most common second factor in enterprise environments.

🫆
Something You ARE
Inherence factors — biometric characteristics
Fingerprint Face recognition Iris/retina scan Voice recognition Vein pattern Gait analysis

Weakness: can't be changed if compromised. False accept/reject rates. Privacy concerns. Deepfake attacks on facial recognition emerging.

📍
Somewhere You ARE (4th Factor)
Location factors — geolocation, network location
GPS location IP geolocation Network (corp Wi-Fi) Geofencing

Used in adaptive/contextual MFA. "User is in the office network — reduce friction." "User in new country — require stronger auth."

MFA Methods — Strength Comparison

TOTP — Time-Based One-Time Password

# How authenticator apps work (Google Authenticator, Authy, MS Authenticator): Setup (once): Server and authenticator app share a secret key QR code scanned → secret stored in app Every 30 seconds: Both server AND app independently compute: TOTP = HMAC(secret_key, current_time_window) Example 6-digit code: 847293 Valid: 30 seconds only Next window: 194827 (completely different) Authentication: User enters: 847293 Server computes: same value → MATCH ✓ Server accepts: 1 window before/after (clock skew) # Why it's better than SMS: No network connection needed (offline capable) No SIM card to steal Standard (RFC 6238) — works with any TOTP app # But still phishable — real-time attack can relay code

FIDO2 / WebAuthn — Phishing-Resistant MFA

# FIDO2 = most secure MFA — why it's phish-resistant: Setup: YubiKey generates: unique key pair (public + private) Public key: stored on server (login.company.com) Private key: NEVER leaves YubiKey hardware Authentication: Server sends: challenge (random value) YubiKey: signs challenge with PRIVATE KEY User: taps YubiKey (physical presence proof) Server: verifies signature with stored PUBLIC KEY Why phishing can't work: Attacker creates fake site: login.company-secure.com User visits fake site → challenge issued YubiKey signs challenge BUT: Key is bound to: login.company.com ONLY (origin binding) Fake site: login.company-secure.com → KEY REFUSES FIDO2: checks domain before signing Phishing site: gets nothing ✓ # Hardware key: YubiKey, Titan Key, Feitian # Software: Windows Hello, Apple FaceID (uses TPM/Secure Enclave) # FIDO2 = gold standard — US government mandates for high-value systems

MFA Fatigue Attack

# MFA fatigue = bombarding user with push notifications until they approve Attacker has stolen username + password for j.smith MFA: push notification to j.smith's phone Attacker triggers: Push #1 at 11:45 PM: j.smith taps DENY Push #2 at 11:46 PM: j.smith taps DENY Push #3 at 11:47 PM: j.smith (annoyed, half asleep) taps APPROVE ← Attacker: logged in — MFA bypassed by user error Real example: Uber breach 2022 — attacker bombarded contractor with WhatsApp MFA pushes at 1am until contractor approved → full network access gained. Defenses against MFA fatigue: 1. Number matching: push shows number, user must match it in app Attacker sends push: app shows "Enter 47" — user must type 47 Attacker doesn't know what number appears → can't coordinate 2. Geographic context: push shows "Login attempt from Moscow, Russia" User thinks: "I'm in Toronto" → clearly deny 3. Limit push rate: max 3 pushes → account locked 4. FIDO2 hardware key: no push → immune to fatigue

Adaptive / Risk-Based MFA

# MFA strength adapted to risk level — not always same friction: Low risk (trusted device, known location, normal hours): User: sarah.chen — office network — Tuesday 9am — known laptop Risk score: LOW MFA: username + password only (SSO session trusted) Medium risk (new location, off-hours): User: sarah.chen — home VPN — Wednesday 11pm Risk score: MEDIUM MFA: password + authenticator TOTP required High risk (new country, new device, sensitive resource): User: sarah.chen — Singapore IP — unknown device — admin console Risk score: HIGH MFA: password + FIDO2 hardware key required Additional: manager notification sent Impossible travel (two countries 30 minutes apart): User: sarah.chen — logged in NY 2pm, login attempt UK 2:30pm Risk score: CRITICAL — impossible travel Action: BLOCK — force full re-verification + alert SOC

05Federation & Identity StandardsCross-Org Trust · SAML · OAuth

Federation extends identity trust across organizational boundaries. Instead of creating separate accounts in every partner's system, your home organization's identity is trusted by partner organizations — one identity works everywhere.

Federation — What It Solves

# Problem: Company A employees need access to Company B's portal Without federation: Company B: creates accounts for all Company A employees Employee has: Company A identity + Company B identity When employee leaves Company A: Company A account: disabled Company B account: STILL ACTIVE (orphaned!) Security risk: ex-employee still has partner access With federation: Company B: "Trusts" Company A's IdP (Okta/Azure AD) Employee: logs in with Company A credentials Company A IdP: vouches for employee to Company B Company B: grants access based on Company A assertion When employee leaves Company A: Company A account: disabled Company B: immediately loses access (trust chain broken) ✓ No orphaned accounts at partner ✓

OAuth 2.0 — Authorization Framework

# OAuth 2.0 — delegated authorization ("grant access to my data") # Common scenario: app wants access to your Google calendar 1. Expense app: "I need your Google Calendar access" 2. Redirect to Google consent screen: "Expense App wants to: ✓ View your calendar events ✗ Not requesting email access ✗ Not requesting contacts" 3. User: clicks "Allow" 4. Google: issues access_token to Expense App 5. Expense App: uses token to read calendar NEVER sees Google password ← this is the key! # OAuth roles: Resource Owner: the user (owns the Google calendar) Client: Expense App (wants access) Authorization Server: Google (issues tokens) Resource Server: Google Calendar API (holds the data) # OAuth 2.0 grant types: Authorization Code: most secure — server-side apps Implicit: deprecated — was for SPAs (insecure) Client Credentials: machine-to-machine (no user) Device Code: TVs, IoT, devices without browsers

Token Types

Access Token

Short-lived credential (minutes to hours). Grants access to specific resources. Presented to API/resource server. Revoked when expired or user logs out.

Refresh Token

Long-lived (days to months). Used to get new access tokens without re-authentication. Stored securely server-side. If stolen — attacker can get access tokens indefinitely.

JWT — JSON Web Token

Self-contained token format — contains identity claims. Three parts: Header.Payload.Signature. Signed by IdP — anyone can verify. Used by OIDC for id_token. Stateless — no server lookup needed.

JWT structure: eyJhbGciOiJSUzI1NiJ9 ← header (base64) .eyJzdWIiOiJzYXJhaCJ9 ← payload (base64) .SflKxwRJSMeKKF2QT4 ← signature (RSA) Payload decoded: {"sub":"sarah.chen", "email":"[email protected]", "roles":["Finance"], "exp":1704067200}

Directory Services — LDAP & Active Directory

# LDAP — Lightweight Directory Access Protocol: Protocol for querying directory services Port: 389 (unencrypted) | 636 (LDAPS — encrypted) Used to: authenticate users, look up group membership Active Directory, OpenLDAP use LDAP for queries # Active Directory — enterprise identity backbone: Domain: company.com Domain Controller (DC): hosts AD database Objects: Users, Groups, Computers, Policies (GPO) Kerberos: authentication within domain LDAP: directory queries # Azure AD (now Microsoft Entra ID): Cloud version of Active Directory Supports: SAML, OIDC, OAuth 2.0 Hybrid: sync on-prem AD to Azure AD (AD Connect) MFA: built-in Azure MFA Conditional Access: adaptive policies # LDAP injection attack: Attacker injects LDAP query syntax into login form Example: username = *)(uid=*))(|(uid=* Bypasses authentication if not sanitized Prevention: input validation, parameterized LDAP queries

Trust Models in Federation

Direct / Bilateral Trust

Company A specifically trusts Company B's IdP. Configured manually. Works for a small number of partners. Simple — two parties agree to trust each other. SAML federation between two organizations.

Brokered Trust (Hub and Spoke)

Trusted third-party broker in the middle. All organizations trust the broker. Broker trusts all organizations. Scale: hundreds of orgs can federate without individual agreements. Examples: government PKI hierarchies, cloud marketplace identity.

06Privileged Access Management (PAM)Vault · JIT · Session Recording

Privileged accounts — admins, service accounts, root users — are the highest-value targets for attackers. PAM protects them through vaulting, just-in-time access, session recording, and privileged access workstations. These accounts need the strongest controls because compromise means full control.

Why Privileged Accounts Are High Risk

# What a compromised admin account can do: Create new admin accounts (persistence) Disable security controls (AV, EDR, logging) Exfiltrate any data (no access restrictions) Encrypt systems (ransomware with no shadow copies) Cover tracks (delete logs) Move laterally to all systems (domain admin = all) # Statistics: 80% of breaches involve privileged credentials Most damaging attacks involve admin account compromise Domain Admin compromise = game over for entire org # Common problems without PAM: Shared admin passwords ("Password123!" for everyone) Passwords never rotated (set once, never changed) No visibility into who used admin when Admin accounts used for daily work tasks Service accounts with admin rights unnecessarily

PAM Core Components

🔐
Password Vault
Stores all privileged credentials encrypted. Admin never sees the actual password — requests access through vault. Vault checks out credential, injects it, checks it back in. Automatic password rotation after each use. Nobody knows the password — only the vault does. Examples: CyberArk, HashiCorp Vault, Delinea (formerly Thycotic).
⏱️
Just-in-Time (JIT) Access
Admin privileges granted only for the duration needed — not permanent. Engineer needs to patch server: request JIT access, get 2-hour admin window, access expires automatically. No persistent admin privileges sitting exposed. Also called: Privileged Elevation and Delegation Management (PEDM).
🎥
Session Recording
Every privileged session recorded — full screen capture + keystroke logging. Forensic evidence: "what did admin do during this session?" Compliance: PCI-DSS, HIPAA require privileged session monitoring. Deterrent: admins know they're being recorded — less insider threat risk. Searchable: find "who ran this command on this server?"
💻
Privileged Access Workstation (PAW)
Dedicated hardened workstation used ONLY for privileged administration. Never used for: browsing, email, Teams, regular work. Separate from regular workstation. Cannot access internet directly. No software installation. Hardware enforced — separate physical machine or VM. Drastically reduces attack surface for admin actions.

Types of Privileged Accounts

Domain Admin (DA)

Highest privilege in Windows environment. Can do anything in the domain. Should be: locked in vault, used via JIT only, never used for daily work. Compromise = game over for entire AD environment.

Local Administrator

Admin on individual machine only. LAPS (Local Administrator Password Solution) — Microsoft tool that sets unique random password on every machine's local admin account. Prevents lateral movement using shared local admin password.

Service Accounts

Non-human accounts — run applications and scheduled tasks. Often over-privileged and never have password rotated. Managed Service Accounts (MSA) — Windows auto-rotates password. Must be inventoried in PAM vault.

Cloud Root / Global Admin

AWS root account or Azure Global Admin. Never used for daily work. No access keys. MFA always required. Locked away — only used in break-glass scenarios. Monitored 24/7 — any use triggers immediate alert.

Emergency / Break-Glass Account

Last resort account used when all other admin access fails. Credential in physical safe. Password known to CISO + CTO only. Any use immediately triggers P1 security alert. Reviewed quarterly — confirm it hasn't been used.

API Keys / Secrets

Non-human privileged credentials. Access cloud APIs, databases, third-party services. Must be stored in secrets manager (HashiCorp Vault, AWS Secrets Manager). Never hardcoded in code. Auto-rotated regularly.

07Access Control ModelsRBAC · ABAC · MAC · DAC

Access control models define how authorization decisions are made — who decides who gets what access, and based on what criteria. Different models suit different environments and sensitivity levels.

The Four Core Access Control Models

DAC
Discretionary Access Control
Resource OWNER decides who can access their resources. Most flexible — owner controls access. Most common in everyday file systems (Windows NTFS, Linux file permissions). Owner of a file can share it with anyone they choose.
Example: Sarah creates file: budget.xlsx Sarah is OWNER → she decides who can access Sarah: grants READ to Finance team Sarah: grants EDIT to her manager Sarah: revokes access from Bob (she decides) # No central authority — owner controls everything
MAC
Mandatory Access Control
System enforces access based on SECURITY LABELS. Owner has no control — security policy decides. Used in military/government — TOP SECRET users can only access TOP SECRET or below. Users cannot share above their clearance. Most restrictive model.
Labels on objects: TOP SECRET | SECRET | CONFIDENTIAL | UNCLASSIFIED Labels on subjects: User clearance level Rule — "read down, write up" (Bell-LaPadula): SECRET user + SECRET file → ALLOWED SECRET user + TOP SECRET file → DENIED (no read up) SECRET user + UNCLASSIFIED file → ALLOWED (read down) # Owner CANNOT override — system enforces always
RBAC
Role-Based Access Control
Access granted based on JOB ROLE not individual identity. Most common enterprise model. User assigned to role → role has permissions → user gets those permissions. Role changes automatically update all permissions. Simple to manage at scale.
Roles defined: Finance-Analyst: READ finance drive, USE accounts payable Finance-Manager: + APPROVE payments, VIEW all reports HR-Staff: READ HR system, EDIT employee records IT-Admin: ADMIN all systems, RESET passwords User assigned role: sarah.chen → Finance-Analyst role Gets: exactly Finance-Analyst permissions Nothing more, nothing less # Change role → all permissions change automatically
ABAC
Attribute-Based Access Control
Most granular and flexible model. Access based on MULTIPLE ATTRIBUTES simultaneously: user attributes (department, clearance, location) + resource attributes (classification, owner) + environment attributes (time, device, location). Used in Zero Trust architectures and cloud environments.
Policy example: ALLOW access IF: user.department == "Finance" AND user.clearance >= "Confidential" AND resource.classification == "Financial" AND device.managed == true AND time.hour BETWEEN 8 AND 18 AND location.country == "Canada" Same user — different result: Sarah, Finance, Monday 9am, managed laptop, Toronto: ALLOWED Sarah, Finance, Sunday 2am, personal phone, Paris: DENIED

Additional Access Control Concepts

Rule-Based AC

Access based on rules/conditions — not role or identity. Firewall rules are rule-based: "allow TCP 443 from any to webserver." Time-based rules: "allow access Monday-Friday 9am-6pm only." Often confused with RBAC — different!

Separation of Duties (SoD)

No single person controls entire critical process. Finance: one person requests, another approves, another executes. Prevents fraud and errors. Cannot have same person create and approve invoices. Also called dual control or two-man rule.

Need-to-Know Principle

Access only to information REQUIRED for specific job function. Even with appropriate clearance level — if you don't need it for your job, you don't get it. Limits insider threat blast radius. Complements MAC in government environments.

Least Privilege

Grant minimum permissions necessary to do the job. No more, no less. Regular users: no admin rights. Admins: use standard account for daily work, admin account only for admin tasks. Review and remove unused privileges regularly.

Time-of-Day Restrictions

Access only during authorized hours. Contractor: Monday-Friday 9am-5pm only. System maintenance account: 2am-4am Saturday only. Prevents off-hours access that could indicate compromise or insider threat.

Location-Based Access

Access restricted to specific geographic locations. VPN required for remote access. Sensitive systems: only from office network. Cloud: conditional access policy requiring trusted location. Geofencing alerts on access from unexpected countries.

08Zero Trust ArchitectureNever Trust · Always Verify

Zero Trust is a security model based on the principle "never trust, always verify" — no user, device, or network is trusted by default, even if inside the corporate network. Every access request is verified before being granted, every time.

Zero Trust vs Traditional "Castle and Moat"

Traditional — Castle & Moat
Trust inside network, distrust outside
VPN = fully trusted once connected
Authentication once → access all
Attacker inside = unrestricted movement
Flat network — lateral movement easy
Zero Trust — Verify Always
No implicit trust based on network location
Every access request verified — always
Least privilege for each access
Micro-segmentation limits lateral movement
Assume breach — always monitoring

Zero Trust Core Principles

# NIST SP 800-207 Zero Trust Tenets: 1. All resources treated as untrusted: Network location = irrelevant Inside the office? → still verify On VPN? → still verify "We don't trust our own network" 2. Least privilege access: Only access needed for current task Time-limited sessions No persistent broad access JIT access for privileged operations 3. Verify explicitly: Check all available signals: Identity: Who is this user? Device: Is this device healthy and managed? Location: Where is this request coming from? Behavior: Does this match normal patterns? Risk: What's the current risk score? 4. Assume breach: Act as if attacker is already inside Micro-segment networks Encrypt all traffic (even internal) Monitor everything continuously Minimize blast radius 5. Continuous evaluation: Not just at login — verify throughout session Risk score changes → re-verify or terminate Sensitive action → step-up authentication

Zero Trust Components

Identity Verification

Strong identity = foundation of ZT. MFA required always. Conditional access policies. Risk-based authentication. Identity is the control plane. Verified on every access request.

Device Trust

Device health verified before access. MDM enrollment required. Patch compliance checked. EDR installed and active. Certificate-based device identity. Unmanaged devices → limited access only.

Network Micro-segmentation

Break flat network into tiny segments. Each segment = separate trust boundary. East-west traffic controlled. Server-to-server access requires explicit policy. Lateral movement becomes nearly impossible.

Application Access

Applications not exposed to network directly. Accessed through secure proxy/gateway. User verified before app connection established. ZTNA (Zero Trust Network Access) replaces VPN. App-level microsegmentation.

Data Protection

Data classified and protected at all locations. Encryption in transit AND at rest. DLP enforces data access policies. Access to sensitive data requires additional verification. Data sovereignty and compliance built in.

Continuous Monitoring

All access logged and analyzed. UEBA detects behavioral anomalies. SIEM correlates across all signals. Real-time risk scoring. Suspicious behavior → immediate re-verification or session termination.

ZTNA — Zero Trust Network Access

# ZTNA replaces traditional VPN: Traditional VPN: User connects → VPN tunnel → access to ENTIRE internal network Risk: user (or malware) can reach all internal systems One compromised endpoint → flat network access ZTNA: User connects → verified → access to SPECIFIC app only Not the network — just the application Even on "trusted" network — access is app-by-app Micro-application perimeter Example: HR staff → ZTNA → HR system only (not finance, not IT) Finance → ZTNA → Finance systems only Developers → ZTNA → Dev servers only (not prod) # ZTNA providers: Cloudflare Access, Zscaler ZPA, Palo Alto Prisma Access Cisco Duo Network Gateway, Google BeyondCorp

Conditional Access — Zero Trust in Practice

# Microsoft Entra ID Conditional Access — ZT enforcement: Policy: "Access to Finance app" Conditions evaluated: Is this user in Finance group? Is device compliant (MDM + patches)? Is MFA satisfied? Is location trusted (office or home VPN)? Is sign-in risk low? All conditions met → ALLOW with standard MFA Device non-compliant → ALLOW web only (no download) Unfamiliar location → ALLOW + require hardware key MFA High risk sign-in → BLOCK + alert security team Unknown device + foreign country → BLOCK immediately # Same user — different decisions based on context # This IS Zero Trust — continuous contextual evaluation
