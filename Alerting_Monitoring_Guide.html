01Alerting & Monitoring OverviewVisibility · Detection

Monitoring is the continuous collection and analysis of data from systems, networks, and applications to detect threats, anomalies, and policy violations. Alerting is the mechanism that notifies the right people when something suspicious is found.

The Core Problem Monitoring Solves

# Without monitoring — you're flying blind: Attacker enters network: You don't know Attacker moves laterally for weeks: You don't know Attacker exfiltrates 50GB of data: You don't know Attacker leaves backdoor: You don't know Average time to detect a breach (industry stats): Without monitoring: 207 days (6.9 months!) With monitoring: days to hours Average cost difference: Breach detected quickly: $3.05M average Breach detected late: $4.86M average Monitoring ROI: obvious # With monitoring — you see everything: Attacker enters network: ALERT — unusual login Attacker moves laterally: ALERT — lateral movement detected Attacker tries to exfiltrate: ALERT — large data transfer blocked

Monitoring Principles

Continuous — Not Periodic

Attacks happen at 3am Sunday. Monitoring must run 24/7/365. Not a weekly check — continuous real-time analysis. SOC (Security Operations Center) staffed around the clock or MDR (Managed Detection & Response) service used.

Centralized Collection

All logs from all sources sent to central platform (SIEM). Logs on local systems can be deleted by attackers — central logs much harder to tamper with. Single pane of glass for analysts.

Baseline First

Must know what "normal" looks like before detecting "abnormal." Baselines built over time — normal login hours, normal traffic volume, normal data transfer rates. Deviations from baseline = alerts.

Monitoring Architecture — What Feeds What

# Data sources → Collection → Analysis → Alert → Response DATA SOURCES: Endpoints: EDR agents → process, file, network activity Network: Firewalls, switches, routers, IDS/IPS → traffic Identity: Active Directory, IAM → logons, privilege use Applications: Web servers, DBs, apps → access, errors Cloud: AWS CloudTrail, Azure Monitor → API calls Email: Gateway logs → spam, phishing, attachments Physical: Badge readers, cameras → physical access COLLECTION: Syslog (UDP 514 / TCP 6514 encrypted) → SIEM Agent-based → EDR platform API integration → Cloud security tools NetFlow → Network monitoring ANALYSIS: Correlation rules → known attack patterns Behavioral analytics → baseline deviations (UEBA) Threat intelligence → known bad IPs/domains/hashes ML/AI → anomaly detection ALERT → ANALYST → RESPONSE

02SIEM — Security Information and Event ManagementHeart of SOC

SIEM is the central platform that collects logs from every source, correlates events across them, and generates alerts when suspicious patterns emerge. It's the single pane of glass for security monitoring — the brain of a SOC.

What SIEM Does — Core Functions

Log Collection & Aggregation

Ingests logs from: firewalls, servers, workstations, network devices, cloud services, applications, identity systems. Normalizes different log formats into common schema. Thousands of events per second from hundreds of sources — all in one place.

Correlation Engine

Applies rules that look for patterns ACROSS multiple log sources simultaneously. One failed login = noise. 100 failed logins across 50 accounts in 10 minutes = password spraying alert. SIEM connects dots humans can't connect at scale.

Alerting

Generates alerts when correlation rules trigger. Prioritizes by severity. Routes to right team. Integrates with ticketing systems. Can page on-call analyst for critical alerts. Reduces noise — only alert on meaningful events.

Dashboards & Visualization

Real-time security dashboards. Geographic maps of login origins. Top threat sources. Event timeline for investigations. Customizable per audience — executive vs analyst views.

Historical Analysis

Search historical logs for investigation. "Show me all events from this IP address in last 90 days." Forensic timeline reconstruction. Threat hunting — proactively search for hidden attackers.

Compliance Reporting

Pre-built compliance reports: PCI-DSS, HIPAA, SOX, ISO 27001. Evidence of monitoring for auditors. Retention policies ensure logs kept for required period. Automated report generation.

SIEM Correlation — The Magic

# Single events that look harmless: Event 1 (AD): john.smith failed login at 02:14 Event 2 (VPN): Connection from Bucharest, Romania at 02:15 Event 3 (File): john.smith accessed finance share at 02:17 Event 4 (Email): john.smith forwarded 500 emails to gmail at 02:19 Event 5 (DLP): Large data upload blocked from john.smith at 02:21 # Each event alone: maybe nothing # SIEM correlates ALL FIVE → pattern: CRITICAL ALERT: Possible Account Compromise + Data Exfiltration → Failed login (credential testing) → Foreign country access (impossible travel?) → Sensitive file access (lateral movement) → Email forwarding (exfil attempt) → DLP block (exfil stopped) → All within 7 minutes → automated attacker behavior # Analyst action: disable john.smith account immediately # Investigation: was john.smith actually in Romania?

03EDR, MDR, and XDREndpoint · Extended · Managed

EDR — Endpoint Detection and Response — goes far beyond traditional antivirus. It monitors endpoint behavior continuously, detects threats by what they DO not just what they are, and can automatically respond to contain threats.

EDR vs Traditional Antivirus

# Traditional Antivirus — signature based: New malware arrives on endpoint AV scans: checks against signature database New malware: NO SIGNATURE → PASSES THROUGH ✗ Malware executes: damage done Detection: only after signature update (days/weeks later) # EDR — behavior based: New malware arrives on endpoint EDR: doesn't check signature — watches BEHAVIOR Malware starts executing: → Spawns cmd.exe from Word ← suspicious → Reads sensitive file ← suspicious → Makes outbound connection ← suspicious → Tries to disable AV ← VERY suspicious EDR: "This behavior pattern = malware" EDR response: KILL PROCESS + QUARANTINE + ALERT ✓ Result: stopped BEFORE damage done zero-day malware stopped by behavior

What EDR Monitors on Each Endpoint

Process Activity

  • Every process created — parent/child relationships
  • Process injection detection
  • Unusual process spawning (Word → PowerShell)
  • Command line arguments recorded
  • Process memory analysis
  • DLL loading and injection

File System Activity

  • File creation, modification, deletion
  • Executable file drops
  • Mass file modification (ransomware indicator)
  • Shadow copy deletion attempts
  • Sensitive file access patterns
  • File hash comparison to threat intel

Network Activity

  • All outbound connections from endpoint
  • DNS queries made by processes
  • Connections to known-bad IP/domains
  • Unusual port usage
  • Data volume transferred
  • C2 callback patterns

Registry & System

  • Registry modifications (persistence mechanisms)
  • Startup item additions
  • Service creation and modification
  • Scheduled task creation
  • User account changes
  • Security tool tampering attempts

EDR Response Actions

# EDR can respond automatically or give analyst options: Automated responses: Kill process: terminate malicious process immediately Quarantine file: move to sandboxed location, isolate from system Block hash: prevent file with same hash from running again Network isolation: cut endpoint from network (keep EDR connection) → analyst can still investigate remotely! Manual analyst actions via EDR console: Live response: run commands on remote endpoint Memory dump: capture RAM for forensic analysis File retrieval: pull specific files for analysis Timeline view: see everything that happened hour by hour Threat hunting: search across ALL endpoints simultaneously "Show me every endpoint that ran this process" "Find all endpoints that contacted this IP"

04Network Monitoring ToolsIDS · IPS · NetFlow · Packet

Network monitoring tools provide visibility into traffic flowing across the network — who is talking to whom, what protocols, what volumes, and whether any of it looks malicious.

IDS / IPS — Already Covered in Depth

IDS — Intrusion Detection System

Out of band — receives copy of traffic. Detects and ALERTS only. Cannot block traffic. No impact on network performance. Good for passive monitoring of critical links.

Placement: SPAN port or network TAP Traffic copy → IDS → alert only Real traffic: flows regardless of IDS

IPS — Intrusion Prevention System

Inline — all traffic passes through. Detects AND BLOCKS. Can drop malicious packets in real time. Risk: false positives block legitimate traffic. Performance impact possible.

Placement: Inline — between zones All traffic → IPS → pass or drop Bad traffic: dropped immediately

NetFlow Analysis

📊
What Is NetFlow?
NetFlow is a network protocol that records metadata about traffic flows — NOT the actual content. Records: source IP, destination IP, ports, protocol, bytes transferred, duration. Lightweight — no deep packet inspection needed. Cisco invented it (also: IPFIX = standardized version, sFlow = sampled version).
# NetFlow record example: Src IP: 10.0.0.47 Dst IP: 185.220.101.34 Src Port: 54321 Dst Port: 443 Protocol: TCP Bytes: 847,293,048 (847 MB) Start: 02:14:33 End: 02:47:12 Duration: 32 minutes # Analyst reads this as: Internal workstation → suspicious external IP 843 MB transferred over 32 minutes At 2am = DATA EXFILTRATION INDICATOR ← investigate immediately # NetFlow can't tell WHAT was transferred # But volume + destination + time = enough for alert # Tools: SolarWinds NTA, Cisco Stealthwatch, ntopng, Kentik

Packet Capture (PCAP)

# Packet capture = capturing actual packet contents Tools: Wireshark (GUI), tcpdump (CLI), NetworkMiner When to use full packet capture: → Incident investigation (what exactly happened?) → Malware analysis (what did it download?) → Protocol debugging → Forensic evidence collection Limitations: → High storage requirements (1Gbps = 450GB/hour!) → Privacy/legal concerns — captures ALL content → Encrypted traffic: see metadata, not content → Usually only deployed on specific segments Common approach: NetFlow everywhere + PCAP on tap for key segments Or: trigger full PCAP capture when alert fires

Network Detection and Response (NDR)

🌐NDR — Network Detection and ResponseBehavioral Network Analysis
Modern evolution of IDS/IPS. Uses ML/AI to detect network anomalies — not just signature matching. Establishes baseline of normal network behavior. Detects: lateral movement, C2 traffic, insider threats, data exfiltration — even in encrypted traffic (by analyzing metadata patterns). Examples: Darktrace, ExtraHop, Vectra AI, Cisco Stealthwatch.
# NDR detects encrypted C2 traffic by behavior: Can't decrypt: HTTPS traffic Can analyze: Connection timing: beaconing every 5 minutes exactly Packet sizes: suspiciously uniform Destination: newly registered domain Certificate: self-signed, unusual attributes NDR: "This looks like C2 beaconing behavior" → ALERT

DNS Monitoring

# DNS = treasure trove of threat intelligence Why monitor DNS: Every C2 connection starts with DNS lookup Malware often uses DNS for covert comms (DNS tunneling) Domain generation algorithms (DGA) detectable Known-bad domains blocked at DNS layer DNS monitoring detects: DNS tunneling: excessively long DNS queries "bWFsd2FyZWRhdGFleGZpbHRyYXRpb24=.evil.com" (base64 encoded data in subdomain!) DGA malware: malware generating random domains "xjk23ndf.com, kdf93kls.net, qwer7823.org" (rotating C2 domains to evade blacklists) Known bad domains: query to known malware C2 domain # DNS security tools: Cisco Umbrella (OpenDNS): cloud-based DNS security Infoblox: enterprise DNS monitoring Pi-hole: lightweight DNS blocker DNSBL: DNS-based blackhole lists

Other Network Monitoring Tools

SNMP Monitoring

Simple Network Management Protocol — monitors network devices (switches, routers, firewalls). Tracks: interface utilization, error rates, uptime, CPU/memory. Alerts on device failure or unusual utilization. SNMPv3 = secure (encrypted + authenticated). SNMPv1/v2 = insecure.

Bandwidth Monitoring

Track network utilization over time. Spike at 3am = potential exfiltration or attack. Unusual protocol traffic = indicator. Tools: PRTG, SolarWinds, Cacti, Grafana + Prometheus.

Wireless Monitoring (WIDS)

Wireless Intrusion Detection System. Detects rogue access points, evil twin attacks, deauth floods, unauthorized clients. Monitors RF spectrum continuously. Enterprise APs often have WIDS built in.

Log Collection Methods

# Three main collection approaches: 1. Syslog (push-based): Devices push logs to central syslog server Protocol: UDP port 514 (unencrypted) TCP port 6514 (TLS encrypted — preferred) Syslog-ng / rsyslog = common syslog servers Simple, universal, supported by all network devices 2. Agent-based (pull-based): Agent installed on each system Agent collects, filters, and forwards logs More reliable — resends if connection lost Examples: Splunk Universal Forwarder Elastic Beats (Filebeat, Winlogbeat) Microsoft Monitoring Agent (MMA) 3. API integration: Cloud services → pull via API AWS CloudTrail → API → SIEM Microsoft 365 → Graph API → SIEM Salesforce → REST API → SIEM

Log Integrity

Why Integrity Matters

Attackers who compromise systems often try to delete or modify logs to cover their tracks. Local logs = easy to tamper with. Centralized logs = much harder. Append-only logs = can't be modified. WORM storage = cannot be changed after written.

Attacker compromises server: Deletes /var/log/auth.log ← local logs gone But SIEM already received those logs: → Logs sent to SIEM seconds after creation → Deletion of local copy doesn't affect SIEM → Evidence preserved ✓

Integrity Methods

  • Centralized syslog — off system immediately
  • Append-only log storage
  • WORM (Write Once Read Many) storage
  • Cryptographic log signing (hash chain)
  • Immutable cloud storage (AWS S3 Object Lock)
  • Separate credentials for log system access

Time Synchronization — Critical for Logs

# NTP — Network Time Protocol — essential for log correlation Problem without NTP: Server A clock: 14:32:15 Server B clock: 14:34:47 (2 min 32 sec ahead) Firewall clock: 14:30:02 (2 min 13 sec behind) Incident timeline reconstruction: 14:32:15 — Server A: user logged in 14:30:02 — Firewall: connection allowed (BEFORE login?) 14:34:47 — Server B: file accessed (AFTER other events?) Timeline makes no sense → investigation fails With NTP — all clocks synchronized: 14:32:15 — Firewall: connection allowed 14:32:15 — Server A: user logged in 14:32:16 — Server B: file accessed Timeline makes perfect sense → attack reconstructed ✓ NTP port: UDP 123 Hierarchy: Stratum 0 (atomic clock) → Stratum 1 server → Stratum 2... 06Alert Types & TuningTP · FP · FN · TN

Generating alerts is easy — generating meaningful alerts is hard. Too many alerts causes alert fatigue. Too few means threats are missed. Tuning the alerting system is a critical ongoing security operations task.

The Four Alert Outcomes

True Positive (TP) ✅

Alert fired AND the threat is real. This is what we want — correctly identified attack. Analyst investigates and confirms malicious activity. Take action to respond.

Alert: "Ransomware behavior detected" Investigation: Confirmed — ransomware running Action: Isolate endpoint, initiate IR Result: CORRECT DETECTION ✓

False Positive (FP) ❌

Alert fired BUT there's no actual threat. Legitimate activity mistaken for attack. Major source of alert fatigue — analysts waste time on non-issues. Too many FPs = analysts start ignoring alerts.

Alert: "Port scan detected from 10.0.0.5" Investigation: That's our vulnerability scanner running its scheduled scan! Action: Tune rule to exclude scanner IP Result: WASTED ANALYST TIME

True Negative (TN) ✅

No alert AND there's no threat. Normal activity correctly identified as benign. This is what happens for the vast majority of traffic. The system working correctly for non-attacks.

False Negative (FN) ⚠️

No alert BUT a real threat exists. MISSED ATTACK. Most dangerous outcome. Attacker operates undetected. Causes: outdated signatures, coverage gaps, evasion techniques. Regular threat hunting helps catch FNs.

Attacker uses novel technique — no rule for it SIEM: no alert generated Reality: attacker active for days Result: BREACH GOES UNDETECTED

Alert Fatigue — The Real Danger

# Alert fatigue is one of the biggest real-world problems: SOC analyst Monday morning: Alert queue: 3,847 unreviewed alerts from weekend Analyst: overwhelmed, starts clicking "dismiss" Reality: 3,800 false positives → safely dismissed 47 real attacks → ALSO dismissed (can't process all) Alert fatigue killed: attacker hidden in noise Real example (Target breach): FireEye security system DID alert → twice Alert buried in thousands of other alerts Security team: overwhelmed, didn't prioritize it Result: breach continued for weeks Solutions to alert fatigue: 1. Tune rules → reduce false positives 2. Prioritize → focus on high-severity first 3. Automate → auto-close obvious false positives 4. Enrich → add context so analyst decides faster 5. SOAR → automate response for common alert types

Alert Types by Detection Method

Signature-Based Alerts

Match known attack patterns — CVE exploits, malware signatures, known bad IPs/domains/hashes. Low false positive rate for known threats. Zero detection of unknown threats. Must be continuously updated.

Example rules: "Alert if packet matches CVE-2017-0144 exploit" "Alert if hash matches known ransomware" "Alert if connection to known C2 IP"

Anomaly / Threshold Alerts

Alert when metrics exceed thresholds or deviate from baseline. Higher false positive rate — legitimate anomalies trigger. Catches unknown attacks. Requires good baseline establishment.

Example rules: "Alert if failed logins > 10 in 5 minutes" "Alert if data transfer > 1GB in 1 hour" "Alert if login from new country for user" "Alert if process count spikes 300%"

Alert Prioritization

🔴 P1 — Critical Confirmed active attack, ransomware, data breach in progress, business-critical system compromised 15 min response
🟠 P2 — High Malware detected, suspicious admin activity, privilege escalation, lateral movement detected 1 hour response
🟡 P3 — Medium Unusual user behavior, policy violations, multiple failed logins, suspicious process execution 4 hour response
🔵 P4 — Low Informational, routine events, low-risk anomalies, compliance-related logging Next business day

Alert Tuning Best Practices

# Practical alert tuning approach: Step 1 — Inventory your alerts: How many alerts per day? What % are false positives? Which rules generate most noise? Which rules generate true positives? Step 2 — Tune the noisy rules: Add exceptions for known-good IPs (vulnerability scanner) Increase thresholds (10 failed logins → 50 for lab systems) Add time conditions (alert only after-hours) Exclude maintenance windows Step 3 — Enrich alerts with context: Add: asset owner, asset criticality Add: user role and department Add: threat intel on source IP Add: previous incident history Result: analyst decides faster with more context Step 4 — Automate obvious ones: Auto-close: known scanner IPs → "false positive" Auto-respond: malware detected → auto-quarantine SOAR playbooks handle routine alerts automatically Step 5 — Regular review: Monthly: review alert volume and FP rate Quarterly: review all active rules Continuous: tune after each false positive

07Computing Resource Activities to MonitorWhat to Watch

Security monitoring must cover specific activities on computing resources. Knowing what to look for — and why — is as important as having the tools.

System Resource Monitoring

CPU Usage

Sudden CPU spikes on server with no user activity = suspicious. Indicators of: cryptomining (sustained high CPU), malware running heavy computation, DoS attack consuming resources, large-scale data processing (exfiltration preparation).

Normal server: 5-20% CPU during business hours 2-5% CPU at night Alert trigger: 3am: CPU jumps to 95% → sustained Process: svchost.exe with unusual parent Cryptominer or malware detected

Memory Usage

Unusual memory consumption can indicate: fileless malware living in RAM, memory injection attacks, data being staged in memory before exfiltration, process hollowing (legitimate process hosting malware).

Disk I/O

High disk read activity suddenly = large-scale file access (staging for exfiltration or ransomware encryption scan). High disk write = data being written (ransomware encrypting files). Monitor for: disk encryption patterns, mass file creation/modification.

Network Utilization

Unexpected outbound traffic spikes = potential exfiltration. Traffic to unusual destinations. Traffic at unusual times (3am data transfer). Protocol anomalies — internal server suddenly generating lots of DNS queries.

Key Activities to Monitor

Account and Authentication Activity
Monitor every login event — success and failure. Failed logins (brute force), logins at unusual hours, logins from new geographic locations, logins to multiple systems rapidly (lateral movement), privileged account usage, service account behavior changes, new account creation, group membership changes.
Suspicious authentication patterns: Password spray: 1 failed login each across 500 accounts Brute force: 500 failed logins against 1 account Pass-the-hash: logon type 3 without password attempt Impossible travel: login from NY then UK 30 minutes later Golden ticket: Kerberos ticket with unusual lifetime
Privileged Activity Monitoring
Admin and privileged actions carry highest risk — monitor them all. Admin logins, use of sudo/runas, changes to security policies, group policy modifications, backup/restore operations, security tool configuration changes, audit policy modifications (attacker covering tracks).
CRITICAL alert triggers: Audit policy disabled → attacker covering tracks AV/EDR stopped/uninstalled → attacker silencing defenses New admin account created at 3am Scheduled task created by non-admin Security group membership changed
File and Data Access Activity
Track access to sensitive files and data stores. Mass file access (ransomware scanning or exfil staging), access to sensitive directories by unusual accounts, bulk file downloads from SharePoint/file servers, file permission changes, sensitive data copied to removable media, database queries returning large result sets.
Process and Application Activity
Monitor which processes run, parent-child relationships, command line arguments. Suspicious: Office app → PowerShell, browser → cmd.exe, script interpreters (Python, PowerShell) running encoded commands, process running from temp folder, process creating network connections that don't normally do so.
HIGH FIDELITY attack indicators: winword.exe → powershell.exe (macro malware) cmd.exe running from %TEMP% (dropped malware) powershell.exe -enc [long base64] (encoded payload) regsvr32.exe /s /n /u /i:http:// (Squiblydoo) rundll32.exe executing suspicious DLL
Network Connection Activity
Every network connection is a potential data path. Monitor for: connections to known-bad destinations, beaconing behavior (connections at regular intervals = C2), unusual protocols or ports, internal systems connecting to internet directly (should go through proxy), DNS queries for newly registered or DGA domains, large outbound data transfers.
Cloud Resource Activity
Cloud environments introduce new activities to monitor. API calls from unusual locations, resource creation/deletion at unusual times, IAM policy changes, S3 bucket permission changes (becoming public!), new root account activity, cost anomalies (cryptomining), unusual data egress charges, service quota changes.
AWS CloudTrail alert triggers: Root account login (should NEVER be used daily) New IAM admin created S3 bucket ACL changed to public Security group rule allows 0.0.0.0/0 CloudTrail logging disabled ← attacker covering tracks KMS key deletion initiated

08SOAR & Security AutomationOrchestrate · Automate · Respond

SOAR — Security Orchestration, Automation, and Response — automates repetitive security tasks and response actions so analysts can focus on complex investigations. Turns hours of manual response into seconds of automated action.

What SOAR Does

# Manual response vs SOAR response: Manual response to phishing alert: Analyst receives alert: 09:14 Analyst reviews email: 09:19 (5 min) Analyst checks URL in sandbox: 09:23 (4 min) Analyst blocks URL on proxy: 09:27 (4 min) Analyst checks who else received it: 09:35 (8 min) Analyst notifies IT to quarantine emails: 09:41 (6 min) Analyst creates ticket: 09:47 (6 min) Total: 33 minutes — one analyst SOAR automated response to same phishing alert: Alert received: 09:14:00 SOAR triggered: 09:14:00 URL reputation checked (VirusTotal API): 09:14:02 URL blocked on all proxies: 09:14:03 All recipients identified (email API): 09:14:04 All copies quarantined (Exchange API): 09:14:05 Ticket created with full context: 09:14:06 Analyst notified with summary: 09:14:07 Total: 7 seconds — zero analyst time Analyst reviews summary → closes ticket or escalates Analyst time saved: 32+ minutes for complex work

SOAR Playbooks — Automated Response Workflows

🎣Phishing PlaybookAutomated
Triggered when phishing email reported or detected.
1. Extract: sender, URLs, attachments from email 2. Check: URL reputation (VirusTotal, URLscan.io) 3. Check: attachment hash against threat intel 4. If malicious: → Block sender domain on email gateway → Block URL on web proxy → Quarantine all copies in organization → Notify recipients (awareness email) → Create incident ticket 5. If suspicious (unknown): → Detonate attachment in sandbox → Analyst review required 6. Document all actions in ticket automatically
🖥️Compromised Endpoint PlaybookAutomated + Manual
Triggered when EDR detects malware or compromise on endpoint.
1. Get asset details: owner, department, criticality 2. Isolate endpoint from network (EDR API) 3. Preserve memory dump and disk image 4. Pull recent process, network, file activity 5. Check: lateral movement from this host? → If yes: isolate affected systems too 6. Notify: asset owner, manager, security team 7. Create P1 ticket with full context 8. Assign to IR analyst for investigation 9. Schedule: forensic analysis, reimaging
🔐Suspicious Login PlaybookAutomated
Triggered on impossible travel, new country login, or multiple failed attempts followed by success.
1. Get user details (AD, HR system) 2. Get login history for comparison 3. Check: is user actually traveling? (check Outlook calendar) 4. Check: is VPN/proxy being used? 5. Calculate: distance between last login and current 6. If impossible travel (>500mph): → Force MFA re-authentication → If MFA fails: disable account → Notify user and manager 7. If genuine concern: disable account, notify 8. Document investigation in ticket

SOAR Products

Splunk SOAR (Phantom)

Most widely deployed SOAR. Tight Splunk SIEM integration. Hundreds of pre-built app connectors. Visual playbook builder. Python-based for custom automation. Strong for large enterprises.

Microsoft Sentinel (built-in)

SOAR capabilities built into Sentinel SIEM. Logic Apps for workflow automation. Native Microsoft 365 and Azure integration. No separate SOAR product needed for Microsoft shops.

Palo Alto XSOAR (Demisto)

Enterprise SOAR with strong threat intel integration. Collaborative investigation features — multiple analysts on same case. Good for MSSPs managing multiple clients.

Threat Hunting — Proactive Monitoring

🔍
What Is Threat Hunting?
Proactive search for threats that have evaded automated detection. Hunters form hypotheses ("What if an attacker is using living-off-the-land techniques?") and manually search through data to find evidence. Finds the false negatives — attacks that slipped through all automated controls. Requires skilled analysts, good data, and time.
# Threat hunting example hypothesis: "Are any attackers using PowerShell to live off the land?" Hunter searches SIEM/EDR for: → PowerShell with -EncodedCommand flag → PowerShell downloading from internet (IEX, WebClient) → PowerShell running Base64-encoded commands → PowerShell with unusual parent processes Finding 1: 3 endpoints running encoded PowerShell at 2am Finding 2: Command decodes to: download + execute from pastebin Result: Active attacker found — not caught by automated detection IR initiated → attacker removed before escalation ✓
