Segmentation
Network · VLAN · Micro-seg · Zero Trust
Limits Lateral Movement
SEGWhy Network Segmentation MattersDefense in Depth

Segmentation divides a network into isolated zones with controlled communication between them. Its primary security value is limiting lateral movement — an attacker who compromises one segment cannot freely reach all others.

# Flat network vs segmented — the security difference: FLAT (no segmentation): IoT thermostat → CAN reach domain controller Marketing laptop → CAN ping finance database Attacker on ANY device: free to reach everything Lateral movement: trivial — one hop to crown jewels SEGMENTED: IoT VLAN: only reaches HVAC controller Corp VLAN: only reaches permitted apps via firewall Finance VLAN: isolated — only finance apps + audit logging Server VLAN: only reachable from authorized sources Attacker compromises IoT thermostat: → Stuck in IoT VLAN ✓ → Cannot reach AD, finance DB, servers → Blast radius: the thermostat # Segmentation is a CORRECTIVE and PREVENTIVE control: Preventive: blocks unauthorized cross-segment traffic Corrective: contains damage when a segment IS breached

// Exam Keywords

Segmentation purpose →Limit lateral movement — contain breaches — reduce blast radius
Flat network risk →No barriers — compromise any device = potential access to everything
Lateral movement →Attacker moving from initial foothold to other systems — segmentation blocks paths

DMZ — The Classic Buffer Zone

🏰
DMZ — Demilitarized Zone
Buffer zone between untrusted internet and trusted internal network. Hosts publicly accessible servers (web, email relay, DNS). Two-firewall DMZ: one between internet and DMZ, another between DMZ and internal network. Compromise of a DMZ server does NOT automatically grant access to internal network — the inner firewall blocks it. One-firewall DMZ (three-legged): single firewall with three interfaces — cheaper but less secure. The compromised web server in the DMZ cannot pivot to the internal AD server because the inner firewall denies it.

// Exam Keywords

DMZ →Buffer zone between internet and internal — houses public-facing servers — compromise ≠ internal access
Two-firewall DMZ →Firewall 1: internet→DMZ / Firewall 2: DMZ→internal — most secure architecture
Guest VLAN →Internet access only — cannot reach corporate resources — IoT isolation
SEGVLANs & Micro-SegmentationLayer 2 · SDN · Workload-Level
VLAN
Virtual LAN — Logical Segmentation
Divides one physical network into multiple isolated broadcast domains. Configured on managed switches via port assignment. Inter-VLAN traffic must pass through Layer 3 device (router/firewall) — security controls applied there. 802.1Q tagging carries VLAN ID across trunk links. VLAN hopping: attacker sends double-tagged 802.1Q frames to jump VLANs — defense: native VLAN != data VLANs, disable DTP. Cost-effective — same physical hardware, multiple logical networks.
Micro-Segmentation
Workload-Level Isolation
Takes segmentation to individual workload level — each server, VM, container has its own security policy. Controls east-west traffic between workloads — not just at VLAN boundary. Software-defined: policy in software, scales to thousands of workloads. VMware NSX, Illumio, Guardicore. Zero Trust networks implement micro-seg. Even if attacker reaches a server, cannot reach adjacent servers without explicit policy allow. More granular than VLANs but more complex to manage.
# VLAN segmentation example — inter-VLAN through firewall only: Switch config: Port 1-20: VLAN 10 (Corporate users) Port 21-30: VLAN 20 (Finance systems) Port 31-40: VLAN 30 (IoT devices) Port 41: VLAN 99 (IT Management trunk) Firewall ACL rules (inter-VLAN): VLAN 10 → VLAN 20: DENY (users can't reach finance directly) VLAN 10 → internet: ALLOW via proxy VLAN 30 → any: DENY except HVAC controller IP VLAN 20 → internet: DENY (finance never touches internet) Result: even inside the "trusted" corporate network, Finance data is protected from all other users IoT devices are completely isolated

// Exam Keywords

VLAN →Logical segment on same physical hardware — inter-VLAN through firewall only — managed switches
VLAN hopping →Double-tagged 802.1Q frame jumps VLANs — fix: native VLAN ≠ data VLANs, disable DTP
Micro-segmentation →Workload-level isolation — east-west control — VMware NSX, Illumio — Zero Trust networks
East-west traffic →Traffic between systems inside the network — micro-seg controls this — traditional firewalls only do north-south
SEGZero Trust SegmentationNever Trust · Always Verify · ZTNA

Zero Trust treats segmentation as identity-based, not network-location-based. Access is granted to specific applications based on verified identity and device health — not based on which network segment you're on.

Traditional Segmentation

Perimeter-based. Inside corporate VLAN = some trust. Segmentation controls which VLAN can reach which VLAN. Once inside a VLAN, relatively free within it. Problem: lateral movement within a VLAN is unrestricted. Weakness: VPN connected = trusted network = access to the whole segment.

Zero Trust Segmentation

Identity-based. Network location is irrelevant. Access decision: who are you + device health + context. ZTNA (Zero Trust Network Access): replaces VPN — grants access to specific app, not network. Micro-seg at workload level. Even inside your segment: you only reach what you're explicitly authorized to reach.

Segmentation in OT / ICS Environments

🏭
Purdue Model — OT Network Segmentation
Industrial Control Systems (ICS) use the Purdue Model to segment OT from IT. Level 0-2: physical process, sensors, PLCs. Level 3: manufacturing operations. Level 4-5: enterprise IT. Each level separated — traffic strictly controlled. Air gap between Level 3 and Level 4: no direct network connection between OT and IT. Data diode: one-way data flow — OT data goes to IT for monitoring but IT cannot send commands to OT. Stuxnet exploited gaps in this segmentation — crossed from internet to air-gapped ICS via USB.

// Exam Keywords

Zero Trust →Identity not location — every request verified — micro-seg at workload level
ZTNA →Zero Trust Network Access — app-level access not network access — replaces VPN
Purdue Model →OT/ICS segmentation — levels 0-5 — air gap between OT and IT networks
Data diode →One-way traffic — OT data goes out to IT — no commands can come back in

Access Control
Identity · Least Privilege · MFA · IAM · PAM
Identity-Based Defence
ACCAccess Control PrinciplesLeast Privilege · Need-to-Know · SoD

Least Privilege

Every user, process, and system gets only the minimum access required to perform its function — nothing more. Reduce blast radius: if account is compromised, attacker only has what the account had. Apply to: user accounts, service accounts, API keys, firewall rules, IAM policies, database permissions.

Need-to-Know

Even with appropriate clearance level, a user should only access data relevant to their current task. Extension of least privilege into data sensitivity. Common in: government / classified, healthcare (HIPAA), finance. "You have Secret clearance but don't need this specific project file."

Separation of Duties

No single person controls a critical process end-to-end. Developer cannot deploy to production without approval. Finance: person who creates payments cannot approve them. Prevents: insider fraud, accidental errors, reduces impact of compromised accounts. Technical enforcement: RBAC, workflow approvals, two-person integrity.

# Least privilege applied everywhere: Web server service account: Needs: read web files, write logs DOES NOT need: admin rights, DB write access, network admin If compromised: attacker gets → web files + logs only Database service account: Needs: SELECT on specific tables app uses DOES NOT need: CREATE, DROP, full DBA access If SQLi: attacker can only read permitted tables Developer in production: Needs: read-only access to troubleshoot DOES NOT need: write access, admin, deployment rights Production changes: via approved pipeline + second approval # Privilege creep: users accumulate permissions over time # Fix: quarterly access reviews — remove what's no longer needed # Certification campaigns: managers attest to what team still needs

// Exam Keywords

Least privilege →Minimum access needed — reduces blast radius — apply everywhere including service accounts
Need-to-know →Even with clearance — only access data relevant to current task
Separation of duties →No single person controls critical process — developer ≠ deployer — creator ≠ approver
Privilege creep →Accumulated permissions over time — quarterly access reviews fix — recertification campaigns
ACCIdentity & Multi-Factor AuthenticationMFA · SSO · Federation
🎫
MFA — The Single Most Effective Access Control
Requiring two or more different authentication factor types before granting access. Microsoft: MFA blocks 99.9% of automated account attacks. Factor types: something you KNOW (password), something you HAVE (TOTP, hardware key), something you ARE (biometric). MFA strength: SMS OTP (weakest — SIM swapping risk) → TOTP app → FIDO2/WebAuthn hardware key (strongest — phishing-resistant, bound to site). Even if attacker steals password: useless without second factor.

SSO — Single Sign-On

Authenticate once → access many applications. Protocols: SAML 2.0 (XML, enterprise federation), OIDC/OAuth 2.0 (web/mobile), Kerberos (on-prem AD). Benefit: reduces password fatigue, centralizes access control, easier to enforce MFA everywhere. Risk: SSO compromise = all apps compromised — make SSO the most protected system with strongest MFA.

Identity Federation

Trust identities from another organization's identity provider. Company A trusts Company B's IdP — B's employees can access A's systems using their own credentials. SAML, OIDC enable federation. B2B partnerships, cloud provider access. Defense: federation agreements define trust boundaries — control what federated users can access.

// Exam Keywords

MFA →Two+ different factor types — blocks 99.9% automated attacks — FIDO2 is phishing-resistant
SMS MFA weakness →SIM swapping can intercept SMS codes — use TOTP app or hardware key instead
FIDO2 / WebAuthn →Phishing-resistant — bound to specific site — hardware security key
SSO →Authenticate once — many apps — SAML / OIDC — must be most protected system

RBAC — Most Common Enterprise

Users assigned to roles → roles have permissions → users inherit. Change a role → all members updated instantly. Roles: Finance-User, IT-Admin, HR-Manager, Developer. Principle of least privilege per role. Problem: role explosion — too many granular roles → management burden.

ABAC — Zero Trust Foundation

"Finance-User in UK, business hours, managed device, can access EU reports." Combines: who (identity attributes), what (resource sensitivity), where/when (environmental context). Most granular — implements complex conditional policies. Foundation of Zero Trust and cloud IAM (AWS IAM conditions, Azure Conditional Access).

ACCPrivileged Access Management (PAM)Admin Accounts · JIT · Session Recording

Admin and privileged accounts are the most targeted credentials in any environment. PAM solutions enforce controls around who can use privileged credentials, when, and how.

# PAM capabilities: Credential vaulting: Store privileged credentials (admin passwords, root keys) in secure vault Users request access → vault provides → session ends → password rotated Admins never see the actual password Even if admin's workstation is compromised: no stored creds to steal Just-in-Time (JIT) access: Admin access granted ONLY when needed → automatically revoked after session No persistent admin accounts sitting idle = no persistent target Workflow: request access → approval → time-limited grant → auto-revoke Session recording: All privileged sessions recorded: every keystroke, screen action Non-repudiation: admin cannot deny what they did Forensics: replay session during investigation Deterrent: "you know everything is recorded" Privileged Access Workstations (PAW): Dedicated hardened workstation ONLY used for admin tasks No internet browsing, email, or general use from PAW Reduces risk of admin credential theft via phishing on admin's regular PC

// Exam Keywords

PAM →Privileged Access Management — vaults credentials — JIT access — session recording
JIT access →Just-in-Time — granted only when needed — auto-revoked — no persistent admin
Credential vault →Admin never sees actual password — vault rotates after use — can't steal what you don't know
PAW →Privileged Access Workstation — hardened dedicated device — admin tasks only — no browsing
ACCAllow Lists & Deny ListsWhitelist · Blacklist · App Control
Allow List (Whitelist)
Default: DENY
Only explicitly approved items permitted. Everything else: blocked. Most secure — unknown = blocked. Use for: application control (AppLocker/WDAC), firewall rules, trusted email senders. Harder to manage: must approve everything upfront. Attacker must get approved — very difficult.
Deny List (Blacklist)
Default: ALLOW
Only explicitly blocked items denied. Everything else: permitted. Less secure — unknown = allowed. Use for: known malicious IPs/domains, AV signatures. Easier to manage: only block known bad. Attacker just needs to use something not yet listed.
🛡️
Application Allow Listing — AppLocker / WDAC
Only approved, digitally signed applications may execute. Windows: AppLocker (Group Policy) and WDAC (Windows Defender Application Control). Any unknown executable — including ransomware — is blocked automatically. Blocks living-off-the-land attacks if specific LOLBins are excluded from approved list. Change management: adding new approved software requires formal change request with security review. Emergency: ECAB approval to add application to allow list.

// Exam Keywords

Allow list →Default deny — most secure — unknown = blocked — AppLocker/WDAC for apps
Deny list →Default allow — known bad blocked — easier but less secure — misses new threats
Application allow listing →Only approved signed apps execute — blocks ransomware + unknown malware

Configuration Enforcement
Baselines · Drift Detection · Compliance Automation
Consistency at Scale
CFGSecurity Baselines & StandardsCIS · STIG · Hardening

A security baseline is a documented, minimum acceptable configuration for a system type. Every system of that type must meet the baseline — deviation requires documented exception with compensating controls.

CIS Benchmarks

Consensus-based configuration standards for 100+ technologies. Level 1: basic security — minimal performance/usability impact. Level 2: more restrictive — may affect usability. Scored: every control has automated pass/fail check. Free download. Covers: Windows, Linux, AWS, Azure, Docker, Kubernetes, network devices. Gold standard for system hardening across all major platforms.

STIG — Security Technical Implementation Guide

US DoD hardening guidance — more restrictive than CIS. Required for DoD systems and many federal contractors. Automated checking: STIG Viewer, Evaluate-STIG. SCAP (Security Content Automation Protocol): machine-readable compliance checking enables automated assessment. XCCDF and OVAL formats for automated scanning. Very specific, prescriptive — leaves little room for interpretation.

# What a baseline defines (examples from CIS Windows L1): Password policy: Minimum length: 14 characters Maximum age: 60 days (optional — NIST now recommends against rotation) Complexity: enabled Account policy: Account lockout threshold: 5 attempts Lockout duration: 30 minutes Audit policy: Account logon: Success and Failure Privilege use: Failure System events: Success and Failure Services to disable: Telnet, FTP, SMBv1, NetBIOS, LLMNR, Print Spooler (on non-printers) Registry settings: LAN Manager authentication: NTLMv2 only (not NTLMv1/LM) UAC: enabled for all administrators

// Exam Keywords

Security baseline →Documented minimum config — every system must comply — exceptions require approval + compensating controls
CIS benchmark →Consensus-based hardening standard — Level 1 (low impact) / Level 2 (more restrictive)
STIG →DoD hardening guides — more restrictive than CIS — federal/military systems
SCAP →Security Content Automation Protocol — machine-readable compliance checking
CFGConfiguration Drift DetectionCMDB · FIM · CSPM

Configuration drift occurs when a system's actual configuration diverges from its approved baseline. Drift can be: accidental (admin made a change without proper process), malicious (attacker modified configuration), or systemic (software update changed settings).

CMDB

Configuration Management Database — authoritative inventory of all configuration items (CIs) and their relationships. Updated after every approved change. Drift: actual config ≠ CMDB record → unauthorized change detected. Every change tracked from baseline through all versions.

FIM — File Integrity Monitoring

Hash every critical file at baseline. Continuously compare current hash to baseline. Any change to config file, binary, or system file → immediate alert. Tools: Tripwire, OSSEC, Wazuh. Critical files: /etc/passwd, web server configs, application binaries. On Windows: key registry keys monitored.

CSPM — Cloud Posture

Cloud Security Posture Management — continuously checks cloud configs against security benchmarks. Alerts when: S3 bucket becomes public, security group allows 0.0.0.0/0, MFA disabled on root account, encryption disabled on storage. AWS Security Hub, Prisma Cloud, Azure Defender. Real-time drift detection for cloud resources.

// Exam Keywords

Configuration drift →Actual config diverges from approved baseline — unauthorized change or accidental modification
FIM →File Integrity Monitoring — hash baseline files — alert on change — Tripwire, OSSEC, Wazuh
CMDB →Configuration Management Database — authoritative record — drift = actual ≠ CMDB
CSPM →Cloud Security Posture Management — continuous cloud config compliance — alerts on misconfig
CFGConfiguration Automation ToolsIaC · Ansible · Group Policy

Infrastructure as Code (IaC)

Define infrastructure in code (Terraform, CloudFormation) — configuration is version-controlled, reviewed, tested. Security benefit: every config change goes through pull request review. Drift prevention: if someone manually changes a resource, IaC can detect and revert (immutable infrastructure). Security scanning: tfsec, Checkov scan IaC code for misconfigs before deployment.

Configuration Management Tools

Ansible, Puppet, Chef: enforce desired system state continuously. If system drifts from desired state → tool automatically corrects. Ansible playbook: "ensure SMBv1 is disabled on all Windows servers" → runs against all servers → non-compliant ones fixed. Agent-based (Puppet) or agentless (Ansible). Idempotent: running twice = same result as running once.

Group Policy (GPO)

Windows: enforce configuration on all domain-joined systems via Active Directory Group Policy Objects. Password policy, screen lock, USB restrictions, firewall rules, software installation, audit settings — all enforced centrally. Cannot be changed by local admin on domain-joined machines (domain policy overrides). Regular GPO audits ensure policies still align with security baseline.

MDM — Mobile Device Management

Enforce configuration on mobile devices and endpoints. Policies: require PIN, enable encryption, install EDR agent, block camera, prohibit sideloading, enforce OS version. Corporate-owned: full MDM control. BYOD: MAM (application management only). Compliance check: device must meet policy before accessing corporate resources.

// Exam Keywords

IaC →Config in code — version controlled — reviewed — drift detection — tfsec/Checkov scan
Ansible / Puppet →Enforce desired state — auto-remediate drift — idempotent
Group Policy →Windows domain config enforcement — centrally managed — overrides local admin
MDM →Mobile config enforcement — compliance before access — MDM = device / MAM = apps only

Hardening

Reduce Attack Surface
HRDOperating System HardeningRemove · Disable · Restrict · Monitor

OS hardening follows one principle: remove or disable everything not needed, secure everything that remains. Hardening reduces attack surface by eliminating potential entry points.

# OS hardening checklist (per CIS Benchmark): Remove unused components: Uninstall: unused packages, compilers on prod, sample apps Remove: default accounts (Guest, default admin names) Delete: help files, documentation not needed at runtime Disable unused services: SMBv1: disable (EternalBlue target) Telnet: disable — use SSH FTP: disable — use SFTP LLMNR / NBNS: disable (credential poisoning target) Print Spooler: disable on non-print servers (PrintNightmare) NetBIOS: disable where not needed Configure secure settings: NTLM: require NTLMv2 minimum (or disable entirely, use Kerberos) UAC: enable for all administrator accounts Windows Firewall: enable with default deny inbound Secure boot: enable in UEFI BitLocker: enable on all drives Audit logging: configure comprehensive audit policy File system permissions: Principle of least privilege on all directories No world-writable directories in system paths Protect: SAM database, shadow passwords # Linux-specific: Root SSH login: disabled (PermitRootLogin no) SSH: key-based auth only (disable password auth) SELinux / AppArmor: enable mandatory access control SUID/SGID binaries: audit and minimize

// Exam Keywords

OS hardening →Remove unused — disable unneeded services — restrict permissions — enable logging
Disable SMBv1 →EternalBlue vector — no legitimate reason to run it — disable immediately
Disable LLMNR/NBNS →Credential poisoning attack vector — disable both on all Windows systems
SELinux / AppArmor →Mandatory access control on Linux — limits what processes can do even if exploited
HRDApplication HardeningWeb Server · Database · Browser

Web Server Hardening

  • Disable directory listing (reveals file structure)
  • Remove: default pages, sample apps, README files
  • Disable verbose error messages (reveals stack traces)
  • Enable security headers: HSTS, CSP, X-Frame-Options, X-Content-Type-Options
  • Run as low-privilege user (www-data not root)
  • TLS 1.2+ only — disable old versions and weak ciphers
  • WAF in front of web application

Database Hardening

  • Rename/delete default accounts (sa, root, admin)
  • Enable authentication — no blank passwords
  • Least privilege DB accounts — no DBA for app accounts
  • Disable xp_cmdshell (MSSQL — OS command execution)
  • Encrypt data at rest (TDE) and in transit (TLS)
  • Enable audit logging of all queries and access
  • Bind to localhost or specific IP — not all interfaces

Browser Hardening (Enterprise)

  • Deploy via Group Policy with security settings
  • Block: known malicious categories (proxy bypass)
  • Disable: Java plugin, Flash (EOL), unnecessary extensions
  • Enable: Safe Browsing, phishing protection
  • Control DNS-over-HTTPS (can bypass DNS filtering)
  • Certificate pinning for internal apps

Change Defaults Everywhere

Default credentials are the #1 IoT and device attack vector. Admin/admin, admin/password — all must be changed immediately on deployment. Default service names reveal software versions. Change: admin portals, SNMP community strings, WiFi PSKs, management interfaces. Document: every non-default credential change in credential vault.

// Exam Keywords

Remove defaults →Default accounts, pages, sample apps — change all default credentials immediately
Security headers →HSTS, CSP, X-Frame-Options, X-Content-Type-Options — web server hardening
xp_cmdshell →MSSQL OS command execution — disable — attack escalation path
TDE →Transparent Data Encryption — database at-rest encryption — hardening control

Isolation
Air Gap · Sandboxing · Quarantine · Deception
Complete Separation
ISOAir Gap & Physical IsolationNo Network Connection · Maximum Security

An air gap is complete physical isolation — no network connection of any kind between the isolated system and any other network. The ultimate isolation technique.

Air Gap

Physical separation — no network cables, no WiFi, no Bluetooth. Used for: classified systems, nuclear plant controls, offline CA root keys, high-value backup systems. Data transfer: only via physical media (USB, CD) under strict procedures. Attack: Stuxnet (2010) crossed an air gap via USB — infected Iranian nuclear centrifuge PLCs. Defense: strict removable media policy, media scanning before use, personnel vetting.

Data Diode

Hardware-enforced one-way data flow. Data can only flow in one direction — physically impossible to send data back. OT to IT: industrial control systems send monitoring data to IT → IT cannot send commands back to OT. Prevents: attacker from using monitoring connection to pivot into OT. Unidirectional Security Gateway (e.g., Waterfall Security, Owl Cyber Defense).

VLAN-Based Isolation

Logical isolation — systems on same physical hardware but in separate broadcast domains. Less absolute than air gap but practical for most environments. IoT VLAN: devices isolated from corporate network. Guest VLAN: internet only. OT VLAN: industrial devices isolated with very restricted access. All inter-VLAN traffic through firewall.

Jump Server / Bastion Host

Hardened intermediary for administrative access to sensitive networks. Admin must authenticate to jump server (MFA required) → then SSH to production servers. Production: only accepts connections from jump server IP. Session recording on jump server. Reduces: attack surface of admin access. Limits lateral movement from admin compromise.

// Exam Keywords

Air gap →Complete physical isolation — no network — Stuxnet crossed via USB — strictest media policy
Data diode →Hardware-enforced one-way data flow — OT → IT but not back — prevents pivot from monitoring
Jump server →Hardened intermediary for admin access — prod only accepts from jump server IP — session recording
ISOSandboxingEmail · Browser · Application

A sandbox is an isolated execution environment where untrusted code can run without affecting the host system or network. If malware executes in a sandbox: the sandbox is damaged, not the production environment.

Email Sandboxing

Detonate email attachments in isolated VM before delivery. Execute the attachment: does it call out to C2? Does it modify system files? Does it spawn suspicious processes? Safe: deliver. Malicious: quarantine. Catches zero-day malware that AV signatures miss. Microsoft Defender ATP, Proofpoint, Mimecast. Small delivery delay (~1-5 min) is the tradeoff.

Browser Isolation

Execute web page rendering in isolated container on remote server — only pixels streamed to user's browser. User clicks link → content renders in cloud sandbox → screenshot sent to user's screen. Malware in web page: executes in sandbox → cannot reach user's device. Remote Browser Isolation (RBI): Menlo Security, Zscaler. Eliminates browser-based drive-by downloads entirely.

Application Sandbox

Isolate applications from each other and from the OS. iOS/Android: each app runs in own sandbox — cannot read other apps' data. Windows: Windows Sandbox (disposable VM for testing). macOS: Gatekeeper + App Sandbox. Browser: each tab in separate process. If one tab is exploited: cannot access other tabs' cookies or passwords.

// Exam Keywords

Email sandbox →Detonate attachments in isolated VM before delivery — catches zero-day malware
Browser isolation (RBI) →Web content renders in cloud sandbox — only pixels sent to user — eliminates drive-by downloads
App sandbox →Each app isolated — cannot read other apps' data — iOS/Android default model
ISOQuarantine & Deception TechnologyIsolate Infected · Honeypot

Endpoint Quarantine

When EDR detects compromise: automatically isolate the endpoint from the network immediately. Quarantined device: cannot communicate with other internal systems — only to the security team's management infrastructure. Stops lateral movement in its tracks. Forensic investigation proceeds on isolated device. Automated quarantine: seconds from detection to isolation — manual would take minutes/hours. Network quarantine: NAC places non-compliant device on remediation VLAN.

Honeypot & Honeynet

Decoy systems placed in isolated network segments to attract and detect attackers. Honeypot: single decoy — any access = alert (zero false positives). Honeynet: network of honeypots. Honeytokens: fake credentials/data — use triggers alert. Honeyfiles: decoy files ("salaries.xlsx") — access triggers alert. DNS sinkhole: redirect malicious domain lookups to controlled server. Value: detect attackers who've already breached perimeter, study techniques, waste attacker time.

// Exam Keywords

EDR quarantine →Auto-isolate compromised endpoint — seconds to isolation — stops lateral movement
NAC quarantine →Non-compliant device → remediation VLAN — enforce posture before full network access
Honeypot →Decoy system — any access = zero-false-positive alert — legitimate users never touch it
Honeytoken →Fake credential/data — use triggers alert — fake AWS key, fake password in dump
ISOIsolation as a Compensating ControlLegacy Systems · EOL · Unpatched

When a system cannot be patched, upgraded, or replaced — isolation becomes the primary compensating control to reduce risk to an acceptable level.

// Scenario — Legacy Medical Device Isolation
# Hospital: patient monitoring device running Windows XP # Cannot be patched — FDA certification required for any change # Cannot be replaced — 18-month procurement process Isolation compensating controls applied: 1. Dedicated VLAN: device on isolated VLAN with no internet 2. Firewall ACL: only allows traffic to/from nursing station IP 3. IDS monitoring: any unusual traffic from device → immediate alert 4. No USB: USB ports physically blocked with port blockers 5. Physical access: server room lock — only biomedical engineering 6. Exception doc: formally documented in risk register, CISO-approved What this achieves: Cannot be reached from internet ✓ Cannot be used to pivot to corporate network ✓ Any attack on the device: detected immediately ✓ Blast radius: only the device itself ✓

PTCPatch Management ProcessIdentify · Test · Deploy · Verify

Patch management is the process of systematically identifying, testing, and applying updates to systems to close known vulnerabilities. Without it, every known CVE is an open door attackers can freely exploit.

# Patch management lifecycle: 1. Discovery: Inventory: know every asset (can't patch what you don't know about) Vulnerability scanner: Nessus, Qualys, OpenVAS — identify CVEs on assets Asset management: CMDB keeps inventory current Shadow IT risk: untracked assets never patched 2. Prioritization (see next tab for full detail): CVSS score + CISA KEV status + asset criticality + exploitability Critical/High on internet-facing: patch within 7-14 days CISA KEV: patch within 2 weeks (federal) or ASAP 3. Testing: Test environment that mirrors production Deploy patch → run functional tests → check compatibility Critical systems: extended testing period Emergency patches: abbreviated test — risk vs risk tradeoff 4. Deployment: Change management: RFC submitted, approved Change window: off-hours, maintenance window Staged rollout: pilot group → broader → all Rollback plan: documented before starting 5. Verification: Re-scan after patching: confirm vulnerability closed Patch not applied: patch didn't install, reboot needed Reboot may be required: kernel patches especially Documentation: CMDB updated, change record closed # Key metric: Mean Time to Patch (MTTP) # Measure per severity level — track trend over time

// Exam Keywords

Patch management lifecycle →Identify → prioritize → test → deploy → verify — ongoing cycle
Asset inventory →Cannot patch what you don't know exists — shadow IT = unpatched assets
Verify after patching →Re-scan to confirm vulnerability closed — patch not active without reboot for kernel patches
PTCPatch PrioritizationCVSS · CISA KEV · Risk-Based
# CVSS score alone is insufficient for prioritization: CVSS 9.8 on internal dev server with no internet exposure: → Lower priority than CVSS 7.2 on public-facing web app Prioritization factors: 1. Exploitability: is exploit code publicly available? 2. Exposure: internet-facing vs internal-only 3. CISA KEV: is it actively exploited in the wild? 4. Asset criticality: crown jewel vs test server 5. CVSS score: severity of the vulnerability itself CISA Known Exploited Vulnerabilities (KEV): Catalog of CVEs actively exploited in real attacks Federal agencies: must patch within 2-3 weeks Best practice: treat KEV as highest priority — patch NOW Reasoning: if attackers are actively using it against real targets every day you don't patch = window of active exploitation Practical SLA targets (industry standard): Critical + internet-facing + KEV: 48-72 hours Critical + internet-facing: 7 days Critical + internal: 14 days High + internet-facing: 14 days High + internal: 30 days Medium: 60-90 days Low: best effort / next major maintenance # Context-based prioritization > pure CVSS score

// Exam Keywords

CISA KEV →Known Exploited Vulnerabilities — actively exploited in wild — patch immediately / within 2 weeks
CVSS →Severity score 0-10 — important but NOT sufficient alone — context determines actual priority
Internet-facing critical →Patch within 7 days — reduces window of exploitation from internet
Risk-based patching →Exploitability + exposure + criticality + CVSS together = prioritization decision
PTCTesting & DeploymentStaged Rollout · Change Window · Rollback

Testing Requirements

  • Test environment mirrors production as closely as possible
  • Functional testing: does patch break existing functionality?
  • Regression testing: run full test suite after patch
  • Compatibility: check critical applications still work
  • Security validation: confirm CVE actually closed (re-scan)
  • Emergency patch: abbreviated testing — documented risk acceptance

Deployment Strategy

  • Staged rollout: 5% pilot → 25% → 50% → all (detect problems early)
  • Change window: off-hours, agreed with business stakeholders
  • Rollback plan: documented before starting — tested
  • Communication: notify users before downtime
  • Monitoring: heightened monitoring after deployment
  • Reboot: kernel patches inactive until reboot — schedule reboot

// Exam Keywords

Staged rollout →Pilot group first — detect problems before affecting everyone
Rollback plan →Required before patching — how to revert if patch breaks something
Reboot requirement →Kernel patches not active until reboot — vulnerability still open until system restarts
Change window →Off-hours maintenance window — agreed with stakeholders — sufficient duration
PTCEOL Systems & Virtual PatchingEnd-of-Life · WAF · IPS Rules
Ideal Solution
Upgrade or Replace EOL System
End-of-life OS/software: vendor no longer provides patches. Every future CVE has no fix available. WinXP, Win7, Server 2003, 2008 — still widely found. Best answer: upgrade to supported version. Second best: replace with modern equivalent. Both require: change management, testing, budget approval.
Compensating: Virtual Patching
WAF/IPS Rules Block the Exploit
Cannot patch the underlying vulnerability? Deploy WAF/IPS rule that blocks the specific exploit pattern at network level. Web app vuln: WAF rule blocks the malicious request. OS vuln: IPS rule blocks the exploit traffic. "Virtual patch" — doesn't fix the vulnerability but prevents exploitation. Temporary — use until real patch available. Document as compensating control.
# EOL system risk management approach: Option 1 (best): Upgrade/replace → real patch available Option 2 (acceptable short-term): Virtual patching WAF rule: blocks exploitation of web app CVE IPS rule: blocks network-based exploit Limitation: only blocks known exploit patterns zero-day variants may bypass Option 3 (compensating if can't patch or virtual patch): Network isolation → reduce exposure Enhanced monitoring → detect if exploited Physical security → reduce attack vectors Documented exception → formal risk acceptance Option 4 (never acceptable): "We'll get to it eventually" with no compensating controls This is how WannaCry spread via unpatched Windows XP/7 in 2017
