01 Secure Baselines Configuration Standard

A secure baseline is a documented minimum-security configuration that every system of a given type must meet. Instead of configuring each system differently, every Windows laptop, every Linux server, every network switch gets the same hardened starting point.

What Is a Baseline?

# Without baseline: Laptop 1: Admin configured it → password policy: 8 chars Laptop 2: Different admin → password policy: 6 chars Laptop 3: No policy set → default Windows settings Server 1: Telnet enabled (forgot to disable) Server 2: Guest account active, default password Result: inconsistent security across entire org ✗ # With secure baseline: ALL laptops deployed from SAME golden image: → Password: 12+ chars, complexity required → Telnet: disabled → Guest account: disabled → Windows Defender: enabled + configured → Firewall: enabled, default-deny inbound → BitLocker: enabled on all drives → USB: controlled by policy Result: every device starts with identical security posture ✓

Baseline Sources — Industry Standards

CIS Benchmarks

Center for Internet Security — most widely used free baseline standards. Covers Windows, Linux, macOS, browsers, cloud services, network devices. Two levels:

  • Level 1 = basic, minimal performance impact
  • Level 2 = comprehensive, may affect usability

NIST SP 800-70

National Checklist Program — US government baseline configurations. SCAP (Security Content Automation Protocol) checklists for automated compliance checking. Required for US federal agencies.

DISA STIGs

Defense Information Systems Agency — Security Technical Implementation Guides. Very strict — military/government grade. Covers OS, apps, network devices. Most detailed available.

Vendor Guides

Microsoft Security Baselines, AWS Security Hub, Azure Security Benchmark. Vendor-specific recommendations for their own products. Regularly updated as threats evolve.

What a Baseline Covers

🔒
Account Policies
Password length, complexity, history, lockout threshold (5 attempts), lockout duration, MFA requirement, disable guest and default accounts
🛡️
Network Settings
Disable unused ports/protocols, enable host firewall, disable Telnet/FTP, require SSH/HTTPS only, disable UPnP, configure NTP time sync
📋
Audit and Logging
Enable security event logging, logon/logoff events, privilege use, object access, policy changes. Log retention minimum 90 days.
⚙️
Software Settings
Auto-update enabled, remove unnecessary software, disable autorun, configure browser security, enable antivirus real-time protection
🔐
Encryption Settings
BitLocker/FileVault required, TLS 1.2+ only, disable SSL/TLS 1.0/1.1, require encrypted protocols (SSH over Telnet, HTTPS over HTTP)

Baseline Enforcement Methods

Group Policy (GPO)

Windows environments — push baseline settings to all domain machines from Active Directory. Enforced at login. User can't override GPO settings.

MDM / EMM

Mobile Device Management for phones and tablets. MDM enforces baseline: screen lock, encryption, approved apps only, remote wipe capability.

Configuration Management

Ansible, Puppet, Chef, Salt — automate baseline enforcement on servers. Any drift from baseline automatically corrected. Infrastructure as Code.

Configuration Drift

# Configuration drift = systems slowly deviate from baseline Day 1: Server deployed from golden image ✓ Week 2: Admin opens port 8080 for quick test — forgets to close Month 3: Developer installs Python 2.7 "temporarily" Month 6: Another admin enables Telnet "just for this one thing" Month 12: Audit shows 47 deviations from baseline ✗ # Continuous compliance monitoring detects drift: Tool scans all servers daily against baseline Server-42 alert: port 8080 open — not in baseline Server-17 alert: Telnet enabled — not in baseline Auto-remediation: closes port, disables Telnet, alerts admin ✓
// SCENARIO — Healthcare System Baseline Enforcement

A hospital's 500 Windows workstations are all deployed from the same CIS Level 1 hardened image via GPO. When a new vulnerability requires disabling SMBv1, IT pushes ONE GPO change and within an hour all 500 workstations are updated — no manual visits required.

# WannaCry vulnerability — SMBv1 must be disabled across all 500 PCs Without baseline/GPO: IT team visits each of 500 workstations Manual registry edit on each one Time: 500 × 10 minutes = 83 hours Risk: some missed, some done incorrectly With GPO baseline enforcement: Admin creates GPO: "Disable-SMBv1" Admin links GPO to All-Workstations OU Next login on each workstation: auto-applied Time: 2 hours (GPO creation + propagation) Coverage: 100% of domain machines ✓

02 Mobile Solutions MDM · MAM · BYOD

Mobile devices leave the controlled corporate environment, connect to unknown networks, and users have significant control over them. Organizations need policies and tools to manage and secure mobile devices without invading user privacy.

Deployment Models — Who Owns the Device?

BYOD — Bring Your Own Device

Employee owns the device. Uses personal phone/laptop for work. Most flexible for employees, most challenging for IT security.

  • Challenge: personal and work data mixed
  • Solution: containerization — work apps in encrypted container
  • Legal: careful about MDM on personal device
  • Policy: acceptable use agreement required

COPE — Corporate-Owned, Personally Enabled

Company owns the device. Employee can use for personal activities within policy. Most common enterprise approach.

  • Company controls the hardware
  • Can fully manage with MDM
  • Employee gets some personal use
  • Remote wipe: destroys ALL data including personal

CYOD — Choose Your Own Device

Employee picks from approved list of devices. Company buys it. Balance between user preference and IT standardization.

  • Limited approved device list
  • IT knows exactly what hardware to support
  • User has some choice

COBO — Corporate-Owned, Business Only

Company owns device, ONLY business use allowed. No personal apps, no personal use. Strictest policy — used for highly sensitive roles.

  • Complete corporate control
  • No personal use whatsoever
  • Common in financial, legal, government

MDM vs MAM vs MEM

MDM — Mobile Device Management: Controls the ENTIRE DEVICE Can: enforce PIN, encryption, wipe whole device Can: control installed apps, camera, GPS Can: push certificates and Wi-Fi configs Risk: on BYOD — employer sees personal data Examples: Microsoft Intune, Jamf, VMware Workspace ONE MAM — Mobile Application Management: Controls only WORK APPS — not entire device Work apps wrapped in secure container Personal apps completely untouched IT can wipe work apps without touching personal data Best for: BYOD — protects privacy Examples: Intune App Protection Policies MEM — Microsoft Endpoint Manager: Combines MDM + MAM + PC management Single console for ALL endpoints Phones, tablets, laptops, desktops Modern enterprise standard

MDM Capabilities — What It Can Do

→

→

→

→

Enforcement Policies

  • Require screen lock PIN/biometric
  • Enforce device encryption
  • Block rooted/jailbroken devices
  • Require minimum OS version
  • Control which apps can be installed
  • Block camera in sensitive areas
  • Enforce VPN for corporate access

Remote Actions

  • Remote wipe — erase entire device
  • Selective wipe — erase work data only
  • Remote lock — lock device immediately
  • Location tracking — find lost device
  • Push software updates remotely
  • Revoke certificates — cut off access
  • Force compliance check

Mobile Specific Threats

Jailbreaking / Rooting

Removes manufacturer security restrictions. Bypasses MDM controls. Disables secure boot and integrity checks. MDM should detect and block access from jailbroken devices.

Sideloading

Installing apps outside official app store. Bypasses Apple/Google vetting process. Apps may contain malware. MDM policy should prohibit sideloading on corporate devices.

SIM Swapping

Attacker convinces carrier to transfer phone number. Bypasses SMS-based MFA completely. All OTP codes go to attacker. Defense: use authenticator app not SMS for MFA.

03 Hardening Reduce Attack Surface

Hardening is the process of securing a system by reducing its attack surface — removing unnecessary components, disabling unneeded services, and configuring security settings. Out-of-the-box systems are designed for convenience, not security.

Default State Problem

# Fresh Windows Server — what's running by default: Services running: 47 services (many unnecessary) Open ports: 135, 139, 445, 3389, and more Admin shares: C$, D$, ADMIN$ — accessible Guest account: exists (may be enabled) SMBv1: may be enabled (WannaCry vulnerable) NTLMv1: may be allowed (weak authentication) # After hardening: Services running: 12 services (only what's needed) Open ports: 443, 22 only (HTTPS + SSH) Admin shares: disabled Guest account: disabled and renamed SMBv1: disabled NTLMv1: blocked — NTLMv2+ only

Hardening by System Type

Operating System Hardening

🚫
Remove Unnecessary Software
Uninstall all software not required for the system's role. Every application = potential vulnerability. Web server doesn't need Office, games, media players.
🔌
Disable Unused Services
Stop and disable services not needed. Telnet, FTP, NetBIOS, Print Spooler on servers that don't print. Each running service = open attack vector.
🔐
Change Default Credentials
Rename/disable Administrator account. Change ALL default passwords. Attackers know every default credential for every major system. This is basic and critical.
📦
Apply All Patches
Update OS, drivers, firmware. Enable automatic updates where possible. Unpatched known vulnerabilities = easiest attack. No excuse for missing critical patches.
🔢
Least Privilege Accounts
Users run as standard user, not admin. Only admin when admin is required. Admin account ≠ daily use account. Separate admin accounts for admin tasks.

Network Device Hardening

Switch Hardening

  • Disable unused switch ports
  • Enable port security (MAC filtering)
  • Disable CDP/LLDP on untrusted ports
  • Enable BPDU guard (prevent rogue switches)
  • Disable DTP (prevent VLAN hopping)
  • Change native VLAN from VLAN 1
  • Use SSH not Telnet for management
  • Enable 802.1X authentication

Router / Firewall Hardening

  • Change default admin credentials
  • Disable remote management on WAN
  • Disable UPnP — it opens ports automatically
  • Enable logging to syslog server
  • Apply ACLs — block RFC1918 from internet
  • Disable unused protocols (SNMP v1/v2)
  • Enable SNMPv3 only if needed
  • Regular firmware updates

Application Hardening

Web Server

  • Remove default pages (test.html, info.php)
  • Disable directory listing
  • Remove server version banner
  • Restrict HTTP methods (GET/POST only)
  • Set security headers (CSP, HSTS, X-Frame)

Database

  • Remove sample databases (Northwind)
  • Disable xp_cmdshell (SQL Server)
  • Rename or disable sa account
  • Restrict DB to local connections only
  • Principle of least privilege on accounts

Browser Hardening

  • Disable Java plugin
  • Block third-party cookies
  • Enable built-in malware protection
  • HTTPS-only mode
  • Block dangerous file types
// SCENARIO — Cisco Router Default vs Hardened
# Default Cisco router — out of box: Console password: none Enable password: none VTY lines: allow all IPs, Telnet enabled SNMP: community "public" — read access for all HTTP server: enabled on port 80 CDP: broadcasting device info to anyone Unused interfaces: up/up — active and exploitable # After hardening: Console password: complex password + timeout 5min Enable password: enable secret (MD5 hash) VTY lines: SSH only, access-list mgmt-IPs only SNMP: SNMPv3 with auth+encryption, or disabled HTTP server: disabled — use SSH for management CDP: disabled on external interfaces Unused interfaces: shutdown command applied

// EXAM KEYWORDS

Hardening →Reduce attack surface — remove unnecessary, disable unused, change defaults
Default credentials →Change immediately — attackers know every default password
Disable unused services →Every running service = attack vector — only run what's needed
Disable unused ports →Switch ports — shut down if not connected
Remove unnecessary software →Every app = potential vulnerability — minimize installed software
BPDU Guard →Prevents rogue switches from being plugged in — switch hardening
xp_cmdshell →SQL Server feature allowing OS commands — disable in hardening
04 Wireless Security WPA3 · 802.1X · EAP

Wireless networks extend the network perimeter into thin air — anyone within range can attempt to connect. Encryption and authentication are critical to prevent eavesdropping and unauthorized access.

Wireless Protocols — Evolution of Security

WEP
❌ Completely Broken — Never Use
RC4 encryption with static keys. Cracked in minutes with freely available tools. IV reuse vulnerability. From 1997 — completely obsolete.
AVOID
WPA
⚠️ Deprecated — Insecure
TKIP encryption — temporary fix for WEP. TKIP also found vulnerable. Only used as WPA/WPA2 transition. Should not be used today.
DEPRECATED
WPA2
✅ Acceptable — Still Widely Used
AES-CCMP encryption — strong. KRACK vulnerability discovered 2017 but patched. Still widely deployed. Personal (PSK) and Enterprise (802.1X) modes.
ACCEPTABLE
WPA3
✅ Best — Current Standard
SAE (Simultaneous Authentication of Equals) handshake replaces PSK. GCMP-256 encryption. Forward secrecy — past traffic safe if key captured. 192-bit Enterprise mode. Protects open networks (Enhanced Open).
RECOMMENDED

Personal vs Enterprise Mode

WPA2/WPA3 Personal (PSK — Pre-Shared Key): Everyone uses the SAME password Simple — good for home/small office Risk: if key compromised → EVERYONE re-keys Insider can capture all traffic WPA2/WPA3 Enterprise (802.1X): Each user has OWN unique credentials Requires RADIUS authentication server User logs in with username + password (PEAP) OR with certificate (EAP-TLS) Benefits: → One user compromised → revoke that user only → Others unaffected → Per-user audit trail → Dynamic encryption keys per session → Supports MFA integration Enterprise mode = required for organizations Personal mode = home/small business only

Wireless Attacks — Must Know

Evil Twin Attack

Attacker creates a fake AP with same SSID as legitimate network. Victims connect thinking it's real. Attacker performs MitM — sees all traffic. Even WPA2 users can be victims if they connect to the fake AP.

Legitimate: "CafeWifi" — real AP Evil Twin: "CafeWifi" — fake AP Victim: connects to stronger signal (fake) Attacker: reads all victim's traffic Defense: VPN, verify AP certificates

Deauth Attack

Sends spoofed deauthentication frames forcing clients to disconnect. When clients reconnect, attacker captures the WPA2 handshake. Then brute-forces the handshake offline to crack the password.

Step 1: Send deauth → client disconnects Step 2: Client reconnects → 4-way handshake Step 3: Capture handshake Step 4: Offline brute force attack Defense: WPA3 SAE makes this much harder

Rogue Access Point

Unauthorized AP connected to corporate network. Employee plugs in personal router for better coverage. Creates uncontrolled wireless entry point. No 802.1X, no monitoring, no corporate security controls.

KRACK Attack

Key Reinstallation Attack — WPA2 vulnerability. Manipulates WPA2 handshake to reinstall already-used encryption keys. Allows decryption of packets. Patched by OS updates — important to keep devices updated.

Wireless Security Best Practices

📡
Use WPA3 (or WPA2-Enterprise minimum)
WPA2 Personal acceptable for home. Organizations must use Enterprise mode with 802.1X and RADIUS. WPA3 for all new deployments.
🔒
Network Segmentation — Separate Guest Wi-Fi
Guest VLAN with no access to corporate network. IoT VLAN isolated from everything. Corporate VLAN with 802.1X. Never mix guests and corporate on same network.
📍
Wireless Intrusion Detection (WIDS)
Monitors Wi-Fi spectrum for rogue APs and evil twins. Alerts on deauth attacks and unauthorized devices. Many enterprise APs have WIDS built in.
📶
Reduce Signal Strength
Adjust AP transmit power to cover only required area. No need for signal to reach parking lot or street. Smaller coverage area = smaller attack surface.
// SCENARIO — Coffee Shop Evil Twin Attack
# Employee working at coffee shop: Legitimate Wi-Fi: "CoffeeShop_Free" (-70 dBm) Attacker's AP: "CoffeeShop_Free" (-65 dBm) ← stronger signal Employee's laptop connects to ATTACKER's AP (stronger signal chosen automatically) Without VPN: Employee accesses corporate email via HTTPS HTTPS encrypted but: DNS queries visible Attacker does SSL stripping on non-HSTS sites Employee credentials for unprotected site: CAPTURED ✗ With Always-On VPN: Employee connects → VPN activates before any traffic All traffic tunneled through corporate gateway Evil twin sees: encrypted VPN tunnel only ✓ Attacker gets: nothing useful ✓

// EXAM KEYWORDS

WEP →Completely broken — never recommend — RC4 weakness
WPA2 →AES-CCMP — acceptable — KRACK patched — Personal vs Enterprise
WPA3 →SAE handshake — best current — forward secrecy — Enhanced Open
Enterprise mode →802.1X + RADIUS — per-user credentials — organizations must use this
Evil twin →Fake AP same SSID — MitM — defense: VPN, certificate verification
Deauth attack →Forces reconnect to capture WPA2 handshake for offline cracking

05 Application Security SDLC · SAST · DAST

Application security means building security into applications from the start — not adding it as an afterthought. Most breaches exploit application vulnerabilities. Secure development practices, code review, and testing catch vulnerabilities before attackers do.

Secure SDLC — Security at Every Phase

1. Requirements Phase
Define security requirements upfront. "System must use TLS 1.2+." "Passwords must be hashed with bcrypt." Security requirements alongside functional requirements.
Design
2. Design Phase
Threat modeling — identify what could go wrong. STRIDE (Spoofing, Tampering, Repudiation, Info Disclosure, DoS, Elevation). Architecture review. Data flow diagrams.
STRIDE
3. Development Phase
Secure coding standards. Code reviews. Input validation. No hardcoded credentials. SAST (Static Analysis) runs automatically in IDE/CI pipeline. Pair programming for sensitive code.
SAST
4. Testing Phase
DAST (Dynamic Analysis) — test running application. Penetration testing. Fuzzing — send unexpected inputs. IAST — real-time testing during QA. Vulnerability scanning.
DAST
5. Deployment Phase
Secure configuration in production. Secrets management — no credentials in code. WAF deployment. Security scanning of container images before deploy.
Deploy
6. Maintenance Phase
Patch management. Vulnerability monitoring. Dependency scanning — check third-party libraries. Bug bounty program. Incident response for discovered vulnerabilities.
Monitor

Testing Methods — SAST vs DAST vs IAST

SAST — Static Application Security Testing: Tests: SOURCE CODE (not running) When: during development, in CI/CD pipeline Finds: SQL injection in code, hardcoded passwords, buffer overflow potential, insecure functions Tools: SonarQube, Checkmarx, Veracode, Semgrep Analogy: reading a recipe for mistakes before cooking Advantage: finds issues early — cheap to fix Limitation: false positives, can't find runtime issues DAST — Dynamic Application Security Testing: Tests: RUNNING APPLICATION When: testing phase, after deployment Finds: XSS, SQLi in real app, auth bypasses, misconfigurations, runtime vulnerabilities Tools: OWASP ZAP, Burp Suite, Nikto Analogy: actually cooking the recipe and tasting it Advantage: tests real behavior, finds runtime issues Limitation: late in process — more expensive to fix IAST — Interactive Application Security Testing: Tests: INSIDE running application (agent-based) When: during QA testing — hybrid approach Finds: combines SAST accuracy + DAST realism Tools: Contrast Security, Seeker Advantage: most accurate — low false positives Limitation: requires instrumented environment

Key Application Security Controls

Input Validation

Validate ALL input before processing. Never trust user input. Whitelist allowed characters. Reject or sanitize everything else. Primary defense against injection attacks.

# Bad — no validation: query = "SELECT * FROM users WHERE id=" + userId # userId = "1 OR 1=1" → dumps all users # Good — parameterized query: query = "SELECT * FROM users WHERE id=?" execute(query, [userId]) # Input treated as data, not code

Secure Coding Practices

  • Never hardcode credentials in source code
  • Use parameterized queries — prevent SQLi
  • Output encoding — prevent XSS
  • Error handling — never expose stack traces
  • Secrets in vault — HashiCorp Vault, AWS Secrets Manager
  • Dependency scanning — check for vulnerable libraries
  • HTTPS everywhere — never HTTP

Code Signing

Digitally sign application code. Users/OS verify signature before running. Proves code came from legitimate developer and hasn't been modified. Required for: iOS apps, Windows drivers, Android apps.

Fuzzing

Send massive amounts of random, unexpected, or malformed input to application. Looking for crashes, unexpected behavior, memory leaks. Finds vulnerabilities that manual testing misses. AFL, libFuzzer, Boofuzz.

// SCENARIO — Dev Pipeline Security Gates
# Secure CI/CD pipeline — security gates at each stage: Developer commits code to Git: Pre-commit hook: secrets scanner runs → Detects AWS key in code: BLOCKED ✗ → Developer removes key, uses vault instead ✓ Code pushed to pipeline: SAST scan (SonarQube): → SQL injection found in login.js line 47: BLOCKED ✗ → Dev fixes to parameterized query ✓ Code passes SAST → deploy to test env: DAST scan (OWASP ZAP): → XSS vulnerability in search field: BLOCKED ✗ → Dev adds output encoding ✓ All gates pass → deploy to staging: Dependency check: → lodash 4.17.4 has known CVE: WARNING ⚠ → Updated to 4.17.21 ✓ All gates passed → deploy to production ✓ # Vulnerabilities caught before production = $0 to fix # Vulnerabilities in production = $500k breach average

// EXAM KEYWORDS

SAST →Tests source code — not running — during development — early
DAST →Tests running application — after deployment — finds runtime issues
IAST →Agent inside running app — most accurate — during QA
Fuzzing →Random/malformed input to find crashes and vulnerabilities
Input validation →Whitelist allowed input — primary defense against injection
STRIDE →Threat modeling framework — Spoofing, Tampering, Repudiation, Info Disclosure, DoS, Elevation
Code signing →Proves code came from legitimate developer — prevents tampering
Secure SDLC →Security at EVERY phase — not just testing at end
06 Sandboxing Isolation · Containment

A sandbox is an isolated, controlled environment where code or files can run without affecting the real system. Think of it as a safe testing zone with walls — malware runs inside, can't escape outside.

How Sandboxing Works

# Without sandbox: Email arrives with suspicious attachment User opens attachment Malware executes on REAL system Accesses real files, real network Steals credentials, encrypts files → BREACH ✗ # With sandbox: Email arrives with suspicious attachment Attachment automatically sent to SANDBOX first Sandbox: isolated virtual environment Malware executes in sandbox Sandbox monitors: file changes, network calls, registry modifications, process spawns Malware tries: connect to C2 server Sandbox logs: outbound connection to evil-c2.ru:4444 Sandbox verdict: MALICIOUS — attachment blocked Real system: never touched ✓ # What sandbox records: Files created: C:\Windows\System32\malware.exe Registry keys: HKLM\Software\Run\malware.exe Network calls: GET evil-c2.ru/payload.bin Processes: cmd.exe spawned from winword.exe → Full behavioral analysis report generated

Types of Sandboxes

Email Sandbox

All email attachments automatically detonated in sandbox before delivery to user. Suspicious behavior = attachment quarantined. Clean = delivered normally. Built into Microsoft Defender for Office 365, Proofpoint, Mimecast.

Email arrives → attachment extracted → detonated in sandbox VM → behavior monitored 3-5 minutes → clean: delivered to user ✓ → malicious: quarantined, alert sent ✗

Browser Sandbox

Every browser tab runs in its own isolated sandbox. JavaScript from one tab can't access another tab's data. Malicious website can't escape browser and access OS. Chrome, Firefox, Edge all sandbox tabs.

NGFW Sandbox

Next-gen firewall detonates suspicious files in cloud sandbox before allowing into network. Unknown file downloaded? Execute in cloud first. Zero-day malware detected before reaching endpoints. Palo Alto WildFire, Fortinet FortiSandbox.

Development Sandbox

Isolated development environment — test code changes without affecting production. Break things safely. Separated from real data. Common in DevOps — dev → staging → production pipeline.

Mobile App Sandbox

iOS/Android sandbox each app completely. App can only access its own data folder. Can't read other apps' data. Must request permissions explicitly. Limits damage from malicious apps.

App A: /data/app_a/ ← only access this App B: /data/app_b/ ← App A cannot see System: requires explicit permission Camera: must request → user approves

Container Sandbox

Docker containers sandbox applications. Each container isolated from others. Limited system call access. Can't access other containers' data. Kubernetes network policies add network isolation.

Sandbox Evasion Techniques

# Sophisticated malware tries to detect sandbox: Time-based evasion: Malware sleeps for 10 minutes before activating Many sandboxes only run for 2-3 minutes Malware appears benign → passes → activates later Defense: longer sandbox analysis time User interaction check: Malware checks: has mouse moved? Any keystrokes? Sandbox: no real user → no movement Malware: "I'm in a sandbox — stay dormant" Defense: simulate user interactions in sandbox Environment fingerprinting: Check: CPU count (VMs often have fewer) Check: RAM amount (VMs often have less) Check: specific sandbox artifacts (VMware registry keys) Defense: hardware-based sandboxes, disguise VM artifacts Network connectivity check: Real system has internet → sandbox may not Malware checks for internet → no internet = sandbox Defense: allow sandbox internet access (monitored)
// SCENARIO — Zero-Day in Email Attachment
# Targeted attack on finance department: Attacker sends: "Invoice_Q4_2024.xlsx" Subject: "Urgent: Outstanding payment review" Step 1 — Email gateway receives attachment: Antivirus scan: CLEAN (zero-day — no signature) Sandbox: automatically detonating attachment... Step 2 — Sandbox execution (3 minutes): Excel opens → runs macro Macro: downloads payload from pastebin.com Payload: executes in memory (fileless) Payload: connects to 185.220.101.45:443 Payload: runs mimikatz → credential harvesting All this recorded by sandbox Step 3 — Sandbox verdict: MALICIOUS Email: quarantined before delivery ✓ Finance staff: never saw the email ✓ Alert: security team notified with full IOCs Step 4 — Threat intelligence shared: 185.220.101.45 added to block list Macro detection rule created All future emails with this payload: auto-blocked ✓

// EXAM KEYWORDS

Sandbox →Isolated environment — code runs safely without affecting real system
Email sandbox →Attachments detonated before delivery — zero-day detection
NGFW sandbox →WildFire (Palo Alto) — cloud sandbox for unknown files
Browser sandbox →Each tab isolated — malicious site can't escape browser

07 Monitoring SIEM · Logs · Alerts

You can't protect what you can't see. Monitoring is the continuous collection, analysis, and alerting on system, network, and security events — to detect threats, investigate incidents, and prove compliance.

What to Monitor — Log Sources

System Logs

  • Authentication events — logon/logoff/failed
  • Account management — created/deleted/modified
  • Privilege use — admin actions
  • Policy changes — GPO, firewall rules
  • System events — startup/shutdown/crashes
  • Object access — file/folder/registry access

Network Logs

  • Firewall allow/deny logs
  • IDS/IPS alerts
  • NetFlow data — traffic patterns
  • DNS queries — detect C2 via DNS
  • DHCP logs — track IP assignments
  • VPN connection logs

Application Logs

  • Web server access logs (Apache/IIS/Nginx)
  • Database query logs
  • Application error logs
  • Authentication logs (failed logins)
  • API request/response logs
  • Email gateway logs

Security Tool Logs

  • Antivirus/EDR detections
  • DLP policy violations
  • WAF alerts
  • Vulnerability scanner results
  • MDM compliance status
  • Certificate events

Key Monitoring Tools

🔍
SIEM — Security Information and Event Management
Central platform that collects logs from everything and correlates them to find patterns no single tool could see. Real-time alerting, historical analysis, compliance reporting. Examples: Splunk, Microsoft Sentinel, IBM QRadar, Elastic SIEM. The heart of a SOC (Security Operations Center).
# SIEM correlation example: Event 1 (AD): user "john" failed login ×5 at 02:15 Event 2 (VPN): john connected from Russia at 02:16 Event 3 (CASB): john downloaded 10,000 files at 02:17 Event 4 (DLP): john uploaded to personal drive at 02:18 Individual: each event looks minor SIEM: correlates all 4 → HIGH PRIORITY ALERT "Possible account compromise + data exfil"
🖥️
EDR — Endpoint Detection and Response
Agent installed on every endpoint monitoring behavior in real time. Records: process creation, network connections, file changes, registry modifications. Detects malware by behavior not just signature. Can automatically contain compromised endpoints. Examples: CrowdStrike Falcon, Microsoft Defender for Endpoint, Carbon Black.
🌐
NDR — Network Detection and Response
Monitors network traffic for anomalies and threats. Uses ML to establish baseline, detect deviations. Catches lateral movement, C2 traffic, data exfiltration. Works even for encrypted traffic (metadata analysis). Examples: Darktrace, ExtraHop, Vectra.
🔭
XDR — Extended Detection and Response
Unifies EDR + NDR + cloud + email into one platform. Single pane of glass for all threats. Automated correlation and response across all vectors. Next evolution beyond SIEM. Examples: Palo Alto Cortex XDR, Microsoft Defender XDR.
📊
UEBA — User and Entity Behavior Analytics
ML-based anomaly detection. Builds baseline of "normal" for each user and entity. Alerts when behavior deviates: user downloading 1000x more than normal, login from impossible location, access at unusual hours. Detects insider threats that signature tools miss.
Normal user baseline: Login: weekdays 8am-6pm, New York Downloads: ~50MB/day Accesses: Finance shared drive UEBA alert triggers when: Login: 3am Sunday from Beijing ← impossible travel Downloads: 47GB in 2 hours ← 940x normal Action: accessing HR, Legal, Finance ← unusual

Log Management Best Practices

Centralized Logging

All logs sent to central syslog/SIEM. Logs on local systems can be deleted by attacker. Central logs harder to tamper with. Use syslog protocol (UDP 514 / TCP 6514 for TLS).

Log Retention

HIPAA: 6 years minimum. PCI-DSS: 1 year (3 months online). GDPR: as long as necessary. Internal: 90-day minimum recommended. Forensic investigations often go back 6-12 months.

Time Synchronization

ALL systems must use NTP (Network Time Protocol) from same source. Logs from different systems must have synchronized timestamps. Without time sync: correlating events from multiple sources is impossible.
