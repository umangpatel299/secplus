APPMemory & Injection VulnerabilitiesBuffer Overflow · UAF · Injection

Memory vulnerabilities occur when applications fail to properly manage memory allocation and deallocation, allowing attackers to overwrite memory regions and redirect code execution. These are among the most severe vulnerability types — many lead directly to remote code execution (RCE).

Buffer Overflow
Writing more data into a buffer than it can hold — overflow overwrites adjacent memory including return addresses and function pointers. Stack overflow: overwrites return address → redirect execution to attacker's shellcode. Heap overflow: corrupts heap metadata or adjacent objects. Classic in C/C++. Defense: memory-safe languages, ASLR, DEP/NX bit, stack canaries, compiler protections.
RCE possibleC/C++ commonASLR+DEP mitigate
Use After Free (UAF)
Using a memory pointer after the memory it points to has been freed and potentially reallocated to something else. Attacker controls what fills the freed memory → controls what happens when pointer is used. Modern exploit technique — bypasses some stack overflow mitigations. Defense: memory-safe languages, tools like AddressSanitizer, careful pointer management.
RCE possibleBrowser exploits common
NULL Pointer Dereference
Application attempts to use a null pointer — typically causes crash (denial of service). In some contexts can be escalated to privilege escalation. Common in kernel-mode code. Defense: null pointer checks before dereference, exception handling, input validation.
DoSPrivilege escalation sometimes
Integer Overflow / Underflow
Arithmetic result exceeds maximum value of integer type → wraps around to small (or negative) value. Security impact: size calculation underflows → too-small buffer allocated → buffer overflow follows. Example: malloc(user_input + 8) where user_input = MAX_INT wraps to 7 → tiny allocation → overflow. Defense: integer bounds checking, safe integer libraries.
Leads to buffer overflowType confusion

Memory Protection Mechanisms

# Defenses against memory corruption attacks: ASLR (Address Space Layout Randomization): Randomizes: memory addresses of stack, heap, libraries Attacker: can't hardcode addresses in shellcode Bypass: information leak vulnerability reveals addresses DEP / NX bit (Data Execution Prevention / No-Execute): Marks memory regions as: executable OR writable — not both Prevents: shellcode in data/stack from executing Bypass: Return-Oriented Programming (ROP) — reuse existing code Stack Canaries: Place a random value (canary) between local vars and return address Check canary before returning: if changed → buffer overflow → abort Bypass: heap overflow, information leak to read canary value CFI (Control Flow Integrity): Validates: indirect branches only go to valid targets Prevents: redirecting execution to arbitrary locations

APPInput Validation VulnerabilitiesSQLi · Command Injection · Path Traversal
SQL Injection (SQLi)
Attacker inserts SQL commands into input that the application uses to build database queries. Classic: username field → enters "' OR '1'='1 --" → bypasses authentication. Impact: dump entire database, modify/delete records, in some cases achieve RCE via xp_cmdshell. Defense: parameterized queries (prepared statements — #1 fix), stored procedures, input validation, WAF, least privilege on DB accounts.
CriticalData theftParameterized queries fix
# Vulnerable query (string concatenation): query = "SELECT * FROM users WHERE name = '" + username + "'" input: ' OR '1'='1' -- query: SELECT * FROM users WHERE name = '' OR '1'='1' --' result: returns ALL users → auth bypass # Fixed with parameterized query: query = "SELECT * FROM users WHERE name = ?" params = [username] # User input treated as data, never as SQL code
Command Injection / OS Injection
Application passes user input to an OS shell command without proper sanitization. Application: runs "ping " + user_input as shell command. Attacker enters: "8.8.8.8; cat /etc/passwd" → runs both ping AND reveals password file. Semicolon (;), pipe (|), backtick (`) all used to chain commands. Defense: avoid shell commands when possible, use APIs instead, sanitize all input, escape shell metacharacters.
RCEHigh severity
Path Traversal / Directory Traversal
Attacker uses "../" sequences in file path to escape the intended directory. Application: serves files from /var/www/uploads/ + filename. Attacker: filename = "../../etc/passwd" → /var/www/uploads/../../etc/passwd → /etc/passwd. Can read any file the application has permission to access. On Windows: ..\ or ..\\ — same concept. Defense: normalize paths, validate against allowed base directory, use safe file path APIs.
File readLFI variant
XML External Entity (XXE)
XML parser processes external entity references in user-supplied XML. Attacker crafts XML with external entity pointing to sensitive file or internal service. Can: read local files (/etc/passwd), SSRF (server-side request forgery), in some cases RCE. Defense: disable XML external entity processing in parser configuration (DOCTYPE processing disabled by default in modern parsers).
File readSSRF possible

APPAccess Control & Auth VulnerabilitiesIDOR · Broken Auth · CSRF
IDOR — Insecure Direct Object Reference
User can access another user's resources by guessing or modifying an object identifier. GET /account?id=1234 → change to 1235 → see another user's account. Missing server-side authorization check. Impact: access any user's data, modify any record, delete any object. Defense: server-side authorization check on EVERY request — "does this authenticated user own object 1235?" — enforce at the API layer.
Horizontal privilege escalationData exposure
Broken Authentication
Weaknesses in authentication implementation: session tokens predictable or improperly managed, missing account lockout (brute force possible), passwords stored in plaintext or weak hash, no MFA, session ID in URL (exposed in logs), session not invalidated on logout. Defense: strong session tokens (CSPRNG), secure cookie attributes (HttpOnly, Secure, SameSite), MFA, account lockout, proper logout handling.
Account takeoverSession hijacking
CSRF — Cross-Site Request Forgery
Tricks authenticated user's browser into making unauthorized requests to a site where they're logged in. Victim visits malicious page → page contains hidden form that auto-submits to bank → bank receives request with victim's session cookie → transfers money. Browser automatically sends cookies. Defense: CSRF tokens (unique per-form random token verified server-side), SameSite cookie attribute (blocks cross-site requests), check Referer header.
Action on behalf of userCSRF token defends

APPCryptographic & Config VulnerabilitiesWeak Crypto · Hardcoded Secrets · Misconfig
Hardcoded Credentials / Secrets
Passwords, API keys, or cryptographic secrets embedded directly in source code. Developer convenience → serious security vulnerability. Common in: IoT firmware, mobile apps, backend services. Anyone with access to code (or binary) → extracts credentials. Special risk: code pushed to public GitHub → credentials exposed immediately. Defense: secrets management (Vault, AWS Secrets Manager), environment variables, secret scanning in CI/CD pipeline (git-secrets, GitHub Advanced Security).
Credential exposureAffects all deployments
Improper Certificate Validation
Application accepts invalid, expired, or self-signed certificates without error — or does not verify hostname matches certificate. Mobile apps often vulnerable: accept any certificate → all traffic vulnerable to MITM with any certificate. Defense: strict certificate validation, certificate pinning (for high-security apps), reject invalid certs, test SSL validation in security review.
MITM possibleMobile apps especially
Security Misconfiguration
Application or server deployed with insecure default settings: debug mode enabled in production, default credentials unchanged, sample applications left installed, verbose error messages revealing internal details, unnecessary features/services enabled, overly permissive CORS policy. Defense: security hardening checklists, CIS benchmarks, regular configuration audits, infrastructure as code with security scanning (tfsec, Checkov).
Very commonOWASP Top 10 #5

APPRace Conditions & Logic FlawsTOCTOU · Business Logic
Race Condition / TOCTOU
Time-of-Check to Time-of-Use: gap between validating a condition and acting on it is exploited. Check: "does user have $100?" (yes). Delay. Use: debit $100 from account. Attacker exploits gap: sends multiple simultaneous requests → balance checked as positive for all → all debit → overdraft. Classic: ATM double-withdrawal attack. Defense: atomic operations, database transactions, locks/mutexes, synchronization primitives.
Concurrency bugFinancial systems high risk
Business Logic Vulnerabilities
Flaws in the application's business rules that allow unintended actions. Examples: apply coupon code multiple times, purchase item for negative price (reverse sign), skip payment step in checkout flow, access admin functions from non-admin path. These are logic errors, not technical memory bugs — automated scanners often miss them. Require manual testing and understanding of intended business flow. Defense: thorough design review, manual security testing, abuse case development.
Hard to automate detectFinancial impact

HWHardware VulnerabilitiesFirmware · Chips · Physical · Side-Channel

Hardware vulnerabilities exist in the physical components of computing systems — processors, firmware, memory chips, and peripherals. They are often the hardest to patch, the most persistent after exploitation, and can undermine all software-level security controls.

Firmware Vulnerabilities

BIOS / UEFI Vulnerabilities
Firmware that initializes hardware before OS loads. Vulnerabilities allow: persistent malware that survives OS reinstall ("firmware implant"), disable Secure Boot to load untrusted OS, modify boot sequence. BootHole (2020): GRUB2 UEFI Secure Boot bypass. Black Lotus (2022): UEFI bootkit — survived OS wipe. Defense: UEFI Secure Boot, Measured Boot (TPM records boot chain), firmware updates, physical access controls.
Survives OS reinstallVery persistent
End-of-Life Firmware / No Patches
Devices whose firmware vendor no longer updates — routers, IoT, industrial controllers. Known vulnerabilities exist with no available fix. Defense: replace EOL devices, if impossible: network isolation + virtual patching + enhanced monitoring as compensating controls.
No patch availableIoT / industrial

Processor / CPU Vulnerabilities

Spectre & Meltdown (Side-Channel)
Spectre (2018): exploits speculative execution to read memory of other processes — affects nearly all modern CPUs. Meltdown (2018): breaks isolation between user applications and OS kernel — reads kernel memory from user space. Both use timing-based side-channel analysis. Impact: a process can read memory it shouldn't have access to — including passwords, encryption keys, other users' data. Defense: CPU microcode patches, OS kernel patches, browser mitigations, hardware isolation (separate physical hardware for highest sensitivity).
Cross-VM memory leakAll modern CPUs affectedVirtualization risk
Rowhammer
Repeatedly accessing (hammering) rows in DRAM causes bit flips in adjacent rows — changing a 0 to a 1 or vice versa in memory the attacker shouldn't control. Used to escalate privileges by flipping a permission bit. Physical hardware flaw in DRAM — not patchable via software alone. Defense: ECC (Error-Correcting Code) memory, memory refresh rate increases, DDR4 improvements.
DRAM flawECC memory mitigates

Physical & Peripheral Vulnerabilities

JTAG / Debug Interfaces

Hardware debug port — used during development to inspect and modify device internals. Often left enabled on production firmware. Attacker with physical access: connect to JTAG → read firmware, extract encryption keys, modify code. Common in: IoT devices, embedded systems, networking equipment. Defense: disable JTAG in production firmware, physical tamper resistance.

Evil Maid Attack

Attacker with brief physical access to unattended device installs: hardware keylogger, malicious bootloader, hardware implant, modified firmware. Bypasses full-disk encryption if attacker can intercept boot before OS loads or modify BIOS. Defense: detect tamper via TPM measured boot, BIOS/UEFI passwords, physical security controls, never leave devices unattended in untrusted environments.

USB Attacks — BadUSB / Rubber Ducky

Malicious USB device impersonates a keyboard (HID — Human Interface Device). Computer trusts keyboard → executes commands automatically at typing speed. BadUSB: reflash USB firmware to act as keyboard. USB Rubber Ducky: purpose-built attack tool. Malicious USB charger: steal data or deliver payload. Defense: USB port controls (disable unauthorized USB), endpoint policies, physical USB blockers for high-security environments.

Cold Boot Attack

RAM retains data for seconds to minutes after power off (remanence effect). Attacker: quickly power off, cool RAM (spray), transplant to attacker machine, read contents. Can extract: full disk encryption keys, passwords, session tokens from RAM. Defense: enable memory encryption features in modern CPUs (AMD SME, Intel TME), enable memory scrubbing on shutdown, physical security.

MOBMobile Device VulnerabilitiesSideloading · Jailbreak · MDM Bypass

Mobile devices carry sensitive data, access corporate systems, and are always with the user — including outside controlled environments. Mobile-specific vulnerabilities include the unique risks from app stores, device modification, and the intersection of personal and professional use.

Mobile Vulnerability Types

Jailbreaking (iOS) / Rooting (Android)
Removing manufacturer/OS restrictions to gain full root/administrator access. Bypasses: app store restrictions, OS sandboxing, security policies, MDM controls. Allows: install unauthorized apps, access any system file, disable security features. Security risk: jailbroken/rooted device bypasses MDM enrollment controls, disables iOS app sandboxing, exposes root-level attack surface. Defense: MDM with jailbreak/root detection — deny access to corporate resources from jailbroken devices.
Bypasses security controlsMDM bypass
Sideloading — Unofficial App Installation
Installing apps from outside the official app store (Play Store / App Store). Android: enabled by default or via settings. iOS: normally restricted (jailbreak required or enterprise certificates). Risk: sideloaded apps not reviewed by Apple/Google — may contain malware. Enterprise certificate abuse: attackers use enterprise developer certificates to distribute malicious apps to iOS devices without jailbreak. Defense: MDM policy restricting sideloading, iOS App Store enforcement, enterprise cert monitoring.
Malware vectorAndroid more exposed
Outdated OS / Unpatched Mobile
Android fragmentation: device manufacturers control updates — many devices never receive OS patches even when available from Google. iOS: Apple pushes updates directly but users delay. Old mobile OS = known CVEs exploitable. Corporate problem: BYOD devices running old OS connect to corporate email/VPN. Defense: MDM policy requiring minimum OS version — block corporate access for out-of-date devices. Android Enterprise can enforce patch levels.
Android fragmentationKnown CVEs
Excessive App Permissions
Apps requesting more permissions than needed — microphone, camera, location, contacts, SMS. Malicious apps: request excessive permissions → access data → exfiltrate. Legitimate apps: unnecessary permissions are still a risk (app compromise → those permissions exploited). Defense: review app permissions before installation, OS-level permission controls, MDM restricts which apps can be installed, principle of least privilege for apps.
Data exposurePrivacy risk
MDM / MAM Bypass & Enrollment Weaknesses
MDM (Mobile Device Management) manages device security policies. MAM (Mobile Application Management) manages apps. Bypass: factory reset removes MDM enrollment (depends on implementation — DEP/ABM can survive reset). Enrollment weakness: user-enrolled devices have less control than corporate-enrolled. Defense: Apple DEP / Android Zero-Touch Enrollment — enrolled at factory → survives reset. Corporate-owned personally enabled (COPE) vs BYOD — COPE has more control.
Policy bypassDEP/Zero-Touch defends

Mobile-Specific Attack Vectors

Rogue Mobile Apps

Malicious apps mimicking legitimate ones. "Bank app" that captures credentials. Trojanized apps: real app bundled with malware. Defense: only official app stores, verify developer identity, check permissions, security tools that scan installed apps.

SMS-Based Attacks

SIM swapping: social engineer carrier to take victim's number → receive their MFA codes. SS7 attacks: telecom protocol vulnerabilities allow intercepting SMS anywhere in the world. Defense: use authenticator app (TOTP) not SMS for MFA, carrier SIM lock PIN.

Unsecured WiFi / Bluetooth

Mobile users frequently connect to open WiFi. Bluesnarfing: unauthorized BT data access. Evil twin: fake corporate WiFi on mobile. Defense: VPN on untrusted networks, disable BT when not needed, MDM enforces VPN policy.

VIRTVirtualization VulnerabilitiesVM Escape · Sprawl · Side-Channel · Containers

Virtualization shares physical resources between multiple isolated environments. Vulnerabilities in the virtualization layer can compromise the isolation that is the fundamental security promise of virtualization.

Core Virtualization Vulnerabilities

VM Escape — Breaking Hypervisor Isolation
A guest VM exploits a hypervisor vulnerability to gain access to the host system or other VMs. CRITICAL — bypasses the fundamental isolation guarantee of virtualization. Real examples: Venom (2015, CVE-2015-3456 — virtual floppy driver), multiple Xen/KVM escape CVEs. If attacker escapes the VM, they own the entire physical host and all VMs running on it. Defense: patch hypervisor immediately (highest priority), run minimal attack surface on hypervisor, dedicated physical hosts for high-sensitivity workloads.
CriticalBreaks isolationPatch hypervisor ASAP
VM Sprawl — Unmanaged Virtual Machines
VMs easy to create → organizations accumulate hundreds of forgotten, unmanaged VMs. Unpatched VMs: running old OS with unpatched CVEs, no EDR, no monitoring. "The forgotten dev VM from 2018 that still has RDP open." Attackers find these via internal scans — easy targets. Defense: VM inventory and lifecycle management, auto-decommission policy, regular audits, tag all VMs with owner and purpose.
Unpatched systemsAttack surface hidden
VM Snapshot Vulnerabilities
Snapshots capture VM state at a point in time — including credentials, certificates, and vulnerabilities from that time. Restored snapshot: re-introduces old vulnerabilities that were since patched, old credentials that may have been rotated, old configurations. "We restored the snapshot from before the breach" — but snapshot may contain the initial foothold too. Defense: snapshot lifecycle policy, patch before restoring, don't hoard old snapshots.
Stale credentialsPatched vulns return
Side-Channel Attacks (Spectre / Cross-VM)
Co-located VMs on same physical host can potentially infer information about each other via shared hardware resources — CPU cache timing, memory buses, branch predictor state. Spectre exploits this in virtualized environments — VM A reads memory of VM B via timing analysis. Cloud shared tenancy: your VM next to competitor's VM on same physical host — timing attacks potentially leak info. Defense: CPU microcode patches, OS patches, separate physical hosts for highest sensitivity, cloud dedicated hosts option.
Cloud multi-tenancy riskMicrocode patches

Container Vulnerabilities

Container Escape

Break out of container isolation to access host OS. Weaker isolation than VMs — containers share host kernel. Vulnerability in container runtime (runc, containerd) can allow escape. Privileged containers: run as root on host — more escape risk. Defense: never run privileged containers unnecessarily, patch container runtime, use Seccomp/AppArmor profiles, consider gVisor (sandboxed container kernel).

Vulnerable Container Images

Base images with known CVEs, outdated packages. "We're running the latest app code on Ubuntu 18.04 base from 2 years ago." Hundreds of known CVEs in the base. Shared images from public registries: may contain malware or vulnerabilities. Defense: regular image scanning (Trivy, Snyk), rebuild images frequently, use minimal base images (distroless, Alpine), private registry with scanning gate.

OSOperating System VulnerabilitiesKernel · Privilege Escalation · EOL · Misconfig

The OS is the foundation of all security — a compromised OS means all applications and data on the system are compromised. Kernel vulnerabilities, unpatched systems, and misconfigured OS settings are among the most exploited attack vectors.

OS Vulnerability Types

Kernel Vulnerabilities — Local Privilege Escalation
Flaws in the OS kernel allow escalation from unprivileged user to root/SYSTEM. Attacker: gains low-privilege shell via web app vulnerability → exploits kernel CVE → achieves full system compromise. Notable: EternalBlue (CVE-2017-0144, SMBv1 RCE — WannaCry), Dirty Cow (Linux kernel race condition → root), PrintNightmare (Windows Print Spooler RCE). Defense: patch OS promptly, disable unused OS services (SMBv1, Print Spooler on non-print servers), endpoint protection that detects exploit behavior.
Local privesc → rootPatch critical
Unpatched / End-of-Life Operating Systems
Systems running after vendor end-of-support date receive no security patches — all future CVEs have no fix. Windows XP: EOL 2014 — still found in industrial environments. Windows 7: EOL January 2020 — still widespread. EternalBlue specifically targeted XP/7. Healthcare: medical devices locked to old OS. Defense: upgrade or replace — if impossible: network isolation, virtual patching (IPS rules), disable vulnerable services, enhanced monitoring as compensating controls.
No patches availableCompensating controls needed
Default Configuration Weaknesses
OS defaults optimized for compatibility and ease of use — not security. Windows: SMBv1 enabled by default (pre-2019), Guest account enabled, NTLM authentication enabled alongside Kerberos, NetBIOS and LLMNR broadcast enabled (NBNS/LLMNR poisoning). Linux: root SSH login enabled, all ports accessible, no firewall configured. Defense: OS hardening per CIS Benchmark (Level 1/2), STIG for government, automated hardening scripts/Ansible playbooks.
Compatibility defaultsCIS benchmark fixes
Privilege Escalation Paths
Multiple OS-level paths to escalate privileges beyond intended: misconfigured SUID/SGID binaries (Linux), weak file/directory permissions on system files, writable paths in PATH variable, DLL hijacking (Windows loads attacker DLL from writable location), unquoted service paths (Windows), scheduled tasks running as SYSTEM owned by user, sudo misconfigurations. Defense: regular privilege audit, CIS benchmark, minimal services, principle of least privilege throughout.
Local privescConfiguration flaw

CLDCloud-Specific VulnerabilitiesMisconfiguration · IAM · Shared Tenancy

Cloud environments introduce unique vulnerabilities tied to shared infrastructure, API-driven management, and the complexity of cloud-native services. Misconfiguration is consistently the #1 cause of cloud breaches.

Cloud Vulnerability Types

Cloud Misconfiguration — #1 Cloud Risk
Incorrectly configured cloud resources expose data or systems. Classic failures: S3 bucket public (companies expose PII to internet), security group allows 0.0.0.0/0 on all ports (any internet connection to database), no MFA on root/admin accounts, storage replication to different region with weaker controls, Lambda function with admin IAM role. Defense: CSPM (Cloud Security Posture Management) — continuously checks against security benchmarks — AWS Security Hub, Prisma Cloud, Azure Defender.
#1 cloud breach causeCSPM detects
Overly Permissive IAM Policies
Cloud IAM roles with AdministratorAccess when ReadOnlyAccess would suffice. Service accounts with far more permissions than needed. Wildcard IAM policies: "Action: *" "Resource: *" — full access to everything. Role chaining: low-privilege role can assume high-privilege role. Defense: least privilege in IAM, regular permission audits, AWS IAM Access Analyzer, remove unused roles and policies, use temporary credentials (STS) for short-lived access.
Privilege escalationVery common
Insecure Cloud APIs
Cloud management APIs exposed to internet without adequate authentication or authorization. API keys embedded in client-side code, mobile apps, or public repositories. No rate limiting — brute force enumeration. Broken object-level authorization in cloud APIs (IDOR equivalent). Defense: never embed API keys in client code, secrets manager for keys, API gateway with auth, rate limiting, regular key rotation, monitor for API key usage anomalies.
Key exposureOWASP API Top 10
Shared Tenancy Risks
Multiple customers share same physical hardware in public cloud. Hypervisor vulnerabilities (Spectre/Meltdown) could allow cross-tenant information leakage. Noisy neighbor: one tenant's resource exhaustion impacts performance of others. Defense: dedicated instances for highest-sensitivity workloads, cloud provider security controls, encryption of all data (even at rest in cloud), encrypt east-west traffic within cloud environment.
Side-channel riskDedicated hosts option
Instance Metadata Service (IMDS) Attacks
Cloud instances have a metadata endpoint (169.254.169.254) that provides: IAM credentials for the instance role, user data scripts (may contain secrets), network configuration, identity information. SSRF vulnerability in application → attacker makes request to IMDS → steals IAM credentials → full cloud account access. Capital One breach (2019): WAF SSRF → IMDS → IAM credentials stolen → 100M records. Defense: IMDSv2 (requires session tokens — prevents SSRF exploitation), restrict IMDS access from sensitive workloads.
SSRF → cloud takeoverIMDSv2 defends

WEBWeb Injection VulnerabilitiesSQLi · SSRF · Template Injection

Web applications accept user input and use it to interact with backend systems. When that input isn't properly validated and neutralized, attackers inject commands that get executed by the backend.

Server-Side Request Forgery (SSRF)

SSRF — Server-Side Request Forgery
Attacker causes server to make HTTP requests to internal/unintended targets. Application: "fetch this URL for me" → attacker provides internal URL (cloud IMDS, internal admin interface, internal database). Server fetches it — bypassing any firewall that blocks direct external access. Capital One: SSRF → IMDS → IAM credentials. Defense: input validation for URLs, block requests to internal IP ranges (169.254.169.254, 10.x, 172.16.x, 192.168.x), IMDSv2, network policy blocking SSRF targets.
IMDS → cloud takeoverInternal access bypass
Server-Side Template Injection (SSTI)
Web frameworks use template engines (Jinja2, Twig, Freemarker). User input directly rendered in template → executes template code. Jinja2: {{7*7}} in input → outputs "49" → code executed. Escalate to RCE: access OS via template sandbox escape. Defense: never pass user input directly to template engine, use safe rendering, sandbox template execution.
RCE possibleTemplate engine specific
Open Redirect
Application takes a URL as parameter and redirects user to it without validation. Legitimate app: /redirect?url=https://company.com/dashboard. Attacker: /redirect?url=https://phishing.com → victim's browser redirected to phishing site. Appears to originate from trusted domain — higher click-through on phishing. Defense: whitelist of allowed redirect destinations, or use relative paths only, never redirect to user-supplied absolute URLs.
Phishing enablerTrusted domain abuse

XSS — Cross-Site Scripting
Inject malicious JavaScript into web pages viewed by other users. Stored XSS: script saved in database → executes for ALL who view that page (comment, profile field). Reflected XSS: script in URL parameter → reflected in response → victim clicks malicious link. DOM-based XSS: script modifies DOM client-side, never touches server. Impact: steal session cookies, redirect to phishing, keylog page content, perform actions as victim. Defense: output encoding (HTML entity encode untrusted data before rendering), Content Security Policy (CSP), HttpOnly cookies (XSS can't read them).

Clickjacking / UI Redress
Attacker overlays a transparent iframe of the target site over a decoy page. Victim: thinks they're clicking "Win a Prize!" → actually clicking "Transfer Money" button on hidden bank page. Defense: X-Frame-Options: DENY (prevents page being loaded in iframe), or Content-Security-Policy: frame-ancestors 'none'.

WEBAPI VulnerabilitiesREST · GraphQL · Broken Auth

Broken Object Level Authorization (BOLA)

API equivalent of IDOR. GET /api/orders/1234 → change to 1235 → access another user's order. API-specific because APIs are designed for programmatic access — attackers automate enumeration. Defense: server-side authorization on every API request — verify user owns the requested object.

Broken Function Level Authorization

Regular user calls admin API endpoint. POST /api/admin/deleteUser — should require admin role. Application only checks on UI, not in API. Attacker: calls API directly — bypasses UI restriction. Defense: enforce authorization in API layer, not just UI.

Excessive Data Exposure

API returns more data than the client needs — expects client to filter. GET /api/user/profile returns SSN, full credit card, internal ID. Client should only show username. But attacker calls API directly → gets all fields. Defense: return only what the client needs — filter at server, not client.

Mass Assignment

API automatically maps request body to object properties. Client sends: {"username": "alice", "isAdmin": true}. API blindly assigns all fields → user becomes admin. Defense: explicit allowlist of fields that can be set via API — never auto-assign all body parameters to model.

SCSupply Chain VulnerabilitiesVendors · Dependencies · Hardware · MSP

Supply chain vulnerabilities arise from trust relationships with third parties — software vendors, hardware manufacturers, cloud providers, managed service providers, and open-source components. Compromising one link can affect all downstream organizations.

Supply Chain Vulnerability Types

Compromised Software Updates
Attacker infiltrates vendor's build/update infrastructure — malicious code inserted into legitimate update that is then signed by the vendor and distributed to all customers. SolarWinds Orion (2020): 18,000 organizations installed backdoored update. 3CX (2023): communications app update contained malware. XZ Utils (2024): backdoor inserted into Linux compression library via social engineering of maintainer. Defense: SBOM, verify update signatures, integrity monitoring, anomaly detection for post-update behavior, network monitoring for new C2 connections after updates.
Signed malwareAffects all customers
Open Source Dependency Vulnerabilities
Modern applications depend on hundreds of open-source libraries. Each library: potential attack surface. Log4Shell (CVE-2021-44228): single vulnerability in logging library used in thousands of applications — mass exploitation globally. Developers often unaware of transitive dependencies (dependencies of dependencies). Defense: Software Composition Analysis (SCA) tools — Snyk, OWASP Dependency Check, GitHub Dependabot — scan for known CVEs in all dependencies including transitive. SBOM makes all dependencies visible.
Log4Shell exampleSCA scanning fixes
Dependency Confusion & Typosquatting
Dependency confusion: attacker publishes malicious package with same name as a private internal package but on public registry (npm, PyPI). Package manager prefers public registry → installs malicious version. Alex Birsan (2021): researcher demonstrated this against 35+ companies including Apple, Microsoft, PayPal. Typosquatting: publish "reqeusts" when developers mean "requests". Defense: private package registries with scope/namespace locking, explicitly specify registry source, verify package digests.
Package manager exploit35+ companies affected
Hardware Supply Chain Tampering
Physical hardware modified before reaching end user. Methods: malicious chip added to server motherboard (Bloomberg SuperMicro claim), firmware modified on hard drives (Equation Group — NSA-linked), malicious BIOS/UEFI injected at manufacturing or in transit. Interdiction: intelligence agencies intercept packages and modify hardware. Defense: trusted suppliers with verification processes, tamper-evident packaging, hardware attestation (TPM measured boot), firmware integrity verification, supply chain diversity.
State-actor levelVery hard to detect
Third-Party Vendor Compromise
Vendors with privileged access to customer environments become attack vectors. Target breach (2013): HVAC vendor → network pivot → POS systems → 40M cards. Kaseya (2021): RMM tool used by MSPs → pushed ransomware to 1,500 orgs. Any vendor with: VPN access, API credentials, admin accounts, or physical access is a supply chain risk. Defense: vendor risk assessment, least privilege for vendor access, monitor vendor connections, incident response plan if vendor is compromised.
Trusted access exploitedTPRM mitigates
