PKIPublic Key Infrastructure OverviewTrust · Identity · Binding

PKI is the framework of policies, procedures, hardware, software, and standards that enables secure electronic communication through asymmetric cryptography. It solves the fundamental trust problem: how do you know the public key you received actually belongs to who you think it does?

The Trust Problem PKI Solves

# Without PKI — the MITM problem: Alice wants to send encrypted message to Bob Alice receives: "Here is my public key: [key]" from Bob Problem: how does Alice KNOW this is Bob's key? Eve could intercept and substitute her own key Alice encrypts with Eve's key → Eve decrypts → reads message → re-encrypts with Bob's key # With PKI: Bob's key is wrapped in a CERTIFICATE Certificate signed by a trusted Certificate Authority (CA) CA vouches: "I verified this key belongs to Bob at company.com" Alice's browser trusts the CA (pre-installed trust store) Chain: Alice trusts CA → CA vouches for Bob → Alice trusts Bob's key PKI answer: binding identity to public key via trusted third party

PKI Components

Certificate Authority (CA)

Trusted third party that issues and signs digital certificates. Verifies identity before issuing. Root CA: highest level of trust. Intermediate CA: issues end-entity certificates. Browsers ship with pre-trusted root CAs.

Registration Authority (RA)

Delegates identity verification for CA. Handles enrollment requests, verifies identity, forwards to CA for signing. Offloads verification work from CA — CA still signs. May be an internal team or organizational role.

Certificate Repository

Public directory where certificates can be retrieved. LDAP directory, HTTP server. Also: Certificate Revocation Lists (CRL) published here. Clients query to verify certificate validity and retrieve certificates for others.

PKIKey Pairs — Public & Private KeysAsymmetric Foundation

PKI is built on asymmetric (public-key) cryptography — mathematically linked key pairs where data encrypted with one key can only be decrypted with the other. This elegant math enables secure communication without prior shared secrets.

Public Key
🔓 Share with everyone
Published freely — in certificate, on website, in directory. Anyone can have it. Used by others TO: encrypt messages to you, verify your digital signature. Cannot be kept secret — the whole point is sharing it widely.

Private Key
🔒 Never leave your device
Held exclusively by owner — never transmitted. Used by YOU TO: decrypt messages others encrypted to you, create digital signatures. Compromise of private key = compromise of everything. Protected by HSM, smart card, or encrypted key store.

Key Operations — What Each Key Does

# Encryption use case (confidentiality): Sender encrypts with recipient's PUBLIC key → only recipient can decrypt Recipient decrypts with their own PRIVATE key → confidential message Analogy: Public key = padlock anyone can close Private key = only key that opens that padlock # Digital signature use case (authentication + integrity + non-repudiation): Sender signs with their own PRIVATE key → only they could have signed Recipient verifies with sender's PUBLIC key → signature is authentic # Critical exam rule: which key for what? Encrypt for someone: their PUBLIC key (anyone can send them encrypted data) Decrypt: YOUR PRIVATE key (only you can decrypt what was sent to you) Sign: YOUR PRIVATE key (proves it came from you) Verify signature: signer's PUBLIC key (anyone can verify your signature) # Key exchange: Problem: asymmetric crypto is SLOW for bulk data Solution: use asymmetric to exchange a SYMMETRIC session key Then: use fast symmetric key for actual data encryption This is how TLS/HTTPS works

Key Storage and Protection

Hardware Security Module (HSM)

Dedicated hardware device for cryptographic operations and key storage. Keys generated inside HSM — never exported in plaintext. FIPS 140-2 Level 3: physical tamper resistance — destroys keys if opened. Uses: CA root key, code signing, high-value certificate private keys. Cloud: AWS KMS, Azure Key Vault (cloud HSM).

TPM — Trusted Platform Module

Chip on motherboard that stores cryptographic keys. Used by BitLocker: encryption key bound to the TPM — disk encrypted, key in hardware. Measured boot: TPM records boot sequence — detects tampering. Cannot extract keys from TPM without physical access to chip. Most modern PCs have TPM 2.0.

Internal vs External PKI

Public PKI (External CA)

Commercial CAs: DigiCert, Let's Encrypt, Sectigo. Trusted by browsers/OS out of the box. Required for: public websites, external-facing services. Cost: commercial CAs charge; Let's Encrypt is free. Validation: CA verifies domain/organization before issuing.

Private PKI (Internal CA)

Organization runs its own CA hierarchy. Used for: internal services, employee certificates, device authentication, code signing. Not trusted by external browsers by default — must push root cert to devices via GPO/MDM. Windows: Active Directory Certificate Services (AD CS). Benefits: full control, no cost per cert, custom validity periods.

PKICertificate RevocationCRL · OCSP · Stapling

When a certificate must be invalidated before its expiry — private key compromised, employee leaves, org changes — revocation mechanisms allow CAs to declare a certificate invalid.

CRL — Certificate Revocation List

A list of serial numbers of revoked certificates, digitally signed by the CA. Published at CDP (CRL Distribution Point) URL in the certificate. Client downloads CRL → checks if serial number is on it. Problems: CRL can be large (megabytes), updated on schedule not in real time, client caches it — delay between revocation and detection.

OCSP — Online Certificate Status Protocol

Real-time revocation checking via HTTP request to OCSP responder. Client sends: serial number. OCSP returns: good/revoked/unknown. Faster and smaller than CRL. Problem: OCSP responder must be available — single point of failure. Privacy: OCSP responder learns which sites you're visiting.

⚡
OCSP Stapling — Best of Both
Server queries OCSP responder itself → caches the signed response → staples it to the TLS handshake. Client receives cert + OCSP response in one handshake. Benefits: faster (no extra client request), better privacy (client doesn't contact OCSP), works even if OCSP responder is temporarily down (cached response). Modern web servers use OCSP stapling — it's the recommended approach.

// Exam Keywords

CRL →List of revoked serial numbers — published by CA — client downloads — can be large/stale
OCSP →Real-time revocation check — client queries responder — faster than CRL — privacy concern
OCSP stapling →Server staples OCSP response to handshake — fast + private + resilient — best approach
Certificate pinning →App only trusts specific cert/key — prevents MITM even with rogue CA

ENCEncryption OverviewConfidentiality · Transform Data

Encryption is the process of transforming readable plaintext into unreadable ciphertext using a mathematical algorithm and a key. Only those with the correct key can decrypt the ciphertext back to plaintext. It is the primary technical control for confidentiality.

Symmetric vs Asymmetric — Core Distinction

Symmetric
One Key
Same key encrypts AND decrypts. Fast — ideal for bulk data. Problem: how do you securely share the key initially? Examples: AES, 3DES, ChaCha20.
Asymmetric
Key Pair
Public key encrypts, private key decrypts (or vice versa for signing). Slow — impractical for bulk data. Solves key distribution. Examples: RSA, ECC, DH.
# How TLS/HTTPS uses BOTH (hybrid encryption): 1. Asymmetric: client uses server's public key to encrypt a random value 2. Both derive: symmetric session key from that value (key exchange) 3. Symmetric: all subsequent data encrypted with fast symmetric key Result: security of asymmetric (key exchange) + speed of symmetric (bulk data) This hybrid approach is used in: TLS, SSH, PGP, S/MIME

// Exam Keywords

Symmetric →One key — fast — bulk data — key distribution problem
Asymmetric →Key pair — slow — solves key distribution — used for key exchange
Hybrid →Asymmetric for key exchange + symmetric for data — TLS, SSH, PGP
ENCSymmetric EncryptionAES · 3DES · ChaCha20

Symmetric Algorithms

AES
Advanced Encryption Standard. CURRENT STANDARD. Block cipher — 128-bit blocks. Key sizes: 128, 192, 256 bits (AES-256 = gold standard). Extremely fast in hardware. Adopted by US government for classified data. Used in: TLS, disk encryption (BitLocker), Wi-Fi (WPA2/3), file encryption.
Block cipherApproved128/192/256-bit key
ChaCha20
Stream cipher. Excellent performance on devices without AES hardware acceleration (mobile, IoT). Used in TLS 1.3 alongside AES. Paired with Poly1305 for authentication: ChaCha20-Poly1305. Google uses it for HTTPS on mobile devices. Modern and secure — good alternative to AES.
Stream cipherApprovedTLS 1.3
3DES
Triple DES — applies DES three times. Legacy algorithm — being phased out. Effectively 112-bit security. Much slower than AES. NIST deprecated 3DES in 2017 — disallowed after 2023. If you see 3DES: replace with AES. Still found in legacy payment systems and old TLS configs.
LegacyDeprecated
DES
Data Encryption Standard. 56-bit key — completely broken. Crackable in under 24 hours. Never use for anything. Replaced by 3DES → AES. If you see DES in a security context: that system is broken.
Broken56-bit — DO NOT USE

Block Cipher Modes

# How block ciphers process multiple blocks of data: ECB (Electronic Codebook) — NEVER USE: Same plaintext block → same ciphertext block Pattern preservation: encrypted penguin image still shows penguin shape Deterministic: easy to detect patterns, replay attacks CBC (Cipher Block Chaining): Each block XOR'd with previous ciphertext before encrypting Randomized by IV (Initialization Vector) — no two same Sequential: cannot parallelize decryption Vulnerable: BEAST, POODLE attacks on TLS — avoid in TLS GCM (Galois/Counter Mode) — RECOMMENDED: Parallelizable: fast Provides: encryption + authentication (AEAD) AES-GCM: used in TLS 1.3, SSH, IPsec Both encrypts data AND verifies integrity in one step CTR (Counter Mode): Turns block cipher into stream cipher Parallelizable Requires: unique nonce per use — reuse is catastrophic

ENCAsymmetric EncryptionRSA · ECC · DH · Key Exchange

Asymmetric Algorithms

RSA
Rivest–Shamir–Adleman. Based on difficulty of factoring large numbers. Key sizes: 2048-bit minimum (secure), 4096-bit for long-term. Slow — used for key exchange and signatures, not bulk data. RSA-2048: widely used in TLS certificates, code signing. RSA-1024: broken — do not use.
2048-bit minWidely deployed
ECC
Elliptic Curve Cryptography. Based on algebraic structure of elliptic curves. Smaller keys for equivalent security: ECC-256 ≈ RSA-3072. Faster and less power-hungry than RSA. Widely used on mobile/IoT. Curves: P-256, P-384, Curve25519 (modern, fast, resistant to timing attacks). Used in: TLS 1.3 key exchange (ECDHE), Bitcoin.
Smaller keysEfficientMobile/IoT
Diffie-Hellman
Key exchange — allows two parties to establish a shared secret over an insecure channel WITHOUT exchanging the secret itself. Not encryption itself — just key exchange. DHE (ephemeral DH): new keys per session → forward secrecy. ECDHE: DH using elliptic curves — faster. Used in TLS for key exchange.
Key exchangeForward secrecy (DHE)

Forward Secrecy — Critical Concept

# Why forward secrecy matters: Without FS (old RSA key exchange): Attacker records encrypted TLS session Later: compromise server's RSA private key Can decrypt ALL previously recorded sessions With FS (DHE/ECDHE): Each session: new ephemeral key pair generated Session key derived from ephemeral keys → discarded after session Compromise server key: cannot decrypt past sessions Each session: only decryptable with that session's ephemeral key # TLS 1.3 mandates forward secrecy: Only allows ECDHE and DHE for key exchange Removed: RSA key transport (no forward secrecy)

ECC →Smaller keys — same security as larger RSA — efficient — mobile/IoT
Diffie-Hellman →Key EXCHANGE only — not encryption — establishes shared secret over insecure channel
Forward secrecy →Ephemeral keys — past sessions safe if key compromised later — TLS 1.3 mandates it

Encryption At Rest vs In Transit

Encryption At Rest

Encrypts stored data — protects if physical media is stolen. Full Disk Encryption: BitLocker (Windows), FileVault (macOS), LUKS (Linux). Database encryption: TDE (Transparent Data Encryption). File-level: VeraCrypt, PGP. Key: store encryption key separately from data — encrypted disk with key on same drive = useless.

Encryption In Transit

Encrypts data moving between systems — prevents eavesdropping. TLS 1.3: HTTPS, SMTP/TLS, IMAP/TLS. SSH: encrypts admin traffic. IPsec VPN: encrypts entire network layer. Never: plain HTTP, Telnet, FTP for sensitive data. Certificate required: TLS relies on PKI for authentication.

OBFObfuscationHide · Obscure · Conceal

Obfuscation is the practice of making something unclear, obscure, or difficult to understand — without necessarily encrypting it. It is security through concealment rather than mathematical strength. NOT a substitute for strong encryption.

Obfuscation Techniques

Data Masking
Replacing real data with realistic but fake data. Used in: non-production environments (dev, test, QA). Credit card: 4532-9812-3456-7890 → 4532-XXXX-XXXX-7890. SSN: 123-45-6789 → XXX-XX-6789. Real customer data never in dev/test → no breach risk. Static masking: one-time transform. Dynamic masking: real-time based on user role (support agent sees masked, billing sees full).
Not reversibleDev/test use
Steganography
Hiding secret data INSIDE another file (image, audio, video, text). The carrier file appears normal — nobody knows there's hidden data. Technique: hide bits in least-significant bits of image pixels. Covert channel: attacker exfiltrates data hidden in images (bypasses DLP). Steganalysis: detecting steganography. Unlike encryption: doesn't draw attention — nobody knows data is hidden there.
Hidden in plain sightCovert channel risk
Tokenization
Replacing sensitive data with a non-sensitive placeholder (token). Token has no mathematical relationship to original — just a lookup table. Used in: payment processing (PCI-DSS) — merchant stores token, not real card number. The token vault (mapping token → real data) is protected. Unlike encryption: token is useless without vault. Even if tokens are stolen: cannot derive original data without vault access.
PCI-DSS commonNo math relationship
Code Obfuscation
Transforming source code to make it harder to reverse-engineer while keeping functionality identical. Techniques: variable renaming (x → a1b2c3), control flow flattening, string encryption, dead code insertion. Used by: malware authors (evade AV detection), software vendors (protect IP). Security implication: obfuscated malware harder to analyze. Does not provide security — determined analyst will reverse it.
Not true securityMalware technique

HSHHashing OverviewOne-Way · Integrity · Fingerprint

A hash function takes input of any size and produces a fixed-size output (digest) that uniquely represents the input. It is a one-way function — you cannot reverse a hash to get the original input. Any change to input produces a completely different hash.

# Hash properties — must know for exam: Deterministic: same input → always same hash One-way: hash → cannot reverse to get input Fixed output: any input size → same length output Avalanche effect: 1 bit change in input → completely different hash Collision resistant: computationally infeasible to find two inputs with same hash # SHA-256 example: input: "Hello" hash: 185f8db32921bd46d35cc2e... (64 hex chars = 256 bits) input: "hello" ← one character different (case) hash: 2cf24dba5fb0a30e26e83b2a... ← completely different! (avalanche) # What hashing provides: Integrity: verify file hasn't changed (compare hashes) Password storage: store hash not password → can't reverse to get password Digital sig: sign the hash (not the data) — efficient Forensics: chain of custody — evidence hash never changes

Collision Attacks

Collision Attack

Finding two different inputs that produce the same hash. "Hash collision." If achievable: attacker can substitute a malicious file for a legitimate one while keeping the same hash. MD5 and SHA-1: collision attacks exist — do not use for security. SHA-256+: currently collision-resistant.

Pre-image Attack

Given a hash, find ANY input that produces that hash. Much harder than collision attack. Current algorithms (SHA-256+) are resistant to pre-image attacks. Password hashes: pre-image attack would allow recovering password from hash — this is why rainbow tables are a concern.

// Exam Keywords

Hash →One-way function — fixed-size output — cannot reverse — integrity verification
Avalanche effect →Tiny input change = completely different hash — proves even small modifications detected
Collision →Two inputs same hash — MD5/SHA-1 vulnerable — reason they're deprecated
HSHHash AlgorithmsMD5 · SHA Family · HMAC
MD5 128-bit BROKEN Cryptographically broken — collisions demonstrated in seconds. Still used for: non-security checksums (file download verification where integrity not security-critical). NEVER use for: password hashing, digital signatures, security-sensitive integrity checks.
SHA-1 160-bit DEPRECATED Collision demonstrated in 2017 (SHAttered attack). Deprecated by NIST. Browsers no longer accept SHA-1 certificates. Still seen in: legacy git commits, older systems. Replace with SHA-256 minimum.
SHA-256 256-bit APPROVED Current minimum standard. Part of SHA-2 family. Used in: TLS certificates, Bitcoin, code signing, S/MIME. Fast and well-studied. Default choice for most integrity applications. SHA-256 is the go-to hash algorithm.
SHA-384 384-bit APPROVED Higher security margin than SHA-256. Used when extra strength needed. Government/classified data.
SHA-512 512-bit APPROVED Highest SHA-2 strength. Actually faster than SHA-256 on 64-bit systems. Used for password hashing as part of SHA-512crypt.
SHA-3 Varies APPROVED Completely different construction from SHA-2 (Keccak sponge). Backup if SHA-2 weaknesses found. Not yet widely deployed but NIST approved.
HMAC Varies KEY-BASED Hash-based Message Authentication Code. Hash + secret key = HMAC. Provides: integrity AND authentication (you need the key to verify). Used in: TLS MAC, API authentication (HMAC-SHA256 JWT), IPsec. Without key: cannot produce valid HMAC.

HSHPassword HashingSalts · bcrypt · Argon2

Passwords must be stored as hashes — never plaintext. But basic hashing is insufficient for passwords. Dedicated password hashing functions add important protections.

# Why basic hash (SHA-256) is wrong for passwords: Problem 1: Rainbow tables Attacker pre-computes hash("password") = 5e884898... Steals database → looks up hash → finds "password" instantly SHA-256 is fast → trillions of lookups/second Problem 2: Identical passwords = identical hashes Alice: password = "summer2023" Bob: password = "summer2023" Database: alice_hash = bob_hash → tells attacker both use same pass # Solutions: 1. Salt: Random string appended to password BEFORE hashing Alice: hash("summer2023" + "a7f3d2") = unique hash Bob: hash("summer2023" + "k8q2r9") = completely different hash! Rainbow table: useless — would need table for every salt value Salt stored alongside hash (not secret — uniqueness is enough) 2. Slow hash functions (work factor): bcrypt: designed to be slow — thousands of iterations attacker: only thousands of attempts/second (not trillions) Argon2: memory-hard — uses lots of RAM → expensive to parallelize PBKDF2: key derivation with iterations → NIST approved scrypt: memory + CPU hard # Password storage rule: NEVER: store plaintext NEVER: use MD5 or SHA-256 alone (too fast) ALWAYS: use bcrypt, Argon2, or PBKDF2 ALWAYS: salt is automatically handled by these functions

// Exam Keywords

Salt →Random value added to password before hashing — defeats rainbow tables — stored with hash
bcrypt →Slow password hash — work factor — thousands of iterations — correct for passwords
Argon2 →Memory-hard password hash — modern winner — current recommendation
Rainbow table →Pre-computed hash lookup table — defeated by salts — attacker tool
SHA-256 for passwords? →WRONG — too fast — use bcrypt/Argon2/PBKDF2 instead

SIGDigital SignaturesAuth · Integrity · Non-Repudiation

Digital signatures provide three things simultaneously: authentication (who sent it), integrity (wasn't modified), and non-repudiation (can't deny sending it). They combine asymmetric cryptography and hashing into one powerful mechanism.

How Digital Signatures Work

# Signing (sender): 1. hash = SHA256(document) // create fingerprint 2. sig = RSA_sign(hash, private_key) // encrypt hash with private key 3. send: document + sig # Verifying (receiver): 4. hash_from_sig = RSA_verify(sig, sender_public_key) // decrypt sig with public key 5. hash_of_doc = SHA256(received_document) // hash what arrived 6. if hash_from_sig == hash_of_doc: ✓ Integrity: document not modified since signing ✓ Authentication: only private key holder could produce this sig ✓ Non-repudiation: signer cannot deny — only they have their private key else: ✗ Document was modified or signature is invalid # Why hash then sign (not sign the document directly)? RSA encryption is SLOW — signing 1GB file directly = too slow SHA-256 of 1GB file = 32 bytes → sign the tiny hash instead Fast regardless of document size

What Digital Signatures Provide

Authentication

Proves the message came from the private key holder. Only the owner of the private key could have produced the signature. Relies on: PKI to bind the private key to a verified identity via certificate.

Integrity

Proves the document wasn't modified after signing. If even one bit changes: hash changes, signature verification fails. Detects: MITM tampering, accidental corruption, unauthorized modification.

Non-Repudiation

Signer cannot deny having signed. Private key is secret — only key holder could produce the signature. Legal weight: digital signatures legally binding in many jurisdictions (eIDAS EU, ESIGN USA).

BLKBlockchainDistributed · Immutable · Consensus

A blockchain is a distributed, append-only ledger where each block contains a cryptographic hash of the previous block, creating an immutable chain. Tampering with any block invalidates all subsequent blocks — making fraud detectable.

# Why blockchain is tamper-evident: If attacker modifies Block 1 transaction: "Bob → Carol 2 BTC" → "Bob → Eve 100 BTC" Block 1's hash: changes completely (avalanche effect) Block 2's "Previous hash" field: now doesn't match Block 1's real hash Block 2 is now invalid → Block 3 is invalid → entire chain after Block 1 invalid Detection: any node in the distributed network checks → mismatch detected Attack: would need to recalculate ALL subsequent blocks AND convince majority of distributed network to accept the tampered chain This is "51% attack" — requires controlling majority of network computing power

Blockchain Security Properties

Immutability

Records cannot be altered retroactively. Each block's hash is part of the next block — tampering invalidates all following blocks. Provides: audit trail that cannot be falsified. Security use: certificate transparency logs, supply chain records, audit trails.

Distributed

No single point of failure or control. Hundreds/thousands of nodes hold full copies. Attacker must compromise majority of nodes simultaneously (51% attack). Provides: high availability + resistance to single-party manipulation.

Transparency

Public blockchains: all transactions visible to everyone. Every transaction verifiable by any participant. Auditability: complete history of every transaction since genesis. Note: privacy concern — no anonymity on public blockchain (addresses pseudonymous, not anonymous).

Security Applications of Blockchain

🔒
Certificate Transparency (CT) Logs
Public, append-only logs of all issued TLS certificates. CAs must submit certificates to CT logs. Browsers check CT logs — certificates not in logs are rejected. Detects: rogue CAs issuing unauthorized certificates (e.g., for google.com without Google's knowledge). DigiNotar 2011: rogue certificates issued — CT logs would have immediately detected this. Security benefit: any unauthorized cert issuance is publicly visible within seconds.
📦
Supply Chain Integrity
Record every step in a supply chain on blockchain — manufacturer to retailer. Immutable record: cannot falsify provenance records. Detect counterfeits: every item has a traceable history. Applications: pharmaceutical (verify drug authenticity), food safety (trace contamination source instantly), electronics (prevent counterfeit components).
