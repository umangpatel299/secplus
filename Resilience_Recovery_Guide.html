01 High Availability Availability · Uptime

High Availability (HA) means designing systems to stay operational continuously by eliminating single points of failure. If one component fails — the system keeps running without users noticing.

Key HA Concepts

Load Balancing

Distributes traffic across multiple servers. No single server overwhelmed. If one fails — others absorb traffic automatically. Users notice nothing.

User request → Load Balancer → Server A (40% load) ✓ → Server B (35% load) ✓ → Server C (25% load) ✓ Server A dies → LB routes to B+C Zero downtime for users

Active/Active vs Active/Passive

Active/Active: ALL nodes handle traffic simultaneously. Best performance + redundancy. Active/Passive: ONE active, ONE standby — takes over only on failure.

Active/Active: Node1 ACTIVE — handling traffic Node2 ACTIVE — handling traffic Both share load Active/Passive: Node1 ACTIVE — handling traffic Node2 STANDBY — waiting Node1 fails → Node2 takes over

Clustering

Multiple servers grouped as one logical unit. Work together, cover for each other. Database clusters most common — primary replicates to secondary in real time.

Redundancy

Duplicate everything critical — dual power supplies, dual NICs, dual ISPs, RAID storage. No single hardware failure takes system down.

Key HA Metrics — Must Know All Three

MTTR

Mean Time To Repair
How long to FIX failed component

Lower = better
Goal: fix fast

MTBF

Mean Time Between Failures
How long until NEXT failure

Higher = better
Goal: last long

MTTF

Mean Time To Failure
Expected lifespan of component

Higher = better
Non-repairable items

# Availability formula: Availability = MTBF / (MTBF + MTTR) # Example: Server fails every 1000 hours (MTBF = 1000) Takes 4 hours to fix (MTTR = 4) Availability = 1000 / (1000 + 4) = 99.6% # To improve availability: Lower MTTR → faster repairs (spare parts, skilled staff) Higher MTBF → better hardware (quality components)

Geographic Considerations

Geographic Dispersal

Spread operations across multiple physical locations. Hurricane hits East Coast → West Coast continues. Earthquake in one city → other city unaffected. Different power grids = different outage risk.

Distance Rules

Too close — same natural disaster, same power grid, same flood plain — affects both.
Too far — replication latency increases, data becomes stale.
Sweet spot — different city, different region, same country.

03 DR Testing Types Prove It Works

An untested DR plan is just a theory. Testing proves resilience actually works — and finds gaps before a real disaster does. Five test types from safest to most realistic.

1. Tabletop Exercise
Discussion only — no systems touched. Team sits in room and talks through disaster scenario verbally. Facilitator presents: "It's 3am, data center on fire — walk me through your response." Finds plan gaps and communication issues. Zero risk, very cheap.
Risk: ZERO Cost: $ Systems affected: NONE Best for: finding plan gaps, testing communication Frequency: Quarterly recommended
2. Walkthrough / Structured Walkthrough
Team physically walks through every DR procedure step by step. Verify procedures are accurate and up to date. Check contact info is correct. Ensure staff knows where documentation is. Still no systems touched.
Risk: ZERO Cost: $ Systems affected: NONE Best for: verifying procedures are current and accurate
3. Simulation Test
Test specific components without full failover. "Let's actually restore this server from backup." "Let's test database failover to replica." Proves specific procedures work in practice.
Risk: LOW Cost: $$ Systems affected: PARTIAL Best for: verifying specific recovery procedures work
4. Parallel Test (Full Interruption — Safe)
Activate backup site while primary STAYS RUNNING. Both sites operational simultaneously. Compare outputs — are they identical? Primary never goes down. Proves backup site actually works without risking real downtime.
Risk: LOW Cost: $$$ Systems affected: BOTH RUNNING Primary: continues normal operations ✓ Backup: activated and verified ✓ Best for: proving backup site works safely
5. Full Interruption / Cutover Test
Actually SHUT DOWN primary site and switch entirely to backup. Real users, real transactions on backup site. Most realistic test possible. If backup fails — you have a real outage. Only do annually at most.
Risk: HIGH Cost: $$$$ Systems affected: PRIMARY SHUT DOWN If backup fails → REAL OUTAGE OCCURS Best for: absolute proof the plan works end-to-end Frequency: Annually maximum

UPS — Uninterruptible Power Supply

Purpose: INSTANT power bridge when grid fails — activates in milliseconds. Bridges gap until generator starts (usually 30-120 seconds).

UPS Types: Standby: switches to battery (tiny delay) Line-interactive: conditions power constantly Online/Double: ALWAYS on battery ← best Grid charges battery Battery powers everything Zero switchover delay Without UPS: 2-min gap = servers crash With UPS: zero interruption ✓

Generator

Purpose: Long-term power for extended outages. Diesel or natural gas. Runs indefinitely with fuel supply.

Key considerations: Fuel supply: how many hours of diesel? Automatic Transfer Switch (ATS): auto-starts on power failure auto-transfers load Testing: run monthly under load Maintenance: oil, filters, coolant Multiple generators: redundancy Fuel contracts: guaranteed delivery # Never assume generator works # Test it regularly under real load

Dual Power Feeds

Critical servers have TWO power supplies from DIFFERENT circuits, UPS units, and generators. Both must fail simultaneously to lose server — astronomically unlikely.

Server PSU1 ← Circuit A ← UPS-A ← Generator-A Server PSU2 ← Circuit B ← UPS-B ← Generator-B If entire Generator-A fails: Server still running on Generator-B ✓

PDU — Power Distribution Unit

Distributes power from UPS/generator to individual servers. Smart PDUs allow remote monitoring and control of each outlet. Can see power consumption per server remotely.

// SCENARIO — Data Center Power Outage
# Major power grid failure — entire city block: 14:32:15 — City power grid fails 14:32:15 — UPS systems activate instantly (milliseconds) All servers continue running on battery 14:32:17 — Monitoring detects grid failure Automatic Transfer Switch signals generators 14:32:45 — Generator A starts and warms up 14:33:10 — Generator A reaches stable output ATS transfers load to Generator A 14:33:12 — UPS batteries begin recharging 14:33:12 — All servers: running normally ✓ Users: zero impact noticed ✓ 17:45:00 — Grid still down — fuel check: Generator A: 280 gallons remaining (8.2 hours) Generator B: 295 gallons remaining (8.7 hours) Action: emergency fuel delivery ordered ✓ # City grid restored 22:15 — 7.7 hours total # Data center: 100% uptime throughout ✓

ATS →Automatic Transfer Switch — auto-starts generator on grid failure
05 Platform Diversity Reduce Blast Radius

Using different technologies, vendors, and platforms so one vulnerability, bug, or failure doesn't affect everything simultaneously. Avoids putting all eggs in one basket.

# No diversity — dangerous: All servers: Windows Server 2022 All network: Cisco only All cloud: AWS only All storage: NetApp only Critical Windows zero-day discovered → ALL servers vulnerable simultaneously ✗ AWS outage → ALL cloud services down simultaneously ✗ # With platform diversity: 50% servers: Windows Server 50% servers: Linux (RHEL/Ubuntu) Network: Cisco core + Juniper edge Cloud: AWS primary + Azure secondary Storage: NetApp primary + Pure secondary Windows zero-day → Linux servers completely unaffected ✓ AWS outage → Azure failover activates ✓

Types of Platform Diversity

OS Diversity

Mix of Windows and Linux. Vulnerability in one OS doesn't affect the other. Critical services duplicated across both OS types where possible.

Vendor Diversity

Multiple vendors for critical components. Cisco + Juniper for network. Palo Alto + Fortinet for firewalls. One vendor's critical bug doesn't compromise everything.

Cloud Provider Diversity

Multi-cloud strategy. AWS primary + Azure secondary. GCP for specific services. Major provider outage doesn't take down everything. Avoids vendor lock-in.

Geographic Diversity

Data centers in different cities, regions, countries. Different power grids, different natural disaster risks, different network routes. One location disaster = others unaffected.

Technology Diversity

Different technologies for similar functions. Not all Microsoft or all open-source. Web servers: Apache + Nginx. Databases: SQL Server + PostgreSQL. Reduces monoculture risk.

Crypto Diversity

Multiple cryptographic algorithms. If one algorithm is broken, not all data is exposed. RSA + ECDSA for signatures. AES + ChaCha20 for encryption.

// SCENARIO — Log4Shell (Log4j) Attack 2021
# Log4Shell: critical Java library vulnerability # Affected: any application using Log4j Company A — No platform diversity: All 47 applications: Java-based All using: Log4j logging library Log4Shell discovered: → ALL 47 applications vulnerable ✗ → Emergency patches for everything → 2-week remediation effort → Some services offline during patching Company B — Platform diversity: 12 applications: Java (use Log4j) 18 applications: Python 10 applications: Node.js 7 applications: Go Log4Shell discovered: → Only 12 Java apps need attention ✓ → 35 apps completely unaffected ✓ → 3-day targeted remediation → No service interruptions → Blast radius: 25% instead of 100%

Trade-offs

Benefits

Reduced blast radius of vulnerabilities. No vendor lock-in. Resilience against targeted attacks. Regulatory compliance. Negotiating leverage with vendors.

Challenges

Increased management complexity. Staff needs expertise across multiple platforms. Higher licensing costs. Integration between different systems harder. More potential failure points.

// EXAM KEYWORDS

Platform diversity →Different vendors/OS/tech — one vulnerability doesn't hit everything
Blast radius →How much is affected by one failure — diversity reduces this
Monoculture risk →All same technology = all fail together
Multi-cloud →AWS + Azure — cloud provider diversity
Trade-off →More secure but more complex — increased management overhead

3-2-1 Backup Rule — Must Know

3 — Three COPIES of data 2 — Two DIFFERENT media types 1 — One copy OFFSITE Example: Copy 1: Production server (local SAN) Copy 2: NAS backup device (local — different media) Copy 3: Cloud backup (offsite) ← most critical If fire destroys building: → Copies 1 and 2: destroyed → Copy 3 in cloud: SAFE ✓ Modern extension — 3-2-1-1-0: +1 — One OFFLINE/AIR-GAPPED copy Ransomware can't reach offline storage! +0 — Zero ERRORS (test restores regularly!) Untested backup = worthless backup

Backup Security

Immutable Backups

Cannot be modified or deleted — even by admins. WORM storage (Write Once Read Many). Ransomware tries to delete → refused. Critical modern protection.

Encrypted Backups

Backup data as sensitive as production data. Encrypt everything. Stolen backup tape → unreadable without key. AES-256 minimum.

Test Restores

Actually restore monthly. Verify data integrity. Measure restore time against RTO. An untested backup is NOT a backup — it's a hope.

07 Continuity of Operations (COOP) BIA · RTO · RPO · MTD

COOP is the plan and capability to continue essential business functions during and after ANY disruption — cyberattack, natural disaster, pandemic, or hardware failure.

Key Metrics — Must Know All Four

RTO — Recovery Time Objective

Maximum acceptable time to restore a service after failure.

"We must be back online within 4 hours of failure" Shorter RTO = more expensive (requires more redundancy) Drives: redundancy planning failover architecture

RPO — Recovery Point Objective

Maximum acceptable data loss measured in time.

"We can lose no more than 1 hour of data" RPO = 1hr → backups every hour RPO = 24hr → daily backups OK RPO = 0 → real-time replication Drives: backup frequency replication strategy

MTD — Maximum Tolerable Downtime

Absolute maximum time before irreversible harm to organization.

Hospital patient monitors: MTD = 30 sec Online retailer Black Friday: MTD = 5 min HR system: MTD = 72 hours CRITICAL RULE: RTO must ALWAYS be LESS than MTD If RTO > MTD → business fails before recovery completes!

BIA — Business Impact Analysis

Identifies critical business functions and the impact of disrupting each one. Foundation of all continuity planning.

BIA output example: Payment system: $50k/hr loss RTO: 15 min RPO: 5 min Email: $5k/hr loss RTO: 4 hrs RPO: 1 hr HR system: $500/hr loss RTO: 48 hrs RPO: 24 hrs Do BIA FIRST — everything else follows from BIA results

RTO vs RPO — Easy Memory Aid

RTO = Recovery TIME Objective → About TIME to restore SERVICE → "How fast back online?" → Determines REDUNDANCY needed RPO = Recovery POINT Objective → About data POINT in time → "How much DATA LOSS is OK?" → Determines BACKUP FREQUENCY Disaster strikes at 3:00 PM: RPO = 1 hour → last backup at 2:00 PM acceptable ✓ RTO = 4 hours → back online by 7:00 PM ✓ MTD = 6 hours → if not online by 9:00 PM → catastrophic

COOP Documents

BCP — Business Continuity Plan

BROAD — covers the entire business. People, processes, facilities, communication, vendors. "How do we keep serving customers during disruption?"

DRP — Disaster Recovery Plan

NARROW — restoring IT systems specifically. Servers, data, applications, infrastructure. "How do we get systems back online?" DRP is a subset of BCP.

IRP — Incident Response Plan

SECURITY incidents specifically. Cyberattacks, breaches, malware. Who to call, steps to contain, evidence preservation, regulatory notifications.

Communication Plan

Who tells who what during crisis. Internal: staff chain of command. External: customers, regulators, media, vendors. Pre-approved messages ready.

Order of Restoration — Critical for Exam

1. Network Infrastructure
Can't do anything without network. Core switches, routers, firewalls first.
2. Security Systems
Restore securely — firewalls, IPS, authentication systems before adding more systems.
3. Directory Services
Active Directory — identity and access. Without this, users can't authenticate to anything.
4. Critical Business Applications
ERP, payment systems, core business apps — highest revenue/operational impact first.
5. Communication Systems
Email, phone — staff can communicate internally and with customers.
6. Supporting Systems
HR, reporting, non-critical apps — restore last when critical systems stable.
