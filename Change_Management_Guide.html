01What Is Change Management?Process · Control · Security

Change management is the formal process for controlling modifications to IT systems, configurations, software, and infrastructure. Every unauthorized or uncontrolled change is a potential security vulnerability — a firewall rule accidentally opened, a patch that breaks MFA, a configuration drift that exposes sensitive data.

Why Change Management Is a Security Control

# Without change management — real consequences: Scenario 1: Developer modifies firewall rule "temporarily" Never documented → never reverted Port 3389 (RDP) opened to internet 3 months later: ransomware via RDP brute force Root cause: unauthorized firewall change Scenario 2: Admin applies patch without testing Patch breaks authentication module All users locked out of production for 6 hours Root cause: no testing → no rollback plan Scenario 3: New feature deployed without change ticket Feature introduces SQL injection vulnerability Customer data exfiltrated Auditor asks: "Was this change reviewed for security?" Answer: No → compliance violation # With change management: All changes: documented → reviewed → approved → tested Security team: reviews every change for risk Rollback plan: defined BEFORE implementation Audit trail: complete record of who changed what when Compliance: SOX, PCI-DSS, HIPAA all require change controls

The Security Connection

Unauthorized Changes = Security Risk

Every change can introduce vulnerabilities, break controls, or create configuration drift. Change management ensures every modification is reviewed for security impact before going live.

Audit Trail = Compliance

SOX requires: IT general controls including change management. PCI-DSS: all changes to cardholder data systems tracked. HIPAA: audit controls for systems holding PHI. Change records = evidence for auditors.

Rollback = Resilience

Every change must have a rollback plan. If the change breaks something → revert to known-good state. Without rollback: bad change → extended outage → availability failure.

02Types of ChangesStandard · Normal · Emergency

Not all changes carry the same risk or require the same level of review. Change type determines the approval path and urgency of the process. Security+ expects you to know all three types and when each applies.

🟢 Low Risk
Standard Change
Pre-approved, routine changes that have been reviewed and approved in advance. The process and risk are well-understood. No need for individual change request review — the change type itself has been pre-authorized. Must be performed exactly as defined — any deviation requires a new change request. Examples: weekly patching cycle per approved patch policy, password resets for users, adding a user to a pre-approved role, routine scheduled maintenance, deploying from an approved standard configuration template.
Pre-approved: Change Advisory Board already approved this TYPE of change Process: documented procedure exists → follow exactly Deviation: ANY deviation → escalate to Normal change Risk: LOW — well-understood, tested, documented Audit: still logged — standard change ≠ untracked change Examples: weekly Windows patches, nightly backup jobs, routine firewall rule template changes
🔵 Normal
Normal Change
Any change that is not pre-approved or an emergency. Follows the full change management lifecycle: request → risk assessment → CAB review → approval → implementation → review. Most common change type in enterprise environments. Timeline: typically reviewed at weekly CAB meetings — plan changes in advance. Security team MUST review all normal changes affecting security controls, network topology, or access management.
Timeline: request → CAB review (typically weekly) → approved → scheduled Risk: MEDIUM — requires assessment and review CAB: Change Advisory Board reviews and approves Examples: new application deployment, network topology change, new firewall rules, major OS upgrades, new user access model, security tool deployment Security review: required for anything touching: - Firewall / network security - Authentication systems - Access control modifications - Security monitoring tools
🔴 Urgent
Emergency Change
Changes required immediately to resolve a critical issue — a production outage, active security incident, or critical vulnerability being actively exploited. Expedited approval — Emergency CAB (ECAB) or designated approvers on call. MUST be documented and reviewed AFTER implementation (post-implementation review). Not an excuse to bypass all controls — still requires some authorization. Security patches for actively exploited CVEs (CISA KEV) qualify as emergency changes.
Triggers: active exploit, production down, critical security incident Process: get verbal/expedited approval → implement → document after ECAB: Emergency Change Advisory Board — subset of CAB available 24/7 for emergency approvals Post-implementation review: REQUIRED after emergency changes Documentation: retroactive but MUST be completed Risk: HIGH — less testing, expedited review, pressure to act fast Examples: hotfix for actively exploited zero-day, blocking attacker IP during incident, emergency password reset after compromise, taking system offline during active attack Risk: emergency changes most likely to introduce new problems Mitigation: get at least verbal approval, document everything

03Business ProcessesRequest → Review → Approve → Implement → Close

The change management business process is a structured lifecycle ensuring every change is properly requested, reviewed, approved, implemented, and documented before reaching production systems.

The Change Lifecycle

1

RFC submitted — describes change, reason, scope, affected systems
›
2

Risk, impact, security review — dependencies, rollback plan defined
›
3
Approve
CAB reviews — approved, rejected, or returned for more info
›
4
Test
Implemented in test environment — security validation, peer review
›
5
Implement
Deployed to production during approved change window
›
6
Review
Post-implementation review — did it work? any issues? lessons learned

Key Process Components

📋
Request for Change (RFC)
The formal document initiating a change. Contains: change description, business justification, scope (what systems affected), risk assessment, rollback plan, implementation steps, testing plan, who is responsible, proposed change window. Without an RFC, no change should be made to production. The RFC is the paper trail that satisfies auditors.
👥
Change Advisory Board (CAB)
Cross-functional committee that reviews and approves Normal changes. Membership: IT operations, security, application owners, business stakeholders, CISO representative. Meets: typically weekly — all changes requiring approval submitted before deadline. Security's role: review all changes for security impact. Any change touching security controls must have explicit security team sign-off. CAB can: approve, reject, request more information, or defer to next meeting.
🕐
Change Window / Maintenance Window
Pre-approved time period when changes may be implemented. Usually: low-traffic hours (nights, weekends), when business impact is minimal. Defined in advance: users notified, support staff available. Emergency changes: may bypass standard window but still need approval. Blackout periods: certain times when NO changes allowed (quarter-end close, major product launches, audit periods). Change windows are agreed with business stakeholders — not unilaterally decided by IT.
🔄
Rollback Plan — Critical Security Requirement
Every change must have a documented rollback procedure: how to revert to the previous state if the change fails or introduces problems. Without rollback: bad change → extended outage → availability failure. Security implication: a failed security control change without rollback could leave systems unprotected. Rollback must be tested too — a rollback that doesn't work is useless. ITIL standard: no change approved without defined rollback.
🧪
Testing Requirements
Changes must be tested in a non-production environment before production deployment. Test environment should mirror production as closely as possible. Security testing: does the change introduce new vulnerabilities? Does it break existing security controls? User acceptance testing (UAT): does it work as intended? Peer review: second set of eyes — especially for security-relevant changes. Testing evidence must be documented and attached to the RFC.

Separation of Duties in Change Management

# Who requests vs who approves vs who implements: Requester: The person/team proposing the change Cannot approve their own change Approver: CAB or designated authority Independent from the requester Cannot be the same person as implementer for critical changes Implementer: The technical person making the change Works during change window Has pre-approved steps to follow exactly Reviewer: Post-implementation reviewer Verifies change was successful Documents lessons learned # Why separation matters for security: Developer submits RFC → security reviews → CAB approves Developer CANNOT: approve their own RFC Admin CANNOT: make a change and mark it approved This prevents: insider threat, accidental errors going unchecked circumventing security review # SOX requirement: separation of duties in change management Developer writes code ≠ person who deploys to production

# Network team submits RFC for "temporary" firewall rule change: RFC submitted: Description: "Open port 8080 on external firewall to allow new vendor tool access during testing" Risk assessment: "Low — vendor testing only" Rollback: "Remove rule when testing complete" Change window: "Any time this week" CAB Review — Security raises concerns: Port 8080: unencrypted HTTP → data exposure risk "Any time this week": no defined end date → rule might stay forever No MFA: vendor accessing internal systems without MFA? No IP restriction: rule allows ANY source to reach port 8080? CAB Decision: REJECTED — returned for revision Required changes: 1. Restrict source IP to vendor's known IP range 2. Define hard cutoff date: rule removed by Friday 3. Require HTTPS (8443) not HTTP (8080) 4. Confirm vendor uses MFA for their side Revised RFC: approved with conditions Monitoring: IDS alert if unexpected sources use the rule Automatic: firewall automation removes rule on Friday Result: change made securely with controls ✓ Without CAB: insecure rule → potential breach vector ✗

04Technical ImplicationsImpact · Dependencies · Downtime

Every change has technical implications that must be assessed before implementation. Security must evaluate: what does this change break, what new risks does it introduce, and what dependencies could be affected?

Allow Lists and Deny Lists Impact

Allow List Changes

Adding to allow list (whitelist): explicitly permits new traffic, software, or access. Change implications: did we verify this is legitimate? Could attacker use this path? Scope: is it too broad? IP range vs single IP? Time-bound or permanent? Removing from allow list: may break legitimate functionality — test first. Example: adding a new vendor IP to firewall allow list → verify it's the right IP, restrict to needed ports only.

Deny List Changes

Adding to deny list (blacklist): blocks new traffic, software, or access. Change implications: will this block legitimate users? False positive risk? Is the block specific enough? Removing from deny list: why was it blocked? Has the risk changed? Example: adding a malicious IP to firewall deny list → will this IP also be used by legitimate traffic? CDNs share IPs — blanket block could break things.

Key Technical Considerations for Every Change

# Technical assessment checklist for any change: 1. Impact Assessment: What systems are directly affected? What downstream systems depend on what's changing? What is the blast radius if this change fails? Business impact: revenue? compliance? safety? 2. Dependencies: What does this change depend on? (must be in place first) What depends on what's changing? (will break after) Dependency mapping: document the chain Example: changing AD password policy Dependencies: all apps using AD auth affected some apps may not support longer passwords cached credentials on endpoints 3. Downtime Assessment: Will this change require service interruption? How long? Planned maintenance window sufficient? Can it be done without downtime? (rolling update?) Notification: users + stakeholders informed in advance? 4. Security Impact: Does this change open new attack surface? Does it affect authentication or authorization? Does it change logging or monitoring coverage? Does it affect data classification or protection? Does it bypass any existing security controls? 5. Restart Requirements: Will services need to restart? (service interruption) Will systems need to reboot? (longer downtime) What is the restart order? (dependencies!) Staged restart: can components be restarted individually? 6. Legacy System Considerations: Will this change affect legacy systems? Legacy: may not support new protocols or configurations Test thoroughly: legacy systems often undocumented Compensating controls: if legacy can't adapt to the change

Configuration Changes — Security Specific

Firewall Rule Changes

Every firewall rule change is a potential security vulnerability. Questions: Is the rule too permissive? Source IP restricted or any? Port specific or wide range? Protocol-specific? Time-limited or permanent? Who requested it — business justification? Will it be reviewed in 90 days? Firewall rules accumulate — periodic review required.

Operating System Updates

OS patches fix vulnerabilities but can break application compatibility. Test before production: especially for critical systems. Patch grouping: test batch before mass deployment. Rollback: OS patch rollback more complex than application. Timing: patch Tuesday for Windows — coordinate with change window. Zero-day patches: may require emergency change process.

Application Updates

New versions may change APIs, behavior, or security model. Test in staging: functional testing + security testing (DAST). Dependency check: does the new version require different libraries? Certificate changes: does the update require new cert infrastructure? Configuration migration: settings may not carry over automatically.

Network Topology Changes

Reconfiguring network segments, adding VLANs, changing routing. Security review: does this create unintended connectivity? ACL review: do existing ACLs still work correctly after topology change? Monitoring: will SIEM/IDS still capture traffic on new paths? Segmentation: does the change maintain or compromise network segmentation?

Downtime Windows — Planning

# Planning downtime for security changes: Communicate early: Users: advance notice (minimum 48-72 hours for planned) Stakeholders: business impact communicated upfront Customers: external notification for significant outages Timing strategy: Business hours: never for production changes Low traffic: typically 2am-5am local time Low risk period: avoid quarter-end, major releases Blackout periods: documented — no changes during these Maintenance window duration: Include: implementation time + testing time + buffer Example: "30-min patch → 2-hour window" (buffer for issues) If not done in window: rollback and reschedule Communication plan: Before: "Maintenance scheduled for Saturday 02:00-04:00" During: status updates every 30 minutes if extended After: "Maintenance complete, all systems normal" OR "Issue encountered, maintenance extended until X" # Emergency vs planned — different notification expectations Emergency: notify as quickly as possible → start → update Planned: notify well in advance → implement at agreed window

// Exam Keywords

Allow list →Explicitly permits — adding requires security review — scope too broad = risk
Deny list →Explicitly blocks — removing requires justification — shared IPs risk blocking legitimate traffic
Dependencies →Map what the change affects and what depends on it — downstream systems
Restart requirements →Service/system restart causes downtime — plan and communicate ahead
Legacy systems →May not support new configurations — test separately — compensating controls if needed
Change window →Off-hours, agreed with business, sufficient duration including buffer and rollback time
05Allow Lists & Deny ListsWhitelist · Blacklist · Application Control

Allow lists and deny lists are fundamental security control mechanisms that explicitly define what is permitted or prohibited. Any change to these lists must go through change management — adding or removing entries has direct security implications.

Allow List (Whitelist) — Explicitly Permit

# Allow list philosophy: DENY by default, permit explicitly Default: DENY everything Allow list entry: "only this specific thing is permitted" Everything NOT on the list: automatically denied Most secure posture — attacker must get on the list to succeed Applications of allow lists in change management: Application control: only approved apps may run (AppLocker/WDAC) Firewall: only listed IPs/ports permitted Email: only approved senders bypass spam filter Software installation: only approved software may be installed Script execution: only signed/approved scripts run (PowerShell) Certificate authorities: only listed CAs are trusted # Change implications for allow lists: Adding entry: security review — is this legitimate? too broad? "Allow ALL IPs to port 443" vs "Allow 203.0.113.0/24 to port 443" More specific = more secure Removing entry: test first — will this break legitimate functionality? "Remove vendor IP from allow list" — is vendor still in use? # Allow list maintenance = change management: Add vendor IP → RFC → security review → CAB → approved Remove expired rule → RFC → test impact → CAB → remove

Deny List (Blacklist) — Explicitly Block

# Deny list philosophy: ALLOW by default, block explicitly Default: PERMIT everything Deny list entry: "this specific thing is blocked" Everything NOT on the list: automatically allowed Less secure posture — attacker just needs to avoid listed items Applications of deny lists in change management: Firewall: known malicious IPs blocked Email gateway: known spam domains/IPs blocked DNS filtering: malicious domains sinkholed AV/EDR: known malware signatures blocked Web proxy: known malicious URL categories blocked # Change implications for deny lists: Adding entry: is this specific enough? Shared CDN IPs? "Block 192.168.0.0/16" — accidentally blocks internal subnets "Block attacker.evil.com" → attacker just changes domain Removing entry: why was it added? Has the threat passed? Removing expired block → re-exposes to previous threat # Deny list weakness: Unknown threats not on list = bypass Attackers know what's on common deny lists and avoid it That's why allow list (default deny) is more secure

Allow List vs Deny List — Security Comparison

Allow List (Whitelist)
Default: DENY
Security: Highest — unknown = blocked
Usability: Harder to manage — must approve everything
Use when: High security needed — known set of approved items
Failure mode: Legitimate item not on list = blocked (inconvenience)
Attacker: Must get approved to succeed — much harder
Deny List (Blacklist)
Default: ALLOW
Security: Lower — unknown = allowed
Usability: Easier — only maintain blocked items
Use when: Large unknown set — can't pre-approve everything
Failure mode: New threat not yet listed = allowed through
Attacker: Just needs to avoid listed items — easier

Application Allow Listing

🛡️
Application Control — AppLocker / WDAC
Allow listing for applications — only approved, signed software may execute. Microsoft: AppLocker (Group Policy) and WDAC (Windows Defender Application Control). Any change to the approved application list = formal change request. Adding new software: RFC → security review (is it from trusted publisher? signed?) → test in non-prod → CAB approval → deploy policy update. Emergency software needed: ECAB approval → add to list → implement → document. Blocks: ransomware (unsigned), unknown malware, unauthorized tools.

Deny list →Default allow — explicit block — less secure — only known bad blocked
Allow list vs deny list →Allow list more secure (default deny) but harder to manage — deny list easier but misses unknowns
Application allow listing →Only approved signed apps run — blocks ransomware/unknown malware — AppLocker/WDAC
List changes need CM →Adding or removing from allow/deny list = change request — security must review
06Restricted Activities & DowntimeFreeze · Blackout · Impact

Change management defines periods and circumstances where changes are prohibited or require elevated approval. Understanding these restrictions is critical for planning security changes appropriately.

Change Freeze / Blackout Periods

# Periods when NO production changes are permitted: Financial blackout periods: Quarter-end close: last 2 weeks of fiscal quarter Year-end close: December 15 - January 5 (example) Reason: financial systems must be stable for reporting SOX implication: changes during close could affect financial integrity Product/business blackout periods: Major product launches: 2 weeks before/after Black Friday / peak shopping season: November Tax season: February-April for financial services Enrollment period: healthcare payers Reason: any outage = massive revenue/reputation impact Security blackout scenarios: Active security audit: no changes that could affect audit scope PCI compliance assessment window: freeze scoped systems Regulatory inspection period: systems locked down # During freeze: Emergency changes ONLY with: ECAB or CISO approval Documented urgent business or security justification Post-implementation review required Notification to compliance/audit team # Planning security work around freezes: Security patches during freeze: only emergency/critical Log4Shell during Christmas freeze: emergency = patch it anyway Routine security updates: defer until after freeze period

Downtime Impact Analysis

High-Availability Systems

99.99% uptime requirement — 52 minutes downtime/year maximum. Even planned maintenance must be within SLA. Approach: rolling updates — one node at a time. Blue/green deployment: run two environments, switch traffic. Zero-downtime migration: database schema changes via expand-contract pattern. Change process: extensive testing, gradual rollout, instant rollback capability.

Downtime Notification Requirements

Internal users: email + ticket minimum 48-72 hours before. Customer-facing systems: public maintenance page, email notification, status page update. Regulatory systems (banking, healthcare): may require advance regulator notification for significant changes. SLA: customer agreements may specify advance notice periods — check contract before scheduling.

Service Restarts and System Reboots

# Restart vs Reboot — different implications: Service restart: Only the application service restarts Typically: 30 seconds to 3 minutes downtime Users: brief interruption, may auto-reconnect Risk: service fails to restart → escalate Example: restarting Apache after config change System reboot: Entire server reboots — all services stop and start Typically: 2-10 minutes downtime (longer for large servers) Required: kernel patches, driver updates, BIOS changes Risk: hardware failure during boot, service dependency order Example: Windows Update requiring restart # Restart order matters — dependency chain: Database server → must be running before app servers start App servers → must be running before load balancer routes traffic Correct order: DB → App server 1 → App server 2 → LB health check Wrong order: start app servers before DB → errors → rollback # Security implication of restarts: Patch without reboot: patch may not be active until reboot vulnerability still exploitable Reboot required: schedule appropriately, don't skip "We'll reboot next quarter" → months of exposure

07Documentation RequirementsDiagrams · Policies · Procedures · Tickets

Change management generates and requires comprehensive documentation at every step. This documentation is the evidence that satisfies auditors, enables troubleshooting, ensures knowledge transfer, and supports rollback operations.

Updating Documentation After Changes

⚠ Critical: Documentation Must Stay Current

Outdated documentation is almost as dangerous as no documentation. If a network diagram shows an old topology, incident responders make wrong decisions. If the firewall policy doc doesn't reflect current rules, security gaps go unnoticed. Every change that modifies a documented asset must update that documentation as part of the change itself — not "we'll do it later."

Key Documentation Types

🗺️
Network Diagrams
Visual representation of network topology — switches, routers, firewalls, servers, VLANs, network zones, IP address ranges. Must be updated whenever: new devices added, network segments change, firewall placement changes, VPN tunnels added. Security use: incident responders use diagrams to understand attack paths. Auditors verify architecture against diagrams. Outdated diagram = wrong incident response decisions. Logical vs physical: both maintained. Stored: configuration management database (CMDB) or Visio/Lucidchart with version history.
📋
Policies and Procedures
Security policies define requirements — must be updated when: regulations change, technology changes, incident analysis reveals gaps. Standard Operating Procedures (SOPs): step-by-step instructions for operational tasks — must match current system configuration. Change management itself has a policy — the policy defines the process. Version control: policies stored in document management system with version history, approval signatures, review dates. If a patch changes how authentication works, the authentication procedure must be updated.
🎫
Change Records / Tickets
Every change generates a change record — the authoritative audit trail. Contains: RFC details, risk assessment, CAB decision with rationale, approver names and timestamps, implementation steps taken, testing results, post-implementation review, any issues encountered, rollback actions if needed, final change status (success/fail/partial). ServiceNow, Jira, BMC Remedy: ITSM platforms that manage change records. Must be retained: SOX requires 7 years, PCI-DSS requires 1 year minimum. Change records link to: incident records if change caused an outage, problem records if root cause analysis needed.
⚙️
Configuration Documentation
Baseline configurations documented before changes — enables rollback and drift detection. Firewall rules: every rule documented with business justification and review date. Server builds: documented standard build configurations (hardening standards applied). CMDB: Configuration Management Database — authoritative record of all CIs (configuration items) and their relationships. Change: CMDB updated after every successful change. Drift: actual configuration diverges from documented baseline = unauthorized change or configuration drift detected by CSPM/SCAP tools.
📑
Security Documentation Specific Requirements
Risk assessments: documented before change, updated after. Security exceptions: formal documentation of accepted risks (time-limited, approved by CISO). Compensating controls: documented when primary control can't be implemented. Audit trails: who approved what, when — immutable log. Test results: penetration test findings for changes to security-relevant systems. SLA documentation: security response times, monitoring coverage — updated when monitoring changes.

Documentation in the Change Lifecycle

# Documentation produced at each change stage: At Request: RFC document (change description, justification, scope) Risk assessment (likelihood × impact rating) Current state documentation (baseline before change) Proposed new state documentation (what it will look like after) At Review: CAB meeting minutes (who attended, discussion, decision) Security review findings and approval/rejection Stakeholder approval signatures At Test: Test plan document Test results (pass/fail for each test case) Security scan results (before and after) Peer review sign-off At Implementation: Change log: exact steps taken, timestamps, who executed Screenshots or output captures of key commands Any deviations from plan documented immediately After Implementation: Updated: network diagrams, runbooks, CMDB Post-implementation review: did it work? any issues? Lessons learned: what to do differently next time Updated baseline: new documented configuration state

08Version ControlGit · Tracking · Rollback · History

Version control is the systematic tracking of changes to files and configurations over time. It provides complete history of who changed what and when, enables rollback to any previous state, and supports collaboration with conflict detection.

Version Control Systems

Git

Distributed VCS — most widely used. Every developer has full repository copy. Branching: work on features without affecting main. Platforms: GitHub, GitLab, Bitbucket, Azure DevOps. For security: stores firewall configs, IaC, security policies, scripts — everything as code with full history.

ITSM Versioning

ServiceNow, Jira: version control for change records and policy documents. Not code-level but document-level. Every edit to a policy: logged with who, when, what changed. Approval workflows: version must be approved before active. Audit trail: complete history for compliance.

Configuration Version Control

Ansible, Terraform, Puppet, Chef: infrastructure-as-code stored in Git. Every infrastructure change: committed with message and author. Rollback: revert commit to previous state. Drift detection: current config vs last committed version. GitOps: Git as single source of truth for infrastructure state.

Version Control for Security Configurations

# Treating security configs like code — benefits: Firewall rules in Git: git log shows every rule change with author and timestamp git diff shows exactly what changed between versions git blame shows who last modified each rule git revert rolls back a bad change in seconds Pull request: security team reviews rule changes before merge Example workflow for firewall change: 1. Create branch: feature/add-vendor-access 2. Edit firewall config: add rule 3. Run automated tests: syntax check, security scan 4. Open pull request → security team reviews 5. Review approved → merge to main 6. CI/CD pipeline: automatically deploys to firewall 7. Post-deploy: automated verification test 8. History: complete trail in Git forever Result: every firewall change is reviewed, tested, auditable Without this: "who added this rule? nobody knows"

Versioning Concepts

Semantic Versioning (SemVer)

Standard versioning format: MAJOR.MINOR.PATCH (e.g., 2.4.1). MAJOR: breaking changes — incompatible with previous. MINOR: new features — backward compatible. PATCH: bug fixes only. Security implications: PATCH releases often contain security fixes — update quickly. MAJOR: may require config/code changes — test carefully. Version pinning: lock to specific version to prevent unexpected updates.

v2.4.1 → v2.4.2 = patch (bug fix — security patch) v2.4.1 → v2.5.0 = minor (new feature — test first) v2.4.1 → v3.0.0 = major (breaking — extensive testing)

Change Rollback via Version Control

Every commit is a rollback point. Bad change deployed → git revert commit → previous state restored. Rollback time: seconds to minutes (vs hours without version control). Test rollback: verify rollback works BEFORE implementing the change. Immutability: committed history cannot be rewritten (git rebase harmful — destroys audit trail).

# Bad firewall change deployed at 14:30: git log → identify commit hash abc1234 git revert abc1234 → creates new commit undoing change git push → deploy rollback # Firewall back to previous state: 2 minutes ✓

Version Control Security Practices

# Security requirements for version control systems: Access control: Not everyone can push to main branch Branch protection: require pull request + approval to merge Signed commits: GPG signatures verify committer identity MFA: required on GitHub/GitLab accounts Secrets management: NEVER commit: passwords, API keys, certificates .gitignore: exclude credential files Pre-commit hooks: scan for secrets before commit Secret scanning: GitHub advanced security detects secrets Committed secret = compromised — change it immediately Audit trail: Git history is immutable (force push = audit trail destruction) Disable force push on protected branches All commits: authenticated — who made this change Repository security: Private repos: for internal configs and code Code review: all changes reviewed before merge Dependency scanning: check imported libraries for CVEs SAST: static analysis in CI/CD pipeline # Version control satisfies compliance requirements: SOX: change management evidence — git log shows what changed when PCI-DSS: all changes to cardholder data systems tracked HIPAA: audit controls — version history shows who modified what

Version Control Lifecycle — Security Workflow

# Complete secure development/config change workflow: Plan → RFC submitted in ITSM system (ServiceNow) Develop: Developer creates feature branch in Git Makes changes → commits with descriptive message Commit message: "CHG-1234: Add vendor firewall rule per RFC" Review: Pull request opened Security team: code review → looks for vulnerabilities Automated: SAST scan, secrets scan, dependency check All checks pass → security approves pull request Test: Merge to staging branch → CI/CD deploys to test environment Automated tests run → functional + security tests Manual testing if required → results documented Approve: CAB reviews change + PR approval as evidence of testing CAB approves → merge to production branch permitted Deploy: Production merge → CI/CD auto-deploys during change window Post-deploy verification → automated smoke tests Document: Git commit = automatic documentation (who, what, when) ITSM ticket updated → link to Git commit hash CMDB updated → new configuration baseline recorded
