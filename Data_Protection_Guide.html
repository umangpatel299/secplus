Data Protection

// CompTIA Security+ SY0-701 — Classification · States · Protection Methods

01 Data Classification — Corporate Foundation

Data classification assigns a sensitivity label to data so the organization knows how to protect it. Different data needs different levels of protection — you can't treat everything the same way.

🔴 Critical / Restricted
What it is Most sensitive data — exposure causes severe, irreversible harm. Highest protection required at all times.
Examples Encryption keys, source code, biometric data, nuclear facility plans, trade secrets, authentication credentials
🟠 Confidential / Private
What it is Sensitive business data — exposure damages company competitively or legally. Shared only on need-to-know basis.
Examples Customer PII, financial records, employee salaries, M&A plans, HR records, medical records, legal strategy
🔵 Internal / Sensitive
What it is Internal use only — not for public. Exposure is embarrassing or mildly harmful but not catastrophic.
Examples Internal policies, org charts, meeting notes, project plans, internal emails, staff directories
🟢 Public
What it is Intended for public consumption. No harm if disclosed. Minimal protection needed.
Examples Marketing materials, press releases, product brochures, public website content, published annual reports
// SCENARIO — Hospital Data Classification

A hospital classifies all its data. When a nurse opens a patient record, the system enforces controls based on the classification automatically.

# Hospital data classification in action Patient medical record: RESTRICTED → Encrypted at rest + in transit → Access log required for every view → Cannot be emailed without encryption → MFA required to access Employee salary data: CONFIDENTIAL → HR dept access only → Cannot be printed without approval → Watermarked on every page Internal shift schedule: INTERNAL → All staff can view → Cannot share externally → No special encryption required Hospital location/hours: PUBLIC → Posted on website → No access controls → Anyone can see it

// EXAM KEYWORDS

Critical →Highest protection — encryption keys, credentials, source code
Confidential →PII, financial, medical, HR — need-to-know basis
Internal →Employees only — org charts, policies, meeting notes
Public →Anyone can see — press releases, marketing, website
Purpose →Different classifications = different controls = right protection for right data
02 Government / Military Classification Government

Governments use a formal, legally defined classification system with strict rules about who can access data and how it must be handled. The US government system is the most commonly tested.

⬛ TOP SECRET
Definition Unauthorized disclosure would cause EXCEPTIONALLY GRAVE damage to national security. Highest classification level.
Examples + Access Intelligence sources/methods, nuclear weapons plans, covert operations. Requires TS clearance + need-to-know + SCI access.
🔴 SECRET
Definition Unauthorized disclosure would cause SERIOUS damage to national security. Second highest level.
Examples + Access Military plans, weapons systems capabilities, foreign government intelligence. Requires Secret clearance + need-to-know.
🟡 CONFIDENTIAL
Definition Unauthorized disclosure would cause DAMAGE to national security. Lowest classified level.
Examples + Access Troop movement data, procurement plans, vulnerabilities in systems. Requires Confidential clearance.
🟢 UNCLASSIFIED
Definition No classification — but may still have handling restrictions (CUI — Controlled Unclassified Information).
Examples + Access Public government reports, general policy documents. CUI = sensitive but unclassified — has specific handling rules.
// SCENARIO — Intelligence Analyst Access
# Analyst with SECRET clearance tries to access: Document: UNCLASSIFIED — Army recruitment brochure Access: GRANTED ✓ — clearance exceeds requirement Document: CONFIDENTIAL — troop movement report Access: GRANTED ✓ — SECRET clears CONFIDENTIAL Document: SECRET — foreign intelligence report Access: GRANTED ✓ — matches clearance level BUT: also needs NEED-TO-KNOW for this specific document Document: TOP SECRET — covert operation plan Access: DENIED ✗ — insufficient clearance Even if analyst "needs to know" — clearance insufficient # Key rule: clearance is necessary but NOT sufficient # Must ALSO have need-to-know for the specific document

Special Handling — SCI and SAP

SCI — Sensitive Compartmented Info

Above Top Secret. Intelligence sources and methods. Access requires Top Secret clearance PLUS special SCI access approval. Stored in SCIFs (Sensitive Compartmented Information Facilities).

SAP — Special Access Program

Even more restricted than SCI. Black programs — their existence may be classified. Very small number of people with access. Includes stealth aircraft programs, advanced weapons.

CUI — Controlled Unclassified Info

Sensitive but NOT classified. Examples: PII in government systems, law enforcement data, financial regulation data. Has specific handling requirements but no classification marking.

SCIF — Secure Facility

Room or facility where classified info processed. No phones allowed. Shielded walls (TEMPEST). Access strictly controlled. Where SCI and TS documents are read and worked on.

// EXAM KEYWORDS

Top Secret →Exceptionally grave damage — highest level — TS clearance required
Secret →Serious damage — military plans, weapons systems
Confidential →Damage — lowest classified level
CUI →Controlled Unclassified — sensitive but not classified, still has rules
Clearance ≠ Access →Must have clearance AND need-to-know — both required
SCIF →Secure facility for classified work — no phones, shielded
03 Three States of Data Must Know All Three

Data exists in three states — and each state has different vulnerabilities and requires different protection. Knowing which state data is in tells you which controls to apply.

💾
Data at Rest
Stored on a device — not moving
Where:
Hard drives, SSDs, databases, backup tapes, USB drives, cloud storage buckets, file servers
Primary threat:
Physical theft of device or storage media
Protection:
Encryption (AES-256), access controls, physical security, BitLocker/FileVault
📡
Data in Transit
Moving across a network
Where:
Emails, web traffic, file transfers, API calls, VPN tunnels, streaming, messaging apps
Primary threat:
Man-in-the-Middle interception — someone reading traffic between A and B
Protection:
TLS/HTTPS, VPN, IPSec, SSH, SFTP, encrypted email (S/MIME)
⚙️
Data in Use
Actively being processed in RAM
Where:
CPU registers, RAM, cache memory, application memory, virtual machines, browser tabs
Primary threat:
Memory scraping, cold boot attacks, process injection, screen capture — must be DECRYPTED to use
Protection:
Secure enclaves (Intel SGX), memory encryption, access controls, endpoint protection
// SCENARIO — Customer Database Journey
# Follow customer data through all three states: AT REST: Customer records stored in database Location: MySQL database on server hard drive State: Data at Rest Threat: Server stolen → data exposed Protection: AES-256 encryption on disk Even if drive stolen → unreadable without key IN TRANSIT: Customer places order online Data travels: Browser → HTTPS → Web Server → DB State: Data in Transit Threat: MitM intercepts credit card number Protection: TLS 1.3 encrypts connection Intercepted traffic = unreadable gibberish IN USE: Server processes payment Data loaded into RAM for processing State: Data in Use Threat: RAM scraping malware (like Target POS attack!) Data decrypted in memory → malware reads it Protection: Secure enclave, memory encryption Limited time in memory, clear after use # All three states need protection simultaneously # Weakest state = weakest link in data security

// EXAM KEYWORDS

At Rest →Stored on device — protect with encryption (AES-256, BitLocker)
In Transit →Moving across network — protect with TLS, VPN, IPSec
In Use →Active in RAM — hardest to protect — secure enclaves, memory encryption
Hardest state →Data in Use — must be decrypted to process — vulnerable in memory
Target breach →BlackPOS attacked Data in Use — scraped RAM of POS terminals
04 Encryption — Primary Data Protection Most Important

Encryption converts readable data (plaintext) into unreadable ciphertext. Even if data is stolen, encrypted data is useless without the decryption key. Primary protection for all three data states.

💾 Encryption at Rest

Full Disk Encryption (FDE): BitLocker (Windows), FileVault (macOS), LUKS (Linux). Encrypts entire drive — if laptop stolen, data unreadable.

Database Encryption: Transparent Data Encryption (TDE) — encrypts database files at OS level. Application reads normally — files on disk encrypted.

File-level encryption: Encrypt individual files — EFS (Windows), GPG, PGP. More granular than FDE.

📡 Encryption in Transit

TLS/HTTPS: Port 443. Encrypts all web traffic. TLS 1.3 = current standard. Never use SSL or TLS 1.0/1.1.

VPN / IPSec: Encrypts all traffic in a tunnel. IKEv2/IPSec = most secure VPN protocol.

SSH: Encrypted remote access. Replaces Telnet. Port 22. Also used for SFTP.

S/MIME / PGP: End-to-end email encryption. Content encrypted before sending.

⚙️ Encryption in Use

Intel SGX: Software Guard Extensions. Creates secure encrypted enclave in RAM. Code and data inside encrypted even from OS.

AMD SME/SEV: Secure Memory Encryption. Encrypts RAM contents. Protects VMs from hypervisor.

Homomorphic Encryption: Compute on encrypted data without decrypting. Still emerging technology.

🔑 Key Management

HSM (Hardware Security Module): Dedicated hardware that stores and manages encryption keys. Keys never leave HSM in plaintext.

Key rotation: Regularly replace encryption keys. Old keys retired. Limits damage if key compromised.

Key escrow: Copies of keys held by trusted third party. Recovery if key lost.

// EXAM KEYWORDS

TLS 1.3 →Current encryption standard for data in transit — HTTPS
AES-256 →Gold standard symmetric encryption algorithm — used for data at rest

TDE →Transparent Data Encryption — database files encrypted at OS level
Intel SGX →Secure enclave — protects data in use / in memory
05 DLP — Data Loss Prevention Prevent Exfiltration

DLP monitors and controls data movement to prevent unauthorized data leaving the organization. It watches what data is being copied, emailed, uploaded, or printed — and stops policy violations.

How DLP Works

Content Inspection

Reads the actual content of files and communications. Recognizes patterns: credit card numbers (16 digits), SSNs (XXX-XX-XXXX), medical record numbers, classified keywords.

Policy Enforcement

When sensitive data detected: BLOCK the transfer, QUARANTINE the file, ALERT the security team, ENCRYPT before allowing, LOG the attempt for audit.

Network DLP

Monitors network traffic — email, web uploads, FTP. Sits inline on network. Can block emails containing credit card numbers before they leave the company.

Endpoint DLP

Software on each device. Controls USB drives, printing, clipboard, screenshots. Blocks copying customer database to USB. Prevents screen capture of sensitive data.

Cloud DLP

Monitors cloud services — Microsoft 365, Google Drive, Salesforce. Prevents employees uploading sensitive files to personal cloud accounts. CASB integration.

Storage DLP

Scans files already stored — finds sensitive data that shouldn't be where it is. "Discovery mode" — locates PII in wrong location without blocking.

// SCENARIO — Healthcare DLP in Action
# Hospital DLP system catches multiple violations: Attempt 1: Nurse emails patient list home Email to: [email protected] Attachment: patient-list.xlsx DLP scans: 487 patient names + DOBs detected DLP action: BLOCKED ✗ — HIPAA violation prevented Alert: sent to security team + nurse's manager Attempt 2: Employee copies to USB drive File: customer_database_backup.sql USB inserted into workstation DLP endpoint: database backup detected on USB transfer DLP action: BLOCKED ✗ USB port write access denied Attempt 3: Accidental — doctor drafts email with chart Email to: [email protected] Attachment: patient-chart.pdf (INTERNAL recipient) DLP scans: PHI detected but recipient is internal DLP action: ALLOWED ✓ — internal healthcare recipient + logged for audit trail Attempt 4: Ransomware exfiltration Malware: bulk downloading 50,000 files rapidly DLP: anomalous bulk file movement detected DLP action: BLOCKED ✗ + ALERT ✗ Process terminated, IR team notified

// EXAM KEYWORDS

DLP purpose →Prevent sensitive data leaving organization unauthorized
Network DLP →Monitors email and web traffic — blocks before leaving network
Endpoint DLP →On each device — controls USB, print, clipboard, screenshots
Cloud DLP →Monitors cloud services — works with CASB
Actions →Block, quarantine, alert, encrypt, log — depends on policy
Scenario tip →"Employee emailed customer list" → DLP should have caught this
06 Access Controls for Data Who Can See What

Access controls ensure only authorized people can access specific data — and only in the way they're permitted. Tied directly to data classification.

Access Control Models — Quick Reference

ModelHow Access DecidedWho ControlsUse Case
DAC — Discretionary Resource OWNER decides Data owner Standard file systems — you share your own files
MAC — Mandatory Security LABELS (clearance levels) System enforces Military/government — clearance + need-to-know
RBAC — Role-Based User's JOB ROLE Administrator Most enterprises — Finance role → finance data only
ABAC — Attribute-Based Multiple ATTRIBUTES (dept + time + location) Policy engine Cloud/Zero Trust — most granular

Data-Specific Access Controls

Need-to-Know Principle

Access only to data REQUIRED for job function. Even with high clearance — can't access data outside their role. HR can't see source code. Developers can't see HR records.

Least Privilege

Minimum permissions necessary. Read-only access when read-only is enough. No admin when user access is sufficient. Reduces blast radius if account compromised.

Separation of Duties

No single person controls entire critical process. Finance: one person requests payment, another approves, another executes. Prevents fraud and errors.

Database Access Controls

Column-level security: hide SSN column from most users. Row-level security: HR staff only see their region's employees. View-based access: show summary not raw data.

Data Masking

Show partial data only: credit card 4321-XXXX-XXXX-1234. SSN XXX-XX-6789. Developers work with masked production data. Support agents see masked card numbers.

Tokenization

Replace sensitive data with non-sensitive token. Original: 4532-1234-5678-9012 → Token: TKN-8823-XKPL. Token maps to original in secure vault. Credit card processing.

// SCENARIO — Bank Access Controls
# Bank implements tiered data access by role: Bank Teller role: Can see: customer name, account balance summary Cannot see: full SSN (masked: XXX-XX-1234) Cannot see: other branch customers Cannot see: internal credit scores Action limit: transactions under $10,000 only Branch Manager role: Can see: all teller data + Can see: full customer profiles (own branch) Can see: internal credit scores Action limit: transactions under $100,000 Fraud Analyst role: Can see: transaction patterns across ALL accounts Cannot see: account holder PII (privacy wall) Can see: anomalous activity flags Read-only: no transaction capability Compliance Officer: Can see: audit logs for everything Cannot modify: any account or transaction Purpose: oversight only — cannot change data # Same database — different views per role # RBAC + column/row level security

// EXAM KEYWORDS

Need-to-know →Access only to data required for specific job function
Data masking →Show partial data — XXXX replacing sensitive digits
Tokenization →Replace real data with token — original in secure vault
MAC →Label-based — clearance level must match classification
RBAC →Role determines access — most common enterprise model
Masking vs Tokenization →Masking = partial display / Tokenization = full replacement with token
07 Other Data Protection Methods Complete Picture

Data Backup and Recovery

3-2-1 Backup Rule

3 copies of data, 2 different media types, 1 offsite. Modern addition: 1 offline/air-gapped (ransomware can't reach it), 0 errors (test restores regularly).

Immutable Backups

Backups that CANNOT be modified or deleted — even by admins. Ransomware tries to delete backups → refused. WORM storage (Write Once Read Many).

Data Integrity

Hashing

SHA-256 hash verifies data hasn't changed. File hash before = file hash after → untampered. Used for: file integrity, digital signatures, password storage.

Digital Signatures

Cryptographic proof data came from specific sender AND hasn't been modified. Non-repudiation — sender can't deny sending. Uses asymmetric cryptography (RSA/ECDSA).

Data Lifecycle Management

Data Retention

How long to keep data. Legal requirements: HIPAA = 6 years minimum. GDPR = only as long as necessary. PCI-DSS = cardholder data purged after transaction. Retention policy defines this.

Data Destruction / Sanitization

Secure deletion when no longer needed. Methods by sensitivity: Clearing (overwrite), Purging (degaussing), Destroying (physical shredding/incineration). Certificates of destruction for compliance.

// SCENARIO — Data Destruction Methods
# Different data requires different destruction: PUBLIC data on old hard drive: Method: Clearing — overwrite with zeros/ones Tool: DBAN (Darik's Boot and Nuke) Suitable for: reusing or selling drive CONFIDENTIAL data on SSD: Method: Purging — manufacturer secure erase SSD note: overwriting may not fully work (wear leveling) Use: ATA Secure Erase command instead Suitable for: decommissioned drives TOP SECRET data on any media: Method: Destroying Options: - Physical shredding (certified shredder) - Degaussing (magnetic destruction) - Incineration - Disintegration Certificate: Certificate of Destruction issued Suitable for: when drive must be disposed of # Simply deleting files ≠ data destruction # Deleted files recoverable with forensic tools # Even formatted drives recoverable!

Information Rights Management (IRM)

What Is IRM?

Technology that controls what recipients can DO with a document — even after they receive it. Controls: view, print, copy, forward, screenshot, edit, expiry date.

# IRM protected document: Document: Q4_Financial_Results.docx IRM policy: Print: DISABLED Copy text: DISABLED Forward: DISABLED Screenshot: DISABLED Expiry: Auto-expires in 30 days Watermark: Shows viewer's name on every page Access: Finance department only Even if forwarded to wrong person: → They cannot open without authorization → Document enforces its own controls

Steganography — Hiding Data in Data

Hiding secret data inside innocent-looking files. A photo may contain hidden text. An audio file may contain hidden data. Used for both watermarking and covert communication.

// EXAM KEYWORDS

Clearing →Overwrite data — suitable for reuse in same org
Purging →Degaussing — more thorough than clearing
Destroying →Physical shredding/incineration — most secure — TOP SECRET
IRM →Controls what recipients can DO with document — print, copy, forward
Immutable backup →Cannot be deleted even by admin — ransomware protection
SSD destruction →Overwriting may not work — use ATA Secure Erase or physical destruction
