ONPOn-Premises ArchitectureFull Control

On-premises (on-prem) means the organization owns and operates all hardware, software, and infrastructure within its own facilities. Physical servers, networking equipment, and storage are purchased, installed, and maintained by the org's own IT staff.

What On-Premises Looks Like

# Typical on-prem enterprise architecture: Physical data center: Servers: owned hardware in racks Storage: SAN/NAS arrays in the building Networking: owned switches, routers, firewalls Power/Cooling: UPS, generators, HVAC — org's responsibility Physical access: badge readers, cameras, locked cages Software: OS: licensed, installed on own hardware Applications: installed on own servers Databases: running on own hardware Directory: Active Directory on own Domain Controllers Security responsibility: 100% organization-owned Physical: organization secures the facility Network: organization manages all firewalls, IDS/IPS Compute: organization patches all OS and apps Data: organization protects and backs up data

On-Premises: Pros and Cons

Advantages

  • Full control over hardware and software
  • Data never leaves your facility
  • Customizable to exact requirements
  • No dependency on internet connectivity
  • Better for air-gapped environments
  • Compliance: some regulations mandate local data
  • Predictable costs (CapEx model)

Disadvantages

  • High upfront capital expense (CapEx)
  • Org responsible for ALL patching and maintenance
  • Physical disaster destroys everything at site
  • Scaling requires buying hardware in advance
  • Security talent burden — no shared expertise
  • Hardware refreshes every 3-5 years
  • DR requires second physical site

// Exam Keywords

On-premises →Org owns ALL hardware — full responsibility for physical + logical security
CapEx model →Large upfront capital expenditure — purchase hardware — depreciate over time
Data sovereignty →Data never leaves your facility — important for regulated industries
ONPOn-Premises Security ControlsDefend All Layers

Physical Security Controls

Physical Access

Badge readers, mantraps/airlocks, biometric locks for server room, CCTV, security guards, visitor logs. Physical = the first line of defense in on-prem.

Environmental Controls

UPS (uninterruptible power supply), generators, HVAC for cooling, fire suppression (clean agent — NOT water), humidity monitoring. Protect from: power failure, fire, flood, heat.

Hardware Security

Cable locks, rack locks, TPM chips for measured boot, BIOS/UEFI passwords, hardware asset tags, decommissioning procedure (drive destruction). Physical access to hardware = game over.

Network Security Controls

# Layered network security — on-prem model: PERIMETER: Firewall: filter ingress/egress IDS/IPS: detect/block threats at perimeter DMZ: semi-trusted zone for public services Web proxy: filter outbound web traffic INTERNAL NETWORK: Network segmentation: VLANs separating systems by function Internal firewall: control segment-to-segment traffic NAC (802.1X): verify device before network access Port security: limit which MACs can use a port ENDPOINT: EDR: behavioral detection on every endpoint Host firewall: Windows Defender Firewall / iptables FDE: Full Disk Encryption (BitLocker / LUKS) Patch management: WSUS / SCCM / Ansible IDENTITY: Active Directory: central identity — Kerberos auth MFA: required for privileged access + remote PAM: vault admin credentials

// Exam Keywords

DMZ →Demilitarized zone — semi-trusted buffer between internet and internal network
Defense in depth →Layered controls — perimeter + network + endpoint + identity — each layer adds protection
Air gap →Physically isolated network — no internet connection — highest on-prem security

CLDCloud Service ModelsIaaS · PaaS · SaaS

Cloud computing delivers on-demand IT resources over the internet. Three service models define how much the provider manages vs how much you manage. This directly determines your security responsibilities.

IaaS — Infrastructure as a Service

# IaaS: provider gives you hardware — you do the rest Provider manages: Physical datacenter, hardware, hypervisor You manage: OS, middleware, runtime, apps, data, security Examples: AWS EC2, Azure VMs, Google Compute Engine Security: YOU are responsible for: OS patching: your job App security: your job Network config: your job (security groups = your firewall) Identity: your job Data protection: your job Most control, most responsibility Use when: need OS-level customization, legacy app migration

PaaS — Platform as a Service

# PaaS: provider manages platform — you manage app + data Provider manages: Physical, hardware, OS, runtime, middleware You manage: Application code, data, user access Examples: AWS RDS, Heroku, Google App Engine, Azure App Service Security: YOU are responsible for: Application security: your job (code vulnerabilities) Data classification: your job Identity (IAM): your job Access controls: your job Moderate control, moderate responsibility Use when: developers want to deploy apps without managing servers

SaaS — Software as a Service

# SaaS: provider manages everything — you use the software Provider manages: Everything except your data and who uses it You manage: Data, user access, configuration Examples: Microsoft 365, Salesforce, Google Workspace, Zoom Security: YOU are responsible for: Data classification: who gets to see what IAM (user accounts): who has access + MFA Configuration: sharing settings, retention Compliance for your data: GDPR/HIPAA still YOUR problem Least control, least management burden Use when: want mature software with no infrastructure management

// Exam Keywords

IaaS →Infrastructure — you manage OS up — most control — AWS EC2, Azure VMs
PaaS →Platform — you manage app + data — AWS RDS, Heroku
SaaS →Software — you manage data and IAM only — M365, Salesforce
Customer ALWAYS owns →Data classification + Identity/Access Management — in ALL three models

CLDShared Responsibility ModelWho Owns What

The shared responsibility model defines who is responsible for security in cloud environments. It's the most critical cloud security concept — misunderstanding it leads to security gaps.

# Shared responsibility: what the customer ALWAYS owns: In ALL service models (IaaS/PaaS/SaaS): Customer ALWAYS responsible for: → Data (classification, protection, compliance) → Identity and Access Management (who can access what) → Client/endpoint devices used to access cloud → Account credentials and MFA # What varies by service model: IaaS PaaS SaaS ───────────────────────── Data: Customer Customer Customer IAM: Customer Customer Customer Application: Customer Customer Provider Runtime: Customer Provider Provider OS: Customer Provider Provider Virtualization:Provider Provider Provider Hardware: Provider Provider Provider Physical: Provider Provider Provider Key exam insight: Misconfiguration is #1 cloud breach cause "The cloud is secure — your config might not be" S3 bucket left public = customer fault, not AWS fault

Cloud Security Architecture Controls

CASB

Cloud Access Security Broker — sits between users and cloud services. Enforces: DLP, access control, threat detection, compliance for SaaS apps. Sees all cloud traffic. Essential for shadow IT discovery. Examples: Microsoft Defender for Cloud Apps, Netskope.

CSPM

Cloud Security Posture Management — continuously checks cloud configs against security benchmarks. Detects: S3 bucket publicly accessible, security group open to internet, no MFA on root account. Examples: AWS Security Hub, Azure Defender, Prisma Cloud.

CWPP

Cloud Workload Protection Platform — protects cloud compute instances (VMs, containers, serverless). EDR equivalent for cloud. Monitors: process execution, network connections, file changes in cloud workloads.

SASE

Secure Access Service Edge — combines SD-WAN + security functions (FWaaS, CASB, ZTNA, SWG) into a single cloud-delivered service. Network security delivered from the cloud. Gartner-coined term for converged cloud security.

ZTNA

Zero Trust Network Access — replace VPN with app-level cloud access. Users access specific apps — not the entire network. Continuously verify: identity + device + context. Cloudflare Access, Zscaler ZPA, Prisma Access.

Cloud HSM

Cloud-based Hardware Security Module for key management. AWS KMS, Azure Key Vault. Encryption keys stored in hardware — never exposed in software. FIPS 140-2 validated. Use for: encrypting data, signing code, CA operations.

VRTVirtual Machines & HypervisorsAbstract Hardware

Virtualization creates software-based emulations of hardware — allowing multiple virtual machines (VMs) to run on a single physical host. The hypervisor is the software layer that manages this abstraction.

Hypervisor Types

Type 1 — Bare Metal (Native)

Runs DIRECTLY on hardware — no host OS underneath. Hypervisor IS the operating system. Most performant and most secure (smaller attack surface). Examples: VMware ESXi, Microsoft Hyper-V, Xen, KVM. Used in: enterprise data centers, cloud providers (AWS uses Xen/KVM).

Type 2 — Hosted

Runs ON TOP OF a host OS. Host OS → Hypervisor → VMs. Less performant — extra layer. Less secure — compromising host OS = compromising all VMs. Examples: VMware Workstation, VirtualBox, Parallels. Used in: development, testing, personal use, labs.

VM Security Concepts

# Key virtualization security concepts: VM Isolation: Each VM runs in isolated memory space VMs cannot read each other's memory (by design) Hypervisor enforces isolation Failure: VM escape (critical vulnerability) Snapshots: Point-in-time capture of VM state + disk Security use: capture before pentest, restore after Security risk: old snapshot may contain vulnerabilities stale credentials, outdated software Best practice: delete old snapshots — not "just in case" storage VM Sprawl: VMs easy to create → hundreds of forgotten VMs accumulate Unmanaged VMs: unpatched, forgotten credentials Defense: VM inventory, lifecycle policy, auto-shutdown Dormant VMs: Powered-off VM: may be outdated when powered on Bring online → immediately vulnerable (no patches applied while off) Defense: patch before powering on + rescan after VM Escape: Guest VM escapes hypervisor → accesses host or other VMs Critical vulnerability in hypervisor — exploits escape isolation Defense: keep hypervisor patched, run on dedicated hardware

// Exam Keywords

Type 1 hypervisor →Bare metal — runs directly on hardware — VMware ESXi, Hyper-V — most secure
Type 2 hypervisor →Hosted — runs on OS — VirtualBox, VMware Workstation — less secure
VM escape →Guest VM breaks out of hypervisor isolation — accesses host/other VMs — critical
VM sprawl →Uncontrolled VM proliferation — unpatched forgotten VMs — security risk
Snapshot risk →Old snapshots contain stale credentials/vulnerabilities — manage lifecycle
VRTContainersDocker · Kubernetes · Isolation

Containers are lightweight isolated environments that package application code and dependencies. Unlike VMs, containers share the host OS kernel — making them faster but with different security boundaries.

VMs vs Containers

# The critical security difference: VM architecture: Physical hardware → Hypervisor → Guest OS (per VM — full OS) → Application Container architecture: Physical hardware → Host OS → Container runtime (Docker) → Container (NO full OS — just app + dependencies) Key difference: containers share the HOST OS kernel VM: separate OS = strong isolation (full boundary) Container: shared kernel = weaker isolation Container risk: Kernel vulnerability → ALL containers on that host affected Container escape → access to host OS Privileged container = dangerous (root on host) Container benefits: Faster startup (seconds vs minutes) Smaller image (MB vs GB) Consistent environment (dev = prod) Immutable: replace not patch

Container Security

Image Security

Scan container images before deployment (Trivy, Snyk). Use minimal base images (Alpine Linux vs full Ubuntu). No secrets hardcoded in images. Private registry for internal images. Image signing to verify authenticity. Rebuild images regularly — don't run stale.

Runtime Security

Never run containers as root if avoidable. Read-only file systems for containers. Network policies between containers (Kubernetes NetworkPolicy). Resource limits — prevent one container consuming all CPU/RAM. Falco: runtime threat detection for containers.

// Exam Keywords

Container vs VM →Containers share host OS kernel — less isolation than VMs — faster but weaker boundary
Container escape →Breaking out of container to host OS — weaker isolation than VM escape
Docker →Most popular container runtime — images, containers, registry
Kubernetes →Container orchestration — manages clusters of containers — RBAC + NetworkPolicy

IoTInternet of ThingsConnected Devices

IoT encompasses physical devices with embedded sensors, software, and connectivity that allow them to collect and exchange data. They range from smart thermostats to industrial sensors — and they expand the attack surface massively.

Why IoT Is a Security Challenge

# IoT fundamentally different from traditional IT: Scale: Traditional IT: hundreds/thousands of devices IoT in enterprise: tens of thousands of devices Smart city: millions of sensors Scale = impossible to manually manage security Resource constraints: IoT devices: low CPU, low RAM, limited battery Cannot run: full OS, EDR agent, antivirus Limited: encryption capability on tiny hardware Result: security is an afterthought in many devices Longevity: Traditional PC: replaced every 3-5 years Industrial sensor: runs for 15-20 years Medical device: runs until end-of-support or decommission Problem: vendor stops updating → stuck with vulnerabilities Physical exposure: IoT devices often in physically exposed locations Smart lock: outside the building (can be stolen/tampered) Parking sensor: embedded in pavement Physical access → often = full compromise Default credentials: IoT ships with: admin/admin, root/root, admin/12345 Many users NEVER change them Mirai botnet: used default credentials to compromise millions of IoT devices → DDoS Dyn DNS → internet outage

// Exam Keywords

IoT attack surface →Massive scale + weak security + default credentials = huge attack surface
Mirai botnet →IoT devices with default credentials → DDoS Dyn DNS → major internet outage 2016
Resource constraints →Low CPU/RAM — cannot run full security stack — lightweight security needed

IoTIoT Controls & Exam SummaryMitigate & Know

IoT Security Controls

Network Segmentation

IoT VLAN isolated from corporate network. NAC: auto-assign IoT devices to IoT VLAN by MAC OUI. Firewall: strict rules — IoT can only reach needed services. Internal attacker on IoT VLAN cannot reach HR system.

Inventory and Discovery

You can't secure what you don't know exists. NAC agentless discovery — MAC OUI fingerprinting. Nmap scans for new devices. Asset database: every IoT device catalogued. Regular audits: remove unauthorized devices.

Change Default Credentials

Policy: all new IoT devices — change default credentials before network connection. Scan for default credentials (Shodan-style internal scan). Remove devices that cannot have passwords changed.

ICSIndustrial Control SystemsOT · SCADA · Safety-Critical

Industrial Control Systems (ICS) and Operational Technology (OT) control physical processes in critical infrastructure — power grids, water treatment, manufacturing, pipelines, and transportation. A cyberattack here can cause physical harm or death.

IT vs OT — The Fundamental Difference

# IT vs OT priorities are OPPOSITE: IT Security priorities: 1. Confidentiality (data protection) 2. Integrity (data accuracy) 3. Availability (uptime) CIA Triad — in that order OT/ICS Security priorities: 1. Availability (systems must keep running!) 2. Integrity (commands must be correct) 3. Confidentiality (less important) AIC Triad — REVERSED from IT Why reversed: Power grid offline → city without power → people die Water treatment down → unsafe water → public health crisis Manufacturing stopped → $millions/hour loss "Just patch it and reboot" → NOT acceptable in OT This matters for exam: OT availability >>> confidentiality

ICS Terminology

SCADA

Supervisory Control and Data Acquisition. Centralized monitoring system for geographically distributed infrastructure. Power grid: SCADA monitors thousands of substations. Water: SCADA controls pumps across a city. Collects data + sends control commands over network.

DCS

Distributed Control System. Controls process within a single facility. Chemical plant, refinery, manufacturing floor. Similar to SCADA but geographically contained. High speed, deterministic control loops.

PLC

Programmable Logic Controller. Industrial computer for controlling physical equipment. Reads: sensors (temperature, pressure, flow). Controls: actuators (valves, motors, pumps). Ladder logic programming. Stuxnet targeted PLCs (Siemens S7).

RTU / IED

Remote Terminal Unit / Intelligent Electronic Device. Field devices — collect data from sensors, send to SCADA. RTU: communicates via radio/cellular to SCADA. IED: in electrical grid — automates switching decisions. Often at remote unmanned locations.

// Exam Keywords

OT priority →Availability FIRST — then integrity — then confidentiality (reverse of IT)
SCADA →Supervisory Control and Data Acquisition — centrally monitors/controls distributed OT
PLC →Programmable Logic Controller — controls physical equipment — Stuxnet target
Safety system →SIS (Safety Instrumented System) — shuts down process if dangerous condition — TRITON targeted these

ICSICS Controls & Exam SummaryDefend OT

ICS Security Controls

Air Gap / Network Segmentation

Complete network isolation (air gap) — ideal but rare. Industrial DMZ: buffer zone between IT and OT. Data diode: one-way data flow — OT → IT but not back. No direct internet to OT. Jump host required for OT access.

OT-Specific Monitoring

Passive network monitoring — no active scanning (breaks PLCs). Claroty, Dragos, Nozomi: OT-specific security tools. Baseline: establish normal OT traffic patterns. Alert on: new device on OT network, unusual commands, protocol anomalies.

Physical Controls

Physical access critical for OT. Lock PLC cabinets. Disable USB ports on HMI/workstations (Stuxnet!). Tamper-evident seals on equipment. Escort policy for vendors in OT areas. Security cameras in control rooms.

IaCInfrastructure as CodeCode-Defined Infra

IaC means defining and managing infrastructure through code files rather than manual configuration. Instead of clicking through a GUI to provision servers, you write code that describes the desired infrastructure state — then apply it automatically.

Why IaC Matters for Security

# The "snowflake server" problem IaC solves: Without IaC: Server 1: Admin A configured manually (2020) Server 2: Admin B configured manually (2021) Server 3: Admin C configured manually (2022) 3 servers, same role → 3 different configurations: Server 1: SSH v2, firewall on, MFA enabled ✓ Server 2: SSH v1 still enabled ← vulnerable Server 3: root login allowed ← vulnerable Each server = unique "snowflake" — inconsistent security Audit: impossible to verify 500 unique configs Breach: attacker finds the misconfigured server With IaC: server.tf defines exact configuration Deployed identically to ALL servers Security config is in code → reviewed + tested Drift detection: alert if server deviates from code All 3 servers: identical, secure, auditable ✓

Immutable Infrastructure

🔄
Replace, Don't Patch
Traditional: patch running server — changes applied over time — unknown state accumulates. IaC immutable approach: when you need to change a server, destroy it and spin up a new one from updated IaC definition. Compromised server? Destroy it. New clean server from IaC: online in minutes. Attacker's persistence wiped away. Drift impossible — every deployment starts from clean definition.

IaCIaC ToolsTerraform · Ansible · Docker

Terraform

Cloud infrastructure provisioning. Defines: VPCs, servers, security groups, IAM policies, databases as code. Multi-cloud: same tool for AWS/Azure/GCP. State file: tracks what currently exists. Plan: preview changes before applying. Security groups as code = reviewable firewall rules.

resource "aws_security_group" "web" { ingress { from_port=443 to_port=443 protocol="tcp" cidr_blocks=["0.0.0.0/0"] } # Port 22: NOT OPEN ← security ✓ }

Ansible

Configuration management — applies desired state to existing systems. Agentless: connects via SSH. Idempotent: running multiple times = same result. Security hardening: one playbook enforces CIS benchmark on all servers. Patch deployment. No drift: Ansible ensures servers stay configured correctly.

Docker / Kubernetes

Containerization = immutable runtime. Dockerfile defines exact environment. Image = immutable artifact. Container at runtime = exact same environment every time. Kubernetes: orchestrates containers at scale. Security: scan images before deployment, RBAC in K8s, NetworkPolicy.

CI/CD Security (DevSecOps)

Security gates in deployment pipeline. SAST: static code analysis before build. Secrets scan: catch API keys in code. Image scan: vulnerability check before deploy. DAST: live security test of deployed app. IaC scan: check Terraform for misconfigs (checkov, tfsec) before apply.
