MALVirus & WormSelf-Replicating
🦠 Virus — Needs Human Action to Spread
Attaches to legitimate files or programs. Cannot spread without human interaction — user must execute the infected file or share it. Types: file infector (attaches to .exe), macro virus (Office documents), boot sector virus (infects MBR), polymorphic (changes signature each replication to evade AV), metamorphic (rewrites itself completely). Key rule: virus CANNOT spread on its own.
Needs user actionAttaches to files
🪱 Worm — Self-Propagates Automatically
Self-replicating malware that spreads across networks by exploiting vulnerabilities — no user interaction required. Scans for vulnerable systems, infects, scans again. WannaCry (2017): exploited EternalBlue SMBv1 → 200,000 systems in 150 countries in hours. Slammer (2003): doubled every 8.5 seconds. Worms can consume all network bandwidth through propagation alone.
Self-propagatesExploits vulnerabilitiesPatch = stops it
# KEY EXAM DISTINCTION: Virus: infected file → user runs/shares it → spreads Cannot jump between systems without human help Worm: exploits open port 445 (SMBv1) → installs itself scans network → repeats → exponential growth WannaCry: 200K systems in hours, no user clicks needed Polymorphic virus: changes signature each copy → evades signature-based AV → requires behavioral detection

MALTrojan, RAT & BackdoorDisguised · Remote Control
🐴 Trojan — Disguised as Legitimate Software
Appears legitimate and useful — conceals malicious payload. Does NOT self-replicate — relies on social engineering. "Free game," "cracked software," "fake antivirus." Can deliver ransomware, spyware, backdoor, downloader. Name from the Trojan Horse — invitation-based deception.
Social engineering deliveryDoes NOT self-replicate
🕹️ RAT — Remote Access Trojan
Trojan that establishes covert remote control channel to attacker's C2 (Command and Control) server. Capabilities: screenshot capture, keylogging, file access, webcam activation, microphone recording, network pivot. Establishes persistence via registry keys, scheduled tasks, services. Notable examples: DarkComet, njRAT, Cobalt Strike (commercial tool widely abused).
Full remote controlC2 callback
🚪 Backdoor
Covert access mechanism allowing attacker to return to compromised system, bypassing normal authentication. May be standalone malware, part of a RAT, or hardcoded in firmware. Critical: if a system was fully compromised → assume backdoor is present → reimage, don't just remove detected malware. The backdoor may survive malware removal.
PersistenceSurvives malware removal

MALRootkit, Bootkit & Fileless MalwareStealth · Persistent · Memory
👁️ Rootkit — Hides Malware from OS
Conceals its presence from the OS and security tools by intercepting system calls at kernel level. The OS reports a clean system while malware runs undetected. Types: kernel-mode (most dangerous — operates in OS kernel), user-mode (hooks API calls), hypervisor rootkit (places OS inside malicious VM — nearly undetectable). Detection is extremely difficult because the rootkit controls what the OS reports. Defense: TPM measured boot, offline scanning from trusted bootable media, known-good baseline comparison.
Hides from OSKernel-level
🥾 Bootkit — Infects Boot Process
Infects the MBR (Master Boot Record) or UEFI bootloader — executes before the OS loads. Persists through OS reinstallation because it's not in the OS partition. UEFI bootkits (Black Lotus 2022) survive even disk replacement. Traditional AV cannot detect or remove (it's not running when AV scans). Defense: UEFI Secure Boot (verifies bootloader signature), TPM measured boot, physical disk replacement + UEFI reflash for confirmed infections.
Pre-OS executionSurvives OS reinstall
👻 Fileless Malware — Lives in Memory Only
Operates entirely in memory — no executable written to disk. Uses PowerShell, WMI, registry entries, or injects into legitimate processes (process hollowing — runs malicious code inside svchost.exe). Traditional AV scans files on disk → finds nothing. Fileless never touches disk → AV blind. Clears itself on reboot but persistence mechanisms reinstall. Defense: AMSI (scans scripts in memory at runtime), behavioral EDR, PowerShell script block logging, memory scanning.
Evades traditional AVMemory + scripts onlyAMSI + EDR detects

MALSpyware, Keylogger, Logic Bomb & BotnetSurveillance · Triggered · Army

🔍 Spyware

Covertly monitors user activity and transmits data to attacker. Captures: browsing, credentials, financial data, screenshots, communications. Often bundled with "free" software. Defense: EDR, monitor outbound data flows, application allow listing.

⌨️ Keylogger

Records all keystrokes → captures passwords, PINs, credit cards, messages. Software keylogger: runs as process. Hardware keylogger: physical device inline between keyboard and USB port — captures before OS sees it. Defense: software = EDR / hardware = physical inspection.

💣 Logic Bomb

Malicious code dormant until specific condition: time, date, event, trigger. Insider threat weapon: "if my employee ID stops appearing in payroll → delete databases." Defense: separation of duties, code reviews, monitor scheduled tasks, PAM.

🤖 Botnet

Network of compromised devices (bots/zombies) controlled by botmaster via C2. Used for: DDoS, spam, credential stuffing, crypto mining. Mirai IoT botnet: 620 Gbps DDoS. Defense: patch devices, DNS sinkhole, C2 traffic blocking.

Logic bomb →Dormant until trigger — insider threat — separation of duties + code review
Botnet →Zombie army — C2 controlled — DDoS/spam/mining — DNS sinkhole disrupts C2

PWDBrute Force, Dictionary & SprayingExhaustive · Wordlist · Low + Slow
Brute Force — Try Every Combination
Systematically try every possible password combination. 8-char alphanumeric = ~218 trillion combinations. Online brute force (live login page): account lockout stops it. Offline brute force (stolen hash): no lockout — pure hardware speed. Modern GPUs can try billions of MD5 hashes per second but only thousands of bcrypt hashes per second — this is why bcrypt matters.
Online: lockout stops itOffline: hardware-limited
Dictionary Attack — Wordlist
Try a list of known common passwords, words, phrases rather than all combinations. RockYou wordlist: 14 million real passwords from 2009 breach — covers most common choices. Hybrid: dictionary word + numbers/symbols (Password123!, Summer2024). Much faster than brute force by targeting human password-choosing patterns.
Wordlist basedTargets human patterns
Password Spraying — 1 Password × Many Accounts
Try a SINGLE common password against MANY accounts, then move to next — staying below lockout threshold per account. "Summer2024!" against 10,000 users → statistically several will use it. Bypasses account lockout — each account only sees 1 failed attempt. Effective when organizations have predictable password patterns (CompanyName + Year + !). Detection: look for distributed failures across many accounts from same source.
Bypasses account lockout1 password × many accounts

Dictionary attack →Wordlist of common passwords — RockYou list — targets human patterns
Password spraying →1 password × many accounts — bypasses lockout — monitor distributed failures
PWDCredential AttacksStuffing · Pass-the-Hash · Kerberoasting
Credential Stuffing — Reuse Leaked Credentials
Use username/password pairs leaked from previous breaches against other services. ~0.1-2% success rate — but billions of credentials available → thousands of accounts compromised. LinkedIn breach credentials → try on corporate VPN, Gmail, banking. Defense: MFA eliminates credential stuffing entirely — stolen credential useless without second factor.
Uses leaked credentialsMFA eliminates entirely
Pass-the-Hash (PtH)
In Windows NTLM authentication, the hash of the password IS the credential — knowing the hash allows authenticating without knowing the plaintext. Attacker uses Mimikatz to dump NTLM hashes from LSASS memory → uses hash to authenticate to other systems → lateral movement. Never needs to crack the password at all. Defense: Windows Credential Guard (isolates LSASS secrets in protected enclave), use Kerberos instead of NTLM, least privilege.
No plaintext neededWindows NTLMCredential Guard
Kerberoasting — Steal & Crack Service Tickets
Kerberos service tickets are encrypted with the service account's password hash. Any domain user can request a ticket for any SPN (Service Principal Name). Attacker requests ticket → takes it offline → brute forces the hash (no lockout — it's offline). Goal: crack service account passwords (often weak, old, never-rotated). Defense: gMSA (Group Managed Service Accounts) — automatic 120-char random passwords — impossible to crack.
Offline crack — no lockoutAny domain user can triggergMSA prevents
Golden Ticket / Silver Ticket
Golden Ticket: forge a Kerberos TGT using the KRBTGT account hash. Gives unlimited domain access — can create tickets for any user, any service. Persists even through password changes for most accounts. Silver Ticket: forge service ticket using service account hash — access that specific service only. Defense: rotate KRBTGT password to invalidate all golden tickets, monitor for unusual Kerberos ticket patterns in SIEM.
Golden: domain-wideForged Kerberos tickets

APPInjection & Input AttacksSQLi · XSS · Command Injection
SQL Injection (SQLi)
Insert SQL commands into user input. Application builds SQL query with attacker input → executes malicious SQL. Classic: ' OR '1'='1 — bypasses login. Union-based: UNION SELECT to extract other tables. Blind SQLi: no data returned but behavior reveals info. Time-based blind: SLEEP() delays confirm conditions. Impact: dump entire database, modify/delete records, bypass auth, sometimes RCE via xp_cmdshell. Defense: parameterized queries (prepared statements) — not input filtering alone. WAF as additional layer.
Data theft / manipulationParameterized queries fix
Cross-Site Scripting (XSS)
Inject malicious JavaScript into web pages. Stored XSS: script saved in database → executes for ALL viewers. Reflected XSS: script in URL parameter → victim clicks malicious link. DOM-based: modifies DOM client-side. Attacks: steal session cookies, redirect to phishing, keylog page, perform actions as victim. Defense: output encoding (HTML entity encode before rendering), Content Security Policy (CSP), HttpOnly cookies (script can't read them).
Session hijackingStored = most severeCSP + output encoding
Command Injection & Path Traversal
Command injection: app passes user input to OS shell — attacker adds ; cat /etc/passwd or | whoami to chain commands → RCE. Path traversal: use ../ to escape intended directory — app serves /uploads/ + filename — attacker uses ../../../../etc/passwd to read system files. Defense: avoid shell commands (use APIs instead), sanitize metacharacters, normalize paths, validate against base directory.
Command injection → RCEPath traversal → file read

SQLi fix →Parameterized queries / prepared statements — NOT just input filtering
Stored XSS →Persisted in DB — executes for ALL viewers — most severe XSS type
XSS defense →Output encoding + Content Security Policy (CSP) + HttpOnly cookies

APPReplay & Session AttacksReplay · Hijack · Fixation
Replay Attack
Capture a valid authentication token or transaction and re-transmit it. Without replay protection: captured bank transfer message re-sent = duplicate transfer. Defense: nonces (numbers used once — unique value per message, server rejects duplicates), timestamps (messages expire after short window), sequence numbers, HTTPS (prevents capture in transit).
Re-transmit captured trafficNonces + timestamps fix
Session Hijacking / Cookie Theft
Steal session cookie and impersonate authenticated user. Methods: XSS to steal cookie, network sniffing on HTTP traffic, session ID exposed in URL (appears in logs/referrer headers), predictable session tokens. Defense: HTTPS only, HttpOnly flag (script can't read cookie), Secure flag (HTTPS only), SameSite cookie attribute (blocks cross-site requests), short session timeouts, re-authenticate for sensitive operations.
Token = account accessHTTPS + HttpOnly + Secure
Session Fixation
Attacker sets a known session ID before user authenticates — after login, user's session has the attacker's known ID. Defense: regenerate session ID on authentication — new session after login. Most modern frameworks do this automatically but custom implementations often miss it.
Preset known session tokenRegenerate ID on login

Session fixation →Preset known session ID — regenerate session ID after authentication fixes
APPPrivilege EscalationVertical · Horizontal · Kernel

Vertical Privilege Escalation

Gain higher privileges: user → admin → root/SYSTEM. Methods: kernel exploit (EternalBlue, PrintNightmare, Dirty Cow), DLL hijacking (Windows loads attacker DLL from writable path), SUID binary abuse (Linux), unquoted service paths (Windows). Defense: patch OS, CIS benchmark, audit SUID binaries, DLL search order protection.

Horizontal Privilege Escalation

Access another user's resources at the same privilege level. Alice accesses Bob's account by changing account ID in URL (IDOR). Broken access control — missing server-side authorization check. Defense: check on every request — "does this authenticated user own this resource?"

// Exam Keywords

Vertical privesc →User → admin/root — kernel exploit, DLL hijacking, SUID abuse
Horizontal privesc →Access peer's account — IDOR, broken access control — server-side auth check fixes
DLL hijacking →Windows loads DLL from writable path → attacker places malicious DLL there
APPApplication DoS AttacksReDoS · XML Bomb · Memory Exhaustion

ReDoS — Regex Denial of Service

Crafted input causes catastrophic regex backtracking — 100% CPU consumption from a single request. Affects applications using certain poorly-written regex patterns. Defense: regex timeouts, input length limits, avoid catastrophic backtracking patterns.

XML Bomb / Billion Laughs

Nested XML entities that exponentially expand in memory. 1KB input → gigabytes in memory → crash. Also: Zip bomb (small compressed file → enormous decompressed). Defense: decompression size limits, XML entity depth limits, request size limits.

Physical Attack Types

🔌 USB Drop Attack / Baiting
Leave malicious USB drives labeled enticingly ("Q4 Salaries", "Project X") in parking lots or common areas. Studies show 45-98% of found USBs get plugged in. Variants: USB Rubber Ducky (emulates keyboard → executes commands automatically), O.MG cable (malicious USB charger). Defense: disable USB ports via endpoint policy, application allow listing prevents unauthorized execution, security awareness training.

💳 Skimming — Payment Terminal Compromise
Install malicious hardware overlay on ATM, gas pump, or POS terminal. Card skimmer reads magnetic stripe data. PIN overlay or hidden camera captures PIN entry. Deep-insert skimmers: inside card slot — invisible from outside. NFC skimming reads contactless cards. Defense: inspect terminals before use, use chip+PIN (skimmer can't easily clone chip), NFC-blocking wallet, tokenized payment (Apple/Google Pay).

NETDenial of Service & DDoSVolume · Protocol · Application
Volumetric DDoS — Flood the Pipe
Flood target's internet connection with traffic until saturated. Amplification: small request generates huge response at victim's IP (spoofed as source). DNS amplification: 64-byte query → 3,000-byte response (47× amplification). NTP amplification: monlist command → 206× amplification. Defense: CDN-based DDoS protection (Cloudflare, Akamai, AWS Shield) absorbs traffic at edge — scrubbing centers.
Saturate bandwidthCDN scrubbing defends
SYN Flood — Exhaust TCP State Tables
Send TCP SYN packets without completing the handshake. Server allocates memory for each half-open connection → state table exhausted → legitimate connections refused. SYN packets are cheap to send (spoofed IP) but expensive for server (allocate state). Defense: SYN cookies — server doesn't allocate state until handshake completes — cryptographic approach eliminates state exhaustion.
Half-open connectionsSYN cookies solve it
Application Layer DDoS (Layer 7) — Slowloris
Send legitimate-looking HTTP requests that consume server resources. Slowloris: open many connections, send partial HTTP headers very slowly — keeps connections open indefinitely — exhausts server's connection pool. HTTP flood: GET requests triggering expensive DB queries. Harder to filter because traffic looks legitimate. Defense: rate limiting, WAF with behavioral analysis, CDN with bot detection, connection timeouts.
Legitimate-looking trafficSlowloris / HTTP flood

NETSpoofing, Poisoning & MITMARP · DNS · BGP · On-Path
ARP Poisoning / ARP Spoofing
ARP is stateless — any device can claim any IP → MAC mapping. Attacker sends gratuitous ARP: "I am 10.0.0.1 (the gateway)" → victim updates ARP cache → all traffic goes to attacker (MITM established). Attacker forwards to real destination — victim notices nothing. Defense: Dynamic ARP Inspection (DAI) on switches — validates ARP against DHCP snooping binding table — only trusted ports can send ARP replies.
LAN traffic redirectDAI on switches
DNS Cache Poisoning
Inject false DNS records into a resolver's cache — all clients using that resolver are misdirected. Attacker responds faster than legitimate DNS → resolver caches attacker's IP for bank.com → all users sent to phishing site. Kaminsky Attack (2008): systematic DNS poisoning technique. Defense: DNSSEC (cryptographically signs DNS records — tampering detectable), randomize source port + transaction ID, DoH/DoT (encrypted DNS queries).
Redirect all clientsDNSSEC prevents
BGP Hijacking & On-Path MITM
BGP hijacking: advertise false BGP routes — redirect internet traffic at global scale. Rogue AS announces more-specific route → BGP prefers it → traffic routed through attacker. Turkey 2014: redirected Google DNS traffic for entire country. Defense: RPKI (Resource Public Key Infrastructure) — validates BGP announcements cryptographically. On-path MITM: attacker between victim and server — reads/modifies all traffic. Defense: TLS encryption, HSTS (browser refuses HTTP downgrade).
BGP: global routingRPKI + TLS + HSTS

NETScanning & Packet SniffingPort Scan · Eavesdrop · Recon

Port Scanning & Enumeration

Identify open ports and running services. Nmap: TCP SYN scan, service version, OS detection. Masscan: internet-wide scanning. Shodan: search engine for internet-connected devices. Reconnaissance phase of attack. Defense: firewall rules restrict visible ports, IDS detects scan patterns, disable unused services.

Packet Sniffing / Eavesdropping

Capture network packets and read their contents. Passive (listen without sending) vs active (ARP poison to redirect). Effective against plain HTTP, Telnet, FTP, unencrypted email. Wireshark / tcpdump are legitimate tools also used by attackers. Defense: encrypt all traffic (TLS, SSH, VPN) — captured ciphertext is useless. Switches segment traffic vs hubs.

CRYCryptographic AttacksBirthday · Downgrade · Side-Channel · Quantum
🎂 Birthday Attack — Hash Collision
Based on the birthday paradox: in a group of 23 people, 50% chance two share a birthday. For hashes: find two different inputs producing the same hash — much faster than expected mathematically. MD5: practical collisions in hours. SHA-1: Google SHAttered (2017) — first real collision found. Impact: substitute malicious document with same hash as legitimate one. Defense: SHA-256 or stronger — longer hash = much harder collision.
MD5/SHA-1 vulnerableSHA-256+ safe
⬇️ Downgrade Attack
Force connection to use older, weaker protocol version. POODLE (2014): force SSL 3.0 instead of TLS → SSL 3.0 has CBC padding oracle → decrypt traffic. BEAST: force TLS 1.0 CBC mode. Logjam: force 512-bit export-grade DH → break key exchange. Defense: disable old protocol versions (SSL 2.0/3.0, TLS 1.0/1.1), set minimum protocol version, HSTS preload prevents SSL stripping downgrade.
POODLE / BEAST / LogjamDisable old versions
🔮 Padding Oracle Attack
When server reveals whether decrypted ciphertext has valid padding, attacker uses that info leak (oracle) to decrypt ciphertext one byte at a time — without the key. POODLE exploited SSL 3.0's CBC padding. Practical: decrypt session cookies → session hijacking. Defense: use AEAD cipher modes like AES-GCM — authenticated encryption has no separable padding. TLS 1.3 removes CBC entirely.
CBC mode vulnerableAES-GCM (AEAD) immune
⏱️ Side-Channel Attack
Extract secret keys by measuring physical side effects — not attacking the math. Timing attack: measure how long decryption takes → different values take different time → infer key bits. Power analysis: monitor power consumption → reveals key operations. Acoustic attack: record CPU sounds during RSA decryption. Spectre/Meltdown: timing via CPU cache. Defense: constant-time implementations (same execution time regardless of input values), HSMs, power isolation.

⚛️ Quantum Threat — Harvest Now, Decrypt Later
Quantum computers (Shor's algorithm) can break RSA and ECC by factoring large numbers efficiently — rendering current public-key crypto broken. Not yet practical at scale but: "harvest now, decrypt later" — nation-states record encrypted traffic today to decrypt once quantum computers arrive. Symmetric keys: quantum halves effective key length (AES-256 → ~128-bit with quantum — still OK). NIST finalizing post-quantum standards: CRYSTALS-Kyber, CRYSTALS-Dilithium. Transition now for long-lived sensitive data.

📊 Frequency Analysis
In monoalphabetic substitution ciphers, each plaintext letter is always replaced by the same ciphertext letter. English: 'e' is most common letter (~12.7%). If ciphertext letter appears 12.7% of time → probably 'e'. Breaks classical ciphers (Caesar, Vigenère). Modern ciphers (AES) are immune — output is statistically random with no pattern. Tested on Security+ to demonstrate why simple substitution fails.
