Diagram 1 — Traditional Networking vs SDN
// The core difference — where does the "brain" live?

❌ Traditional Networking

1
Each router/switch has its OWN brain (control plane)
2
Admin must log into EACH device separately
3
Change one rule = update every device manually
4
Inconsistent configs, human errors, slow response

✅ SDN (Software Defined)

1
ONE central controller is the ONLY brain
2
Devices just FORWARD packets — no decisions
3
Change one rule = instantly updates ALL devices
4
Consistent, fast, automated security response

// EXAM KEYWORDS

Control Plane →The DECISION maker — where to send traffic (SDN removes this from devices)
Data Plane →The FORWARDER — actually moves packets (stays in devices)
SDN Controller →Centralized brain — manages all network devices from one place
Diagram 2 — SDN Three-Layer Architecture
// Applications talk to controller, controller talks to devices

// THREE LAYERS — EXAM READY

Application Layer →Security apps, firewall rules, load balancer logic — defines POLICY
Control Layer →SDN Controller — the brain — translates policy to network rules
Infrastructure Layer →Physical devices — just forward packets — no decisions
Northbound API →Apps talk to Controller (what to do)
Southbound API →Controller talks to devices (how to do it) — OpenFlow protocol
Diagram 3 — Professor Messer's Diagram Explained
// The SDN controller is invisible — managing everything you see

✅ Traffic Flow (Normal User)

1
User request hits Internet
2
Outer Firewall/IPS checks it — legitimate ✅
3
Load Balancer routes to Web Server A or B
4
Web Server needs data → Internal Firewall checks
5
Database returns data → Response sent to user

❌ Attack Attempt

1
Attacker sends malicious request
2
Outer Firewall/IPS detects attack signature ❌ BLOCKED
3
If bypassed → gets to web server
4
Tries to reach database → Internal Firewall BLOCKS ❌
5
Database never reachable from internet — safe ✅
Diagram 4 — Hospital SDN Example
// SDN dynamically assigns devices to correct security zones

// KEY SECURITY PRINCIPLE

Medical Zone →Life-critical devices — isolated, cannot be reached from guest network
Staff Zone →Employees — access medical records but not guest internet
Guest Zone →Internet only — ZERO access to medical devices or staff systems
SDN Benefit →All zone rules enforced automatically — no manual switch config per device
Diagram 5 — SDN Attack Response
// How SDN responds to an attack — traditional vs SDN

❌ Traditional — Attack Response

1
Attack detected at 3:00 AM
2
Alert triggers — admin woken up
3
Admin logs into Firewall 1 manually
4
Adds block rule to Firewall 1
5
Logs into Switch 1... then Switch 2...
6
Logs into Firewall 2... Router 1...
7
45 minutes later — all devices updated
8
Attacker was active for 45 minutes ❌

✅ SDN — Attack Response

1
Attack detected at 3:00 AM
2
SDN Controller sees anomaly automatically
3
Controller pushes block rule to ALL devices
4
All firewalls, switches, routers updated
5
Takes 3-5 SECONDS total
6
Attacker quarantined to isolated VLAN
7
Alert sent to admin with full details
8
Attacker contained in seconds ✅

// EXAM TIPS — SDN SECURITY

SDN benefit →Centralized management, dynamic security policies, faster threat response
SDN risk →Controller = single point of failure AND single point of attack
Control Plane →Separated from devices — lives in SDN Controller only
Data Plane →Stays in devices — just forwards packets as told
Micro-segmentation →SDN enables this easily — granular security between workloads
OpenFlow →Protocol SDN controller uses to talk to network devices
