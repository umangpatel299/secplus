01 Asset Management Overview Full Lifecycle

Asset management is the systematic process of tracking and managing every hardware, software, and data asset throughout its entire lifecycle — from the moment it's acquired to the moment it's disposed of. You can't secure what you don't know you have.

Why Asset Management Matters for Security

# The fundamental security problem: Unknown asset = unmanaged asset = unsecured asset Company has 500 known laptops → all patched ✓ Company also has 47 FORGOTTEN laptops → never patched ✗ Attacker finds forgotten laptop via network scan Exploits 3-year-old unpatched vulnerability Uses it as entry point → lateral movement begins # Shadow IT example: Developer spins up AWS instance for testing Forgets about it → never registered in asset inventory Never patched, never monitored, never secured Attacker finds it via Shodan (internet scanner) Breach originates from asset nobody knew existed ✗ # With proper asset management: Every asset registered at acquisition Every asset assigned to an owner Every asset monitored continuously Every asset patched and secured Every asset properly disposed of No unknown assets = no forgotten attack surfaces ✓

02 Asset Acquisition Purchase · Verify · Register

Acquisition is the first and most critical phase — security must be considered before a single purchase is made. Buying the wrong thing, from the wrong vendor, or without proper verification creates security risks that persist for years.

Acquisition Process — Step by Step

📋
Step 1 — Business Requirements & Security Review
Before purchasing: define what's needed, who will use it, what data it will touch. Security team reviews: does this meet security requirements? What risks does it introduce? This prevents buying insecure products.
🔍
Step 2 — Vendor Assessment / Supply Chain Verification
Evaluate vendor security posture. Do they have SOC 2 certification? Do they use secure development practices? Have they had breaches? For hardware: buy ONLY from authorized resellers — not grey market. Remember: Cisco fake switches came from unauthorized vendors.
✅
Step 3 — Purchase & Receipt Verification
When hardware arrives: verify serial numbers match purchase order. Check physical tamper-evident seals intact. Verify firmware version before connecting to network. For software: verify digital signatures on installers.
🏷️
Step 4 — Registration in Asset Inventory
Immediately record in CMDB (Configuration Management Database): serial number, model, purchase date, warranty expiry, cost, assigned department, physical location, responsible owner. This is when the asset officially exists in the organization.
🔒
Step 5 — Baseline Security Configuration
Before deployment: apply secure baseline configuration. Change default credentials. Apply all available patches. Install required security agents (antivirus, EDR, MDM enrollment). This is the "golden image" application phase.

Supply Chain Security — Critical for Exam

# Why supply chain matters during acquisition: Scenario 1 — Grey market hardware: Company buys "Cisco switches" from eBay seller Price: 40% cheaper than authorized reseller Switches arrive → look legitimate → installed Reality: counterfeit switches with modified firmware (like the 2020 fake Cisco Catalyst case) Backdoor built into hardware before arrival Company has no idea Prevention: Buy ONLY from Cisco-authorized resellers Verify serial numbers on Cisco's website Check firmware matches official release hash Physical inspection: holographic labels, serial format Scenario 2 — Malicious software supply chain: Developer downloads popular npm package Package has typosquatted name "lodash-secure" (real package: "lodash") Typosquatted package: contains credential stealer Developer installs → credentials sent to attacker Prevention: Verify exact package names Check download counts and publish dates Use private package registry Software composition analysis (SCA) tools

Acquisition Security Checklist

Hardware Checklist

  • Purchase from authorized reseller only
  • Verify serial numbers on manufacturer site
  • Check tamper-evident seals on receipt
  • Verify firmware version vs official release
  • Compare firmware hash vs vendor's published hash
  • Register in asset inventory immediately
  • Apply baseline before any network connection

Software Checklist

  • Download only from official vendor site
  • Verify digital signature of installer
  • Check SHA-256 hash vs vendor published
  • License key registered and documented
  • Note version — vulnerability tracking later
  • Record all dependencies (SBOM)
  • Scan with antivirus before installation

03 Asset Assignment & Ownership Owner · Classification · Tagging

Every asset must have a named owner who is accountable for its security. Unowned assets are nobody's responsibility — which means nobody patches them, nobody monitors them, nobody notices when they're compromised.

Asset Ownership Roles

Data Owner

Senior business executive responsible for a DATA ASSET. Decides classification level. Determines who has access. Accountable for regulatory compliance. Approves access requests.

Customer database: Data Owner: Chief Revenue Officer → Classifies as CONFIDENTIAL → Approves who can access → Accountable for GDPR compliance → Reviews access quarterly

Data Custodian

IT staff who MANAGES and MAINTAINS the data on behalf of the owner. Implements the owner's classification decisions. Performs backups, applies patches, manages permissions technically.

Customer database: Data Custodian: Database Administrator → Implements encryption (owner decided) → Performs daily backups → Applies security patches → Manages database accounts → Runs access control lists

Asset Owner (Hardware)

Department or individual responsible for a physical asset. Responsible for its security, maintenance, and return. Signs for the asset. Notifies IT if lost, stolen, or damaged.

System Owner

Manager responsible for an information SYSTEM (collection of hardware + software + data). Signs off on system security plan. Authorizes system to operate (ATO — Authorization to Operate).

Asset Tagging Methods

Physical Tags

Barcode or QR code label physically attached to hardware. Scanned during audits. Tamper-evident — removal shows asset has been tampered with. Printed with: asset ID, department, serial number.

Asset Tag: AST-2024-00847 Barcode: ||||||||||||||| Dept: Finance Serial: 5CG23456789

RFID Tags

Radio Frequency Identification — passive tags that respond when scanned. Scanners can detect assets within range without direct line of sight. Used for: server room audits, large warehouses, data centers.

GPS Tracking

Active tracking for mobile assets — company vehicles, high-value equipment. Continuous location data. Alert when device leaves authorized geofenced area. Used for: fleet management, high-value laptop tracking.

Asset Classification at Assignment

# When assigning an asset, classify it: Hardware classification questions: What data will this device process? → Medical records: HIGH sensitivity → Email only: MEDIUM sensitivity → Public display kiosk: LOW sensitivity Who will use it and where? → Travels internationally: higher risk → Office-only desktop: lower risk What happens if it's lost? → Contains PII: breach notification required → Contains public data: minor inconvenience Classification drives security controls: HIGH sensitivity laptop: ✓ BitLocker mandatory ✓ MFA required ✓ VPN for all traffic ✓ Screen privacy filter ✓ GPS tracking enabled LOW sensitivity kiosk: ✓ Local user only (no domain) ✓ Kiosk mode browser ✓ No data stored locally ✓ Physical security sufficient

04 Hardware Asset Management Physical Devices

Hardware assets include every physical device in the organization. Managing hardware means knowing where every device is, what it contains, who owns it, and whether it's secure.

Hardware Inventory Categories

End-User Devices

  • Laptops and desktops
  • Smartphones and tablets
  • Thin clients
  • Workstations
  • Monitors (may have USB hubs)

Network Infrastructure

  • Routers and switches
  • Firewalls and IPS appliances
  • Wireless access points
  • Load balancers
  • VPN concentrators

Server Infrastructure

  • Physical servers
  • Storage arrays (NAS/SAN)
  • Backup appliances
  • HSMs (key management)
  • Out-of-band management (iDRAC/ILO)

Peripherals

  • Printers (contain hard drives!)
  • USB drives and external drives
  • Webcams and headsets
  • Scanners and copiers
  • KVM switches

IoT Devices

  • IP cameras and door locks
  • HVAC and building systems
  • Smart TVs and displays
  • Medical devices
  • Industrial sensors

Removable Media

  • USB flash drives
  • External hard drives
  • Backup tapes
  • CDs/DVDs
  • SD cards

Key Hardware Asset Security Concerns

# Printers — often forgotten but high risk: Modern printers have: → Hard drives storing copies of printed documents → Network interfaces (often poorly secured) → Embedded web servers (default passwords!) → Memory buffers containing recent print jobs If printer not in asset inventory: → Never patched (firmware vulnerabilities) → Default credentials never changed → Hard drive never wiped before disposal → Contains months of sensitive printed documents Solution: Register ALL printers in CMDB Change default admin credentials Enable firmware auto-update Include in vulnerability scan scope Wipe hard drive before decommissioning # USB drives — massive data leak risk: Untracked USB drives: → Employee pockets USB drive when leaving → USB drive lost → sensitive data exposed → USB found in parking lot → plugged in → malware Solution: Encrypt ALL organizational USB drives Register each USB in inventory (serial number) Use DLP to track what's copied to USB Disable USB ports on endpoints where not needed Use managed encrypted drives (IronKey, Kingston)

Hardware Tracking Tools

CMDB — Configuration Management Database

Central database of ALL assets and their relationships. Records: hardware specs, software installed, network connections, owner, location, patch status. Examples: ServiceNow, Freshservice, Device42, Lansweeper.

Network Discovery Tools

Automatically scan network to find connected devices. Compares discovered devices vs CMDB — rogue/unauthorized devices flagged. Examples: Nmap, Nessus, Qualys, Lansweeper, SolarWinds.

Discovery scan finds: 847 devices CMDB has: 823 devices registered Delta: 24 unregistered devices found! Action: investigate each unknown device

05 Software Asset Management Licenses · Versions · Compliance

Software asset management tracks every application, license, and software component in the organization. Unauthorized software is a security risk. Unlicensed software is a legal risk. Outdated software is a vulnerability risk.

Software Inventory Categories

Licensed Commercial Software

  • Microsoft 365, Adobe Creative Cloud
  • Security tools (Splunk, CrowdStrike)
  • ERP systems (SAP, Oracle)
  • Database software (SQL Server, Oracle DB)
  • Operating system licenses

Track: license keys, expiry dates, seat counts, version numbers

Open Source Software

  • Linux distributions
  • Apache, Nginx web servers
  • MySQL, PostgreSQL databases
  • Programming language runtimes
  • Third-party libraries (npm, pip, Maven)

Track: version, known CVEs, license type (GPL, MIT, Apache)

SaaS Applications

  • Salesforce, HubSpot (CRM)
  • Slack, Teams (communication)
  • GitHub, Jira (development)
  • Zoom, Webex (conferencing)
  • Dropbox, Box (file sharing)

Track: subscription cost, data stored, SSO integration, admin access

Shadow IT Software

  • Unapproved SaaS tools employees self-sign-up for
  • Personal apps used for work data
  • Pirated or cracked software
  • Downloaded utilities (not approved)

Detected by: CASB, endpoint agents, browser extensions scanning

License Management — Security Implications

# License compliance failures: Under-licensing (more users than licenses): Company has 100 SQL Server licenses Actual installations: 147 servers Risk: vendor audit → massive fines Microsoft audit: $2.3M compliance penalty (real case) Also: may be running unsupported versions to avoid cost Over-licensing (unused licenses): Company paying for 500 Adobe licenses Only 200 actually used Risk: financial waste Unused accounts → attack surface if credentials shared Expired licenses: Security software license expired: → Antivirus stops updating definitions → Vulnerability scanner loses access → SIEM stops ingesting logs → Security gap opens without warning Solution — License Management Tools: Snow Software, Flexera, Ivanti, ServiceNow SAM → Automated discovery of all installed software → Compare installs vs purchased licenses → Alert before license expires → Track software utilization → Identify unused software for reclamation

SBOM — Software Bill of Materials

📄
What Is SBOM?
A complete inventory of ALL software components in an application — including every third-party library, open source component, and dependency. Like a nutrition label for software. Required by US Executive Order 14028 (2021) for software sold to federal government.
# Why SBOM matters — Log4Shell example: CVE-2021-44228 (Log4Shell) discovered December 2021 Affects: Apache Log4j library versions 2.0-2.14.1 Company WITHOUT SBOM: "Do we use Log4j?" IT team: "I don't think so... maybe?" Days of manual searching across all applications Some applications missed → remain vulnerable Breach via overlooked Log4j instance Company WITH SBOM: "Do we use Log4j?" SBOM query: search component:"log4j" version:"<2.15" Result: 47 applications use vulnerable Log4j Listed by application name and version Patching: targeted, fast, complete All 47 patched within 48 hours ✓

Authorized vs Unauthorized Software

Application Whitelisting

Only pre-approved software allowed to run. Everything else blocked automatically. Extremely effective against malware — unknown executables simply don't run. Tools: AppLocker (Windows), Carbon Black, Bit9.

Policy: only run approved applications User tries to run: HackerTool.exe AppLocker: NOT IN WHITELIST → BLOCKED ✗ Malware installed: BLOCKED → can't execute ✗

Application Blacklisting

Block specific known-bad applications. Everything else runs freely. Less effective than whitelisting — attackers just rename malware. Still useful for blocking specific unwanted software (BitTorrent, gaming apps).

06 Data Asset Management Classification · Location · Retention

Data is often the most valuable asset — and the hardest to track. Unlike hardware, data can be in multiple places simultaneously, copied silently, and lost without anyone knowing. Managing data assets means knowing what data exists, where it lives, who can access it, and when it must be deleted.

Data Asset Inventory — What to Track

Structured Data

Data in databases — organized in rows and columns. SQL Server, MySQL, PostgreSQL, Oracle databases. Customer records, financial transactions, employee data, product inventory.

  • Track: database name, server, schema
  • Track: data classification level
  • Track: who has database access
  • Track: encryption status

Unstructured Data

Files not in database format — documents, spreadsheets, PDFs, images, emails, presentations. Hardest to track — often scattered across file servers, SharePoint, cloud drives, email.

  • Track: file share locations
  • Track: SharePoint/OneDrive sites
  • Track: email archives
  • Discovery tools: Microsoft Purview

Credentials and Keys

Passwords, API keys, certificates, encryption keys, SSH keys. Extremely high value — compromise = attacker impersonates the system. Must be stored in dedicated vault.

  • Never in code repositories
  • HashiCorp Vault, AWS Secrets Manager
  • Rotate regularly
  • Track expiry dates

Intellectual Property

Source code, designs, formulas, patents, trade secrets, research data. Often most valuable organizational asset. Limited access, version control, DRM protection.

  • Track: code repositories (GitHub)
  • Track: design files, CAD drawings
  • Track: research databases
  • Strict access controls

Data Location Tracking

# The challenge: data exists in MANY places simultaneously Customer PII example — where does it live? Production: ✓ CRM database — tracked, encrypted, controlled ✓ ERP system — tracked, access controlled Shadow copies — often missed: ? Marketing spreadsheet on SharePoint — exported from CRM ? Email to vendor with customer list attached ? Backup tape in offsite storage ? Developer test database (real data copied for testing!) ? Analytics platform — connected to CRM ? Support team's Zendesk — customer emails archived ? Employee's personal OneDrive — "backup" they made Data discovery tools find all locations: Microsoft Purview, Varonis, BigID → Scan all storage locations → Find PII, credit card numbers, SSNs → Map where sensitive data actually lives → Compare vs where it SHOULD be → Alert on data in wrong location Developers using production data for testing = major risk Solution: data masking/synthetic data for test environments

Data Retention and Legal Hold

Retention Policies

How long data must be kept before deletion. Driven by regulation:

  • HIPAA: medical records = 6 years minimum
  • PCI-DSS: transaction logs = 1 year
  • SOX: financial records = 7 years
  • GDPR: only as long as necessary
  • IRS: tax records = 7 years

Auto-delete when retention period expires — reduces risk and storage costs.

Legal Hold

When litigation is anticipated — normal retention/deletion policies SUSPENDED. All relevant data must be preserved exactly as-is. Deleting data under legal hold = obstruction of justice.

Legal hold triggered: → All relevant email: preserved → All documents: preserved → Backup tapes: do NOT overwrite → Database logs: preserved Normal auto-delete: SUSPENDED Duration: until litigation resolved Tool: Microsoft Purview eDiscovery

07 Monitoring & Tracking Assets Continuous Visibility

Asset management isn't a one-time activity — it requires continuous monitoring to track changes, detect unauthorized modifications, verify compliance, and identify assets approaching end of life.

What to Monitor

Patch Status

Is every asset running the latest security patches? Unpatched systems = known vulnerabilities. Continuous tracking via vulnerability scanners and SCCM/Intune patch compliance dashboards.

Patch compliance dashboard: Critical patches: 847/847 systems (100%) ✓ High patches: 823/847 systems (97%) Outstanding: 24 systems need patching ! Oldest patch gap: Server-12: 45 days overdue

Compliance Status

Is every asset meeting its security baseline? BitLocker enabled? EDR installed? MDM enrolled? Antivirus up to date? Non-compliant devices flagged and remediated automatically.

License Expiry

Track when software licenses expire. Alert 90 days before expiry. Security software expiring = gap in security coverage. Expired licenses = legal risk. Auto-renewal or procurement triggered.

End of Life Tracking

Track when hardware and software reach end of manufacturer support. EOL = no more security patches = permanent vulnerability. Plan replacement 12-18 months before EOL date.

EOL alerts: Windows Server 2012: EOL Oct 2023 → urgent! iPhone 8 (iOS 16): EOL → replace cycle Cisco 2960 switches: EOL Jan 2025 → plan now

Asset Monitoring Tools

🔍
Vulnerability Scanners
Nessus, Qualys, Rapid7 — continuously scan all registered assets for known vulnerabilities. Compare to CVE database. Prioritize by CVSS score. Flag assets with critical unpatched CVEs. Integration with CMDB — scan unknown devices = alert.
📡
Network Access Control (NAC)
Controls network access based on device health and registration. Only registered, compliant devices get network access. Unknown device plugs in → quarantine VLAN. Non-compliant device → limited access until remediated. Cisco ISE, Forescout, Aruba ClearPass.
📊
SIEM Asset Correlation
SIEM correlates security events WITH asset data. "Which asset generated this alert? Who owns it? Is it in scope for compliance? When was it last patched?" Asset context transforms alerts from noise into actionable intelligence.
🛡️
Endpoint Detection and Response (EDR)
Agent on every managed endpoint continuously monitors behavior. Reports back to CMDB: software installed, patches applied, compliance status, suspicious activity. Integration with CMDB shows asset security health in real time.

08 Asset Disposal Sanitize · Decommission · Document

Disposal is the riskiest phase — data that survives on improperly disposed hardware can lead to massive breaches. Simply deleting files or formatting drives is NOT secure disposal. The sensitivity of data stored determines the disposal method required.

Data Sanitization Methods — Must Know All Three

1. Clearing (Overwriting)
Reuse within org

Overwrite all storage locations with zeros, ones, or random data. Makes data unrecoverable by normal means. Multiple-pass overwriting (DoD 5220.22-M standard = 7 passes). Suitable when device will be reused within same organization.

Tools: DBAN (Darik's Boot and Nuke), Eraser, Windows Reset Method: write zeros → write ones → write random → verify Suitable for: HDDs being reassigned to another department NOT for: SSDs (wear leveling may not overwrite all cells) NOT for: highly classified data disposal
2. Purging (Degaussing / Cryptographic Erase)
Sell / Donate

More thorough than clearing — renders data unrecoverable with laboratory equipment. Degaussing: powerful magnetic field destroys magnetic media. Cryptographic erase: destroy the encryption key (if drive was encrypted — data becomes unreadable).

Degaussing: → Exposes HDD/tape to strong magnetic field → Permanently destroys magnetic data → Also destroys drive firmware → drive unusable → NOT suitable for SSDs (not magnetic) Cryptographic Erase (CE): → If drive encrypted: simply destroy the key → Data becomes permanently unreadable instantly → Works for SSDs, HDDs, USBs → Fastest secure erase method → ATA Secure Erase command for SSDs
3. Destroying (Physical Destruction)
Highly sensitive data

Physical destruction of the storage media. Absolute guarantee that data cannot be recovered. Required for Top Secret / classified data. Certificate of Destruction issued by certified vendor.

Methods: Shredding: industrial shredder reduces to tiny particles Incineration: burn in certified incinerator Disintegration: shred into particles < 2mm Melting: smelt the metal components Pulverizing: hammer/crush into pieces Who requires this: → Government classified data (TS/SCI) → Healthcare with highly sensitive records → Financial institutions per policy → Any org with broken/EOL encrypted drives Certificate of Destruction: legal proof of secure disposal NIST 800-88: US government standard for media sanitization

Special Disposal Considerations

Cloud Asset Disposal

When terminating cloud resources: delete all data first, terminate instances, verify storage deleted. Cloud providers may retain data briefly after deletion. Request cryptographic erasure or deletion confirmation. For compliance: document deletion with timestamps.

AWS S3 bucket disposal: 1. Download/archive needed data 2. Delete all objects 3. Delete bucket 4. Verify with AWS CLI: aws s3 ls 5. Check no orphaned snapshots/AMIs 6. Document deletion date + account

Third-Party Disposal

Using certified disposal vendor (ITAD — IT Asset Disposition): verify they have NAID (National Association for Information Destruction) certification. Get written certificate of destruction for every asset. Do NOT assume vendor handles data securely without verification.
