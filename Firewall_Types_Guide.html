Firewall Types

// CompTIA Security+ SY0-701 — Visual Guide with Scenarios

01 Packet Filtering Firewall Stateless · Basic

The simplest and oldest firewall type. Examines each packet independently and makes allow/deny decisions based only on header information — source IP, destination IP, ports, and protocol. Has no memory of previous packets.

// SCENARIO — Small ISP Edge Router

A small internet provider uses packet filtering on their edge router. They block all inbound Telnet (port 23) and FTP (port 21) traffic. Simple, fast, low cost.

# Example packet filter rules ALLOW src:any dst:any port:443 proto:TCP # HTTPS in ALLOW src:any dst:any port:80 proto:TCP # HTTP in DENY src:any dst:any port:23 proto:TCP # Block Telnet DENY src:any dst:any port:21 proto:TCP # Block FTP DENY src:any dst:any port:any proto:any # Block everything else # ATTACK: Hacker spoofs source port 80 Packet: src_port:80 dst_port:1234 payload:MALWARE Result: ALLOWED ❌ # firewall sees port 80 = web = ok # doesn't know this wasn't requested!

// PROS

  • Extremely fast
  • Low resource usage
  • Simple to configure
  • Good for basic edge filtering

// CONS

  • No state tracking
  • Easily bypassed by port spoofing
  • No application awareness
  • Cannot detect attacks in payload

// EXAM KEYWORDS

Stateless →Each packet evaluated independently — no memory
Checks →Source/destination IP, port, protocol only
Weakness →Port spoofing bypasses it easily
Use →Simple edge filtering, low-cost environments
02 Stateful Inspection Firewall Stateful · Standard

Tracks the state of every network connection in a state table. Knows if a packet is part of an established, legitimate session — or an unsolicited intrusion attempt.

// SCENARIO — Corporate Office Network

Company employees browse the web. Stateful firewall tracks their outbound connections. When the web server responds, firewall recognizes the return traffic as part of an established session and allows it. Random unsolicited packets from the internet are blocked — no matching state found.

# Employee visits google.com Employee → SYN → google.com:443 State: NEW added to table Employee ← SYN-ACK ← google.com:443 State: ESTABLISHED Employee → ACK → google.com:443 Connection established ✓ # Google response packets google.com:443 → Employee:54321 ALLOWED ✓ # matches state table # Random attacker packet (no prior connection) evil.com:1337 → Employee:54321 BLOCKED ✗ # no matching state

// PROS

  • Tracks connection state
  • Blocks unsolicited traffic
  • Better than packet filtering
  • Handles TCP/UDP well

// CONS

  • Cannot inspect packet contents
  • No application awareness
  • State table can be exhausted (DoS)
  • Cannot detect app-layer attacks

// EXAM KEYWORDS

State table →Tracks all active connections (NEW, ESTABLISHED, RELATED)
Blocks →Unsolicited inbound packets — no matching state
Better than →Packet filtering — can't be spoofed with port tricks
Still misses →Application layer attacks — SQL injection inside HTTP
03 Application Layer Firewall (Layer 7) Deep Inspection

Goes beyond headers — reads the actual content of packets. Understands application protocols like HTTP, FTP, DNS, SMTP. Can identify what is being done, not just where traffic is going.

// SCENARIO — School Network Content Filtering

A school deploys an application layer firewall. Students can access educational sites but the firewall reads HTTP requests and blocks social media, gaming, and adult content — even when accessed on port 80/443.

# Student tries to access YouTube HTTP GET: Host: www.youtube.com App Firewall reads: "youtube.com → Entertainment category" Policy: Entertainment = BLOCKED during school hours Result: BLOCKED ✗ — redirect to block page # Student accesses Khan Academy HTTP GET: Host: www.khanacademy.org App Firewall reads: "khanacademy.org → Education category" Policy: Education = ALLOWED Result: ALLOWED ✓ # Web application attack attempt HTTP GET: /search?q=<script>document.cookie</script> App Firewall: XSS PATTERN DETECTED → BLOCK ✗

// PROS

  • Reads packet contents
  • Application protocol awareness
  • Content-based filtering
  • Detects app-layer attacks

// CONS

  • Slower — deep inspection cost
  • Cannot see encrypted traffic
  • More expensive than stateful
  • Complex to configure

// EXAM KEYWORDS

Layer 7 →Application layer — reads actual content of packets
DPI →Deep Packet Inspection — looks inside the payload
Detects →SQL injection, XSS in HTTP traffic
Limitation →Cannot inspect encrypted HTTPS without SSL inspection
04 Next-Generation Firewall (NGFW) Enterprise · All-in-One

The modern standard. Combines stateful inspection, application awareness, user identity, built-in IPS, SSL inspection, and threat intelligence into one unified platform.

// SCENARIO — Financial Institution

A bank deploys Palo Alto NGFW. It identifies that an employee is using Dropbox (personal cloud storage) to upload files — even though Dropbox uses HTTPS port 443. The NGFW identifies the application, checks the user's identity against Active Directory, and blocks the upload per data loss prevention policy.

# Employee uploads file to personal Dropbox Traffic: HTTPS port 443 → dropbox.com Traditional FW sees: "Port 443 HTTPS — ALLOW" ❌ misses it NGFW sees: App identified: Dropbox Personal Storage User: john.smith (from Active Directory) Department: Finance SSL decrypted: file contains financial data Policy: Finance dept → no personal cloud storage Result: BLOCKED ✗ + alert to security team # Zero-day malware in HTTPS response NGFW: SSL decrypt → extract file → send to sandbox Sandbox: executes file in isolated VM Result: Malicious behavior detected → BLOCK ✗

// PROS

  • All-in-one security platform
  • App + user + content aware
  • Built-in IPS — no separate device
  • SSL inspection capability
  • Threat intel integration

// CONS

  • Very expensive
  • Complex to manage
  • SSL inspection = performance hit
  • Privacy concerns (SSL decrypt)
  • Requires skilled staff

// EXAM KEYWORDS

App-aware →Identifies apps by behavior not just port (Dropbox on 443)
User identity →Integrates with Active Directory — rules per user/group
SSL inspection →Decrypts HTTPS to inspect — authorized MitM
IPS built-in →No separate IPS device needed — integrated
Examples →Palo Alto, Fortinet, Check Point, Cisco Firepower
05 WAF — Web Application Firewall Web Apps Only

Specifically protects web applications. Sits in front of web servers and inspects every HTTP/HTTPS request for application-layer attacks like SQL injection, XSS, and CSRF.

// SCENARIO — E-Commerce Website Attack

An online store deploys Cloudflare WAF. Attackers attempt SQL injection to steal the customer database. The WAF reads the HTTP request body, identifies the malicious SQL pattern, and blocks it before it reaches the web server.

# SQL injection attempt on login form POST /login HTTP/1.1 Host: shop.example.com Body: username=admin'--&password=anything WAF intercepts → reads body → SQL pattern detected Rule: SQLi signatures matched Result: BLOCKED ✗ → 403 Forbidden returned # XSS attack in product review POST /review HTTP/1.1 Body: review=<script>steal(document.cookie)</script> WAF: XSS pattern in request body Result: BLOCKED ✗ → sanitized or rejected # DDoS — 5000 requests/second from one IP WAF: rate limit triggered (max 100 req/s per IP) Result: Rate limited + CAPTCHA challenge # Legitimate customer purchase POST /checkout — normal headers, normal payload WAF: no patterns matched → ALLOWED ✓

// PROS

  • Protects OWASP Top 10
  • Rate limiting + DDoS mitigation
  • Cloud WAF scales easily
  • Protects legacy web apps

// CONS

  • Web apps only — not full network
  • False positives can break apps
  • Needs tuning for each application
  • Cannot protect non-HTTP services

// EXAM KEYWORDS

WAF purpose →Protects web applications — HTTP/HTTPS traffic only
Protects against →SQL injection, XSS, CSRF, path traversal — OWASP Top 10
Placement →In front of web servers — between internet and web servers
Cloud WAF →Cloudflare, AWS WAF, Akamai — no hardware needed
WAF vs NGFW →WAF = web only / NGFW = entire network — use BOTH
06 UTM — Unified Threat Management SMB · All-in-One

A single appliance combining multiple security functions — designed for small and medium businesses that need comprehensive security without dedicated enterprise-grade devices for each function.

// SCENARIO — Small Law Firm (50 employees)

A 50-person law firm can't afford separate firewall, IPS, antivirus gateway, and VPN appliances. They deploy one SonicWall UTM. It handles everything — firewall rules, intrusion prevention, antivirus scanning of downloads, VPN for remote lawyers, and email spam filtering.

# All handled by ONE UTM device: Firewall: blocks unauthorized inbound traffic IPS: detects and blocks intrusion attempts Antivirus: scans email attachments and downloads VPN: lawyers work remotely through secure tunnel Content: blocks social media during work hours Email filter: spam and phishing emails blocked # Lawyer receives phishing email with .exe attachment UTM email filter: scans attachment UTM antivirus: malware detected in attachment Result: email quarantined ✗ — lawyer notified # Remote lawyer connects from home UTM VPN: authenticates lawyer → encrypted tunnel ✓ Lawyer works as if in office

// PROS

  • Single device — easy to manage
  • Lower cost than separate devices
  • Comprehensive protection
  • Good for SMBs

// CONS

  • Single point of failure
  • Less granular than NGFW
  • Performance bottleneck
  • Not ideal for large enterprise

// EXAM KEYWORDS

UTM →All-in-one appliance — firewall + IPS + AV + VPN + content filter
Target →Small and medium businesses (SMBs)
vs NGFW →UTM = SMB, simpler / NGFW = enterprise, more granular
Risk →Single point of failure — one device does everything
07 Host-Based Firewall Per Device

Software firewall running directly on an individual device. Protects that single device regardless of what network it's connected to — corporate, home, or public Wi-Fi.

// SCENARIO — Remote Worker Malware Containment

A remote employee accidentally installs malware. The malware tries to connect to an attacker's command-and-control server. The host-based firewall detects an unknown application making outbound connections and blocks it — containing the malware.

# Malware installed on remote worker's laptop Malware attempts: evil-c2.ru:4444 (outbound) Host Firewall outbound rule: "Unknown application attempting outbound connection" App: malware.exe → evil-c2.ru:4444 Status: application not whitelisted Result: BLOCKED ✗ — malware contained Alert: sent to security team Malware: cannot phone home → cannot receive commands Data safe — attacker can't control malware ✓ # Employee at coffee shop — attacker on same Wi-Fi Attacker tries: nmap scan on employee laptop Attacker sees: all ports filtered/closed Host FW: blocking all unsolicited inbound ✓ # Network firewall not present — but host FW protects

// PROS

  • Protects device anywhere
  • Works on any network
  • Controls per-application
  • Built into modern OS

// CONS

  • Protects only one device
  • Disabled by malware sometimes
  • Management at scale complex
  • User can disable it

// EXAM KEYWORDS

Host-based →Software on individual device — protects that device only
Key benefit →Protects device even OFF corporate network
Examples →Windows Defender Firewall, iptables, macOS firewall
Defense in depth →Network FW + Host FW = two independent layers
08 Cloud Firewall — FWaaS Cloud · No Hardware

Firewall delivered as a cloud service. No hardware to buy or manage. Traffic is routed through the provider's global network for inspection. Perfect for remote workforces and cloud-first organizations.

// SCENARIO — 500-Person Remote-First Company

A fully remote tech company has employees in 30 countries. No central office means no hardware firewall makes sense. They deploy Cloudflare Gateway (FWaaS). All employee internet traffic routes through Cloudflare's nearest data center, inspected and filtered before reaching the internet — wherever the employee is.

# Employee in Toronto accesses internet Traffic → nearest Cloudflare PoP (Toronto) Cloudflare: inspects, filters, applies policy Clean traffic → destination # Employee in India — same policy Traffic → nearest Cloudflare PoP (Mumbai) Same policy applied automatically Consistent security globally ✓ # Malicious site attempt Employee visits phishing site Cloudflare DNS: known malicious domain Result: BLOCKED ✗ — DNS-level block # Traditional hardware firewall comparison Remote employee → bypasses HQ firewall entirely FWaaS employee → firewall follows them everywhere ✓

// PROS

  • No hardware costs
  • Protects remote workers
  • Global coverage
  • Scales instantly
  • Always updated

// CONS

  • Needs internet to work
  • Traffic through third party
  • Ongoing subscription cost
  • Vendor dependency

// EXAM KEYWORDS

FWaaS →Firewall as a Service — cloud delivered, no hardware
Best for →Remote workforces, cloud-first organizations
Examples →Cloudflare Gateway, Zscaler, AWS Network Firewall
Zero Trust →FWaaS integrates naturally with Zero Trust architecture
